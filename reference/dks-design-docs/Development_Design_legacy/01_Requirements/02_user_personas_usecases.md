# SRS-02. 사용자 페르소나 및 핵심 유스케이스 정의

> **목적**: 시스템이 누구를 위해 작동하며, 어떤 시나리오에서 어떤 가치를 제공해야 하는지(Usecase) 명확히 하기 위해 핵심 타겟 그룹(페르소나)을 정의하고 시스템과의 상호작용 흐름을 묘사합니다.

## 1. 핵심 사용자 페르소나 (User Personas)

### 🧑‍🔧 페르소나 A: 현장 설비 엔지니어 (운영자)
* **목적**: 긴급한 장비 고장이나 점검 시, 수많은 도면 속에서 원하는 장비의 도면과 계통을 가장 빠르게 찾아보고 대응하기 위함.
* **불만 사항**: 기존에는 CAD 프로그램이 없으면 도면을 열 수 없고, 수백 장의 PDF나 DWG 속에서 특정 밸브 번호를 일일이 검색하는 것이 고통스러웠음.
* **플랫폼 활용**: **글로벌 검색**을 통해 AI 챗봇에게 "어제 에러 난 냉각수 펌프(P-101)와 연결된 밸브가 있는 도면을 찾아줘"라고 묻고, 도면 진입 시(웹 CAD) 곧바로 해당 객체가 **빨간색으로 점멸(Highlight)**되는 하이브리드 검색 기능을 적극 활용함. 잘못된 부분은 **마크업 커밋**으로 현장에서 바로 지시선과 코멘트를 남김.

### 📐 페르소나 B: 도면/설계 관리자
* **목적**: 플랜트 및 사업장 내의 모든 도면의 최신 버전을 관리하고, 시공사나 설계사에서 온 수정본이 이전 버전과 무엇이 다른지 즉시 검증하기 위함.
* **불만 사항**: 누가, 언제, 도면의 어느 선을 지웠는지 도무지 알 수 없어 휴먼 에러가 빈번하게 발생했음.
* **플랫폼 활용**: 도면을 업로드하여 **Git-like 버전 관리(Commit)**를 수행함. 신규 도면이 올라오면 기존 버전(V1)과 신규 버전(V2)의 기하학적 차이를 파악하는 **Auto Diff 모드**를 켜서 추가된 선(초록색)과 삭제된 선(빨간 점선)을 1초 만에 시각적으로 대조하여 검수함.

### 🧠 페르소나 C: 지식 엔지니어 / AI 관리자
* **목적**: 단순 도면을 지식 그래프(TypeDB)로 변환하여 에이전트의 RAG 추론 능력을 극대화함.
* **불만 사항**: 기존에는 도면 정보와 설비 매뉴얼(문서)이 분리되어 있어 다차원적 연결(계통 추론)이 불가능했음.
* **플랫폼 활용**: 백엔드 파이프라인이 자동 추출한 Handle ID가 TypeDB의 엔티티로 정상 편입되는지 관리하고, 프론트엔드의 `entity_mappings` 캐시 구조가 원활히 작동하는지 모니터링함.

## 2. 주요 시스템 유스케이스 (Use Cases)

```mermaid
usecaseDiagram
    actor "현장 엔지니어 (Persona A)" as A
    actor "도면 관리자 (Persona B)" as B
    
    package "자체 웹 CAD 도면관리 플랫폼" {
        usecase "도면 파일 업로드 및 커밋 생성" as UC1
        usecase "Auto Diff (도면 차분 대조)" as UC2
        usecase "도면 뷰어 (Pan/Zoom/Layer)" as UC3
        usecase "객체 측정 및 자석 스냅" as UC4
        usecase "마크업 커밋 및 주석 작성" as UC5
        
        usecase "글로벌 챗봇 설비 검색" as UC6
        usecase "도면 내 타겟 하이라이팅" as UC7
    }

    B --> UC1
    B --> UC2
    B --> UC3
    
    A --> UC3
    A --> UC4
    A --> UC5
    A --> UC6
    
    %% AI와 연계된 워크플로우
    UC6 -.-> |세션 전이 파라미터 전달| UC7
    UC3 -.-> |뷰어 연동| UC7
    UC1 -.-> |백엔드 파싱 및 캐싱| UC2
```

## 3. 핵심 시나리오 상세 (하이브리드 검색 흐름)

1. **상황 발생**: 현장 엔지니어가 모바일 패드나 PC로 DKS 홈 화면에 접속함.
2. **글로벌 질의 (UC6)**: "제1공장 소방배관 도면 찾아줘." → 시스템이 TypeDB RAG 추론 후 도면 목록 반환.
3. **세션 전이 및 로딩 (UC3, UC7)**: 엔지니어가 도면을 클릭하면 웹 브라우저에 Three.js 뷰어가 로딩되며, URL 쿼리로 전달받은 `highlight` 파라미터(Handle ID)를 사용해 즉각 해당 배관 객체로 화면이 줌인(Zoom-to-fit) 됨.
4. **치수 확인 및 피드백 (UC4, UC5)**: 엔지니어가 `flatbush` 기반 자석 스냅 도구를 켜고 밸브 간 거리를 측정(UC4)한 뒤, 도면이 현장과 다르다는 것을 발견하고 그 위에 펜으로 동그라미를 치며 "배관 위치 1m 우측으로 이동 요망" 이라는 마크업 커밋(UC5)을 발생시킴.
5. **검수 (UC2)**: 추후 도면 관리자가 해당 마크업을 확인하고, 설계사가 보내온 신규 수정본 도면(V2)을 올려 Auto Diff로 기존 V1 도면과 겹쳐보며 정말로 1m 우측으로 선이 수정(초록색 렌더링)되었는지 눈으로 검증함.
