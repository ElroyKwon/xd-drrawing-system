# 이슈 워크플로우 명세

## ① 목적
- [[ACC]] Build 이슈 모듈의 "상태 흐름 + 권한 + 알림" 동작을 동등 재현하기 위한 단일 명세.
- 이 문서가 소유하는 주제 = **상태전이 규칙 / 행위 권한 / 알림·멘션 / audit 기록**. 폼 필드·핀 배치·목록 UI는 [[07-2_이슈관리]]로 위임(DRY).

## ② 배경 / 전제
- 핵심 스코프: [[DWG]] 원본 미편집. 이슈는 2D 뷰어 위 "오버레이" 핀으로 생성 → 워크플로우는 데이터 레이어에서 동작.
- 1차 범위 = 2D 시트 한정. 3D/Bridge 범위 밖.
- 차별점: 이슈 핀을 픽셀 좌표뿐 아니라 **설비 엔티티 ID**(Phase 0 도면 파싱 산출물)에도 바인딩. Phase 2 [[TypeDB]] 온톨로지 연동 시 영향도/AI채팅 후크 예정 — 본 문서는 상태/권한/알림에 한정하고, 엔티티 바인딩은 후크 지점만 명시.

## ③ 상세 목차

## 상태 머신
> 이슈 라이프사이클 상태와 전이 규칙. ACC Build 기준 상태 집합부터 확정.

### 상태 정의
> 각 상태의 의미·진입조건·종료조건을 표로. ACC 매핑: Draft/Open/Answered/Closed/Void.
- `Draft` (작성중) → `Open` (열림) → `Answered`(답변/검토요청) → `Closed`(완료) / `Void`(무효)
- ❓ ACC의 "Answered" 단계를 그대로 둘지, 단순화(Open→Closed)할지 청주 운영 합의 필요(TBD)

### 전이 다이어그램
> 상태 전이 그래프(허용 화살표만). mermaid stateDiagram로 채울 것.
- 역전이 허용 범위 정의: `Closed→Open`(재오픈) 허용 여부 TBD

### 전이 가드(조건)
> 각 전이의 선행조건. 예: Open→Closed 시 필수 필드(담당자·답변) 채워짐 검증.
- 필수 필드 충족 / 첨부 유무 / 확인자 승인 여부 등 가드 규칙 나열

### 상태별 허용 액션 매트릭스
> 상태 × 액션(편집/코멘트/첨부/재배치/삭제) 가능 여부 표. 권한 절과 교차.

## 유형 템플릿 / 2단 계층
> ACC 이슈의 "Type(유형) > Subtype(하위유형)" 2단 분류 체계. 유형별로 폼/워크플로우 변형.

### 유형/하위유형 카탈로그
> 청주 사업장 도면관리에 필요한 유형 목록 초안. ACC 기본값 참고.
- 예시(초안): 설계오류 / 시공불일치 / 간섭(Clash) / 정보요청(RFI) / 안전 — TBD 현장 합의

### 유형별 필드 오버라이드
> 유형마다 필수/선택 필드가 달라지는 규칙. 폼 정의는 [[07-2_이슈관리]] 참조, 여기선 워크플로우 영향만.

### 유형별 기본 담당/확인자 매핑
> 유형 선택 시 자동 배정되는 기본 담당자·확인자 규칙(있으면). TBD 운영정책.

### 유형별 상태머신 변형
> 특정 유형이 표준 상태머신과 다른 흐름을 갖는지(예: RFI는 Answered 필수). 없으면 "표준 동일" 명시.

## 권한 / 담당·확인자
> 역할(Role) × 액션 권한. 담당자(Assignee)/확인자(Reviewer/Watcher) 역할 동작 정의.

### 역할 정의
> 프로젝트 역할 집합. 상세 권한모델은 [[06_권한모델]] 소유 — 여기선 이슈 한정 역할만.
- 생성자(Creator) / 담당자(Assignee) / 확인자(Reviewer) / 관찰자(Watcher) / 관리자(Admin)

### 행위별 권한 매트릭스
> 액션(생성/편집/상태변경/배정/Close/Void/삭제) × 역할 표. 핵심 산출물.
- ❓ Close 권한을 담당자에게 줄지, 확인자/생성자 전용으로 둘지 TBD

### 담당자 배정 규칙
> 단일/복수 담당자 허용 여부, 재배정 권한, 미배정 상태 처리.

### 확인자 승인 게이트
> Closed 진입 전 확인자 승인이 필요한 워크플로우(옵션) 정의. 상태 가드와 연동.

### 권한 후크 — 엔티티 단위(Phase 2)
> 향후 설비 엔티티 소유 분야(전기/기계 등)에 따른 자동 권한 분기 가능성 메모. 1차 범위 밖, 후크 지점만.

## 알림 / 멘션
> 상태전이·배정·코멘트 발생 시 알림 트리거와 채널.

### 알림 트리거 이벤트
> 알림을 발생시키는 이벤트 목록. 예: 배정됨/상태변경/코멘트추가/멘션됨/마감임박.

### 수신자 결정 규칙
> 이벤트별 수신 대상(담당자/생성자/확인자/관찰자/멘션대상) 매핑 표.

### 멘션(@) 처리
> 코멘트 내 `@사용자` 파싱 → 관찰자 자동 추가 + 알림. 파싱 규칙·중복 억제.

### 전달 채널
> 인앱 토스트/알림센터, 이메일 등. 1차 = 인앱 우선.
- ❓ 이메일 발송 1차 포함 여부 TBD. 실시간 인앱은 [[WebSocket]] 후보 / 폴링 대안 결정 필요

### 알림 빈도/묶음
> 스팸 방지(다건 변경 묶음, 동일 이벤트 디바운스) 정책. 1차 단순화 가능.

## audit 기록
> 모든 상태전이·권한행위·필드변경의 불변 이력.

### audit 이벤트 스키마
> 기록 단위 필드: who/when/action/from-state/to-state/issueId/diff. 표로 확정.

### 기록 대상 액션 범위
> 어떤 액션까지 audit에 남길지(상태전이·배정·필드변경·코멘트·첨부). 과도 기록 방지.

### 이력 조회/표시
> 이슈 상세의 "활동 타임라인" 렌더 규칙. 표시 UI는 [[07-2_이슈관리]] 위임, 여기선 데이터 계약.

### 불변성 / 보존
