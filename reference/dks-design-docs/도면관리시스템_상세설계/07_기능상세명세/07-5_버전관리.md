# 버전 관리 명세

## 1. 목적
- [[DWG]]/[[PDF]] 도면의 **버전세트(version set) 모델·증가 규칙·이력·복원** 동작을 단일 소유 주제로 정의.
- ACC Build의 "최신 버전이 곧 현재 시트" 모델을 동등 재현. 원본 [[DWG]]는 **편집하지 않음** — 새 업로드 = 새 버전, 기존 버전은 불변(immutable) 보존.
- 이 문서는 **버전 데이터/상태 전이만** 소유. 뷰어 렌더는 [[07-2_시트목록]], 두 버전 시각 diff는 [[07-6_시트비교]], 폴더형 저장소 정책은 [[07-1_파일CDE]]에 위임.

## 2. 배경 / 전제
- 흐름 전제: [[DWG]] → 백엔드 파싱 → 기하 JSON → 웹 2D 뷰어 → 마크업/이슈 오버레이. 버전은 **파싱 산출물 단위**로 고정됨(원본+기하 JSON+썸네일이 1버전 1세트).
- 범위: **2D 시트 한정**. 3D BIM/Bridge 버전 관리 범위 밖.
- 불변성: 한번 생성된 버전은 수정·삭제 안 함(감사 추적). "되돌리기"는 삭제가 아니라 **과거 버전을 새 최신으로 승격(복원)**.
- 차별점(향후): 버전 간 변경을 픽셀 diff뿐 아니라 **설비 엔티티 ID 단위 변경**으로도 추적 가능(영향도). 본 문서는 hook 지점만 명시, 상세는 Phase 2 [[TypeDB]] 트랙에 위임. ❓확정 대기.

---

## 3. 버전세트 모델
> 무엇을 채울지: 버전세트의 데이터 구조·식별 체계·불변 규칙을 정의.

### 3.1 핵심 개념 정의
> 시트(논리 도면) vs 버전(시점 스냅샷) vs 파일(물리 업로드) 3계층 관계 명확화. 1 시트 ↔ N 버전 ↔ 각 버전 1 원본파일.

### 3.2 버전 엔티티 스키마
> 채울 항목(필드 후보): `version_id`, `sheet_id`(시트 연결), `version_no`(정수 증가), `original_file_ref`, `geometry_json_ref`, `thumbnail_ref`, `uploaded_by`, `uploaded_at`, `status`(current/superseded), `checksum`, `parse_status`. 표 형태로 채울 것.

### 3.3 버전 번호 체계
> 단조증가 정수(V1, V2, …) 채택 여부. 시맨틱 버전/리비전 문자(Rev A) 병기 필요성 검토. ❓ACC는 정수형 — 청주 현장 도면 관행(Rev 문자) 수용할지 TBD.

### 3.4 불변성·보존 규칙
> 생성 후 read-only. 원본·기하 JSON·썸네일 묶음 보존. 삭제 정책(soft delete only)과 보존 기간은 [[07-1_파일CDE]] 위임.

## 4. 업로드 시 버전 증가
> 무엇을 채울지: 신규 업로드가 어떤 조건에서 "새 버전"이 되는지, 매칭·증가·승격 전이를 정의.

### 4.1 버전 증가 트리거
> 같은 시트로 식별되는 파일 재업로드 시 version_no +1. 신규 도면번호면 V1 신규 시트 생성. 분기 결정 로직 기술.

### 4.2 동일 시트 매칭 규칙
#### 4.2.1 매칭 키
> 무엇으로 "같은 시트"를 판정할지: 도면번호(Phase 0 파싱 태그) 우선 + 파일명 규칙 보조. ❓도면번호 누락 도면의 fallback(수동 매칭 UI) TBD.
#### 4.2.2 충돌·모호 매칭 처리
> 다중 후보·중복 도면번호 시 사용자 확인 모달(신규 시트 vs 기존 시트 버전). 빈상태/토스트는 공통 UI 위임.

### 4.3 업로드 → 파싱 → 버전 확정 파이프라인
> 단계: 업로드 수신 → 체크섬·중복검사 → 백엔드 파싱(기하 JSON 생성) → 썸네일 → 버전 레코드 commit → current 승격. 파싱 진행 상태는 [[WebSocket]]로 클라이언트 푸시. 실패 시 버전 미확정(파싱 실패 격리).

### 4.4 current 승격·이전 버전 supersede
> 신규 버전 commit 성공 시에만 current 플래그 이동, 직전 current는 superseded. 트랜잭션 경계·동시 업로드 경합 처리 명시. ❓낙관적 락 vs 큐 직렬화 TBD.

### 4.5 중복 업로드 처리
> 동일 checksum 재업로드 시 신규 버전 생성 안 함(no-op) + 안내 토스트. 정책 확정 필요 ❓.

## 5. 이력 / 복원
> 무엇을 채울지: 버전 타임라인 조회와 과거 버전 복원(승격) 동작을 정의.

### 5.1 버전 이력 조회
#### 5.1.1 이력 데이터·정렬
> version_no 내림차순, 각 행: 번호·업로더·일시·상태·변경요약. 우측 인스펙터 패널/데이터테이블 공통 UI 재사용(컬럼토글·내보내기 위임).
#### 5.1.2 이력 항목별 액션
> 보기(해당 버전 뷰어 열기) / 다운로드(원본) / 비교([[07-6_시트비교]]로 이동) / 복원. 액션 권한은 §7 위임.

### 5.2 복원(과거 버전 → 최신 승격)
#### 5.2.1 복원 동작 정의
> 과거 버전을 **삭제 없이** 새 version_no로 복제·승격(append-only). 원리: "롤백"이 아닌 "과거 내용을 신규 버전으로 재발행". ❓복사 승격 vs current 플래그만 이동 — 결정 대기.
#### 5.2.2 복원 후 오버레이 처리
> 핀/마크업/이슈는 버전 종속. 복원 시 기존 오버레이 이관 여부 정의 필요 ❓. 엔티티 ID 바인딩 오버레이는 버전 무관 유지 가능(차별점) — Phase 2 연동 hook.
#### 5.2.3 확인·감사 로그
> 복원은 파괴적 인식 → 확인 모달 + 감사 로그(누가·언제·어느 버전을). 토스트 피드백.

### 5.3 버전 삭제 정책
> 원칙: 물리 삭제 금지(soft delete). current는 삭제 불가, superseded만 숨김 가능 여부 검토. 보존 정책 [[07-1_파일CDE]] 위임.

## 6. 시트-버전 연결
> 무엇을 채울지: 시트 목록·뷰어·오버레이가 어느 버전을 참조하는지 바인딩 규칙 정의.

### 6.1 "현재 시트 = current 버전" 바인딩
> 시트 목록([[07-2_시트목록]])과 뷰어는 기본적으로 current 버전 렌더. 과거 버전 명시 선택 시에만 superseded 로드.

### 6.2 오버레이(핀·마크업·이슈)의 버전 귀속
#### 6.2.1 픽셀 좌표 오버레이
> 좌표는 특정 버전 기하 좌표계 기준 → 버전별 귀속. 버전 전환 시 좌표 재투영/표시 정책 명시. 마크업 상세는 마크업 명세 위임.
#### 6.2.2 엔티티 ID 바인딩 오버레이 (차별점)
> 설비 엔티티 ID 묶임 오버레이는 버전 변경에도 추적 가능 — 좌표만 갱신. Phase 2 [[TypeDB]] 영향도 트랙 연동 지점만 표기, 상세 위임.

### 6.3 버전 선택 UI 연동
> 뷰어 상단 버전 드롭다운(V3 ▾ / 최신·과거 표시). 시트비교 진입 시 from/to 버전 전달 계약([[07-6_시트비교]]).

### 6.4 API 데이터 계약(개요)
> 엔드포인트 후보: `GET /sheets/{id}/versions`(이력), `GET /versions/{id}`(메타+기하 ref), `POST /sheets/{id}/versions`(업로드 트리거), `POST /versions/{id}/restore`(복원). 표준 응답 포맷(success/data/error/meta) 준수. 상세 스키마 TBD.
