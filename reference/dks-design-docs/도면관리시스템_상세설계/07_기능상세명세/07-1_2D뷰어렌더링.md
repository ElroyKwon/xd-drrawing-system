# 2D 뷰어 렌더링 명세

## ① 목적
- 청주 [[도면관리 시스템]]의 핵심 표면 = 웹 2D 도면 뷰어. 본 문서는 "렌더 베이스 동작"만 소유.
- 범위: 백엔드 파싱 산출물(기하 JSON 또는 변환 [[PDF]]/타일 이미지)을 웹 캔버스에 그려, 그 위에 오버레이를 얹을 "좌표 안정적 베이스 레이어" 제공.
- 비범위(위임): 마크업 도형 → [[07-2_마크업오버레이]], 이슈 핀 → [[07-3_이슈핀]], 버전 diff → 시트비교 문서.

## ② 배경 / 전제
- ★스코프 제약: [[DWG]] 원본 미편집. 흐름 = DWG → 백엔드 파싱 → 기하 JSON → 웹 2D 뷰어 렌더 → 오버레이. CAD 편집기 아님, "뷰어 + 주석/이벤트 오버레이".
- 정밀 CAD 편집(osnap 스냅, 좌표 편집, SHX 폰트 정밀 렌더) 범위 밖.
- 1차 범위 = 2D 시트 한정. 3D BIM 뷰어/Bridge 범위 밖.
- 벤치마크: [[Autodesk Construction Cloud]](ACC) Build의 시트 2D 뷰어. 동등 재현이 1차 목표.
- DKS 차별점: 오버레이 좌표가 픽셀뿐 아니라 "설비 엔티티 ID"에도 바인딩(Phase 2 [[TypeDB]] 연동). 본 문서는 그 바인딩이 의존할 "안정적 좌표계" 책임을 명확히 한다.

## 렌더 방식 선택 ([[PDF.js]] / [[OpenSeadragon]] / Canvas)
> 백엔드 산출물 형태에 따라 렌더 파이프라인이 갈린다. 후보 3종 비교표 + 1차 결정 + 폴백을 채운다.

### 후보 비교
> 산출물·정밀도·줌·오버레이 좌표 정합·구현 난이도 5축 비교표.

#### A안: 백터 기하 JSON → Canvas 직접 렌더
> 파싱 결과(선/호/폴리라인/텍스트)를 Canvas 2D로 직접 그림. 엔티티 ID 보존에 가장 유리(DKS 무기). 텍스트/해치 재현 비용·정밀도 한계 메모.

#### B안: 백엔드에서 DWG → [[PDF]] 변환 → [[PDF.js]] 렌더
> 변환 충실도 높고 구현 빠름. 단 PDF는 픽셀/벡터 혼합 → 엔티티 ID 역추적 어려움. 별도 좌표 매핑 테이블 필요 여부 메모.

#### C안: 래스터 타일(DZI 등) → [[OpenSeadragon]]
> 초대형 시트 줌/팬 성능 최강. 단 벡터성 상실, 오버레이는 좌표 변환층 위에만. 대형 도면 폴백 후보.

### 1차 결정 방향
> 권장 조합(예: 기본 A안 + 대형/충실도 필요 시 B/C 폴백) 가정과 근거. ❓ 백엔드 파싱 산출물 실제 포맷 확정 전까지 TBD.

## 타일 / 줌 레벨
> 큰 시트를 끊김 없이 보여줄 타일·레벨 전략. 메모리/네트워크와 직결.

### 줌 레벨 정의
> min/max 줌, fit-to-screen 기준, 레벨당 스케일 비. ACC 체감 수준 목표(과장 수치 금지, TBD).

### 타일링 적용 조건
> A안(벡터)은 타일 불필요할 수 있음 / C안은 필수. 도면 크기·엔티티 수 임계값에서 타일 전환할지 결정 트리.

### 가시 영역 컬링
> 뷰포트 밖 엔티티/타일 미렌더. 벡터 A안에서 공간 인덱스(쿼드트리 등) 도입 여부.

#### LOD (Level of Detail)
> 축소 시 가는 선/소형 텍스트 생략·단순화 규칙. 줌인 시 점진 상세화.

## 좌표 변환
> ★오버레이 정합의 심장. 모든 마크업/핀/이벤트가 의존하는 단일 좌표 규약. 이 절은 타 문서에 절대 위임 불가(SoT).

### 좌표계 정의 3종
> ① 도면 모델좌표(파싱 원본 단위) ② 정규화 시트좌표(0~1 또는 시트 px) ③ 화면 px. 세 좌표 간 변환 행렬 명세.

### 저장 기준 좌표
> 오버레이(마크업/핀)는 어느 좌표로 영구 저장하나 — 줌/리사이즈에 불변해야 함. → 정규화 시트좌표 권장 가정. 회전/플립 도면 처리 ❓TBD.

### 변환 함수 시그니처
> modelToScreen / screenToModel / screenToSheet 등 순수 함수 인터페이스 스텁. 정밀도(부동소수) 주의.

#### 엔티티 ID ↔ 좌표 바인딩 (DKS 무기)
> 클릭 지점 → 엔티티 ID 역조회(히트테스트) 규약. 벡터 A안 전제. 이 매핑이 Phase 2 [[TypeDB]] 영향도/AI채팅 바인딩의 진입점임을 명시. PDF/타일 폴백 시 ID 바인딩 제약 메모.

## 성능 / 메모리
> 대형 시트·다수 오버레이에서도 부드러운 팬/줌 유지 목표.

### 성능 목표 (잠정)
> 초기 로드·팬/줌 프레임·오버레이 동시 표시 개수 목표치. 실측 전 TBD, 과장 금지.

### 렌더 루프 전략
> requestAnimationFrame 기반 dirty-rect 재렌더, 오버레이 레이어 분리(베이스 캔버스 vs 오버레이 캔버스) 합성 방식.

### 메모리 관리
> 타일/엔티티 캐시 상한, 화면 밖 자원 해제, 대형 도면 메모리 상한 가드.

#### 실시간 이벤트 반영
> [[WebSocket]] 이벤트(타 사용자 마크업/이슈/설비 이벤트)를 재렌더 없이 오버레이 레이어만 갱신하는 경로. 상세 프로토콜은 별도 문서 위임.
