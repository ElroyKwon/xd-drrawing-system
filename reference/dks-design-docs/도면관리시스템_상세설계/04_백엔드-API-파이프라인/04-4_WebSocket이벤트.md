# WebSocket / 이벤트 API

## ① 목적
- 도면관리 시스템에서 **실시간 단방향/양방향 이벤트 채널** 정의 (소유 주제: 변환상태·뷰어 동기화·협업)
- REST는 요청/응답, [[WebSocket]]은 "서버 push + 다중 클라이언트 broadcast" 담당 — 경계 명확화
- 채울 것: 어떤 이벤트가 언제 누구에게 흐르는가, 스키마, 채널 구조

## ② 배경 / 전제
- ★스코프: [[DWG]] 원본 편집 없음. 이벤트는 "뷰어 + 오버레이(마크업/이슈/이벤트)" 위에서만 발생 → CAD 협업 편집(좌표 lock 등) 범위 밖
- 1차 = 2D 시트 한정. 3D 동기화 없음
- 변환 파이프라인은 비동기(DWG→파싱→기하 JSON) → 진행상태를 polling 대신 push로 전달 (소유: [[04-1_변환파이프라인]], 여기선 "이벤트 표면"만)
- 협업 = ACC Build 수준의 "동시 열람 + 마크업/이슈 실시간 반영". 실시간 공동 편집(CRDT/OT)은 1차 범위 밖 → TBD
- DKS 차별점: 이벤트 payload에 픽셀 좌표뿐 아니라 **설비 엔티티 ID** 동반 가능 (Phase 2 [[TypeDB]] 온톨로지 바인딩 대비) → 스키마에 `entityId?` 예약 필드만 선반영

## 변환 진행 피드
- 의도: 업로드~렌더 가능 상태까지 비동기 변환 진행률을 구독자에게 push. 클라는 프로그레스바/토스트로 표현
- 소유 경계: 변환 "로직"은 [[04-1_변환파이프라인]] 소유. 여기선 "상태를 이벤트로 내보내는 표면"만 정의

### 변환 상태 머신
- `queued → parsing → rendering → ready` / 실패 시 `failed`
- 각 전이마다 이벤트 1건 emit. payload에 `progress(0~100)`, `stage`, `sheetId`
- 채울 것: 단계별 % 산정 기준(파싱 가중치 vs 렌더 가중치) ❓TBD

### 구독 범위
- 프로젝트 단위 채널 vs 시트 단위 채널 — 업로드 중인 사용자만? 같은 프로젝트 전원? ❓TBD
- 채울 것: 대량 일괄 업로드 시 이벤트 throttle/배치 전략

### 실패·재시도 통지
- `failed` 이벤트에 `errorCode`, `message`, 재시도 가능 여부 플래그
- 채울 것: 부분 실패(다중 시트 중 일부) 표현 방식

## 뷰어 이벤트 (선택 / 하이라이트)
- 의도: 2D 뷰어 상호작용을 다른 패널/사용자에게 전달. "어떤 객체를 선택/하이라이트했는가"
- 전제: 뷰어 렌더는 [[PDF.js]]/Canvas 기반 오버레이([[03_프론트엔드-뷰어]] 소유) — 여기선 이벤트 계약만

### 선택(selection) 이벤트
- payload: `sheetId`, 선택 대상 식별자 — `geometryId`(픽셀/기하) + `entityId?`(온톨로지, Phase 2)
- 용도: 우측 인스펙터 패널 동기화, 이슈 핀 포커스
- 채울 것: 다중 선택 / 선택 해제 이벤트 형태

### 하이라이트 / 포커스 이벤트
- 검색 결과·영향도 분석 결과를 뷰어에 강조 표시하라는 명령형 이벤트
- DKS 무기: "엔티티 ID 기준 하이라이트" — ACC엔 없는 영향도 시각화의 입구 (상세는 [[05_이슈-마크업-데이터모델]] 위임)
- 채울 것: 하이라이트 스타일 토큰(색/펄스) 합의 ❓TBD

### 뷰포트 동기화 (선택 기능)
- 발표 모드: 한 사용자의 pan/zoom을 따라보기. 1차 필수 아님 → ❓TBD(우선순위 낮음)

## 협업 동기화
- 의도: 같은 프로젝트/시트를 보는 사용자 간 오버레이 변경(마크업·이슈·핀)을 실시간 반영
- 전제: 데이터 모델/충돌 규칙은 [[05_이슈-마크업-데이터모델]] 소유. 여기선 "전파 채널"만

### 채널 / 룸 구조
- 룸 키 = `project:{id}:sheet:{id}` 형태(안) — 입장/퇴장 시 presence 갱신
- 채울 것: 인증·권한(룸 join 전 ACL 검사) 흐름, REST 세션과 연계

### 마크업 / 이슈 broadcast
- 생성·수정·삭제 이벤트를 룸 내 broadcast → 다른 클라 오버레이 갱신
- payload는 "변경 델타"인가 "전체 객체"인가 ❓TBD (1차는 단순 전체 객체 push 권장)
- 채울 것: REST 저장 성공 후 emit 순서(저장→broadcast) 보장 규칙

### Presence (접속자 표시)
- 현재 같은 시트 보는 사용자 아바타. join/leave/heartbeat
- 채울 것: heartbeat 주기, 끊김 감지 timeout

### 충돌 처리
- 1차 정책: last-write-wins + 서버 타임스탬프. 실시간 공동 편집 미지원 명시
- 채울 것: 동시 동일 이슈 수정 시 UX(토스트로 "갱신됨" 알림) ❓TBD

## 이벤트 스키마
- 의도: 모든 이벤트가 공유하는 공통 봉투(envelope) + 타입별 payload 규격. REST 표준 응답과 별도 (push 전용)

### 공통 봉투 (envelope)
- 필드(안): `type`, `room`, `ts`, `actor`(userId), `payload`
- 채울 것: 버전 필드 / 멱등 키 필요 여부

### 이벤트 타입 카탈로그
- 카테고리별 type 문자열 네이밍 규칙(안): `conversion.*`, `viewer.*`, `markup.*`, `issue.*`, `presence.*`
- 채울 것: 전체 타입 표 (type / 방향(S→C, C→S) / payload 요약) — 팀이 1행씩 채움
  - 예) `conversion.progress` · S→C · `{sheetId, stage, progress}`
  - 예) `markup.created` · C→S→broadcast · `{sheetId, markup, entityId?}`

### entityId 예약 필드 (Phase 2 대비)
- 온톨로지 바인딩을 위해 selection/하이라이트/마크업/이슈 payload에 `entityId?` 선반영
- 1차에선 null 허용·무시. [[TypeDB]] 연동 시 활성화 (상세 위임: 향후 AI/온톨로지 설계 문서)

### 연결·재연결 규약
- 핸드셰이크, 인증 토큰 전달 위치, 끊김 시 재구독·미수신 이벤트 보충 방식 ❓TBD
- 채울 것: heartbeat/ping-pong 주기, 백오프 재연결 전략
