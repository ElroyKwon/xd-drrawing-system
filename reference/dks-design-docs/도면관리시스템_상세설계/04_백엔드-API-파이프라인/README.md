# 백엔드/API/파이프라인 개요

## 1. 목적
> 채울 것: 이 문서가 04 백엔드 폴더의 **인덱스(진입점)** 임을 명시. 세부 설계는 하위 문서로 위임, 여기서는 지도만 제공.

- 청주 사업장 [[도면관리]] 시스템의 백엔드 계층 전체 골격 한눈에
- 핵심 전제 재확인: [[DWG]] **원본 비편집** — 파싱 → 기하 JSON 산출 → 웹 2D 뷰어 렌더 → 마크업/이슈/이벤트 **오버레이**
- 백엔드 책임 경계 = 파싱 파이프라인 + 오버레이 데이터 [[REST API]] (CAD 편집 엔진 아님)

## 2. 배경/전제
> 채울 것: 스코프 제약과 Phase 구분을 1차 진실로 고정. 상세 근거는 위임.

- 벤치마크: [[ACC Build]] (Autodesk Construction Cloud) 동등 재현 — 시트/마크업/이슈/시트비교/파일([[CDE]])/사진
- 범위 안: **2D 시트 한정**. 범위 밖: 3D BIM 뷰어/Bridge, 정밀 CAD 편집(osnap·좌표편집·[[SHX]] 폰트 정밀렌더)
- Phase 0 산출물 존재: 도면 파싱 결과(분야·도면번호·태그) → 본 파이프라인이 재사용·확장
- Phase 2 차별점: [[TypeDB]] 온톨로지 연동 — 핀/마크업/이슈를 픽셀이 아닌 **설비 엔티티 ID**에 바인딩 (상세는 [[06_온톨로지-바인딩]]에 위임)

## 3. 구성
> 채울 것: 04 폴더 하위 문서 인덱스 + 컴포넌트 책임 1줄 요약. 이 절이 문서의 "소유 주제"의 핵심.

### 3-1. 하위 문서 맵
> 채울 것: 04 폴더 내 각 상세 문서 링크 + 한 줄 책임. 파일 생성 시 갱신.

- [[04a_파싱-파이프라인-상세]] — DWG 인입~기하 JSON 산출 전 과정 (TBD: 파일 생성)
- [[04b_API-엔드포인트-카탈로그]] — 모듈별 REST 엔드포인트 명세 (TBD)
- [[04c_실시간-이벤트]] — [[WebSocket]] 기반 마크업/이슈 협업 동기화 (TBD)
- [[04d_스토리지-전략]] — 원본/산출물/오버레이 저장 위치 분리 (TBD)

### 3-2. 백엔드 컴포넌트 (논리 구성)
> 채울 것: 각 컴포넌트의 단일 책임. 기술 스택 확정값은 ❓.

| 컴포넌트 | 책임 | 비고 |
|---|---|---|
| 파싱 워커 | DWG → 기하 JSON 변환 | 비동기 큐, ❓엔진 |
| 시트 API | 도면 목록·버전·메타 제공 | 표준 response |
| 오버레이 API | 마크업·이슈·핀·이벤트 CRUD | 엔티티 ID 바인딩 |
| 실시간 게이트웨이 | 협업 이벤트 push | [[WebSocket]] |
| 파일/CDE API | 폴더형 문서 저장소 | ACC 파일 모듈 대응 |

### 3-3. 데이터 흐름 (상위)
> 채울 것: DWG 인입 → 파싱 → 산출물 → 뷰어 → 오버레이 한 줄 다이어그램. 상세는 04a로 위임.

- `DWG 업로드 → [파싱 워커] → 기하 JSON + 메타 → [스토리지] → 시트 API → 뷰어([[PDF.js]]/Canvas) → 오버레이 API ↔ 실시간 게이트웨이`

## 4. 파싱 파이프라인 개요
> 채울 것: 단계 목록과 책임 경계만. 알고리즘/포맷 상세는 [[04a_파싱-파이프라인-상세]]로 위임 (DRY).

### 4-1. 단계 (stage)
> 채울 것: 각 단계 입력→출력. 실패 처리·재시도는 04a.

1. 인입(ingest) — 업로드 수신, 형식 검증, 버전 등록
2. 변환(convert) — DWG → 중간표현 (❓ 변환 엔진: ODA / Forge 추출물 / 자체)
3. 기하 추출(extract) — 엔티티·레이어·도면틀 → 기하 JSON
4. 메타 추출(meta) — 분야·도면번호·태그 (Phase 0 산출물 재사용)
5. 색인(index) — 검색·필터용 메타 적재
6. 공개(publish) — 뷰어가 읽을 산출물 확정

### 4-2. 산출물 계약 (output contract)
> 채울 것: 기하 JSON 스키마는 [[05_데이터모델]] 소유 → 여기선 링크만. 좌표계 기준만 명시.

- 좌표계: 도면 단위 기준, 뷰어 픽셀 매핑 규칙 ❓
- 스키마 정의 위치: [[05_데이터모델]] (위임)

### 4-3. 비동기 처리
> 채울 것: 큐 방식·상태머신·진행률 통지. 대용량 DWG 타임아웃 전략 TBD.

- 작업 상태: `queued → parsing → ready → failed`
- 진행률/완료 통지 채널: [[WebSocket]] 또는 폴링 ❓

## 5. API 설계 원칙 (표준 response)
> 채울 것: 전역 CLAUDE.md §5 원칙을 본 프로젝트에 구체화. 엔드포인트 전체 목록은 [[04b_API-엔드포인트-카탈로그]]로 위임.

### 5-1. 통신 방식
> 채울 것: REST 기본 + 보조 채널 정의.

- Frontend ↔ Backend: [[REST API]] 기본
- 실시간 협업(마크업·이슈·핀): [[WebSocket]] 보조
- 대용량 산출물: 파일 경로/서명 URL 전달 ❓

### 5-2. 세분화 (Granular Endpoints)
> 채울 것: 단일 책임 엔드포인트 + Orchestrator 분리. 예시만, 전체는 04b.

- 1 엔드포인트 = 1 책임, 독립 호출 가능
- 복합 플로우(예: 도면 버전 발행 + 색인 + 통지)는 상위 Orchestrator가 세부 API 순차 호출

### 5-3. 표준 응답 구조
> 채울 것: 전 엔드포인트 공통 envelope. 에러 코드 표는 TBD.

```json
{
  "success": true,
  "data": { },
  "error": null,
  "meta": { "timestamp": "ISO8601" }
}
```

- 에러 코드 체계: ❓ (도메인 prefix 규약 TBD)

### 5-4. 오버레이-엔티티 바인딩 규약
> 채울 것: 마크업/이슈/핀이 가지는 위치 참조의 이중 표현(픽셀 좌표 + 설비 엔티티 ID). 온톨로지 상세는 위임.

- 위치 표현: `{ sheetId, geometry(픽셀/도면좌표), entityId? }`
- `entityId`는 Phase 2 [[TypeDB]] 엔티티 참조 — Phase 1에선 nullable
- 바인딩 의미·영향도 질의: [[06_온톨로지-바인딩]]에 위임

## 6. 결정 대기 항목 (TBD / ❓)

- ❓ 실시간 채널: [[WebSocket]] 전용 vs SSE/폴링 혼용
- ❓ 스토리지: 원본/산출물/오버레이 물리 분리 위치
- ❓ 에러 코드 체계 및 인증/권한 모델(프로젝트 스위처 연동)
