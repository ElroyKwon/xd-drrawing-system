## 1. 목적

- 청주 도면관리 시스템의 **시트(도면) 목록·상세·버전·썸네일** 제공 REST 엔드포인트 명세.
- [[ACC]] Build의 "시트" 모듈(도면목록 + 2D뷰어 진입)을 동등 재현하기 위한 백엔드 계약 정의.
- 소유 범위 = 시트/버전 리소스 한정. 마크업·이슈·시트비교 API는 별도 문서로 위임.

> ★전제: [[DWG]] 원본은 **편집하지 않는다**. 흐름 = DWG → 파싱 → 기하 JSON → 웹 2D 뷰어 렌더 → 오버레이. 본 API는 "뷰어가 읽을 시트/버전 메타 + 렌더 산출물 포인터"만 제공. CAD 편집 엔드포인트 없음.

## 2. 배경/전제

- 1차 범위 = **2D 시트 한정** (3D BIM/Bridge 범위 밖).
- [[DKS]] Phase 0 도면 파싱 산출물(분야·도면번호·태그)이 이미 존재 → 시트 메타의 1차 소스.
- 차별점: 시트/버전 응답에 ACC엔 없는 **설비 엔티티 ID 바인딩** 필드를 선택적으로 포함 가능(Phase 2 [[TypeDB]] 온톨로지 연동 대비). 본 문서는 필드 자리만 예약, 채움은 TBD.
- 표준 응답 포맷(success/data/error/meta)은 전역 규약 준수 → [[04-3_REST-API명세]] 공통 절 참조(여기서 재정의 금지).

## 3. 상세 목차

### 3.1 시트 목록/상세

#### 3.1.1 GET /api/projects/{projectId}/sheets — 시트 목록
- 의도 메모: 도면목록 화면용. 페이지네이션·정렬·필터(분야/도면번호/태그/상태) 쿼리 파라미터 정의. 데이터테이블 컬럼토글 대응 필드셋.
- 채울 것: 쿼리 파라미터 표(page, size, sort, filter[discipline], filter[tag], q), 응답 항목 스키마.

#### 3.1.2 GET /api/projects/{projectId}/sheets/{sheetId} — 시트 상세
- 의도 메모: 뷰어 진입 시 단건 조회. 현재 버전 포인터 + 기하 JSON URL + 썸네일 URL + 엔티티 바인딩 요약.
- 채울 것: 상세 응답 스키마(메타 + currentVersionId + renderArtifacts).

#### 3.1.3 시트 메타 스키마
- 의도 메모: id, sheetNumber(도면번호), title, discipline(분야), tags[], status, currentVersionId, updatedAt.
- 채울 것: 필드 표 + 타입 + 출처(파싱 산출물 vs 사용자입력) 매핑.
- ❓ entityIds[] (설비 엔티티 ID 배열) 포함 여부 — Phase 2 결정 대기 → TBD.

#### 3.1.4 정렬·필터·검색 규약
- 의도 메모: 공통 데이터테이블 동작과 일치시킬 정렬키/필터 연산자. 전역 규약 있으면 링크 위임.

### 3.2 버전세트

#### 3.2.1 GET /api/sheets/{sheetId}/versions — 버전 목록
- 의도 메모: 버전 히스토리 패널/시트비교 진입용. 버전 번호·업로드자·업로드시각·변환상태.
- 채울 것: 응답 항목 스키마, 정렬(최신순 기본).

#### 3.2.2 GET /api/sheets/{sheetId}/versions/{versionId} — 버전 상세
- 의도 메모: 특정 버전의 기하 JSON/썸네일 포인터 + 변환 메타. 시트비교(diff) 시 A/B 버전 각각 조회.
- 채울 것: 상세 스키마.

#### 3.2.3 버전 모델/세트 개념
- 의도 메모: "버전세트(version set)" = 동일 도면번호의 버전 묶음. 단조 증가 버전번호. currentVersion 승격 규칙.
- 채울 것: 버전번호 채번 규칙, current 포인터 갱신 시점.
- ❓ 버전 삭제/롤백 허용 여부 → TBD(감사 추적 영향).

#### 3.2.4 버전 간 관계
- 의도 메모: 마크업/이슈가 특정 버전에 고정(pin)되는지, 시트 단위로 따라오는지. 상세 정책은 [[04-3-2_마크업-이슈-API]]로 위임.

### 3.3 업로드/변환 트리거

#### 3.3.1 POST /api/projects/{projectId}/sheets/upload — 업로드 + 변환 착수
- 의도 메모: DWG/[[PDF]] 업로드 → 파싱·변환 잡 등록. 멀티파트 업로드 vs presigned URL 방식 선택.
- 채울 것: 요청 본문, 파일 제약(확장자/용량), 응답(생성된 sheetId/versionId + jobId).
- ❓ 동기 vs 비동기 변환 — 비동기 잡 큐 가정(권장) → 확정 TBD.

#### 3.3.2 변환 잡 상태 조회 / 진행 알림
- 의도 메모: GET /api/jobs/{jobId} 폴링 + 실시간 진행 푸시. [[WebSocket]] 채널로 변환 진행률/완료 토스트.
- 채울 것: 잡 상태 enum(QUEUED/PARSING/RENDERING/DONE/FAILED), 진행률 페이로드.

#### 3.3.3 변환 파이프라인 연계
- 의도 메모: 본 API는 트리거·상태만. 실제 DWG→기하 JSON 변환 로직은 [[04-2_DWG-파싱-파이프라인]]로 위임(DRY).

#### 3.3.4 실패/재시도 처리
- 의도 메모: 변환 실패 시 에러코드 표준, 재시도 엔드포인트(POST .../versions/{versionId}/reconvert).
- ❓ 자동 재시도 횟수 정책 → TBD.

### 3.4 썸네일

#### 3.4.1 GET /api/sheets/{sheetId}/thumbnail — 시트 썸네일
- 의도 메모: 시트 목록 카드/데이터테이블 미리보기. 사이즈 파라미터(?size=sm|md|lg).
- 채울 것: 응답(이미지 바이너리 vs URL 리다이렉트), 캐시 헤더(ETag/Cache-Control).

#### 3.4.2 버전별 썸네일
- 의도 메모: GET /api/sheets/{sheetId}/versions/{versionId}/thumbnail. 시트비교 썸네일 대비용.

#### 3.4.3 썸네일 생성 시점/저장
- 의도 메모: 변환 잡 완료 시 생성. 저장 경로/스토리지 규약, 미생성 시 빈상태 플레이스홀더.
- ❓ 썸네일 포맷(PNG/WebP) 및 해상도 단계 → TBD.
