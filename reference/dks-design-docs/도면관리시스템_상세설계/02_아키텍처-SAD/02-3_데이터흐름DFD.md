# 데이터 흐름도 (DFD)

## ① 목적
- 청주 사업장 [[도면관리 시스템]]의 **데이터 이동 경로**를 단일 SoT로 정의
- 소유 범위 = **업로드 → 변환 → 렌더 → 오버레이** 한 줄기. 그 외 주제(컴포넌트 구조·배포·보안)는 [[링크]]로 위임
- 핵심 전제 재확인: **[[DWG]] 원본을 편집하지 않는다.** 본 시스템은 *CAD 편집기*가 아니라 *뷰어 + 주석/이벤트 오버레이*

## ② 배경 / 전제
- 흐름 골격: `DWG → 백엔드 파싱 → 기하 JSON → 웹 2D 뷰어 렌더 → 마크업/이슈/이벤트 오버레이`
- 1차 범위 = **2D 시트(도면) 한정**. 3D BIM 뷰어/Bridge는 범위 밖
- 정밀 CAD 편집(osnap, 좌표 편집, [[SHX]] 폰트 정밀 렌더)은 범위 밖 — 과거 Development_Design 초안 가정 폐기
- 벤치마크 = [[ACC Build]] (시트/마크업/이슈/시트비교 모듈)
- 차별점 = 오버레이를 **픽셀 + 설비 엔티티 ID** 양쪽에 바인딩 → 향후 [[TypeDB]] 온톨로지 연동(Phase 2)
- Phase 0 산출물 존재: 도면 파싱(분야·도면번호·태그) 결과 재사용

## 도면 업로드 → 파싱 → JSON 적재
> 흐름 1. 원본 보존 + 비편집 변환 파이프라인. 본 절이 "변환 산출물 계약"의 SoT.

### 업로드 진입
- 외부 엔티티(사용자) → 업로드 프로세스 → 원본 저장소([[CDE]] 파일 영역)
- 채울 것: 허용 포맷([[DWG]]/[[DXF]]/[[PDF]]?), 최대 용량, 중복/버전 처리 규칙(파일명 vs 도면번호 키)
- ❓ TBD: PDF 도면도 1차 지원 대상인가, DWG 전용인가

### 변환 작업 디스패치
- 업로드 완료 이벤트 → 변환 큐 → 워커(파서) 비동기 처리
- 채울 것: 동기/비동기 경계, 작업 상태 모델(대기/변환중/완료/실패), 진행률 통지 채널([[WebSocket]] vs 폴링)
- ❓ TBD: 변환 엔진 선택(자체 파서 / [[ODA File Converter]] / 외부 라이브러리)

### 기하 추출 → 기하 JSON
- 파서가 도면 엔티티(라인/폴리라인/원/호/텍스트/블록)를 **기하 JSON**으로 직렬화
- 채울 것: JSON 스키마 골격(레이어·바운딩박스·좌표계·단위), 좌표 정규화 규칙(월드→뷰 변환 파라미터)
- 채울 것: 텍스트는 *형상*이 아닌 *문자열 + 위치*로만 보존(정밀 폰트 렌더 미지원 전제)
- ❓ TBD: 대형 도면 타일링/LOD 분할 여부

### 메타 추출 (Phase 0 재사용)
- 분야·도면번호·태그 → 도면 메타 테이블 적재
- 채울 것: Phase 0 산출물 ↔ 본 적재 스키마 매핑, 설비 엔티티 후보 태그의 보존 위치
- 위임: 메타 스키마 상세 → [[03_데이터모델]]

### 적재 & 인덱싱
- 기하 JSON → 객체 저장소 / 메타 → DB → 검색 인덱스
- 채울 것: 저장 위치 분리(blob=기하, RDB=메타·관계), 버전별 적재 키, 무결성 체크
- ❓ TBD: 기하 JSON 저장소(파일시스템 / S3 호환 / DB blob)

## 뷰어 렌더 흐름
> 흐름 2. 적재된 기하 JSON을 웹 2D 뷰어가 읽어 화면에 그리는 읽기 경로.

### 시트 열기 요청
- 사용자 시트 선택 → API → 기하 JSON + 메타 응답
- 채울 것: 응답 페이로드 구성(기하 URL vs 인라인), 캐시 헤더, 권한 체크 지점

### 클라이언트 렌더
- 뷰어가 기하 JSON 파싱 → Canvas/SVG 렌더 → 팬/줌 인터랙션
- 채울 것: 렌더 기술 선택([[Canvas]] 2D / [[PDF.js]](PDF 경로) / [[WebGL]]), 좌표 변환(월드↔스크린), 레이어 토글
- ❓ TBD: 대형 도면 가상화/타일 로딩 전략

### 뷰포트 상태
- 줌 레벨·팬 오프셋 → 오버레이 좌표 변환 기준 제공
- 채울 것: 뷰포트→오버레이 좌표 동기화 규약(오버레이 흐름과의 공유 계약)

## 이슈/마크업 저장 흐름
> 흐름 3. 렌더된 시트 *위에* 얹는 주석·이슈의 쓰기 경로. **원본 불변, 오버레이만 영속화.**

### 오버레이 작성 (클라이언트)
- 마크업(펜·도형·화살표) / 이슈 핀 → 뷰포트 좌표로 그림
- 채울 것: 입력→정규화 좌표 변환(줌 무관 저장), 마크업 도형 모델, 이슈 핀 ↔ 상세폼 결합
- 위임: UI 인터랙션 상세 → [[_ACC-Build-화면분석-재현설계]]

### 좌표 + 엔티티 바인딩 (★차별점)
- 오버레이 위치를 **(a) 정규화 픽셀좌표** + **(b) 설비 엔티티 ID**(가능 시) 둘 다로 저장
- 채울 것: 엔티티 ID 해석 방법(클릭 지점 ↔ 기하/메타 역참조), 바인딩 실패 시 픽셀-only 폴백
- ❓ TBD: 엔티티 매칭을 클라이언트 hit-test로 할지 서버 조회로 할지

### 영속화
- 마크업/이슈 → API → 오버레이 저장소(원본과 분리된 테이블)
- 채울 것: 저장 스키마(시트ID·버전·좌표·엔티티ID·작성자·상태), 이슈 상태/필터 필드
- 위임: 스키마 상세 → [[03_데이터모델]]

### 동기화 / 협업
- 저장 결과 → 같은 시트 보는 사용자에게 반영
- 채울 것: 실시간 통지([[WebSocket]]) vs 새로고침, 충돌/동시편집 정책
- ❓ TBD: 1차에 실시간 협업 포함 여부 (ACC 동등 재현 우선순위 결정 필요)

### 버전/비교 연계
- 시트 버전 교체 시 오버레이의 버전 추종 규칙
- 채울 것: 마크업이 특정 버전에 고정인지, 엔티티ID 기반으로 버전 이월되는지
- 위임: 시트비교 diff 로직 → [[시트비교-설계]] (별도 문서)

## 이벤트 (향후 AI 하이라이트) 흐름
> 흐름 4. **Phase 2** [[TypeDB]] 온톨로지 연동. 본 절은 *연결 지점*만 정의, 상세는 위임.

### 이벤트 인입
- 외부 이벤트(알람/AI 채팅 질의 결과) → 하이라이트 대상 엔티티 ID 산출
- 채울 것: 이벤트 소스 종류, 엔티티 ID 표준(흐름 3의 바인딩 ID와 동일 키 사용)

### 엔티티 → 좌표 역해석
- 엔티티 ID → 해당 시트/뷰포트 위치 역조회 → 뷰어 하이라이트 지시
- 채울 것: 엔티티가 여러 시트에 등장할 때 선택 규칙, 미렌더 시트 처리
- ❓ TBD: 역해석을 흐름 1 적재 시 사전 계산할지 질의 시 계산할지

### AI 채팅 바인딩
- AI 질의 ↔ 영향도/검색 결과 ↔ 뷰어 하이라이트 왕복
- 위임: 온톨로지·AI 채팅 상세 → [[Phase2-온톨로지연동-설계]], [[데이터-지식-스튜디오]]
