# 02-6 보안 아키텍처

## ① 목적

청주 사업장 도면관리 시스템의 **보안 설계 단일 진실 위치(SoT)**.
이 문서가 소유하는 범위는 **권한(인증/인가) · 도면 원본 유출방지 · 전송/저장 보안 · 감사 로깅** 뿐. 그 외(데이터 모델 상세, API 스펙, 배포 토폴로지)는 [[링크]]로 위임.

## ② 배경 / 전제

- 시스템 흐름 = [[DWG]] 원본 → 백엔드 파싱 → 기하 JSON → 웹 2D 뷰어 렌더 → 마크업/이슈/이벤트 오버레이. **DWG 원본은 편집하지 않음**.
- 보안 관점 핵심 자산 2종: **(a) DWG 원본 파일**(파싱 전, CDE 저장소), **(b) 파싱 산출물 기하 JSON**(뷰어 전송용). 둘의 보호 등급·전송 경로가 다름 → 본 문서에서 구분.
- 도면 = 진실원천(DKS 컨텍스트). 유출 시 사업장 설비 배치 노출 → **유출방지가 본 시스템 보안의 1순위 동인**.
- 1차 범위 = 2D 시트 한정. Phase 2의 [[TypeDB]] 온톨로지 연동·AI 채팅 시 추가되는 권한 표면은 본 문서에서 "확장 지점"으로만 표기, 상세는 Phase 2 설계로 위임.
- 납품처 = 청주 사업장(온프레미스 가능성). 인터넷 격리망/폐쇄망 여부가 전송보안 설계를 좌우 → ❓ 결정 대기.

## 도면 원본 보호 (서명 URL)

> 채울 것: DWG 원본과 기하 JSON이 클라이언트로 새지 않게 하는 핵심 방어. 본 시스템 보안의 1순위.

### 자산 분류 / 노출 경계
> 채울 것: 무엇이 브라우저까지 내려가고, 무엇이 서버에만 남는가.
- DWG 원본: **클라이언트로 절대 직접 전송 안 함** (다운로드는 명시적 권한 + 감사 로깅 동반) — 원칙 확정 대기 ❓
- 기하 JSON: 뷰어 렌더용으로 전송됨 → 이것 자체가 도면 정보. 보호 필요.

### 서명 URL (Pre-signed URL)
> 채울 것: 단기 만료 서명 URL로 파일 접근 제어. 스토리지 종류에 따라 방식 분기.
- 만료 시간: 짧게(예: 분 단위) — 구체값 TBD
- 발급 조건: 인가 통과 후에만 발급, 발급 자체를 감사 로그에 기록
- 스토리지: S3 호환 오브젝트 스토리지 presigned vs 자체 토큰 게이트웨이 — ❓ 온프레미스 환경 따라 결정

### 워터마킹 / 추적
> 채울 것: 유출 발생 시 출처 추적 수단. 뷰어 렌더 위 사용자ID/타임스탬프 오버레이.
- 뷰어 동적 워터마크(사용자명·시각): 적용 여부 TBD
- 다운로드 산출물 워터마크/표식: TBD

### 다운로드 / 내보내기 통제
> 채울 것: 데이터테이블 내보내기·도면 PDF 출력 등 반출 경로 통제.
- 내보내기 권한 분리, 반출 이벤트 감사 기록
- [[PDF.js]] 뷰어 인쇄/저장 차단 수준 — 브라우저 한계 명시 + 서버측 통제 우선

## 전송 / 저장 암호화

> 채울 것: 데이터 in-transit / at-rest 보호. 폐쇄망 여부가 설계를 가른다.

### 전송 구간 암호화 (In-Transit)
> 채울 것: 클라이언트↔서버, 서버↔스토리지 구간 TLS.
- HTTPS/TLS 1.2+ 강제, HTTP 리다이렉트 차단
- [[WebSocket]] 보안: 실시간 마크업/이벤트 동기화 채널은 WSS + 토큰 인증
- 내부 서비스 간 통신 암호화 여부 ❓ (온프레미스 단일 호스트면 불요할 수 있음)

### 저장 구간 암호화 (At-Rest)
> 채울 것: DWG 원본·기하 JSON·DB·첨부 사진 저장 시 암호화.
- 오브젝트 스토리지 SSE / 디스크 암호화: 적용 범위 TBD
- DB 저장 데이터(이슈·마크업·사용자) 암호화 — 민감도 따라 선택, 상세 스키마는 [[02-4_데이터모델]]
- 비밀번호 해싱(bcrypt/argon2 등) — 알고리즘 확정 대기

### 비밀/키 관리
> 채울 것: TLS 인증서·서명키·DB 자격증명 보관.
- 시크릿 저장소(env vault / OS keystore) 방식 TBD
- 키 회전 주기 TBD
