# 시스템 아키텍처 설계서 (SAD) 개요

## 1. 목적

- 청주 사업장 납품용 **도면관리 시스템**의 전체 아키텍처를 정의하는 상위 인덱스 문서.
- 본 문서는 **아키텍처 문서 인덱스**만 소유. 세부 설계(데이터모델·뷰어·API·배포 등)는 하위 문서로 위임 (DRY).
- 독자: 백엔드/프론트엔드 개발자, PM, 검수 담당. 첫 진입점 역할.

## 2. 배경 / 전제

- **벤치마크**: [[Autodesk Construction Cloud]](ACC) Build 제품 동등 재현 — 37화면 분석은 [[_ACC-Build-화면분석-재현설계]] 참조.
- **★핵심 스코프 제약** (모든 하위 문서가 전제):
  - [[DWG]] 원본을 **편집하지 않음**. 흐름 = DWG → 백엔드 파싱 → 기하 JSON → 웹 2D 뷰어 렌더 → 마크업/이슈/이벤트 **오버레이**.
  - "CAD 편집기"가 아니라 **"뷰어 + 주석/이벤트 오버레이"** 시스템. 정밀 CAD 편집(osnap, 좌표편집, [[SHX]] 폰트 정밀렌더)은 **범위 밖**.
  - 1차 범위 = **2D 시트(도면) 한정**. 3D [[BIM]] 뷰어/Bridge는 범위 밖.
- **DKS 차별점**: 도면=진실원천. Phase 0 파싱 산출물(분야·도면번호·태그) 보유. ACC엔 없는 **"설비 엔티티 ID" 단위** 핀/마크업/이슈 바인딩이 무기. AI 채팅/지식질의는 Phase 2 [[TypeDB]] 온톨로지 연동으로 추가.

## 3. SAD 구성

> 본 절: SAD 문서군이 어떤 단위로 쪼개지고, 각 문서의 책임 경계를 1줄로 정의. 책임 중복 금지.

### 3.1 문서군 분할 원칙

- 한 문서 = 한 아키텍처 관심사. 교차 주제는 [[링크]]로 위임.
- 모든 하위 문서는 §2 스코프 제약을 전제로 상속.

### 3.2 버전 / 변경 이력

- SAD 버전 관리 정책, 주요 변경 로그 채울 자리. (TBD: 버전 채번 규칙)

## 4. 하위 문서 인덱스

> 본 절: 실제 세부 설계 문서로 가는 라우팅 테이블. 각 문서가 "소유한 주제"와 "다루지 않는 것"을 명시.

### 4.1 데이터 / 파싱 계층

- **[[DWG 파싱 파이프라인 설계]]** — DWG → 기하 JSON 변환, 분야/도면번호/태그 추출, Phase 0 산출물 스키마. ❓파서 엔진 선정 (TBD).
- **[[도면 데이터 모델]]** — 시트/버전/엔티티/태그 관계 모델. 설비 엔티티 ID 체계.

### 4.2 프론트엔드 / 뷰어 계층

- **[[2D 뷰어 아키텍처]]** — 기하 JSON 렌더 엔진([[Canvas]]/[[SVG]]/WebGL 중 선택 ❓), 줌·팬·레이어 토글, 마크업/핀 오버레이 좌표계.
- **[[앱셸 / 공통 UI 설계]]** — 모듈 사이드바·프로젝트 스위처·데이터테이블·우측 인스펙터·모달폼·토스트·빈상태. ACC 공통 UI 패턴 재현.

### 4.3 백엔드 / API 계층

- **[[백엔드 서비스 설계]]** — 모듈별 도메인 서비스(시트·마크업·이슈·시트비교·파일·사진). 표준 response format(CLAUDE.md §5).
- **[[API 명세]]** — Granular Endpoints + Orchestrator. REST 기본, 실시간 이벤트는 [[WebSocket]].

### 4.4 모듈별 기능 설계 (ACC 재현)

- **[[마크업 모듈 설계]]** — 펜·도형·화살표 오버레이.
- **[[이슈 모듈 설계]]** — 핀 + 상세폼 + 목록 + 필터. 핀을 픽셀 + 설비 엔티티 ID 양쪽 바인딩.
- **[[시트비교 모듈 설계]]** — 버전 오버레이 diff.
- **[[파일 / CDE 모듈 설계]]** — 폴더형 [[CDE]].
- (사진/홈 대시보드 위젯 등은 위 모듈 설계 내 또는 별도 분리 ❓)

### 4.5 Phase 2 (온톨로지 / AI)

- **[[온톨로지 연동 설계]]** — [[TypeDB]] 엔티티 바인딩, 영향도 분석, AI 채팅을 뷰어/이슈에 바인딩. 1차 범위 외, 인터페이스 훅만 1차에서 예약.

### 4.6 배포 / 운영

- **[[배포 아키텍처]]** — 청주 사업장 온프레미스/네트워크 구성, 빌드·배포 파이프라인. ❓인프라 제약 (TBD).

## 5. 아키텍처 뷰포인트

> 본 절: 4+1 등 뷰포인트별로 "어느 하위 문서가 그 뷰를 책임지는지" 매핑. 본 문서는 매핑만, 내용은 위임.

### 5.1 논리 뷰 (Logical)

- 도메인 모듈 구조(시트·마크업·이슈·비교·파일·사진) + 엔티티 모델. → [[도면 데이터 모델]], 4.4 모듈군.

### 5.2 프로세스 뷰 (Process)

- 핵심 런타임 흐름: DWG 업로드 → 파싱(비동기 ❓) → 뷰어 렌더 → 오버레이 실시간 동기화([[WebSocket]]). → [[DWG 파싱 파이프라인 설계]], [[API 명세]].

### 5.3 개발 뷰 (Development)

- 프론트/백엔드 코드 구조, 모듈 경계, 공통 UI 컴포넌트 재사용. → [[앱셸 / 공통 UI 설계]], [[백엔드 서비스 설계]].

### 5.4 물리 / 배포 뷰 (Physical)

- 서버·스토리지·네트워크 토폴로지. → [[배포 아키텍처]].

### 5.5 시나리오 뷰 (Use-case)

- 대표 시나리오: 도면 열람 → 이슈 핀 등록 → 버전 비교. 화면 단위는 [[_ACC-Build-화면분석-재현설계]] 참조.
