# ★ 뷰어 + 오버레이 아키텍처

## 1. 목적
- 본 시스템의 심장부 정의: **렌더 베이스(도면 그림) + 오버레이 레이어(마크업·이슈핀·이벤트)** 의 분리 모델.
- "이 시스템은 CAD 편집기가 아니라 **뷰어 + 주석/이벤트 오버레이**" 라는 핵심 전제를 아키텍처 수준에서 못 박는 것이 이 문서의 책임.
- 소유 주제 = **렌더 파이프라인 + 레이어 합성 + 좌표 변환 + 엔티티 바인딩 훅 + 상태 동기화**. 그 외(파싱/이슈 폼/데이터테이블)는 위임.

## 2. 배경 / 전제
- 흐름: [[DWG]] 원본 → 백엔드 파싱 → 기하 JSON → 웹 2D 뷰어 렌더 → 그 위에 마크업/이슈/이벤트 **오버레이**. (파싱 상세는 [[02-3_백엔드-파싱-파이프라인]] 위임)
- **DWG 원본은 절대 편집하지 않는다.** 모든 사용자 입력은 원본과 분리된 오버레이 데이터로만 저장.
- 정밀 CAD 편집(osnap 스냅, 좌표 편집, SHX 폰트 정밀 렌더)은 **범위 밖**. 과거 Development_Design 초안의 CAD-editor 가정 폐기.
- 1차 범위 = **2D 시트 한정**. 3D BIM 뷰어/Bridge 제외.
- 차별점: 핀/마크업/이슈를 픽셀 좌표뿐 아니라 **설비 엔티티 ID** 에도 바인딩 → ACC엔 없는 온톨로지 연동 기반 (Phase 2, [[TypeDB]]). 본 문서는 그 **바인딩 훅(자리)** 만 설계, 온톨로지 로직은 위임.

---

## 3. 렌더 베이스(타일/벡터) 선택지
> 무엇을 채울지: 도면 그림 자체를 화면에 그리는 방식 3안을 비교하고 1안 선정 근거를 기록.

### 3.1 후보 A — 래스터 타일(Tile)
- 백엔드에서 도면을 PNG 타일 피라미드로 사전 렌더 → 줌 레벨별 타일 서빙. 대형 도면·저사양 클라이언트에 유리. 벡터 정보 손실.

### 3.2 후보 B — 벡터 렌더(기하 JSON 직접 그리기)
- 파싱된 기하 JSON을 클라이언트 Canvas/SVG로 직접 렌더. 줌 무손실·엔티티 hit-test 용이(엔티티 바인딩에 직결). 대형 도면 성능 부담.

### 3.3 후보 C — 하이브리드(벡터 + PDF.js)
- [[PDF.js]] 로 PDF 변환본을 베이스 렌더 + 엔티티 메타는 별도 JSON 레이어. 변환 단순·텍스트 검색 가능. 벡터 hit-test 정밀도 제한.

### 3.4 비교표 / 선정
- 무엇을 채울지: 성능·hit-test·구현비용·엔티티 바인딩 적합성 4축 비교표 + 1차 선정안.
- ❓ TBD: 청주 납품 도면 평균 엔티티 수·최대 시트 크기 측정 후 A/B/C 확정. (측정 전 잠정 = 후보 B 우선 검토)

#### 3.4.1 렌더러 라이브러리 후보
- 무엇을 채울지: Canvas2D / WebGL([[PixiJS]] 등) / SVG 중 선택. 엔티티 수 임계점 기준.

## 4. 오버레이 레이어 모델(마크업/이슈핀/이벤트)
> 무엇을 채울지: 베이스 위에 쌓이는 레이어 스택과 각 레이어의 책임·z-order·토글 정의.

### 4.1 레이어 스택(z-order)
- 무엇을 채울지: 하단→상단 순서 정의. 잠정: 베이스(도면) → 비교 diff → 이벤트 → 마크업 → 이슈핀 → 선택/호버 하이라이트.

### 4.2 마크업 레이어
- 펜·도형·화살표·텍스트. 벡터 객체로 저장(픽셀 굽기 금지). 편집/삭제 가능. 상세 도구 UX는 [[마크업-모듈-설계]] 위임, 여기선 **데이터 모델·렌더 합성**만.

### 4.3 이슈핀 레이어
- 핀 = 좌표 + 이슈 ID 참조. 클릭 시 우측 인스펙터 연동. 핀 데이터 모델·앵커 방식만 소유, 이슈 폼/목록은 [[이슈-모듈-설계]] 위임.

### 4.4 이벤트 레이어
- 무엇을 채울지: 실시간 이벤트(설비 알람·상태) 오버레이 표시 모델. 엔티티 바인딩과 직결. ❓ TBD: 이벤트 소스 정의(DKS 연동 범위).

### 4.5 레이어 토글 / 가시성
- 무엇을 채울지: 레이어별 on/off·투명도, 사용자 필터(내 마크업만 등) 상태 보관 위치.

### 4.6 오버레이 데이터 모델(공통)
- 무엇을 채울지: 모든 오버레이 객체 공통 스키마 — id, layerType, geometry(도면좌표), entityRef(nullable), author, createdAt, versionAnchor.

## 5. 좌표계 / 변환(도면좌표 ↔ 화면)
> 무엇을 채울지: 오버레이가 줌/팬에도 도면에 "붙어있게" 만드는 변환 체계. 본 문서의 가장 까다로운 절.

### 5.1 좌표계 정의
- **도면좌표(World/모델 단위)** vs **화면좌표(Screen/px)**. 오버레이 geometry는 항상 도면좌표로 저장(줌 무관 안정성).

### 5.2 변환 행렬
- 무엇을 채울지: pan/zoom 변환 행렬(scale, translate) 정의. World→Screen, Screen→World 역변환.

### 5.3 줌/팬 동작과 레이어 재합성
- 무엇을 채울지: 변환 변경 시 베이스·오버레이 동시 재투영. 타일 베이스(후보 A) 채택 시 정수 줌 레벨 스냅 처리.

### 5.4 핀/마크업 앵커링
- 무엇을 채울지: 핀은 도면좌표 고정, 표시 크기는 화면 px 고정(역스케일) 등 앵커 정책.

### 5.5 ❓ 단위/원점
- ❓ TBD: 파싱 산출물의 단위(mm/도면단위)·원점 기준 확정 ([[02-3_백엔드-파싱-파이프라인]] 출력 스펙 의존).

## 6. 엔티티 바인딩 훅(향후 온톨로지)
> 무엇을 채울지: 오버레이를 픽셀이 아닌 **설비 엔티티 ID** 에 묶는 자리(hook). Phase 2 [[TypeDB]] 연동 전 **인터페이스만** 확정. 우리의 차별 무기.

### 6.1 entityRef 필드
- 모든 오버레이 객체에 `entityRef`(nullable) — 분야·도면번호·설비태그(Phase 0 파싱 산출물) 참조.

### 6.2 hit-test → 엔티티 해석
- 무엇을 채울지: 화면 클릭 → 도면좌표 → 근접 기하 → 엔티티 ID 후보 반환 흐름. 벡터 렌더(후보 B)일 때 정밀도 ↑.

### 6.3 Phase 2 연동 경계
- 무엇을 채울지: entityRef를 [[TypeDB]] 온톨로지 엔티티로 resolve하는 경계 정의. **본 문서는 호출 지점만**, 온톨로지/AI채팅 로직은 [[온톨로지-바인딩-설계]] 위임.
- ❓ TBD: Phase 1에서 entityRef를 빈 필드로만 둘지, Phase 0 태그까지 채울지 결정.

## 7. 상태 동기화
> 무엇을 채울지: 오버레이의 로컬 편집 상태 ↔ 서버 영속 상태 ↔ 멀티유저 실시간 동기화 모델.

### 7.1 클라이언트 로컬 상태
- 무엇을 채울지: 뷰어 상태(현재 변환·선택·레이어 토글)와 미저장 마크업의 메모리 모델.

### 7.2 서버 영속 / 저장 트리거
- 무엇을 채울지: 오버레이 CRUD API 호출 시점(즉시 저장 vs 명시 저장). API 표준 응답은 전역 규칙 따름.

### 7.3 실시간 협업(멀티유저)
- 무엇을 채울지: 동일 시트 동시 열람 시 핀/마크업 실시간 반영. [[WebSocket]] 기반 broadcast 잠정.
- ❓ TBD: Phase 1에 실시간 협업 포함 여부(ACC 평가판 동등성 vs 범위 축소). 미포함 시 폴링/새로고침으로 대체.

### 7.4 버전 앵커링(시트비교 연계)
- 무엇을 채울지: 오버레이가 특정 시트 버전에 묶이는 방식(versionAnchor). 시트비교 diff 레이어와의 관계는 [[시트비교-모듈-설계]] 위임.
