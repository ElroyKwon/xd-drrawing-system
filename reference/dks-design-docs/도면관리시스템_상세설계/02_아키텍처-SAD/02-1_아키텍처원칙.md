# 아키텍처 원칙

> 청주 사업장 납품 "도면관리 시스템"의 **설계 의사결정 기준**을 소유하는 문서.
> 구체 컴포넌트 배치·기술 선정은 위임([[02-2_시스템구성도]]), 여기서는 "왜 그렇게 결정하는가"의 판단 기준만 다룬다.

## 1. 목적
- 본 문서는 SAD 전반에서 반복 인용할 **4대 아키텍처 원칙**을 정의.
- 설계 충돌 시 "어느 원칙이 우선하는가"의 판정 근거 제공.
- (채울 것) 각 원칙을 "결정 규칙(Rule) + 근거(Rationale) + 적용 예" 3단으로 명문화.

## 2. 배경 / 전제
- 1차 목표 = [[Autodesk Construction Cloud]](ACC) Build 제품의 **동등 재현** (37화면 분석 기준 → [[_ACC-Build-화면분석-재현설계]]).
- ★핵심 스코프 제약 (모든 원칙의 상위 전제):
  - [[DWG]] 원본을 **편집하지 않는다**. 흐름 = DWG → 백엔드 파싱 → 기하 JSON → 웹 2D 뷰어 렌더 → 마크업/이슈/이벤트 **오버레이**.
  - 정밀 CAD 편집(osnap 스냅·좌표 편집·[[SHX]] 폰트 정밀 렌더)은 **범위 밖**. (과거 Development_Design 초안의 CAD-editor 가정 폐기.)
  - 1차 범위 = **2D 시트 한정**. 3D BIM 뷰어/Bridge 범위 밖.
- 차별점 = ACC엔 없는 "[[온톨로지]] 엔티티 단위" 검색·영향도·AI채팅을 향후([[Phase 2]]) [[TypeDB]] 연동으로 뷰어/이슈에 바인딩. (도면=진실원천, Phase 0 파싱 산출물 존재.)

## 3. 상세 목차 (팀 작성 골격)

### 3.1 데이터 주권 / 온프레 우선
> 청주 사업장 납품 = 고객 인프라 내 운영 전제. 외부 클라우드 의존 최소화 기준.

#### 3.1.1 결정 규칙
- 도면 원본·파싱 산출물·이슈/마크업 데이터는 **고객 온프레 경계 내** 저장.
- (채울 것) 외부 SaaS/클라우드 API 사용 가능 조건과 금지선 명문화.

#### 3.1.2 근거
- 도면 = 보안 자산. 데이터 반출 정책·망 분리 가능성 반영. (채울 것: 청주 사업장 보안 요건 확인 후 기술.)

#### 3.1.3 AI/온톨로지 데이터 경계
- Phase 2 [[TypeDB]]·AI채팅도 온프레 우선. LLM 호출 시 외부 모델 사용 여부 ❓ TBD.

### 3.2 뷰어-오버레이 분리
> 본 시스템 정체성 = "CAD 편집기 아님, 뷰어 + 주석/이벤트 오버레이". 가장 중요한 구조 원칙.

#### 3.2.1 레이어 분리 규칙
- **베이스 레이어**(불변): DWG 파싱 기하 JSON → 2D 렌더. 읽기 전용.
- **오버레이 레이어**(가변): 마크업·이슈 핀·이벤트. 베이스와 **물리적 분리** 저장·렌더.
- (채울 것) 두 레이어의 좌표계 일치·정합 방식 → 상세는 [[03-1_뷰어-오버레이-설계]]에 위임.

#### 3.2.2 렌더 기술 경계
- 2D 렌더 = [[Canvas]]/[[WebGL]] 기반 (기하 JSON), 텍스트/PDF 시트는 [[PDF.js]] 후보. ❓ TBD: 렌더 스택 확정.
- 정밀 스냅/좌표 편집 미구현 = **의도적 비범위** (스코프 제약 재확인).

#### 3.2.3 앵커링(Anchoring) 원칙 — 차별점
- 오버레이 객체는 픽셀 좌표뿐 아니라 **설비 엔티티 ID**에도 바인딩 가능 (ACC 대비 우리의 무기).
- (채울 것) 픽셀-앵커 vs 엔티티-앵커 이중 바인딩 모델 개요. 상세 위임.

### 3.3 API 세분화 / 오케스트레이션
> 글로벌 규칙 §5(Granular Endpoints + Orchestrator) 적용 기준.

#### 3.3.1 단일 책임 엔드포인트
- 1 API = 1 책임. 파싱/렌더/마크업/이슈/비교를 각각 독립 호출 가능하게.
- (채울 것) 모듈별 세부 API 후보 목록 → 명세는 [[04-1_API명세]](예정)에 위임.

#### 3.3.2 오케스트레이터 계층
- 복합 플로우(예: 시트 업로드 → 파싱 → 기하 JSON 생성 → 썸네일)는 상위 Orchestrator가 세부 API 순차/병렬 호출. 세부 API 직접 재구현 금지.

#### 3.3.3 표준 응답 구조
- `{ success, data, error, meta }` 일관 적용 (글로벌 규칙 §5 응답 표준).

#### 3.3.4 실시간 채널
- 파싱 진행률·동시 마크업 반영은 [[WebSocket]] 후보. ❓ TBD: 폴링 vs WebSocket 결정.

### 3.4 단순성 우선
> 글로벌 규칙(Karpathy 4원칙 §2) 반영. ACC 동등 재현 범위에서 과설계 차단.

#### 3.4.1 비범위 선언으로 단순화
- CAD 편집·3D·정밀 폰트 렌더 제외 → 구현 복잡도 의도적 절감.
- (채울 것) "재현하지 않을 ACC 기능" 명시 목록.

#### 3.4.2 기존 자산 재사용 (DRY)
- Phase 0 파싱 산출물(분야·도면번호·태그) 재사용. 신규 파서 중복 구현 금지.
- 공통 UI(앱셸·데이터테이블·인스펙터·위젯카드·모달·토스트)는 공유 컴포넌트화 → 상세 위임.

#### 3.4.3 Phase 경계 단순성
- Phase 1 = 뷰어+오버레이+ACC 7모듈 동등 재현. Phase 2 = 온톨로지/AI. 경계 흐리지 않기.
