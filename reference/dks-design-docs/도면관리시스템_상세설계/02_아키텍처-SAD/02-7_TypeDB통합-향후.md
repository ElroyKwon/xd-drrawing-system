## 목적

Phase 2에서 도입할 [[TypeDB]] 온톨로지 연동의 **접점(seam)을 미리 예약**하는 문서. 1차(Phase 1) 구현은 [[ACC]] Build 동등 재현에 집중하되, 향후 "온톨로지 엔티티 단위" 검색·영향도·AI채팅을 뷰어/이슈에 바인딩할 수 있도록 **설계 가드만** 박아둠.

- 이 문서가 소유하는 것: **Phase 2 통합 지점의 위치·계약·막지 말아야 할 설계 제약**.
- 이 문서가 소유하지 **않는** 것: 온톨로지 스키마 자체, AI채팅 UX, 파싱 파이프라인 → 각각 [[링크]] 위임.

> 전제(전 문서 공통): DWG 원본 비편집. 흐름 = [[DWG]] → 백엔드 파싱 → 기하 JSON → 웹 2D 뷰어 → 마크업/이슈 **오버레이**. Phase 2는 이 오버레이 좌표를 **설비 엔티티 ID**에 추가로 묶는 것이 차별점.

---

## 배경 / 전제

- Phase 0 도면 파싱 산출물(분야·도면번호·태그)이 **이미 존재** → 엔티티 후보의 1차 원천.
- [[ACC]] Build엔 없는 무기 = 핀/마크업/이슈를 **픽셀이 아닌 엔티티 ID에도** 결박.
- Phase 1은 TypeDB **없이** 완결되어야 함(온톨로지 미준비 시에도 제품 동작).
- 결합도 원칙: Phase 1 코드는 TypeDB를 **모름**. 통합은 어댑터/옵셔널 필드로만.

## 통합 지점(검색/영향도/채팅)

Phase 1 기능에 **나중에 끼워 넣을 3개 접점**. 각 접점은 Phase 1에서 "비어 있어도 동작"하는 옵셔널 확장으로 설계.

### 검색 통합 지점
- Phase 1: 시트/이슈/파일 **텍스트·메타 검색**(도면번호·태그·분야).
- Phase 2: 동일 검색창에 **엔티티 검색 모드** 추가 → TypeDB 질의 결과를 시트/핀으로 역참조.
<!-- 채울 것: 검색 API 응답에 entityRefs[] 옵셔널 필드 예약 위치 -->

#### 검색 API 계약 예약
- 응답 표준(`success/data/error/meta`) 유지. `data.results[].entityRefs` 옵셔널 배열만 향후 추가.
<!-- 채울 것: entityRef 최소 형태 = {entityId, type, confidence} TBD -->

### 영향도 통합 지점
- Phase 1: 없음(ACC 미보유 기능).
- Phase 2: 엔티티 선택 → "이 설비가 참조된 시트/이슈/마크업" 역색인 + 온톨로지 관계 traversal.
<!-- 채울 것: 영향도 = (오버레이 역색인) ∪ (TypeDB 관계) 합집합 모델 -->

#### 역색인 사전 적재
- Phase 1부터 오버레이 저장 시 **entityId 컬럼을 NULL 허용으로 미리 보유** → Phase 2에 backfill만.

### AI채팅 통합 지점
- Phase 1: 없음. Phase 2: 뷰어/이슈 컨텍스트를 채팅에 주입, 답변 근거를 핀·시트로 하이라이트.
- 채팅↔뷰어 양방향은 [[WebSocket]] 채널로 상정(연결은 Phase 2).
<!-- 채울 것: 채팅 컨텍스트 페이로드 = {projectId, sheetId, selection:{entityIds[], bbox}} TBD -->

> 채팅 UX·프롬프트·온톨로지 질의 변환은 본 문서 범위 밖 → [[데이터-지식-스튜디오]] / 향후 AI채팅 설계 문서로 위임.

## 엔티티↔도면객체 매핑 경계

핵심 난제 = "파싱된 기하 객체 / 오버레이 핀"을 "온톨로지 설비 엔티티"에 어떻게 묶는가. **경계(누가 무엇을 소유)**만 여기서 확정, 매핑 알고리즘은 위임.

### 매핑 키 정의
- 후보 키: 도면 태그 텍스트(예: 장비 태그) ↔ 엔티티 `naturalId`.
- 1:1 / 1:N / N:M 가능성 명시 → 매핑은 **다대다 허용** 전제.
<!-- 채울 것: 매핑 키 우선순위(태그>좌표근접>수동) TBD -->

### 매핑 책임 경계
- **도면관리 시스템(본 프로젝트)**: 오버레이 객체에 `entityId` 슬롯 제공 + 수동 결박 UI.
- **DKS/온톨로지 측**: 엔티티 마스터·관계 그래프 소유, 자동 매핑 제안 제공.
<!-- 채울 것: 경계선 다이어그램(우리 DB ↔ TypeDB) 1장 -->

#### 신뢰도/검증 모델
- 자동 매핑은 `confidence` 동반, 임계값 미만은 **수동 확인 큐**로.
- ❓ 매핑 충돌(같은 객체 ↔ 복수 엔티티) 해소 정책 = TBD.

### 좌표 동기화 경계
- 오버레이 좌표는 **뷰어 좌표계**가 진실. 엔티티는 ID 참조만, 좌표 미소유.
- 버전 변경(시트비교) 시 entityId는 유지, bbox만 갱신 → [[시트비교]] 설계와 정합 확인 필요.
<!-- 채울 것: 버전 변경 시 매핑 승계 규칙 -->

## 지금 막지 않을 설계 가드

Phase 1에서 **하지 말아야 할 결정**(Phase 2 진입로를 막는 것들) 목록. 위반 시 리뷰 차단.

### 데이터 가드
- 오버레이(핀·마크업·이슈) 레코드에 `entityId NULL 허용` 필드 **미리 보유**(Phase 1 미사용).
- 식별자는 **불투명 UUID** 사용 → 외부 엔티티 ID와 충돌·결합 회피.
- 검색·이슈 API 응답에 옵셔널 `entityRefs[]` **추가 여지**만 남김(지금 구현 X).

### 결합도 가드
- Phase 1 코드는 [[TypeDB]] 드라이버·스키마를 **import 금지**.
- 통합은 별도 **온톨로지 어댑터 모듈**(인터페이스만 Phase 1에 stub) 경유.
<!-- 채울 것: 어댑터 인터페이스 시그니처 초안 -->

### UI 가드
- 검색창/인스펙터 패널은 **모드 추가가 가능한 컴포넌트 구조**(엔티티 탭 자리 예약).
- 핀 상세폼에 "연결된 설비" 섹션을 **숨김 슬롯**으로 자리만.

### 안티 가드(과설계 금지)
- Phase 1에서 TypeDB 스키마 설계·질의 코드 **작성 금지**(YAGNI).
- 엔티티 자동 매핑 알고리즘 **선구현 금지** → 슬롯만 예약.
