# 이슈 스키마

## 1. 목적
- [[ACC]] Build "이슈" 모듈 동등 재현 위한 데이터 모델 정의
- 소유 범위 = **이슈 본체 + 부속(핀/댓글/첨부/참조/활동로그)** 한정
- 도면/마크업/사용자/권한 등은 본 문서 소유 아님 → [[링크]]로 위임

## 2. 배경 / 전제
- **핵심 스코프**: [[DWG]] 원본 편집 안 함. 백엔드 파싱 → 기하 JSON → 2D 뷰어 렌더 → 그 위에 이슈 핀을 **오버레이**. 이슈는 뷰어 좌표(또는 엔티티 ID) 위에 얹히는 메타데이터.
- 1차 범위 = **2D 시트 한정**. 3D BIM 핀 좌표계는 범위 밖 → TBD(Phase 2 이후).
- **차별점(우리 무기)**: ACC는 핀을 픽셀 좌표에만 묶음. 우리는 핀을 **설비 엔티티 ID**에도 바인딩 가능 → 향후 [[TypeDB]] 온톨로지 영향도/AI채팅 연결. 본 문서는 그 바인딩 컬럼만 정의(질의 로직은 위임).
- DB 종류·ORM·ID 전략(UUID vs serial)은 상위 [[03-2_도메인스키마]] 공통 규약 따름 → 본 문서는 컬럼 의미만.

## 3. 상세 목차 (스키마 골격)

### Issue (본체)
> 채울 것: 이슈 1건의 전체 컬럼 정의표 + 인덱스 후보.

#### 식별/소속
> id, project_id, sheet_id(FK→[[03-2-1_도면]]), issue_number(프로젝트 내 표시번호), title.
- `issue_number`: 사람이 보는 순번 ❓ 프로젝트 전역 vs 시트별 — TBD

#### 상태(Status) 머신
> ACC 상태값 매핑: Open / In Review(Answered) / Closed / Void. 상태 전이 규칙 1줄.
- enum 값 확정 ❓ ACC 평가판 실제 라벨 재확인 필요 — TBD
- 상태 전이 이력은 본체에 안 둠 → [[#ActivityLog (audit)]]에 기록

#### 유형 계층 (Type / Subtype)
> issue_type_id + 상위 카테고리 2계층 구조. 마스터 테이블 분리.
- 유형 마스터(category → type) 별도 테이블 ❓ 하드코딩 vs 관리화면 제공 — TBD

#### 담당/책임자
> assignee(담당자), watchers(참조인 다대다), created_by, root_cause(선택).
- assignee 단일 vs 복수 ❓ ACC는 단일 — 단일로 가정
- watchers는 조인테이블 → 부속 절에서 정의

#### 확인자(Reviewer) / 워크플로
> 검토 승인 흐름: assignee 답변 → reviewer 확인 → Closed.
- 확인자 필수 여부 ❓ ACC 옵션 — TBD

#### 마감(Due) / 기한
> due_date, 우선순위(priority), 마감 임박/초과 플래그(파생, 저장 안 함).

#### 엔티티 바인딩 (★차별 컬럼)
> linked_entity_id(설비 엔티티), entity_binding_type(pin좌표 only / entity only / both).
- Phase 0 파싱 산출물(분야·도면번호·태그)과 조인 키 정합성 ❓ — TBD
- 질의/영향도 로직은 본 문서 밖 → [[TypeDB]] 연동 설계로 위임

### IssuePin (핀 좌표)
> 채울 것: 이슈를 시트 위에 고정하는 핀 좌표 테이블.

#### 좌표 모델
> pin_id, issue_id(FK), sheet_id, sheet_version_id, x, y(뷰어 모델좌표계).
- 좌표 기준 = **뷰어 정규화 좌표** vs 도면 월드좌표 ❓ — 뷰어 좌표 권장(렌더 일관성). TBD 확정.

#### 버전 추종
> 시트 새 버전 업로드 시 핀 위치 처리: 유지 / 재배치요청 플래그.
- [[03-2-3_마크업]] 버전 추종 규칙과 동일 정책 공유 → 중복정의 금지, 마크업 문서 참조.

#### 멀티 핀
> 이슈 1건당 핀 0~N개 허용 여부.
- ACC = 이슈당 핀 1개 기본 → 1:1 가정, 1:N은 TBD

### Comment / Attachment / Reference (부속)
> 채울 것: 이슈에 매달리는 3종 부속 테이블.

#### IssueComment
> comment_id, issue_id(FK), author, body(텍스트), created_at, edited_at.
- 멘션(@user) / 마크다운 지원 ❓ — TBD
- 답글 트리(중첩) vs 평면 ❓ — 평면 가정

#### IssueAttachment
> attachment_id, issue_id(FK), file_ref(→[[CDE]] 파일), thumbnail, mime, size.
- 첨부 실체는 파일 모듈 소유 → 여기선 **참조키만**. 저장소 설계는 [[CDE]]/파일 스키마로 위임.
- [[사진]] 모듈 사진을 이슈에 첨부하는 경로 ❓ — TBD

#### IssueReference (교차 참조)
> ref_id, issue_id, ref_type(sheet/markup/file/other_issue), ref_target_id.
> ACC "참조" = 이슈를 다른 객체에 연결. 마크업(주석) ↔ 이슈 연결 핵심 채널.
- 마크업-이슈 연결은 [[03-2-3_마크업]]과 양방향 ❓ — 소유 측 결정 TBD

#### Watcher (참조인 조인)
> issue_id × user_id 다대다. 알림 대상.

### ActivityLog (audit)
> 채울 것: 이슈 전 생애주기 감사 로그. ACC "활동(Activity)" 탭 재현.

#### 로그 엔트리 모델
> log_id, issue_id, actor, action_type, field_changed, old_value, new_value, at.

#### 액션 유형
> created / status_changed / assigned / commented / attached / due_changed / reopened ...
- enum 범위 확정 TBD

#### 표시 vs 저장
> 활동 탭 렌더는 본 로그를 시간역순 조회. UI는 [[_ACC-Build-화면분석-재현설계]] 인스펙터 패널 참조.
- 시스템 전역 이벤트 로그와의 경계 ❓ → [[03-2-5_이벤트활동]]와 책임 분리(이슈 한정 vs 전역). TBD
