# 프로젝트/사용자/권한 스키마

## ① 목적
- 청주 [[도면관리 시스템]]의 **계정·프로젝트·멤버·역할** 도메인 테이블 정의(소유 범위 = 이 4개 영역 한정).
- [[ACC Build]] 동등 재현을 위한 **멀티 프로젝트 + 역할 기반 접근제어([[RBAC]])** 골격 확정.
- 다른 도메인(시트/이슈/마크업/파일)은 본 스키마의 `project_id`·`user_id`를 FK로 참조 → 여기서 단일 정의(SoT).

## ② 배경 / 전제
- **스코프 제약**: DWG 비편집. 본 문서는 데이터/권한 계층만 다룸 — 뷰어/오버레이 권한 *판정 지점*만 명시, 렌더링 로직은 위임.
- ACC 계정 모델 차용: **Account(Hub) → Project → Member(Role)** 3계층. 청주 단일 사업장이나 향후 다(多)사업장 대비 Hub 분리 유지.
- 권한은 **모듈 단위(시트/이슈/마크업/파일/사진) × 액션(view/create/edit/delete)** 매트릭스로 표현.
- Phase 2 [[TypeDB]] 온톨로지 연동 시 **엔티티 단위 권한**(특정 설비 엔티티 ID 범위 접근)으로 확장 예정 → 본 스키마는 그 후크만 남김(상세는 위임).
- 인증(로그인/세션/SSO)은 본 문서 범위 밖 → [[03-2-6_인증-세션]](TBD)으로 위임. 여기선 `user` 신원 테이블만 소유.

## ③ 상세 목차

## Hub / Project
- 계정(Hub)과 프로젝트 엔티티. 멀티테넌시 경계선.

### hub (계정/사업장 루트)
- 컬럼 채울 것: `id`, `name`, `region`(청주), `status`, `created_at`. ACC의 Account 대응.
- 단일 사업장이라도 row 1개 보장 → 향후 확장 시 스키마 무변경.

### project
- 컬럼 채울 것: `id`, `hub_id`(FK), `name`, `number`, `status`(active/archived), `start_date`, `created_by`, `created_at`.
- 메모: 모듈 활성화 플래그(시트/이슈/사진 on/off)를 여기 둘지 별도 `project_module` 테이블로 뺄지 → ❓TBD.

#### project_setting
- 프로젝트별 설정(기본 분야 코드, 도면번호 규칙, 단위). 1:1 또는 KV row.
- 메모: DKS Phase 0 파싱 산출물의 **분야·도면번호 체계**를 여기에 매핑(파싱 규칙은 위임).

## User / Member
- 전역 사용자 신원 + 프로젝트별 소속(멤버십) 분리.

### user (전역 신원)
- 컬럼 채울 것: `id`, `email`, `display_name`, `company_id`(FK, nullable), `status`(active/invited/disabled), `created_at`.
- 메모: 인증 자격증명(비번/토큰)은 여기 두지 않음 → 인증 도메인으로 위임.

### project_member (소속 = 조인 테이블)
- 컬럼 채울 것: `id`, `project_id`(FK), `user_id`(FK), `role_id`(FK), `invited_by`, `joined_at`, `status`.
- 메모: 한 user가 여러 project에 서로 다른 role로 소속 → **(project_id, user_id) 유니크**.

#### invitation (초대 상태)
- 초대 흐름: pending → accepted/expired. 이메일 기반 초대.
- 메모: ACC 멤버 초대 모달 대응. 만료/재발송 정책 ❓TBD.

## Role / Permission
- 역할 정의와 모듈×액션 권한 매트릭스. RBAC 핵심.

### role
- 컬럼 채울 것: `id`, `hub_id`(또는 system 전역), `name`, `is_system`(기본역할 보호), `description`.
- 기본 역할 후보: **Admin / Member / Viewer**(3종). ACC의 세분 역할 대비 단순화 — 청주 1차는 3종으로 충분한지 ❓TBD.

### permission (액션 카탈로그)
- 컬럼 채울 것: `id`, `module`(sheet/issue/markup/file/photo), `action`(view/create/edit/delete/export).
- 메모: 마스터 데이터(시드). 화면 접근 게이트의 단위.

### role_permission (매트릭스 = 조인)
- 컬럼 채울 것: `role_id`(FK), `permission_id`(FK). 다대다.
- 메모: 기본 역할별 시드 매트릭스 표를 여기에 채움(Admin=전체, Member=view+create+edit, Viewer=view).

#### permission_check (적용 지점 메모)
- 백엔드 권한 판정 위치 열거: API 미들웨어 / 뷰어 오버레이 액션(핀·마크업 생성) / 내보내기.
- 메모: 프론트는 UI 게이팅만, **최종 판정은 백엔드** 원칙 명시. 구현 상세는 위임.

#### 엔티티 단위 권한 후크 (Phase 2)
- [[TypeDB]] 온톨로지 연동 시 "설비 엔티티 ID 범위" 권한 확장 자리.
- 메모: 본 스키마는 `role`에 `scope_type`(project/entity) 컬럼만 예약 → 상세는 [[03-2-5_온톨로지-바인딩]](TBD)에 위임. ❓예약 컬럼 도입 여부 TBD.

## 회사(Company)
- 외부 협력사/발주처 구분용 조직 엔티티. ACC의 Company 대응.

### company
- 컬럼 채울 것: `id`, `hub_id`(FK), `name`, `type`(owner/contractor/vendor), `created_at`.
- 메모: user·project_member에서 소속 회사 표기/필터(데이터테이블 필터 컬럼)에 사용.

#### project_company (참여사)
- 프로젝트에 참여하는 회사 목록(조인). 발주처/시공사/벤더 묶음.
- 메모: 1차 청주에서 실제로 필요한지 ❓TBD — 단일 사업장이면 보류 가능.
