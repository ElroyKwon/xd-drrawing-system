# 도메인 스키마 개요

> 03-2 도메인스키마 폴더의 **인덱스(목차) 문서**. 개별 테이블 정의는 각 모듈 하위 문서가 소유하고, 이 문서는 "테이블 그룹의 지도"만 담당한다.

## 1. 목적

- 청주 [[도면관리 시스템]]의 관계형 스키마를 **모듈(테이블 그룹) 단위로 인덱싱**하는 단일 진입점.
- 각 모듈 스키마 정의는 위임([[링크]]) — 여기서는 모듈 목록 · 명명 규칙 · 공통 컬럼만 확정.
- 채울 내용: 본 README가 "소유한 주제"는 ① 모듈 인덱스 ② 명명 규칙 ③ audit 공통 컬럼. ERD 전체도/물리 DDL은 위임.

## 2. 배경 / 전제

- 핵심 제약: [[DWG]] 원본 **비편집**. 흐름 = DWG → 백엔드 파싱 → 기하 JSON → 웹 2D 뷰어 렌더 → 마크업/이슈/이벤트 **오버레이**. 따라서 스키마는 "CAD 형상"이 아니라 "뷰어 위 주석·이벤트·메타"를 저장.
- 1차 범위 = **2D 시트 한정**. 3D BIM 엔티티 테이블은 범위 밖.
- 벤치마크 = [[ACC Build]] 모듈(홈/시트/마크업/이슈/시트비교/파일/사진). 모듈 경계가 곧 테이블 그룹 경계의 1차 후보.
- DKS 차별점: 핀/마크업/이슈를 픽셀 좌표뿐 아니라 **설비 엔티티 ID**에도 바인딩. Phase 2에서 [[TypeDB]] 온톨로지와 조인 — 관련 컬럼(`entity_ref`)은 본 스키마에 미리 예약하되 상세는 위임.
- 채울 내용: 위 전제가 각 모듈 테이블 설계에 주는 제약을 1줄씩 명시.

## 3. 스키마 모듈 목록

> 각 행 = 하나의 테이블 그룹. "소유 문서" 컬럼은 실제 정의가 사는 곳(미작성 시 TBD).

### 3.1 핵심 도메인 모듈

#### 3.1.1 프로젝트 / 조직 (`prj_*`)
- 프로젝트, 멤버, 역할/권한, 프로젝트 스위처 메타.
- 채울 내용: `prj_project`, `prj_member`, `prj_role` 등 테이블 목록 + 소유 문서 링크.

#### 3.1.2 도면 / 시트 (`sheet_*`, `dwg_*`)
- 업로드된 [[DWG]] 파일, 파싱 산출 기하 JSON 참조, 시트 메타(분야·도면번호·태그), 버전.
- 채울 내용: 원본 파일 vs 파생 시트 vs 기하 JSON 저장 경계 + 버전 테이블.

#### 3.1.3 뷰어 오버레이 — 마크업 (`markup_*`)
- 펜·도형·화살표·텍스트 주석. 시트 좌표계 기준 좌표 + 스타일.
- 채울 내용: 마크업 종류 enum, 좌표 표현(시트 로컬 좌표), 작성자.

#### 3.1.4 이슈 (`issue_*`)
- 핀 위치, 상세 폼 필드(제목/상태/담당/기한), 코멘트, 첨부.
- 채울 내용: 이슈 본문 + 핀 좌표 + `entity_ref`(설비 엔티티 바인딩, Phase 2) 예약.

#### 3.1.5 시트 비교 / 버전 diff (`compare_*`)
- 버전 오버레이 비교 결과/세션 메타. (diff 알고리즘 산출물 저장 여부 ❓)
- 채울 내용: 비교 세션을 저장하는지 vs 온디맨드 계산인지 결정 후 테이블화.

#### 3.1.6 파일 / CDE (`file_*`, `folder_*`)
- 폴더형 [[CDE]] 트리, 파일 메타, 권한.
- 채울 내용: 폴더 트리 표현(인접리스트 vs path) + 파일-시트 연결.

#### 3.1.7 사진 (`photo_*`)
- 현장 사진, 메타데이터, 이슈/시트 연결.
- 채울 내용: EXIF 보관 범위 + 연결 외래키.

### 3.2 횡단 / 보조 모듈

#### 3.2.1 활동 / 이벤트 로그 (`event_*`, `activity_*`)
- 뷰어 위 이벤트 오버레이, 실시간([[WebSocket]]) 푸시 대상 이벤트.
- 채울 내용: 이벤트 종류 분류 + 실시간 브로드캐스트 대상 플래그.

#### 3.2.2 알림 / 토스트 (`notify_*`)
- 사용자 알림, 읽음 상태.
- 채울 내용: 알림-이벤트 매핑 규칙.

#### 3.2.3 첨부 / 미디어 (`attachment_*`)
- 이슈·마크업·사진 공통 첨부 저장(다형 연결).
- 채울 내용: 다형(polymorphic) 연결 방식 결정 후 정의.

#### 3.2.4 온톨로지 바인딩 (`entity_ref_*`) — Phase 2
- 핀/마크업/이슈 ↔ 설비 엔티티 ID 매핑. [[TypeDB]] 측 ID 보관.
- 채울 내용: Phase 2 — 매핑 테이블만 예약, 상세는 [[온톨로지 연동 설계]]에 위임.

## 4. 명명 규칙

> 전 모듈 공통. 위반 시 리뷰 반려 기준.

### 4.1 테이블
- 소문자 + 스네이크케이스. `<모듈접두사>_<엔티티>` (예: `issue_comment`).
- 모듈 접두사는 §3 괄호 표기 사용. 신규 접두사 추가 시 본 문서 §3 갱신 필수.

### 4.2 컬럼
- 스네이크케이스. PK = `id`. FK = `<참조테이블단수>_id` (예: `sheet_id`).
- Boolean = `is_*` / `has_*`. 타임스탬프 = `*_at`. 좌표 = `*_x`, `*_y` (시트 로컬 좌표계).
- 채울 내용: 좌표계 단위/원점 정의는 [[좌표계-렌더링 설계]]에 위임, 여기선 컬럼 명명만.

### 4.3 enum / 코드값
- DB enum vs 코드 테이블 ❓ — 결정 후 본 절에 규칙 확정.

### 4.4 인덱스 / 제약
- 인덱스명 `idx_<table>_<cols>`, FK `fk_<table>_<ref>`.

## 5. 공통 컬럼 (audit)

> 모든 도메인 테이블이 상속하는 표준 컬럼 세트.

### 5.1 표준 audit 컬럼
- `id` (PK) — 타입 ❓ (BIGINT auto vs UUID).
- `created_at`, `updated_at` (timestamptz).
- `created_by`, `updated_by` (FK → 멤버/사용자).
- 채울 내용: 위 컬럼의 정확한 타입·기본값·트리거 적용 방식.

### 5.2 소프트 삭제
- `deleted_at` nullable (soft delete 채택 여부 ❓ — 채택 시 전 테이블 적용).

### 5.3 멀티테넌시 / 프로젝트 스코프
- 프로젝트 단위 격리: `project_id` 컬럼을 도메인 테이블에 둘지 ❓ (전역 vs 프로젝트 스코프 구분 기준 정의 필요).

### 5.4 적용 예외
- 로그/이벤트 테이블 등 audit 일부 미적용 케이스 목록.
