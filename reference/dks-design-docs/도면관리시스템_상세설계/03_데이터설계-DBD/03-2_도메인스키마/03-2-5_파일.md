# 파일 스키마

> 소유 범위: **Folder / File / FileVersion 테이블** + 파일 단위 공유상태·이슈연결.
> [[ACC]] Build "파일" 모듈(폴더형 [[CDE]])의 데이터 모델 재현. 도면 본체(시트) 자체의 기하·렌더 메타는 [[03-2-1_시트]]가 소유 — 본 문서는 "파일 컨테이너/버전 관리"만 다룸.

## 목적
- 청주 사업장 [[도면관리]] 시스템의 파일 저장소(CDE) 데이터 스키마 정의.
- (메모) [[DWG]]/[[PDF]] 등 원본 파일을 **편집 없이** 보관·버전관리하는 컨테이너 모델. 파싱 산출물(기하 JSON)은 별도 — 여기선 "원본 파일 + 버전 이력 + 폴더 트리"만.

## 배경/전제
- (메모) 핵심 스코프: 원본 비편집. 흐름 = 파일 업로드(원본 보존) → 백엔드 파싱 → [[03-2-1_시트]] 생성. File은 **불변 원본**, 변경은 새 FileVersion으로만.
- (메모) ACC "파일" 모듈 = 폴더 트리 + 파일 목록(데이터테이블) + 버전 드롭다운 + 우측 인스펙터. 3D BIM/Bridge 범위 밖 → 폴더·파일·버전 3테이블로 충분.
- (메모) DKS 차별점: 파일/시트를 "설비 엔티티 ID"에 바인딩 가능 — 본 문서는 연결 키만 노출, 온톨로지 본체는 [[TypeDB]] 스키마(Phase 2)로 위임.

## Folder(트리)
- (메모) CDE 폴더 계층. 자기참조 트리(parent_id). ACC 폴더형 CDE 재현.

### 컬럼 정의
- (메모) 채울 항목: `id`, `project_id`, `parent_id`(self-FK, root=NULL), `name`, `path`(materialized path 또는 ltree ❓), `depth`, `created_by`, `created_at`, `updated_at`, `is_deleted`(소프트삭제).

### 트리 표현 전략
- (메모) ❓ adjacency list(parent_id only) vs materialized path vs closure table — 조회 패턴(브레드크럼/하위전체) 기준으로 결정. **TBD**.

### 폴더 권한/공유
- (메모) 폴더 단위 공유상태는 아래 "공유상태/이슈연결" 절에서 통합 정의(중복 방지).

#### 제약/인덱스
- (메모) `(project_id, parent_id, name)` 유니크(동일 폴더 내 동명 금지). path 인덱스.

## File
- (메모) 논리 파일 = 버전들의 컨테이너. 실제 바이트는 FileVersion이 보유. 파일명·현재버전 포인터·폴더 소속.

### 컬럼 정의
- (메모) 채울 항목: `id`, `project_id`, `folder_id`(FK), `name`(표시명), `file_type`([[DWG]]/[[PDF]]/이미지 등 enum), `current_version_id`(FK→FileVersion), `status`(공유상태 enum, 아래 절), `created_by`, `created_at`, `updated_at`, `is_deleted`.

### File ↔ Sheet 관계
- (메모) 1 File(DWG/PDF) → N [[03-2-1_시트]](파싱 결과). 연결 방향(시트가 file_id를 보유)·생성 시점 명시. **본 문서는 File 측 1:N 존재만 선언**, 시트 컬럼은 시트 문서 소유.

#### 제약/인덱스
- (메모) `(folder_id, name)` 유니크 ❓(동명 허용 여부 결정). `current_version_id` nullable 여부(업로드 직후 상태).

## FileVersion
- (메모) 불변 버전 레코드. 새 업로드 = 새 행. 롤백은 current_version_id 포인터 변경(행 삭제 아님).

### 컬럼 정의
- (메모) 채울 항목: `id`, `file_id`(FK), `version_no`(1부터 증가), `storage_key`(오브젝트 스토리지 경로/키), `byte_size`, `checksum`(sha256, 중복/무결성), `mime_type`, `original_filename`, `uploaded_by`, `uploaded_at`, `parse_status`(미파싱/파싱중/완료/실패 enum), `parse_artifact_ref`(기하 JSON 위치 — 본체는 시트 문서 소유), `comment`(버전 메모).

### 버전 채번/롤백
- (메모) `version_no` 채번 규칙(file 단위 시퀀스). 롤백 시 새 버전 생성 vs 포인터 이동 ❓ — ACC는 "set as current" 방식 추정. **TBD**.

### 파싱 상태 연동
- (메모) `parse_status` 상태 전이. 파싱 완료 시 시트 생성·[[WebSocket]]으로 진행상황 푸시(실시간 로그). 파싱 파이프라인 상세는 별도 문서로 위임.

### 시트비교(버전 diff)와의 관계
- (메모) ACC "시트비교" = 두 FileVersion(또는 그 파생 시트)의 오버레이 diff. 본 문서는 **비교 대상 식별 키(version_no, file_id)만 제공**, diff 알고리즘/렌더는 시트비교 설계 문서로 위임.

#### 제약/인덱스
- (메모) `(file_id, version_no)` 유니크. `checksum` 인덱스(중복 업로드 감지). uploaded_at 정렬 인덱스.

## 공유상태/이슈연결
- (메모) ACC CDE 핵심: 파일/폴더의 공유 라이프사이클 + 파일↔이슈 양방향 연결.

### 공유상태(status) enum
- (메모) ❓ ACC 단계 매핑 확인 필요: WIP / Shared / Published / Archived 류. 청주 납품 운영 단계와 합의. **TBD** — 평가판 화면에서 실제 라벨 재확인.

### 상태 전이 규칙
- (메모) 누가(역할) 어떤 전이를 트리거하는지. 권한 모델은 [[03-2-6_권한]](존재 시) 또는 공통 권한 문서로 위임 ❓.

### File ↔ Issue 연결
- (메모) [[03-2-4_이슈]]가 file_id/file_version_id를 참조하는 방향이 정설(이슈 측 FK 소유). 본 문서는 **File 입장에서 "연결된 이슈 N건" 역참조 뷰**만 정의. 핀이 픽셀이 아닌 시트/엔티티에 묶이는 상세는 이슈·시트 문서 소유.

### File ↔ 설비 엔티티(DKS 차별점)
- (메모) 향후 File/FileVersion을 [[TypeDB]] 온톨로지 설비 엔티티 ID에 바인딩(영향도·AI채팅 바인딩 무기). Phase 2. 연결 키 컬럼(`entity_ref` ❓) 위치는 시트 vs 파일 레벨 결정 **TBD**.
