# 마크업 스키마

## ① 목적
- 웹 2D 뷰어 위에 그려지는 펜·도형·화살표·텍스트 등 "마크업 객체"를 영속 저장하는 단일 테이블 스키마 정의.
- 소유 범위 = **마크업 객체 저장 테이블 그 자체**. 마크업 *렌더링/툴 UX*는 위임, 마크업이 묶이는 *이슈*도 위임.

## ② 배경 / 전제
- [[DWG]] 원본은 편집하지 않는다. 마크업은 파싱된 기하 JSON 위에 떠 있는 **오버레이 레이어**일 뿐 — 원본 도면에 병합되지 않음.
- 렌더링은 [[PDF.js]] 계열 캔버스/SVG 레이어 위에 좌표 기반으로 그림. 마크업 좌표는 **시트 정규화 좌표계**(원본 도면 단위 아님)로 저장.
- 1차 범위 = 2D 시트 한정. 정밀 [[CAD]] 편집(osnap·좌표 편집·[[SHX]] 폰트 정밀 렌더)은 범위 밖.
- 벤치마크 = [[ACC Build]] 마크업 모듈(펜·도형·화살표·텍스트·하이라이트).
- 차별점([[DKS]]): 마크업을 픽셀뿐 아니라 **설비 엔티티 ID**에도 묶을 수 있음 → 향후 [[TypeDB]] 온톨로지 검색/영향도 연동의 앵커. (본 테이블은 그 바인딩 컬럼만 소유, 온톨로지 자체는 위임.)

## ③ 상세 목차 (채울 골격)

## Markup(type / geometry / style)
> 마크업 1행 = 1 마크업 객체. 타입별 기하·스타일을 어떻게 한 테이블에 담을지 결정.

### type 열거
> 지원 마크업 종류 enum. ACC 벤치마크 기준 후보 나열.
- `pen`(자유선) / `line` / `arrow` / `rect` / `ellipse` / `polygon` / `text` / `highlight` / `cloud`(리비전 클라우드) — TBD: 1차 릴리스 포함 범위 확정
- 빈상태/미지원 타입 처리 정책 메모 채울 것

### geometry 저장 방식
> 타입마다 기하 형태가 달라 컬럼 고정이 어려움 → 저장 전략 결정 필요.
- 후보 A: `geometry JSON`(좌표 배열·점 목록) 단일 컬럼 — 유연, 쿼리 약함
- 후보 B: 정규화 컬럼(x/y/w/h) + 보조 JSON — pen/polygon은 점배열 JSON 불가피
- ❓ 결정 대기: JSON 단일 vs 하이브리드. 좌표 단위 = **시트 정규화 0~1** vs **렌더 픽셀** (정규화 권장 — 줌/DPI 독립)

### style 저장 방식
> 색·선두께·투명도·채움·폰트 등 표현 속성.
- 후보: `style JSON`(stroke/strokeWidth/fill/opacity/fontSize/fontFamily) 단일 컬럼
- 텍스트 마크업의 `text` 내용 컬럼 위치(geometry vs 별도 `content`) TBD

### 인덱스 / 제약
> 조회 패턴 기준 인덱스 설계.
- 시트별 마크업 로드가 핵심 쿼리 → `sheet_id` 인덱스 필수 (→ ## 시트 연결)
- soft-delete 여부(`deleted_at`) TBD

## 소유자 / 타임스탬프
> 누가·언제 만들고 고쳤는지. 감사·필터·정렬의 근거.

### 소유자 필드
> 작성자 식별 및 권한 표시.
- `author_id`(FK→사용자) / `author_name` 스냅샷 보관 여부 TBD
- 편집 권한 모델(본인만 vs 프로젝트 멤버)은 위임 → [[03-3_권한설계]] (TBD 경로 확인)

### 타임스탬프 필드
> 생성/수정 시각.
- `created_at` / `updated_at` (UTC 저장, 표시 변환은 프론트)
- 실시간 협업 시 last-write-wins vs 버전 충돌 정책 ❓ — [[WebSocket]] 동시편집 범위와 연동(1차 포함 여부 TBD)

### 상태 / 가시성

- `visible` 토글, 레이어 그룹핑 필요 여부 TBD

## 시트 연결
> 마크업이 어느 시트(도면)·어느 버전 위에 있는지. 오버레이의 앵커.

### 시트 / 버전 FK
> 마크업은 특정 시트 버전에 고정. 버전 바뀌면 좌표 정합성 문제 발생.
- `sheet_id`(FK→시트) 필수 — 정의는 [[03-2-2_시트]] 소유
- `sheet_version` 바인딩: 특정 버전 고정 vs 최신 추종 ❓ ([[시트비교]] diff 시 마크업 이월 정책과 직결)

### 페이지 / 영역
> 멀티페이지 시트 대비.
- `page_no`(다중 페이지 PDF/도면 시) 필요 여부 TBD

### 엔티티 바인딩 (DKS 차별 앵커)
> 픽셀 좌표 외에 "설비 엔티티 ID"에도 마크업을 묶는 선택적 링크. Phase 2 온톨로지 연동의 진입점.
- `entity_id` (nullable, Phase 0 파싱 산출물의 태그/도면번호 참조) — 본 테이블은 **컬럼만 소유**, 해석/영향도는 [[TypeDB]] 측 위임
- 1:1 vs 1:N(마크업 하나가 복수 엔티티 참조) TBD

### 이슈 연결
> 마크업이 이슈 핀에 종속되는 경우.
- 마크업→이슈 FK 방향·소유권은 [[03-2-4_이슈]]가 소유 → 여기서는 위임, 중복 정의 금지
