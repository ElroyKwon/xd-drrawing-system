# 시트/버전 스키마

## ① 목적
- 청주 도면관리 시스템의 **도면 시트(Sheet)**와 **버전세트(VersionSet)** 데이터 모델 정의
- [[ACC Build]] "시트" 모듈(도면목록 + 2D 뷰어 + 버전 오버레이 diff)을 동등 재현할 스키마 골격 확정
- 이 문서가 **소유**: 시트 엔티티, 버전/버전세트 엔티티, 시트↔파일 연결, 공종·태그 분류
- 위임(타 문서 소유): 마크업/이슈/핀 → [[03-2-3_파일-폴더]] 인접 도메인, 픽셀↔엔티티 바인딩 → 별도 [[03-2-4_엔티티-바인딩]] (TBD)

## ② 배경 / 전제
- ★스코프 제약: [[DWG]] 원본을 편집하지 않음. 흐름 = DWG → 백엔드 파싱 → 기하 JSON → 웹 2D 뷰어 렌더 → 오버레이. **CAD 편집기 아님, 뷰어+주석 시스템.**
- 1차 범위 = **2D 시트 한정**. 3D BIM/Bridge 범위 밖.
- DKS Phase 0 도면 파싱 산출물(분야·도면번호·태그) **이미 존재** → 본 스키마가 이를 수용·정규화
- 차별점: 시트가 단순 픽셀이 아니라 향후 [[TypeDB]] 온톨로지 **설비 엔티티 ID**에 바인딩 가능한 구조여야 함(Phase 2 대비 필드 예약)

## ③ 상세 목차

## Sheet
- 도면 1매 = 1 Sheet. 시트 목록/2D 뷰어/이슈 핀의 좌표 기준 단위.
  → 채울 내용: 시트 엔티티의 정체성·표시·렌더 소스 필드

### Sheet 핵심 필드
- `sheet_id`(PK), `project_id`(FK), `sheet_number`(도면번호), `title`(도면명), `discipline`(공종)
  → 채울 내용: 각 필드 타입·제약·예시값 표

### Sheet 식별/넘버링 규칙
- `sheet_number` 유일성 범위 = 프로젝트 내 vs 버전세트 내 ❓
  → 채울 내용: Phase 0 파싱된 도면번호 체계와 매핑 규칙

### Sheet 렌더 소스 (기하 JSON)
- DWG 파싱 산출물 포인터: `geometry_json_uri`, `bbox`, `units`, `scale`
  → 채울 내용: 뷰어가 로드하는 기하 JSON 스키마 위치·좌표계 정의(픽셀 vs 도면좌표)

#### 좌표계 정의
- 오버레이(핀·마크업) 좌표 = 도면좌표 정규화 권장(줌·DPI 독립)
  → 채울 내용: 좌표 변환 규약, 뷰어 viewBox 매핑

### Sheet 상태/메타
- `status`(검토중/승인/폐기), `created_at`, `updated_at`, `tags[]`
  → 채울 내용: 상태 전이 규칙, ACC 시트 상태값과 대조

## Version / VersionSet
- VersionSet = "한 번의 업로드/발행 묶음", Version = 그 안의 개별 시트 리비전.
  → 채울 내용: 버전 모델 ACC 대조(시트비교 diff의 데이터 근거)

### VersionSet
- `version_set_id`(PK), `project_id`, `label`(예: 2026-06 1차발행), `published_at`, `source_file_id`
  → 채울 내용: 발행 단위 메타, 누가/언제/어떤 파일에서

### Version (Sheet Revision)
- `version_id`(PK), `sheet_id`(FK), `version_set_id`(FK), `rev`(A/B/C 또는 1/2/3), `geometry_json_uri`
  → 채울 내용: 시트별 리비전 레코드, 이전 rev 포인터(`prev_version_id`)

#### 현재버전 포인터
- 시트의 "현재 표시 버전" = `sheet.current_version_id` 또는 latest 규칙 ❓
  → 채울 내용: 단일 진실 위치(SoT) 결정, 조회 성능 고려

### 버전 비교(diff) 데이터 요건
- 시트비교 화면 = 두 Version의 geometry JSON 오버레이 diff
  → 채울 내용: diff에 필요한 정렬키(동일 sheet_number 매칭), 추가/삭제/이동 분류 메모. 알고리즘은 위임 → [[03-3_시트비교-설계]] (TBD)

## Sheet-File 관계
- 1 DWG/[[PDF]] 파일 → N Sheet 추출 가능(다중 레이아웃/멀티페이지).
  → 채울 내용: 파일과 시트의 카디널리티, 추출 추적성

### 카디널리티 / FK
- `version.source_file_id` → File 엔티티. File 상세는 위임 → [[03-2-3_파일-폴더]]
  → 채울 내용: File↔VersionSet↔Sheet 3자 관계 ERD 조각

### 추출 추적(provenance)
- `extracted_from`(파일+레이아웃 인덱스+파싱 잡 ID)
  → 채울 내용: 재파싱 시 시트 동일성 유지 키, 파싱 잡 메타 링크

### 재업로드/재파싱 시 시트 동일성
- 같은 도면번호 재업로드 → 신규 Version으로 귀속 vs 신규 Sheet ❓
  → 채울 내용: 매칭 규칙(도면번호 기준), 충돌 처리

## 공종 / 태그
- 공종(discipline) = 분야 분류(건축/구조/전기/기계 등), 태그 = 자유/Phase0 산출 키워드.
  → 채울 내용: ACC 필터·DKS 검색의 분류 축 정의

### Discipline 코드 체계
- Phase 0 파싱 분야값 → 표준 코드 매핑 테이블
  → 채울 내용: 코드 enum, ACC 시트 필터 항목과 대조

### Tag 모델
- `tag`(라벨), `source`(파싱/수동), 시트당 N태그(M:N)
  → 채울 내용: 태그 테이블·조인 테이블 스키마, 데이터테이블 필터/내보내기 연동

### 온톨로지 엔티티 바인딩 (Phase 2 예약)
- 시트/태그 → [[TypeDB]] 설비 엔티티 ID 참조 필드 **미리 예약**(nullable)
  → 채울 내용: `entity_refs[]` 예약 필드, 바인딩 상세는 위임 → [[03-2-4_엔티티-바인딩]] (TBD)
