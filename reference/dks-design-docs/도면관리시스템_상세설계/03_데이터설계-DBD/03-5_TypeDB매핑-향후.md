# TypeDB 온톨로지 매핑 (향후)

> 소유 범위: **V3.2 스키마 연동 (Phase 2)** 한정.
> 관계형 DB 설계(테이블/컬럼)는 [[03-1_논리데이터모델]] 소유 → 본 문서는 위임. 본 문서는 RDB ↔ [[TypeDB]] 그래프 간 "매핑·동기화"만 다룸.

## 1. 목적

- Phase 2 차별점 정의: [[ACC]] Build엔 없는 **온톨로지 엔티티 단위** 검색·영향도·AI채팅을 뷰어/이슈에 바인딩.
- Phase 1(뷰어+주석 오버레이) 데이터를 [[TypeDB]] [[V3.2 스키마]]에 매핑하는 규칙 확정.
- ⚠️ 의도 메모: "왜 그래프 DB인가"는 단순 검색이 아니라 **영향도(traversal)·AI 근거 바인딩** 때문임을 1~2줄로 명시.

## 2. 배경 / 전제

- **핵심 스코프 제약**: [[DWG]] 원본 미편집. 흐름 = DWG → 백엔드 파싱 → 기하 JSON → 웹 2D 뷰어 → 마크업/이슈/이벤트 오버레이. CAD 편집기 아님.
- Phase 0 파싱 산출물(분야·도면번호·태그)이 이미 존재 → 본 매핑의 입력 소스. 상세는 [[03-4_파싱산출물-스키마]] 위임.
- Phase 2는 **향후(후행)** 작업. Phase 1 RDB가 진실원천(SoT), TypeDB는 파생 그래프(읽기/추론 전용 우선 검토).
- 3D BIM / Bridge 범위 밖 → 2D 시트 엔티티만 매핑.
- ❓ TBD: V3.2 스키마 본체(`.tql`) 위치·버전 고정 여부. (본 문서는 매핑 규칙만, 스키마 정의는 외부 소유)

## 3. V3.2 매핑 개요

> 의도 메모: 양쪽 모델(RDB ↔ TypeDB)의 책임 경계와 매핑 방향(단방향/양방향)을 한 장으로.

### 3.1 매핑 책임 경계
- RDB가 소유하는 것 / TypeDB가 소유하는 것 구분표를 채울 것 (오버레이=RDB SoT, 추론·관계=TypeDB).
- ❓ TBD: 마크업/이슈 좌표·픽셀 데이터를 TypeDB에 넣을지 vs RDB에만 두고 엔티티ID 링크만 보낼지.

### 3.2 V3.2 스키마 계층 요약
- 4계층 온톨로지 중 본 시스템이 닿는 계층만 발췌(설비 엔티티·도면·태그). 전체는 [[데이터-지식-스튜디오]] 위임.
- 채울 것: 사용할 entity/relation/attribute type 목록 (도면, 설비, 태그, 분야 등).

### 3.3 식별자 전략
- RDB PK ↔ TypeDB key attribute 대응 규칙. (예: `sheet_id`, `entity_id`)
- ❓ TBD: 엔티티 ID 발급 주체 — 파싱 단계 부여 vs 매핑 단계 부여.

## 4. 도면객체 → 엔티티 매핑

> 의도 메모: Phase 0/1의 각 객체를 V3.2 type으로 1:1 또는 1:N 매핑하는 표가 본 절의 핵심 산출물.

### 4.1 매핑 대상 인벤토리
- 매핑 소스 객체 열거: 도면(시트), 분야, 도면번호, 파싱 태그, 설비 심볼, 마크업, 이슈 핀.
- 각 항목 → 대응 TypeDB type → 카디널리티 표 골격을 채울 것.

#### 4.1.1 도면/시트 → `drawing` 엔티티
- 시트 메타(번호·분야·버전) → attribute 매핑.

#### 4.1.2 파싱 태그/설비 심볼 → 설비 엔티티
- ★차별점: 핀/마크업/이슈를 픽셀이 아닌 **설비 엔티티 ID**에 묶음. 그 연결고리가 여기서 정의됨.
- ❓ TBD: 태그 문자열 → 정규 설비 엔티티 해소(resolution) 규칙. 동음이의·오타 처리.

#### 4.1.3 이슈 핀 / 마크업 → 오버레이 관계
- 이슈·마크업을 `drawing` + `entity`에 동시 연결하는 relation 설계. 픽셀좌표는 RDB, 엔티티링크는 TypeDB.

### 4.2 관계(Relation) 매핑
- 도면-설비 포함, 설비-설비 영향도, 이슈-설비 연관 등 relation type 목록·역할(role) 정의.
- 영향도 질의(1-hop/N-hop) 예시 시나리오 1~2개 기재.

### 4.3 매핑 미해결/손실 항목
- TypeDB로 표현 불가하거나 손실되는 Phase 1 데이터 명시(예: 자유 펜 마크업 픽셀).

## 5. 업서트 / 동기화 규칙

> 의도 메모: RDB 변경을 TypeDB에 반영하는 트리거·일관성·충돌 규칙. Phase 2 운영 안정성의 핵심.

### 5.1 동기화 방향·시점
- 단방향(RDB→TypeDB) 우선 가정. 양방향 필요 여부 검토.
- 트리거 후보: 파싱 완료 이벤트 / 이슈·마크업 저장 시 / 배치. ❓ TBD: 실시간 vs 배치.
- 실시간일 경우 전달 채널은 [[WebSocket]] 또는 이벤트 큐 — 채울 것.

### 5.2 업서트 키·멱등성
- key attribute 기반 upsert(match-insert) 규칙. 재실행 안전성(멱등) 보장 방법.
- 삭제 처리: hard delete vs tombstone.

### 5.3 충돌·정합성
- RDB-TypeDB 불일치 감지 및 복구(재동기화) 절차.
- ❓ TBD: 정합성 검증 주기·기준(row count, checksum 등).

### 5.4 초기 적재(Bulk Load)
- Phase 0 기존 파싱 산출물 일괄 적재 절차·순서(스키마 → 엔티티 → 관계).

### 5.5 AI 채팅 바인딩 훅(연계 지점만)
- 향후 AI 질의가 TypeDB 추론 결과를 뷰어/이슈로 되돌리는 진입점만 표시. 상세 설계는 별도 문서 위임.
