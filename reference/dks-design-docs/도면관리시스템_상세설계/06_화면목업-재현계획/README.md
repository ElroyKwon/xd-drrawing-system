# 화면 목업 재현 계획 개요

## 목적(화면 1:1 목업)

청주 사업장 납품용 [[도면관리 시스템]]의 화면을 [[ACC]] Build 제품과 1:1 수준으로 재현하기 위한 **목업 산출 방법론·도구·범위**를 정의한다.

- 이 문서가 소유: "어떻게(방법) · 무엇으로(도구) · 어디까지(범위) 목업을 만드는가"
- 이 문서가 소유하지 않음 → 위임:
  - 화면별 상세 스펙 → [[_ACC-Build-화면분석-재현설계]]
  - 실제 구현 아키텍처([[DWG]]→파싱→기하 JSON→2D 뷰어) → [[도면관리시스템_상세설계]] 아키텍처 절
- 전제(★핵심 스코프): 목업은 "**뷰어 + 주석/이벤트 오버레이**" UI를 그린다. CAD 편집기 화면이 아니다. 정밀 [[osnap]] 스냅·좌표 편집·[[SHX]] 폰트 정밀 렌더 화면은 목업 대상이 아님.
  > 의도 메모: Phase 1 목업 = 2D 시트 한정. 3D BIM 뷰어/Bridge 화면 제외. "온톨로지 엔티티 바인딩"은 향후(Phase 2) 부가 UI로 별도 표기.

## 배경/전제

> 의도 메모: 목업이 따라야 할 입력·제약·우선순위를 1~2줄씩 명시.

### 입력 자산
- 벤치마크 = [[ACC]] Build 37화면 분석 → [[_ACC-Build-화면분석-재현설계]]
- 모듈 범위: 홈 / 시트 / 마크업 / 이슈 / 시트비교 / 파일([[CDE]]) / 사진
- 공통 UI 컴포넌트: 앱셸, 데이터테이블, 우측 인스펙터, 위젯카드, 모달폼, 빈상태, 토스트

### 전제 / 제약
- 목업은 정적·반응형 화면 표현이 1차 목표. 실데이터 연동([[WebSocket]] 등)은 구현 단계 책임.
- 차별점 표기: 핀/마크업/이슈를 픽셀 좌표뿐 아니라 "설비 엔티티 ID"에도 묶는 UI 흔적을 남긴다(Phase 2 [[TypeDB]] 온톨로지 연동 대비).
  > 의도 메모: 목업 단계에서는 "엔티티 ID 배지/링크" 같은 시각 자리표시자만. 동작은 향후.

## 제작 방법(도구 선택지)

> 의도 메모: 목업 제작 방식 후보를 나열하고, 선택 기준·권장안을 채운다. 빈 표 금지 — 후보는 실제 평가.

### 후보 A — 디자인 툴 정적 시안
- Figma/이미지 기반. 빠른 합의·비개발자 리뷰 용이.
- 한계: 인터랙션(데이터테이블 정렬·핀 드래그) 표현 약함.

### 후보 B — HTML/CSS 정적 프로토타입
- 실제 마크업으로 앱셸·테이블·인스펙터를 그림. 구현 코드로 이어지기 쉬움.
- 2D 뷰어 영역은 자리표시자(placeholder) 또는 정적 캡처로 대체.
  > 의도 메모: 실 뷰어 렌더([[PDF.js]]/Canvas)는 구현 단계. 목업에선 영역·툴바·핀만 표현.

### 후보 C — 인터랙티브 프론트 목업
- 컴포넌트(React/Vue 등) + 가상 데이터로 동작 시연. CLAUDE.md §4 Frontend-First 부합.
- 한계: 제작 비용 큼.

### 선택 기준 / 권장
- 기준: 합의 속도 · 구현 전환성 · 인터랙션 충실도 · 제작 비용
- ❓ TBD: 모듈별로 방법 혼용? (예: 홈·파일=B, 시트·마크업·이슈=C)
- ❓ TBD: 채택 프론트 스택 미정 → [[도면관리시스템_상세설계]] 기술스택 결정에 종속.

#### 가상 데이터 규약
> 의도 메모: 실제 API 스펙으로 전환 가능한 가상 데이터 구조 규약(도면번호·분야·태그·엔티티 ID 필드 포함). Phase 0 파싱 산출물 필드와 정렬.
