# 개발 준비 / 기술스택 개요

> 청주 사업장 납품 "도면관리 시스템" 개발 착수 준비 **인덱스**.
> 이 문서가 소유하는 주제 = "착수 전 무엇을 준비하고 무엇을 확인할 것인가". 개별 모듈 설계는 각 폴더로 위임.

---

## 1. 목적

- 개발 착수 직전 **준비 상태 점검 + 기술스택 선정 진입점** 제공
- 흩어진 결정(스택·환경·산출물·전제)을 한 페이지에서 "어디를 보면 되는가"로 라우팅
- 채울 항목: 본 인덱스가 가리키는 하위 문서 맵 + 착수 게이트 체크리스트

## 2. 배경 / 전제 (모든 하위 문서 공통)

- **1차 목표**: [[ACC]] Build 동등 재현 (홈/시트/마크업/이슈/시트비교/파일/사진). 상세 → [[_ACC-Build-화면분석-재현설계]]
- **★핵심 스코프 제약 (불변)**:
  - [[DWG]] 원본 **편집 금지**. 흐름 = DWG → 백엔드 파싱 → 기하 JSON → 웹 2D 뷰어 렌더 → 마크업/이슈/이벤트 **오버레이**
  - 정밀 CAD 편집(osnap 스냅, 좌표 편집, [[SHX]] 폰트 정밀 렌더) = **범위 밖**
  - 1차 = **2D 시트 한정**. 3D BIM 뷰어/Bridge = 범위 밖
- **DKS 차별점**: 핀/마크업/이슈를 픽셀 좌표뿐 아니라 **설비 엔티티 ID**에 바인딩. [[Phase 0]] 도면 파싱 산출물(분야·도면번호·태그) 기보유. Phase 2 [[TypeDB]] 온톨로지 연동. 상세 → [[데이터-지식-스튜디오]]
- 메모: 이 절은 "전제 요약"만. 스코프 정본은 상위 설계 인덱스 문서 소유 → [[링크]] 위임

## 3. 구성 (하위 문서 맵)

> 메모: 각 항목 = 별도 문서/폴더로 위임. 본 절은 "무엇이 어디에 있는가" 라우팅 테이블만 유지(DRY).

### 3.1 기술스택 선정

#### 3.1.1 프론트엔드 스택
- 2D 뷰어 렌더링 후보: [[PDF.js]] / Canvas / [[SVG]] / WebGL 중 선정 근거. 메모: 기하 JSON 오버레이 좌표계 정합성 기준으로 비교표 채울 것
- 앱셸·데이터테이블·인스펙터 등 공통 UI 컴포넌트 프레임워크(예: React/SolidJS) 선정. 메모: ACC 공통 UI 패턴(앱셸/테이블/모달/토스트) 재현 가능성 기준
- ❓TBD: 상태관리·라우팅 라이브러리

#### 3.1.2 백엔드 스택
- DWG 파싱 파이프라인 언어/런타임(기하 JSON 추출). 메모: Phase 0 파싱 산출물과 포맷 연속성 확인
- API 서버 프레임워크, 실시간 채널([[WebSocket]] — 마크업/이슈 협업 동기화). 메모: 단일 진실 위치는 API 설계 문서로 위임
- ❓TBD: DWG 파서 라이브러리/엔진 선정 (라이선스 포함)

#### 3.1.3 데이터 / 저장소
- 메타·이슈·마크업 RDB, 파일/도면 객체 저장소([[CDE]] 폴더 구조), 기하 JSON 캐시 전략
- Phase 2: [[TypeDB]] 온톨로지 스토어 (엔티티 ID 바인딩). 메모: Phase 1엔 엔티티 ID 컬럼 자리만 확보(스키마 후방호환)

#### 3.1.4 인프라 / 배포
- 청주 사업장 납품 형태(온프렘/사내망 여부), 컨테이너·CI 전략. 메모: 망 분리·보안 요건 확인 후 채움
- ❓TBD: 배포 대상 환경 사양 (사업장 제공 서버 스펙)

### 3.2 개발 환경
- 리포지토리 구조(모노레포 여부), 로컬 개발 환경, 린트/포맷/테스트 도구. 메모: CLAUDE.md TDD·SOLID 원칙 반영
- 실시간 로그 패널(개발/디버깅용, CLAUDE.md §4). 메모: 프로덕션 비활성화 토글

### 3.3 산출물 / 인터페이스 의존성
- Phase 0 도면 파싱 산출물 스키마(분야/도면번호/태그) 입력 계약. 메모: 실제 샘플 1건으로 필드 확정
- 기하 JSON 포맷 명세 위임 → 뷰어 설계 문서 [[링크]]
- 설비 엔티티 ID 카탈로그(향후) 위임 → 온톨로지 설계 문서 [[링크]]
