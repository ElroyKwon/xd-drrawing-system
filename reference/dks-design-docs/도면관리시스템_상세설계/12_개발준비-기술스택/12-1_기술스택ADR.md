# 기술스택 결정 (ADR)

> 청주 사업장 납품용 [[도면관리]] 시스템의 FE/BE/DB/뷰어 라이브러리 기술 선택을 **결정 기록(ADR)** 형태로 누적하는 문서.
> 본 문서는 "무엇을 왜 골랐는가"만 소유한다. 아키텍처 다이어그램·데이터 모델·API 명세는 각 소유 문서로 위임.

---

## 1. 목적

- 핵심 기술 선택(FE/BE/DB/뷰어)을 ADR 단위로 기록 → 나중에 "왜 이렇게 골랐나" 추적 가능하게.
- 의도 메모: 각 ADR이 "결정/대안/근거/영향"을 담도록 양식 고정. 번복 시 새 ADR로 supersede.

## 2. 배경 / 전제

- ★스코프 제약: [[DWG]] **원본 편집 없음**. 흐름 = DWG → 백엔드 파싱 → 기하 JSON → 웹 2D 뷰어 렌더 → 마크업/이슈/이벤트 **오버레이**.
- 즉 CAD 편집기가 아니라 "**뷰어 + 주석/이벤트 오버레이**" 시스템. 정밀 CAD 편집(osnap, 좌표 편집, [[SHX]] 폰트 정밀 렌더)은 범위 밖.
- 1차 범위 = **2D 시트 한정**. 3D [[BIM]] 뷰어/Bridge 범위 밖.
- 벤치마크 = [[ACC Build]] 37화면 → 상세는 [[_ACC-Build-화면분석-재현설계]] 위임.
- [[DKS]] 차별점: 핀/마크업/이슈를 픽셀뿐 아니라 "**설비 엔티티 ID**"에 바인딩 → 향후 [[TypeDB]] 온톨로지 연동(Phase 2). 기술 선택 시 이 바인딩 확장성을 평가 축에 포함.
- 의도 메모: 이 전제가 모든 ADR의 평가 기준. "편집기 가정"으로 회귀한 선택은 기각.

---

## 3. ADR 양식

> 모든 결정은 아래 고정 양식으로 작성. 한 결정 = 한 H3 블록.

#### 3.1 양식 필드
- 의도 메모: `ADR-NNN | 제목 | 상태(제안/수락/기각/대체됨) | 컨텍스트 | 결정 | 대안 | 근거 | 영향/리스크 | 날짜` 필드 나열.

#### 3.2 상태 전이 규칙
- 의도 메모: 제안→수락→(대체됨). 번복은 삭제 금지, 새 ADR이 기존을 supersede 표기.

#### 3.3 번호 체계
- 의도 메모: ADR-001부터 연번. 영역 접두 쓸지(FE-/BE-/DB-/VW-) ❓ TBD.

## 4. 프론트엔드 (React 등 TBD)

### 4.1 ADR-FE-001 프레임워크 선택
- 의도 메모: 후보 [[React]] / [[SolidJS]] / Vue 비교. ACC 재현(데이터테이블·인스펙터·모달 다수) → 컴포넌트 생태계/채용 용이성 가중치.
- ❓ TBD: SolidJS(자사 SoftPLC 트랙 경험 있음) vs React(생태계) — 팀 역량 기준 결정 대기.

#### 4.1.1 평가 기준
- 의도 메모: 생태계·뷰어 라이브러리 호환·캔버스 성능·팀 숙련도 4축.

### 4.2 상태관리 / 데이터 페칭
- 의도 메모: 서버상태(목록·필터) vs 클라이언트상태(뷰어 도구·선택) 분리. TanStack Query 등 후보.
- ❓ TBD: 실시간 마크업/이슈 동기화에 [[WebSocket]] 직결 vs 폴링.

### 4.3 캔버스 / 렌더 레이어
- 의도 메모: 2D 뷰어 위 오버레이를 [[Canvas]] vs [[SVG]] vs WebGL 중 무엇으로 그릴지. 핀/도형 수천 개 시 성능 고려.
- 상세 렌더 구조는 [[뷰어-아키텍처]] 문서로 위임(생성 예정).

### 4.4 디자인 시스템 / UI 키트
- 의도 메모: 공통 UI(앱셸·데이터테이블·토스트·빈상태) 재현 → 헤드리스 키트 vs 풀 키트. LS사우타 디자인 가이드 반영.

## 5. 백엔드 (파서 C#/.NET, API 언어 TBD)

### 5.1 ADR-BE-001 DWG 파서 런타임
- 결정(잠정): [[DWG]] 파싱은 **[[.NET]]/C#** — RealDWG/ODA 등 .NET 친화 라이브러리 생태계 때문.
- 의도 메모: 라이브러리 라이선스·DWG 버전 지원 범위·기하 추출(선/원/호/텍스트/레이어) 가능성 검증.
- ❓ TBD: 파서 라이브러리 구체 선정([[ODA]] Drawings SDK / Teigha / 기타) — 라이선스 비용·납품 제약 확인.

#### 5.1.1 파서 출력 = 기하 JSON 스키마
- 의도 메모: 파서가 내보낼 JSON(엔티티·레이어·도면번호·태그) 구조. 상세 스키마는 [[기하JSON-스키마]] 문서 위임.

### 5.2 ADR-BE-002 API 서버 언어
- 의도 메모: 파서(.NET)와 API 서버를 같은 .NET으로 통합 vs 분리(파서=.NET 워커, API=별도). 후보 .NET / Go / Node.
- ❓ TBD: 단일 .NET 통합이 운영 단순. 그러나 향후 [[TypeDB]] 클라이언트 드라이버 성숙도(언어별) 확인 후 결정.

#### 5.2.1 파싱 워커 vs 동기 API 경계
- 의도 메모: DWG 파싱은 무거움 → 비동기 잡 큐. API와 워커 분리 경계 정의.

### 5.3 실시간 채널
- 의도 메모: 마크업/이슈/이벤트 오버레이 실시간 반영용 [[WebSocket]] 서버 위치(API 내장 vs 별도). 상세는 [[실시간-동기화]] 위임.

### 5.4 파일 스토리지 / [[CDE]]
- 의도 메모: 원본 DWG·변환 산출물(기하 JSON·썸네일) 저장소. 폴더형 CDE 재현 → 오브젝트 스토리지 vs 파일시스템. 납품 환경(온프레) 제약 반영.

## 6. 뷰어 라이브러리 후보 비교

> ★우리는 "DWG 직접 렌더 뷰어"가 아니라 "**파서 산출 기하 JSON을 그리는 자체 뷰어**"가 기본 전제. 외부 뷰어 라이브러리는 보조 후보로만 비교.

### 6.1 ADR-VW-001 렌더 방식 결정
- 의도 메모: (A) 백엔드가 DWG→[[PDF]] 변환 후 [[PDF.js]]로 렌더 vs (B) 기하 JSON을 [[Canvas]]/WebGL 자체 렌더. 오버레이·엔티티 바인딩 용이성으로 비교.

#### 6.1.1 후보별 비교표 (채울 칸)
- 의도 메모: 후보 행 = PDF.js / 자체 Canvas 렌더 / 자체 WebGL / (참고)외부 상용뷰어. 열 = 오버레이 정밀도·엔티티ID 바인딩·성능·라이선스·구현비용.
- ❓ TBD: PDF 경유 시 "엔티티 ID 단위 클릭/하이라이트"가 어려움 → [[DKS]] 차별점과 충돌. 자체 기하 렌더 우세 가설, 검증 필요.

### 6.2 시트 비교(버전 diff) 렌더
- 의도 메모: 두 버전 오버레이 diff를 같은 렌더 레이어에서 처리할지. 상세 알고리즘은 [[시트비교-설계]] 위임.

### 6.3 좌표계 / 엔티티 바인딩
- 의도 메모: 픽셀 좌표 ↔ 도면 좌표 ↔ 설비 엔티티 ID 매핑. 향후 [[TypeDB]] 엔티티와 연결되는 핵심 접점. 상세는 [[엔티티-바인딩-모델]] 위임.

## 7. DB (RDB + TypeDB)

### 7.1 ADR-DB-001 1차 RDB
- 의도 메모: 시트·마크업·이슈·사용자·버전 등 트랜잭션 데이터 저장. 후보 [[PostgreSQL]] / SQL Server(.NET 친화). 납품 환경 라이선스 고려.
- ❓ TBD: 온프레 납품 시 SQL Server 라이선스 비용 vs PostgreSQL 무료.

#### 7.1.1 공간/기하 데이터 처리
- 의도 메모: 핀 좌표·도형 영역을 RDB에 어떻게(JSON 컬럼 vs PostGIS). 검색 패턴 기준 결정.

### 7.2 ADR-DB-002 Phase 2 [[TypeDB]] 온톨로지
- 의도 메모: 설비 엔티티·분야·도면번호·영향도를 [[TypeDB]] 온톨로지로. Phase 0 파싱 산출물(분야·도면번호·태그)을 입력으로.
- ★Phase 1에서는 도입 안 함. 단, RDB 스키마가 향후 엔티티 ID 바인딩을 막지 않도록 설계(전방 호환). 상세 온톨로지 모델은 [[온톨로지-데이터모델]] 위임.

#### 7.2.1 RDB ↔ TypeDB 동기화 경계
- 의도 메모: 진실원천 분담(트랜잭션=RDB, 의미/관계=TypeDB). 동기화 방향·시점 ❓ TBD.

### 7.3 캐시 / 검색 인덱스
- 의도 메모: 도면 목록 필터·전문검색 필요 시 인덱스 전략. 후보 RDB 인덱스로 충분 vs 별도 검색엔진. 초기엔 RDB 우선 가설.
