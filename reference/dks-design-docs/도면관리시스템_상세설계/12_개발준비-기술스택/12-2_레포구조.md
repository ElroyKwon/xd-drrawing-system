# 레포 구조

## ① 목적
- 청주 도면관리 시스템(ACC Build 재현)의 **소스 코드 조직 규약** 정의.
- 프런트(2D 뷰어 + 오버레이) / 백엔드([[DWG]] 파싱 → 기하 JSON) / 공유 타입을 **한 레포에서 일관 관리**.
- 신규 합류자가 "이 코드가 어디 있고 어디에 새 파일을 둘지" 1분 내 판단할 수 있는 결정 트리 제공.

## ② 배경 / 전제
- **핵심 스코프 제약**: DWG 원본 미편집. 흐름 = DWG → 백엔드 파싱 → 기하 JSON → 웹 2D 뷰어 렌더 → 마크업/이슈/이벤트 **오버레이**. CAD 편집기 아님.
- 1차 범위 = **2D 시트 한정**. 3D BIM / Bridge 범위 밖.
- Phase 2의 [[TypeDB]] 온톨로지 연동(엔티티 단위 검색·영향도·AI채팅)은 **패키지 경계 단계에서 자리만 예약**, 1차 구현 대상 아님.
- 본 문서는 **레포 구조/디렉토리 규약만** 소유. 기술 선정 근거·빌드 파이프라인은 [[12-1_기술스택]]·[[12-3_빌드-배포]]로 위임.

## ③ 상세 목차

## 레포 전략 (모노/멀티)
> 단일 레포 vs 멀티 레포 선택과 근거를 확정. 공유 타입(기하 JSON 스키마, 이슈/마크업 DTO)이 프런트·백엔드를 가로지르므로 모노레포 가정으로 출발.

### 선택지 비교
> 모노레포 / 멀티레포 / 폴리레포 후보를 표로 비교(공유 타입 동기화 비용·배포 독립성·CI 복잡도 기준).

### 권장안 + 근거
> 권장: **모노레포**(프런트·백엔드·공유 타입 동시 변경 잦음, 단일 납품 단위). 근거 3~4줄 채울 것.

### 워크스페이스 매니저
> pnpm workspace / npm workspace / Turborepo 중 택1 후보 메모. ❓ TBD — [[12-1_기술스택]]의 런타임 결정에 종속.

## 디렉토리 레이아웃
> 최상위 트리를 한눈에. 각 디렉토리 1줄 책임 명시.

### 최상위 트리(초안)
> 아래는 채워나갈 골격. 실제 도구 확정 후 조정.

```
repo/
├─ apps/
│  ├─ web/         # 2D 뷰어 + 오버레이 SPA (홈/시트/마크업/이슈/비교/파일/사진)
│  └─ api/         # 백엔드: DWG 파싱·기하 JSON 제공·이슈/마크업 영속화
├─ packages/
│  ├─ shared/      # 공유 타입(기하 JSON 스키마·DTO·enum)
│  ├─ viewer/      # 2D 렌더 코어(좌표계·레이어·줌/팬) — UI 비종속
│  └─ overlay/     # 마크업/핀/이슈 오버레이 모델(픽셀+엔티티ID 바인딩)
├─ services/
│  └─ dwg-parser/  # DWG → 기하 JSON 변환 워커(분리 가능성 대비)
├─ tools/          # 스크립트(시드·마이그레이션·코드젠)
├─ docs/           # 설계/운영 문서 stub (SoT는 vault)
└─ <repo-config>   # 워크스페이스/린트/포맷 설정
```

#### apps/ vs packages/ vs services/ 구분 규칙
> apps=배포 단위 / packages=재사용 라이브러리(빌드만, 단독배포X) / services=독립 프로세스 가능 후보. 경계 모호 시 packages로 시작 후 승격.

#### 모듈별 폴더 매핑(ACC 7모듈)
> 홈·시트·마크업·이슈·시트비교·파일·사진을 `apps/web` 내부 어떤 폴더 규약(feature 폴더 등)에 매핑할지. 공통 UI(앱셸·데이터테이블·인스펙터·위젯카드·모달·빈상태·토스트)는 별도 `ui/` 영역.

### 디렉토리 명명 규칙

## 패키지 경계
> 패키지 간 **의존 방향**을 고정해 순환 의존 차단. 단일 진실 위치(SoT) 원칙 적용.

### 의존 그래프(허용 방향)
> 규칙 초안: `shared` ← (viewer, overlay, api, web). `web` → (viewer, overlay, shared). `viewer`는 UI/도메인 비의존(순수 렌더). `overlay`는 `shared`만 의존. 역방향·순환 금지.

### shared 패키지 책임
> 기하 JSON 스키마, 이슈/마크업/이벤트 DTO, 좌표/단위 타입. **프런트·백엔드 공통 진실원천**. UI·DB 코드 반입 금지.

### viewer / overlay 분리 근거
> viewer=좌표·레이어·렌더만, overlay=주석/핀/이슈 데이터 모델. 분리로 렌더 엔진 교체([[PDF.js]] 등) 시 오버레이 영향 최소화.

### Phase 2 온톨로지 패키지 예약
> `packages/ontology-binding`(가칭) 자리만 예약 — 핀/마크업/이슈를 픽셀 좌표뿐 아니라 **설비 엔티티 ID**에 묶는 어댑터. [[TypeDB]] 연동·영향도·AI채팅의 프런트 접점. 1차 미구현, 인터페이스 stub만. ❓ TBD.

## 컨벤션
> 파일·코드·커밋 규약. 깊이 한도와 위치 결정 트리.

### 파일 배치 결정 트리
> "새 파일 어디 둘까" 1초 룰: 공유 타입→shared / 렌더→viewer / 주석모델→overlay / 화면→apps/web feature / 파싱→services. 망설이면 apps/web 내 임시 feature 후 승격.

### 깊이 한도
> 폴더 깊이 4~5 한도. 추가 분류는 폴더가 아닌 **파일명 규약**으로(vault §11 골자 준용).

### import 경로 규칙
> 워크스페이스 별칭(`@app/*`, `@shared/*` 등) vs 상대경로. 패키지 간은 별칭 강제, 내부는 상대.

### 코드 스타일 / 린트 위임
> 린터·포매터·네이밍 상세는 [[12-1_기술스택]] 또는 별도 컨벤션 문서로 위임(DRY). 여기선 위치만 명시.

### 커밋 / 브랜치 규약
> Conventional Commits 채택 여부, 브랜치 전략. ❓ TBD — [[12-3_빌드-배포]]와 정합 필요.
