## 목적

청주 사업장 납품용 [[도면관리 시스템]] 개발의 **Phase별 로드맵·마일스톤·산출물·게이트**를 한 곳에서 정의.
- 이 문서가 소유하는 단일 주제 = **일정 & 마일스톤**(언제·무엇을·어떤 순서로).
- 기술 스택/아키텍처/모듈 상세 설계는 각 소유 문서로 위임([[링크]]).

## 배경 / 전제

- 1차 목표 = [[Autodesk Construction Cloud]](ACC) Build 제품 **동등 재현**(벤치마크 37화면 분석 존재).
- ★핵심 스코프: **CAD 편집기 아님**. 흐름 = [[DWG]] → 백엔드 파싱 → 기하 JSON → 웹 2D 뷰어 렌더 → 마크업/이슈/이벤트 **오버레이**.
- 범위 밖: 정밀 CAD 편집(osnap·좌표편집·[[SHX]] 폰트 정밀렌더), 3D [[BIM]] 뷰어/Bridge.
- 1차 = **2D 시트 한정**.
- [[DKS]] 차별점: 핀/마크업/이슈를 픽셀이 아닌 **설비 엔티티 ID**에도 바인딩 → Phase 2 [[TypeDB]] 온톨로지 연동(영향도·AI채팅). Phase 0 도면 파싱 산출물(분야·도면번호·태그) 기보유.
- 본 로드맵은 일정 **골격**이며 확정 날짜·인력·기간은 대부분 TBD(과장 수치 금지).

## 마일스톤(M1~Mn)

전체 진행을 게이트 단위로 끊어 정의. 각 M은 "완료 판정 기준(DoD)"과 "산출물"을 가진다.

| 마일스톤 | 명칭 | 핵심 산출물 | 완료 판정(DoD) | 목표 시점 |
|---|---|---|---|---|
| M0 | 기반 셋업 | 앱셸·라우팅·프로젝트 스위처 | 빈 모듈 네비게이션 동작 | TBD |
| M1 | 뷰어 코어 | DWG 파싱 → 기하 JSON → 2D 렌더 | 샘플 도면 웹 렌더 성공 | TBD |
| M2 | 시트 모듈 | 도면 목록 데이터테이블 + 뷰어 연동 | 시트 열람·전환 | TBD |
| M3 | 마크업/이슈 | 오버레이 주석 + 이슈 핀/폼/목록 | 마크업·이슈 CRUD | TBD |
| M4 | 비교/파일/사진 | 버전 오버레이 diff·CDE·사진 | 3개 보조 모듈 동작 | TBD |
| M5 | Phase 1 출하 | 청주 납품 빌드 | 인수 테스트 통과 | TBD |
| M6+ | Phase 2 AI | 엔티티 바인딩·온톨로지·AI채팅 | 별도 게이트 정의 | TBD |

> 각 행의 "목표 시점"은 인력/기간 확정 후 채운다. 의존 순서(M0→M1→M2…)는 고정.

#### 마일스톤 게이트 규칙
- 메모: 다음 M 착수 전 직전 M의 DoD·데모·리뷰 통과 필요. 게이트별 데모 시나리오는 TBD.

## Phase1 도면관리 범위

ACC Build 동등 재현 = Phase 1. 모듈 단위로 착수 순서와 의존을 정의.

### 모듈별 착수 순서 / 의존도

| 순서 | 모듈 | 의존 선행 | 비고 |
|---|---|---|---|
| 1 | 앱셸/공통 UI | - | 사이드바·스위처·데이터테이블·인스펙터·토스트·빈상태 |
| 2 | 시트(도면목록+2D뷰어) | 뷰어 코어(M1) | 진실원천 모듈 |
| 3 | 마크업(펜·도형·화살표) | 뷰어 | 오버레이 레이어 |
| 4 | 이슈(핀+상세폼+목록+필터) | 뷰어·마크업 | 엔티티 ID 바인딩 훅 선반영 |
| 5 | 시트비교(버전 오버레이 diff) | 시트 | 버전 모델 필요 |
| 6 | 파일(폴더형 CDE) | 앱셸 | [[CDE]] 폴더 트리 |
| 7 | 사진 | 앱셸 | 보조 |
| 8 | 홈(위젯 대시보드) | 위 전부 | 집계 위젯이라 후순위 |

> 상세 화면/필드는 각 모듈 설계 문서가 소유. 여기선 순서·의존만.

#### 뷰어 코어 선결 과제
- 메모: 렌더러 선정([[PDF.js]] vs Canvas/SVG 직접) 및 파싱 파이프라인은 기술스택 문서가 소유 → 여기선 M1 게이트 일정만.

#### 공통 UI 우선 처리
- 메모: 앱셸/데이터테이블/인스펙터/모달/토스트/빈상태를 M0에 몰아 표준화 → 이후 모듈 재사용(DRY).

### Phase 1 범위 경계(In/Out)
- In: 2D 시트 뷰어, 마크업, 이슈, 비교, 파일, 사진, 홈.
- Out: CAD 편집, 3D/BIM, 정밀 스냅·좌표편집, AI채팅(→Phase 2).
- 메모: 경계 변경 시 본 절 갱신 + 마일스톤 영향 재평가.

## Phase2 AI 범위

ACC에 없는 [[DKS]] 차별 기능. Phase 1 안정화 후 별도 게이트(M6+).

### AI/온톨로지 단계

| 단계 | 내용 | 선행 |
|---|---|---|
| P2-a | 엔티티 ID ↔ 핀/마크업/이슈 바인딩 정착 | Phase 1 이슈 모듈 |
| P2-b | [[TypeDB]] 온톨로지 연동(설비 엔티티·관계) | Phase 0 파싱 산출물 |
| P2-c | 엔티티 단위 검색·영향도 분석 | P2-b |
| P2-d | AI 채팅/지식질의 뷰어·이슈 바인딩 | P2-c |

#### 엔티티 바인딩 선반영
- 메모: Phase 1 이슈/마크업 데이터모델에 `entityId` 필드를 미리 둔다(후행 마이그레이션 회피). 스키마는 데이터모델 문서가 소유.

#### 실시간/스트리밍 연동
- 메모: AI 응답·협업 갱신에 [[WebSocket]] 활용 여부 TBD → 기술스택 문서와 정합.

### Phase 2 범위 경계
- 메모: Phase 2는 본 로드맵에서 골격만. 상세 일정은 Phase 1 출하 후 재기획.

## 리스크 / 의존성

로드맵을 흔들 수 있는 항목. 발견 즉시 추가·갱신.

### 기술 리스크

| 리스크 | 영향 | 대응(잠정) |
|---|---|---|
| DWG 파싱 정확도/누락 | M1 지연 | 샘플 도면 조기 검증, 파서 후보 비교 |
| 대용량 도면 렌더 성능 | 뷰어 UX | 타일링/지연로딩 검토(TBD) |
| 한글/특수 폰트 표시(편집 아닌 표시) | 가독성 | 폰트 폴백 정책 TBD |
| 버전 diff 정합성 | M4 | 비교 알고리즘 사전 PoC |

### 일정/조직 의존성
- ❓ 투입 인력 수·역할 분담 미확정 → 전체 기간 산정 불가(TBD).
- ❓ 청주 납품 인수 일정/기준 미확정.
- 의존: Phase 0 파싱 산출물 품질 = Phase 2 전제.
- 메모: 외부 라이브러리 라이선스(뷰어 등) 검토는 기술스택 문서로 위임.
