# 청주 온프레 인프라

## 1. 목적
- 청주 사업장 납품용 [[도면관리시스템]]의 **현장 온프레미스(on-prem) 배포 구성**을 정의.
- 폐쇄망/제한망 가정 하에 [[ACC]] Build 동등 재현 시스템을 자체 서버에 올리기 위한 HW·네트워크·서버 역할 골격 제공.
- 이 문서가 소유하는 주제 = **물리/논리 배포 토폴로지**. 컨테이너 오케스트레이션·CI/CD 절차는 [[10-1_배포-개요]]에 위임.

## 2. 배경 / 전제
- 핵심 스코프 제약: DWG **편집 없음**. 흐름 = [[DWG]] → 백엔드 파싱 → 기하 JSON → 웹 2D 뷰어 렌더 → 마크업/이슈/이벤트 **오버레이**.
  - → 인프라 함의: 무거운 CAD 엔진 불필요. 파싱은 **배치/비동기 워커**, 뷰어 렌더는 클라이언트([[PDF.js]]/Canvas) 부담. 서버는 정적 JSON 서빙 + API + 실시간 채널 중심.
- 1차 범위 = **2D 시트 한정**. 3D BIM 뷰어/Bridge 범위 밖 → GPU 서버 불요(Phase 1).
- 네트워크 전제: 청주 사업장 **사내망/폐쇄망 가능성**. 외부 클라우드 의존 최소화(전 구성 온프레 완결). ❓ 인터넷 아웃바운드 허용 여부 TBD.
- DKS 차별점: 핀/마크업/이슈를 픽셀뿐 아니라 **설비 엔티티 ID**에도 바인딩. Phase 2 [[TypeDB]] 온톨로지 서버가 추가되므로 **확장 슬롯**을 토폴로지에 미리 예약(§"서버 역할" 참조).

## 3. 하드웨어 / 네트워크

### 3.1 서버 사양(가정)
- 단일 물리 서버 또는 사내 가상화(VM) 1~2대 가정. ❓ 물리 vs VM(현장 인프라팀 표준) TBD.
- 산정 기준 항목(채울 것): vCPU / RAM / 디스크(도면 원본+기하 JSON+첨부 사진 용량) / 네트워크 대역.
- 파싱 워커 부하 산정 메모: 동시 파싱 작업 수 × 평균 DWG 크기 기준. ❓ 청주 초기 도면 수량/평균 용량 TBD → 산정 입력값 필요.

### 3.2 네트워크 구성
- 망 구분(사내망/DMZ/폐쇄망) 및 접근 경로 다이어그램을 채울 것.
- 포트/방화벽: HTTP(S), API, [[WebSocket]](실시간 마크업/이벤트), DB 포트 — 허용 목록 정리.
- 도메인/인증서: 사내 도메인 + 사설/사내 CA 인증서 가정. ❓ HTTPS 강제 여부·인증서 발급 주체 TBD.

#### 3.2.1 외부 연동 경계
- 폐쇄망 시 외부 LLM/AI API 호출 불가 → Phase 2 AI 채팅은 **온프레 LLM 또는 별도 게이트웨이** 필요. 상세는 [[20_백엔드-파싱]] 아닌 별도 AI 문서로 위임(TBD 링크).

### 3.3 스토리지 / 파일 레이아웃
- 원본 [[DWG]] 보관 영역(불변, 진실원천) + 파생 기하 JSON 캐시 + 첨부(사진/이슈 첨부) 분리.
- [[CDE]] 폴더형 파일 모듈의 물리 저장 위치 매핑(권한·버전 보관).
- 백업 대상 경로 식별(§5 연계).

## 4. 서버 역할

### 4.1 역할 분해(논리)
- 각 역할의 책임 1줄씩 채울 것. Phase 1 최소 구성 → 추후 분리 가능하도록 경계 명시.

#### 4.1.1 웹/API 서버
- 프론트 정적 자산 서빙 + REST API(시트/마크업/이슈/파일/사진). 표준 응답 포맷은 [[10-1_배포-개요]] 규약 따름.

#### 4.1.2 파싱 워커
- DWG → 기하 JSON 변환 배치. 큐 기반 비동기. 동시성/재시도/실패 격리 정책 채울 것.

#### 4.1.3 실시간 채널 서버
- [[WebSocket]] 기반 마크업/이슈/이벤트 실시간 동기화. ❓ API 서버 내장 vs 분리 TBD.

#### 4.1.4 데이터 저장소
- 메타/마크업/이슈 RDB + 기하 JSON·첨부 객체 스토리지/파일시스템. 스키마는 [[20_백엔드-파싱]]·데이터모델 문서에 위임.

### 4.2 Phase 2 확장 슬롯 (예약)
- [[TypeDB]] 온톨로지 서버 노드 1개 예약. 설비 엔티티 ID ↔ 핀/마크업/이슈 바인딩 질의 담당.
- 온프레 AI 채팅 게이트웨이(LLM 호스팅 또는 프록시) 슬롯 예약.
- 메모: Phase 1 토폴로지에서 이 두 슬롯의 **포트/리소스 자리만** 비워두고 실제 배포는 Phase 2.
