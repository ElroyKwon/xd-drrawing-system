# CI/CD 파이프라인

## ① 목적
- 청주 사업장 납품용 도면관리 시스템의 **빌드·테스트·배포 자동화** 단일 소유 문서.
- 코드 push → 자동 빌드 → 게이트 통과 → 온프레 배포까지의 흐름 정의.
- *주의:* 배포 대상 토폴로지·서버 스펙은 [[10-1_배포-아키텍처]] 위임. 테스트 케이스 설계는 [[09_테스트-전략]] 위임. 본 문서는 "자동화 파이프라인" 그 자체만 소유.

## ② 배경 / 전제
- 납품처가 **온프레(폐쇄망) 청주 사업장** → 클라우드 SaaS CI 직접 배포 불가. 인터넷 단절 환경 전제.
- 스코프 제약: 시스템은 [[DWG]] 비편집 "뷰어 + 오버레이". 빌드 산출물 = ① 프론트(2D 뷰어 SPA) ② 백엔드(파싱·기하 JSON API) ③ 파서 워커. → 멀티 아티팩트 파이프라인.
- 컨테이너 기반 가정([[Docker]]). 레지스트리·오케스트레이션 상세는 [[10-2_인프라-구성]] 위임.
- 1차 범위 = 2D 시트 한정. Phase 2([[TypeDB]] 온톨로지 연동) 배포는 향후 별도 잡으로 확장 (현재 TBD).

## ③ 상세 목차

## 파이프라인 단계
> 채울 것: source → build → test → package → deploy 전체 스테이지 정의 + 트리거 조건.

### 트리거 / 브랜치 전략
> 채울 것: main/release/feature 브랜치별 어떤 잡이 도는지. PR 시 vs 머지 시 차이.
- ❓ TBD: 형상관리 도구(Git 호스팅) — GitLab(self-host) vs Gitea vs 사내 표준? 폐쇄망이라 self-host 필수.
- ❓ TBD: CI 러너 — GitLab CI / Jenkins / GitHub Actions(self-host runner) 중 택1.

### 빌드 스테이지
> 채울 것: 3개 아티팩트(프론트 SPA / 백엔드 API / 파서 워커) 빌드 명령·산출물.

#### 프론트엔드 빌드
> 채울 것: 2D 뷰어 SPA 번들. [[PDF.js]]·캔버스 렌더 의존성 빌드 시 정적 자산 포함.
- 산출물: 정적 번들 → Docker 이미지(nginx) 또는 백엔드 정적 서빙. TBD 결정.

#### 백엔드 빌드
> 채울 것: 파싱 API + 기하 JSON 서빙 서버 빌드. [[WebSocket]](실시간 마크업·이슈 동기화) 의존 포함.
- ❓ TBD: 백엔드 런타임 스택 미확정 → 빌드 명령 확정 후 기입.

#### 파서 워커 빌드
> 채울 것: DWG → 기하 JSON 변환 워커 이미지. CAD 파싱 라이브러리 라이선스·네이티브 의존성 주의.
- ❓ TBD: DWG 파싱 엔진(ODA / Teigha 등) 라이선스 → 폐쇄망 빌드 시 오프라인 설치 가능 여부.

### 패키징 / 아티팩트
> 채울 것: 빌드 결과를 어떤 형태로 묶어 배포처로 옮기는가.

#### 컨테이너 이미지 빌드·태깅
> 채울 것: 이미지 네이밍·태그 규칙(semver / commit-sha). 멀티스테이지 Dockerfile 위치.

#### 오프라인 아티팩트 번들
> 채울 것: 폐쇄망 반입용 — 이미지 tar + DB 마이그레이션 + 설정 묶음. USB/내부망 반입 절차.
- ❓ TBD: 폐쇄망 반입 매체·승인 절차(고객사 보안정책) 확인 필요.

#### 사설 레지스트리 푸시
> 채울 것: 사내/현장 Harbor 등 사설 레지스트리 푸시. 상세 인프라는 [[10-2_인프라-구성]] 참조.

## 테스트 게이트
> 채울 것: 어떤 검증을 통과해야 다음 스테이지로 진행하는가(게이트 = 실패 시 파이프라인 중단). 케이스 설계 자체는 [[09_테스트-전략]] 소유, 여기선 "게이트 배치·통과 기준"만.

### 정적 검사 게이트
> 채울 것: lint·타입체크·포맷·보안 스캔(의존성 취약점). 빌드 전 1차 차단.

### 단위·통합 테스트 게이트
> 채울 것: 파서 정확도(샘플 DWG → 기대 기하 JSON), 오버레이 좌표 매핑, 이슈/마크업 CRUD API 테스트.
- 핵심 회귀: DWG 좌표계 ↔ 웹 뷰어 픽셀 변환, 핀이 **설비 엔티티 ID**에 올바르게 바인딩되는지(DKS 차별점, Phase 2 대비 회귀 보호).

### 커버리지 / 품질 임계
> 채울 것: 커버리지 하한선·신규 코드 기준. 통과 못 하면 머지 차단.
- ❓ TBD: 커버리지 임계 수치 — 팀 합의 후 기입.

### E2E / 스모크 게이트
> 채울 것: 배포 직전 핵심 플로우(도면 업로드→파싱→뷰어 렌더→마크업→이슈 생성) 스모크 자동 검증.

## 배포 전략 (온프레)
> 채울 것: 폐쇄망 현장에 무중단/저위험으로 올리는 방식. 클라우드형 블루-그린 대비 온프레 제약 반영.

### 환경 분리 (dev / staging / 현장)
> 채울 것: 사내 dev·staging → 청주 현장(prod) 승급 경로. 현장은 망분리라 수동 게이트 개입 전제.

### 배포 방식
> 채울 것: 롤링 vs 블루-그린 vs 재기동. 컨테이너 오케스트레이터 위에서의 무중단 여부.
- ❓ TBD: 오케스트레이터(Docker Compose / k3s 등) — 현장 운영 인력 역량 고려해 단순화 우선 검토.

### DB 마이그레이션
> 채울 것: 스키마 변경 자동 적용·순서 보장. 마크업·이슈·버전 데이터 손실 방지.
- 향후 [[TypeDB]] 온톨로지 스키마는 별도 마이그레이션 트랙 (Phase 2, TBD).

### 롤백 절차
> 채울 것: 직전 이미지 태그로 즉시 복귀 + DB 다운 마이그레이션 가능 여부. 폐쇄망 무인 시간대 대비.

### 배포 승인 게이트 (수동)
> 채울 것: 현장 prod 승급 전 책임자 수동 승인. 폐쇄망 반입·작업 윈도우 협의 반영.

### 헬스체크 / 배포 검증
> 채울 것: 배포 후 자동 헬스체크(파서 워커·API·뷰어 응답) + 스모크 통과 시 완료 처리.
