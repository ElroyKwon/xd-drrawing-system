# 컨테이너 전략

## 1. 목적
- 청주 사업장 납품용 [[도면관리 시스템]]의 3개 런타임([[DWG]] 파서 / AI / 웹)을 재현 가능하고 격리된 컨테이너 이미지로 빌드·배포하는 전략 정의.
- 이 문서가 소유하는 것 = **파서/AI/웹 컨테이너의 이미지 구성·빌드 절차·구성/시크릿 주입**만. 오케스트레이션·네트워크·CI/CD 파이프라인은 [[링크]]로 위임.

## 2. 배경 / 전제
- 시스템 흐름: [[DWG]] → 백엔드 파싱 → 기하 JSON → 웹 2D 뷰어 렌더 → 마크업/이슈/이벤트 오버레이. **CAD 편집기 아님 = 뷰어+주석 시스템** → 파서 컨테이너는 "읽기·변환 전용"이며 무거운 CAD 편집 의존성 불필요.
- 1차 범위 = 2D 시트 한정. 3D BIM 렌더 툴체인은 이미지에 포함하지 않음.
- AI 채팅/지식질의는 Phase 2([[TypeDB]] 온톨로지 연동). 1차 이미지에서는 AI 컨테이너를 **선택적/비활성 프로파일**로 둘지 결정 대기(❓ §아래).
- 차별점(핀/마크업/이슈를 설비 엔티티 ID에 바인딩)은 Phase 2 AI 컨테이너가 [[TypeDB]]와 통신하는 지점에서 구현 — 본 문서는 그 컨테이너의 "껍데기"만 정의, 로직은 위임.

## 3. 컨테이너 분할 개요
- 3개 이미지: **parser**(DWG→기하 JSON 변환 워커) / **api+web**(앱 셸·뷰어·이슈/마크업 API) / **ai**(Phase 2, 온톨로지 질의).
- 단일 책임 원칙: 한 이미지 = 한 런타임. 파서의 변환 부하가 웹 응답성에 영향 주지 않도록 분리.
- ❓ TBD: api와 web을 한 이미지로 묶을지(SPA 정적 서빙 동거) vs 분리할지 — 프론트 빌드 산출물 서빙 방식 확정 후 결정.

## 이미지 구성

### parser 이미지
- (채울 내용) [[DWG]] 파싱 런타임 베이스 결정: 변환 라이브러리/CLI 선정 결과 + 기하 JSON 출력 포맷 연결.
- (채울 내용) 폰트/[[SHX]] 정밀 렌더는 범위 밖 → 관련 패키지 **제외**해 이미지 슬림화.

#### 베이스 이미지 / 런타임
- (채울 내용) 언어 런타임(예: Python/Node/네이티브 변환툴 중 택1) + slim/alpine 여부 + 변환 의존성 목록. ❓ 변환 엔진 미확정 → TBD.

#### 시스템 의존성
- (채울 내용) 변환 CLI가 요구하는 OS 패키지만 명시. 불필요 GUI/X11 의존 배제.

#### 비루트 실행
- (채울 내용) non-root user 지정, 작업 디렉터리·임시 변환 디렉터리 권한.

### api+web 이미지
- (채울 내용) 앱 셸(모듈 사이드바·프로젝트 스위처)·2D 뷰어·이슈/마크업 API 서빙 런타임.
- (채울 내용) 뷰어 렌더는 [[PDF.js]]/Canvas 계열 클라이언트 자산 → 정적 번들로 포함.

#### 백엔드 런타임
- (채울 내용) API 서버 런타임/버전. 이슈·마크업·시트비교 엔드포인트 호스팅.

#### 프론트엔드 정적 자산
- (채울 내용) SPA 빌드 산출물 위치 + 정적 서빙 주체(API 동거 vs 분리). [[WebSocket]] 실시간(이벤트/협업 핀) 경로 노출 여부 명시.

### ai 이미지 (Phase 2)
- (채울 내용) 온톨로지 질의/AI 채팅 런타임. [[TypeDB]] 클라이언트 의존성 포함.
- ❓ TBD: 1차 납품 시 빌드 대상에서 제외 vs 빈 스텁으로 포함 — Phase 2 시작 시점 확정.

#### 모델/런타임 의존성
- (채울 내용) 추론 클라이언트·임베딩 의존성. 모델 가중치는 이미지 미포함(볼륨/외부) 원칙 명시.

## 멀티스테이지 빌드

### 빌드 스테이지 전략
- (채울 내용) 각 이미지의 build 스테이지(의존성 설치·컴파일·프론트 번들)와 runtime 스테이지 분리 — 빌드 도구 최종 이미지 미포함.

#### parser 멀티스테이지
- (채울 내용) 변환 엔진 빌드/다운로드 스테이지 → 런타임 복사. 캐시 가능 레이어 순서(의존성 먼저, 소스 나중).

#### api+web 멀티스테이지
- (채울 내용) 프론트 빌드 스테이지(node) → 정적 산출물만 백엔드 런타임 스테이지로 복사. 백엔드 의존성 캐시 레이어.

### 레이어 캐싱 / 이미지 슬림화
- (채울 내용) 의존성 매니페스트 우선 복사로 캐시 적중률 확보. 최종 이미지 크기 목표는 측정 후 기입(TBD, 허구 수치 금지).

### 빌드 재현성
- (채울 내용) 베이스 이미지 다이제스트 핀, 의존성 lockfile 고정. 빌드 인자(버전 태그) 규약.

## 구성 / 시크릿

### 환경 변수 / 구성 주입
- (채울 내용) 환경별(개발/청주 납품) 구성 분리 방식 — 런타임 환경변수 주입 원칙. 이미지에 환경값 하드코딩 금지.

#### 구성 카탈로그
- (채울 내용) 각 컨테이너가 요구하는 변수 표(이름·용도·필수여부). DB 접속, 저장소 경로, [[WebSocket]] 엔드포인트 등. 채울 항목만 자리표시.

### 시크릿 관리
- (채울 내용) DB 자격증명·[[TypeDB]] 접속정보·외부 AI 키 주입 경로. 이미지·git 미포함 원칙.
- ❓ TBD: 시크릿 주입 방식(오케스트레이터 secret vs 파일 마운트 vs 외부 vault) — 납품 환경 제약 확인 후 결정.

#### 청주 납품 환경 제약
- (채울 내용) 폐쇄망/인터넷 차단 여부에 따른 이미지 반입·레지스트리 전략. ❓ 망 환경 미확인 → TBD.

### 영속 데이터 / 볼륨 경계
- (채울 내용) 이미지에 넣지 않는 것 = DWG 원본·기하 JSON 캐시·업로드 파일·모델 가중치 → 볼륨/외부 스토리지로 위임. 상세는 [[10_배포-인프라]] 스토리지 절로 위임.
