# 배포/인프라 개요

## ① 목적
- 청주 사업장 납품용 [[도면관리 시스템]]의 **배포·운영 인프라 전체 그림**을 한 장에 제시
- 이 문서는 **배포 문서 인덱스(Index)**만 소유 — 세부 구성/절차는 하위 문서로 위임([[DRY]])
- 독자: 현장 구축 담당, 인프라 엔지니어, 검수 담당

## ② 배경/전제
- **스코프 제약 재확인**: 본 시스템은 [[DWG]] 편집기가 아님. 흐름 = `DWG → 백엔드 파싱 → 기하 JSON → 웹 2D 뷰어 렌더 → 마크업/이슈/이벤트 오버레이`
  - 인프라 관점 함의 → 무거운 CAD 엔진 불필요, 단 **파싱 워커**(배치형)와 **뷰어 정적 자산 서빙**이 핵심 부하 지점
- **1차 범위 = 2D 시트 한정**. 3D BIM 뷰어/Bridge 범위 밖 → GPU 서버 등 고사양 요건 없음
- **납품 형태 = 청주 온프레미스(on-prem)**. 폐쇄망 가능성 고려 → 외부 클라우드 의존 최소화, 오프라인 설치 패키지 전제
- **Phase 2(향후)**: [[TypeDB]] 온톨로지 연동(AI 채팅/지식질의) → 별도 서비스로 추가 배포. 본 문서는 그 **확장 슬롯**만 표시, 상세는 [[온톨로지-연동]]으로 위임

> ❓TBD: 청주 사업장 망 구성(인터넷 격리 여부·반입 정책) — 확정 시 본 문서 ② 갱신

## ③ 상세 목차

### 구성
> 의도: 시스템을 이루는 배포 단위(컨테이너/서비스)와 그 관계를 인덱스 수준으로 나열. 각 컴포넌트의 상세 스펙은 별도 문서로 위임.

#### 컴포넌트 인벤토리
> 의도: 배포되는 서비스 목록을 표로. 각 행 = [이름 / 역할 / 기술스택 / 상태(필수·선택)].
- 웹 프론트(앱셸 + 2D 뷰어, [[PDF.js]]/Canvas 기반) — 정적 자산
- API 백엔드(시트·마크업·이슈·파일 CRUD)
- **파싱 워커**(DWG → 기하 JSON 변환, 배치/큐 소비)
- 실시간 채널([[WebSocket]] — 마크업·이슈·이벤트 동기화)
- 데이터베이스(메타/이슈/마크업 영속)
- 오브젝트 스토리지(DWG 원본·기하 JSON·사진·시트 비교 산출물)
- (Phase 2) [[TypeDB]] 온톨로지 서비스 — 확장 슬롯
> ❓TBD: 각 항목 기술스택 확정(언어·DB 종류). 현재 미확정 → [[기술스택-결정]]에서 채움

#### 데이터 흐름 / 의존 관계
> 의도: 업로드→파싱→렌더→오버레이 경로와 서비스 간 호출 방향을 1개 다이어그램(텍스트/mermaid)으로. 동기 vs 비동기(큐) 경계 명시.

#### 스토리지 레이아웃
> 의도: 무엇을 DB에 / 무엇을 오브젝트 스토리지에 두는지 경계. DWG 원본은 불변(immutable) 보관, 기하 JSON은 파생 캐시라는 점 기록.
- 원본 도면 = 진실원천(SoT) → 불변 보관, 버전별 유지(시트 비교·이슈 추적 근거)

#### 엔티티 바인딩 데이터(차별점)
> 의도: 핀/마크업/이슈를 픽셀 좌표뿐 아니라 "설비 엔티티 ID"에 묶는 매핑 데이터의 저장 위치·키 구조 메모. Phase 0 파싱 산출물(분야·도면번호·태그)과의 연결 지점. 상세는 [[엔티티-바인딩]]으로 위임.

#### 외부 의존성
> 의도: 폰트/지도/인증 등 외부 의존 항목 나열 + 온프레/폐쇄망에서의 대체 방안. (SHX 폰트 정밀 렌더는 스코프 밖이므로 폰트 의존 최소.)

---

### 배포 환경 (청주 온프레)
> 의도: 청주 현장 서버에 실제로 올리는 방법·요건을 인덱스화. 절차 상세는 하위 런북으로 위임.

#### 하드웨어 요건
> 의도: 최소/권장 서버 사양 표(CPU·RAM·디스크·네트워크). 파싱 워커가 피크 부하원임을 반영. 동시 사용자 규모 가정 명시.
> ❓TBD: 청주 동시 접속자 수·도면 총량(시트 수) — 사이징 입력값 미확정

#### 소프트웨어 스택 / 런타임
> 의도: OS·컨테이너 런타임([[Docker]] 등)·리버스 프록시 등 기반 SW 버전 핀(pin). 오프라인 이미지 반입 전제.

#### 토폴로지 / 네트워크
> 의도: 단일 서버 올인원 vs 분리 배포 결정. 포트·도메인·내부 통신 경계. 폐쇄망 시 외부 통신 0 가정.

##### 단일 서버(올인원) 안
> 의도: 청주 1차 납품 기본안 — 모든 서비스 1대 집약. 장단점 메모.

##### 분리 배포(확장) 안
> 의도: 부하/이중화 필요 시 파싱 워커·DB 분리. Phase 2 TypeDB 추가 시 확장 경로.

#### 설치 / 배포 절차
> 의도: 오프라인 패키지 → 설치 → 기동 → 검증의 단계 인덱스만. 실제 명령/런북은 [[설치-런북]]으로 위임.

#### 백업 / 복구
> 의도: DB·오브젝트 스토리지(특히 DWG 원본·이슈/마크업) 백업 주기·복구 절차 골격. 진실원천 보존 우선순위.

#### 보안 / 접근제어
> 의도: 인증 방식·역할(권한) 모델·HTTPS·망 격리 항목 인덱스. 상세 권한 매트릭스는 [[권한-모델]]로 위임.
> ❓TBD: 청주 계정 체계(자체 vs 사내 SSO 연동)

#### 모니터링 / 로깅
> 의도: 헬스체크·로그 수집·파싱 실패 알림 등 운영 가시성 항목. 개발 단계 실시간 로그 패널과의 관계 메모.

#### 운영 / 업데이트
> 의도: 무중단 여부·버전 업그레이드·롤백 전략. 온프레 수동 갱신 전제.
