# AI/온톨로지 연동 (Phase 2) 개요

## 목적
- 청주 도면관리 시스템의 **향후(Phase 2) AI 채팅·지식질의** 진입점 인덱스
- 1차([[ACC]] Build 동등 재현)에서는 **구현하지 않음**. 다만 1차 데이터 모델이 Phase 2를 막지 않도록 "가드"를 명시
- 이 문서는 **인덱스/위임 허브**다. 실제 설계는 하위 문서가 소유 (DRY)

> ★전제 재확인: 본 시스템은 [[DWG]] 편집기가 아니라 **뷰어 + 주석/이벤트 오버레이**. AI는 "도면을 그리는" 주체가 아니라 "도면 위 엔티티/이슈/마크업을 질의·추론"하는 주체.

## 배경/전제
- DKS 차별점 = ACC엔 없는 **온톨로지 엔티티 단위** 검색·영향도·AI 채팅
- 도면 = 진실원천(SoT). Phase 0에서 이미 **분야·도면번호·태그 파싱 산출물** 확보
- 우리의 무기 = 핀/마크업/이슈를 **픽셀이 아니라 "설비 엔티티 ID"** 에도 바인딩 → AI가 "이 [[CDU]] 관련 이슈 다 보여줘" 같은 엔티티 질의 가능
- Phase 2 지식 백엔드 후보 = [[TypeDB]] 온톨로지 (4계층). 연동 방식은 TBD

## Phase 2 범위
> 무엇을 채울지: Phase 2에서 "켜질" 기능 묶음과 그 경계를 확정. 1차 범위와 명확히 선 긋기.

### 포함(예정)
- 자연어 도면 질의 ("청주 동 2층 냉동기 계통도 어디?")
- 엔티티 기반 검색·필터 (설비 ID·분야·태그 교차)
- 영향도 추론 (한 엔티티 변경/이슈 → 연관 설비·도면 파급)
- 이슈/마크업 컨텍스트 요약 (엔티티에 묶인 이벤트 타임라인)

### 제외(Phase 2에서도 범위 밖)
- DWG 원본 편집·CAD 생성
- 3D BIM 추론
- ❓ 멀티 프로젝트 횡단 추론 — Phase 2 vs Phase 3? (TBD)

### AI 표면(UI 바인딩 지점)
> 무엇을 채울지: 채팅이 어느 화면에 붙는지. 전역 채팅 패널 vs 뷰어 인스펙터 내장 vs 둘 다.
- 전역 채팅 패널 (앱셸 우측 인스펙터 재활용 검토 → [[_ACC-Build-화면분석-재현설계]])
- 뷰어 내 "엔티티 클릭 → AI 질의" 컨텍스트 메뉴
- ❓ 채팅 응답에서 도면/이슈로 딥링크 점프 방식 (TBD)

## 지금 준비할 가드
> 무엇을 채울지: 1차에서 코드 한 줄 더 쓰지 않되, Phase 2를 "막지 않기 위한" 최소 데이터/구조 약속. 과잉 추상화 금지.

### 데이터 모델 가드
- 핀/마크업/이슈 레코드에 **`entity_id` 필드(nullable)** 자리만 확보 → 1차엔 비워도 됨
- Phase 0 파싱 산출물(분야·도면번호·태그)과의 **조인 키** 합의 ❓ (도면번호? 시트 UUID? TBD)
- 기하 JSON 스키마에 엔티티 라벨/레이어 메타 **보존** (버리지 말 것)

### 인터페이스 가드
- 검색/필터 API를 "텍스트+태그"로 설계하되, 추후 **엔티티 질의 파라미터 추가 가능**한 형태로 (스키마는 [[02_시트-2D뷰어]]·[[04_이슈오버레이]]가 소유)
- 실시간 채널([[WebSocket]]) 이벤트에 type 필드 → 향후 `ai_*` 이벤트 확장 여지

### 비-가드(지금 하지 말 것)
- TypeDB 스키마 작성·온톨로지 모델링 → Phase 2 진입 시
- LLM 프롬프트/RAG 파이프라인 설계 → Phase 2
- 벡터 인덱싱·임베딩 저장소 → Phase 2
