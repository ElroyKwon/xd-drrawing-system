# 챗봇-뷰어 이벤트 버스

## 1. 목적
- 이 문서가 소유하는 단일 주제 = **대화(챗봇) ↔ 도면 2D 뷰어 양방향 하이라이트**의 메시지 계약과 트리거 흐름.
- AI 채팅 답변에 등장한 설비/도면 엔티티를 뷰어에서 시각적으로 강조하고(챗봇→뷰어), 역으로 뷰어에서 사용자가 선택한 객체를 대화 컨텍스트로 올린다(뷰어→챗봇).
- ※ 채워야 할 것: 본 버스가 "왜 직접 호출이 아니라 이벤트 버스인가"의 한 줄 근거(느슨한 결합·다중 구독자·재생 가능성).

## 2. 배경/전제
- ★스코프 제약: [[DWG]] 원본 편집 없음. 흐름 = DWG → 백엔드 파싱 → 기하 JSON → 웹 2D 뷰어 렌더 → 그 위 오버레이. 본 버스는 **렌더된 뷰어 + 오버레이 레이어**만 조작한다(지오메트리 자체는 불변).
- 1차 범위 = 2D 시트 한정. 3D/Bridge 범위 밖.
- 챗봇·온톨로지 질의 엔진 자체는 본 문서 범위 밖 → [[08_AI-온톨로지연동]]에 위임(DRY). 본 문서는 **"그 사이를 잇는 메시지 계약"만** 소유.
- 차별점: ACC에는 없는 [[TypeDB]] 온톨로지 "설비 엔티티 ID" 기반 하이라이트. 픽셀 좌표가 아닌 엔티티 ID로도 강조 대상 지정 가능(우리의 무기).
- ※ 채워야 할 것: Phase 1(픽셀/핸들 기반)과 Phase 2(엔티티 ID 기반)의 적용 시점 구분 메모.

## 3. 상세 목차

## 이벤트 버스 구조
- 버스의 형태·전달 매체·이벤트 카탈로그를 정의하는 핵심 절.

### 버스 토폴로지
- 메모: 챗봇 패널 ↔ 뷰어가 같은 SPA 내부인지(인-앱 pub/sub) vs 별도 컨텍스트인지([[WebSocket]] 경유)에 따른 분기 정리.
- ❓ TBD: 단일 페이지 내 in-memory event emitter로 충분한가, 아니면 멀티 탭/멀티 유저 동기화까지 고려해 서버 중계가 필요한가.

### 이벤트 타입 카탈로그
- 메모: 표준 이벤트명 + 페이로드 스키마 표. 후보 — `chat:highlightRequest`, `chat:focusEntity`, `viewer:selectionChanged`, `viewer:entityClicked`, `viewer:highlightApplied`(ack).
- 메모: 모든 이벤트 페이로드 공통 필드 = `sheetId`, `source`(chat|viewer), `correlationId`.

#### 페이로드 스키마 (공통 규약)
- 메모: ① 좌표 기반(`{x,y,w,h}` 뷰어 좌표계) ② 핸들 기반(`{dwgHandle}`) ③ 엔티티 기반(`{entityId}`) 세 가지 식별 방식 중 택1 또는 우선순위 명시.
- ❓ TBD: 좌표계 단위 — 뷰어 내부 픽셀 vs 도면 월드좌표(파싱 기하 JSON 기준). 한쪽으로 표준화 필요.

### 구독/발행 생명주기
- 메모: 뷰어 마운트 시 구독 등록, 시트 전환 시 구독 해제·재등록, 챗봇 세션 종료 시 정리(메모리 누수 방지).
- ❓ TBD: 뷰어가 아직 로드 전인데 챗봇 이벤트가 먼저 도착한 경우 버퍼링/재생 정책.

## Handle/엔티티 ID 전달
- 챗봇이 "무엇을" 강조하라고 가리킬 때의 식별자 체계와 해석 책임을 정의.

### 식별자 3계층
- 메모: ① [[DWG]] 엔티티 Handle(파싱 시 보존) ② 뷰어 렌더 객체 ID(기하 JSON 인덱스) ③ 온톨로지 엔티티 ID([[TypeDB]]) — 세 계층의 매핑 테이블이 핵심.
- 메모: Phase 0 파싱 산출물(분야·도면번호·태그)에 Handle↔태그 매핑이 이미 존재 → 재사용. 별도 파싱 금지(DRY).

#### Handle → 렌더객체 해석
- 메모: 백엔드 파싱 단계에서 `dwgHandle → renderId` 룩업 테이블을 기하 JSON에 동봉할지, 런타임 조회할지 결정.

#### 엔티티 ID → Handle 역참조
- 메모: 온톨로지가 "이 펌프(entityId)는 어느 도면 어느 Handle에 그려졌나"를 응답해야 챗봇→뷰어 하이라이트 가능. [[엔티티-ID-바인딩]]에 매핑 소유, 본 문서는 "전달 계약"만.
- ❓ TBD: 1:N 케이스 — 하나의 엔티티가 여러 시트/여러 Handle에 등장할 때 어느 시트를 열고 어디를 강조할지 선택 규칙.

### 식별자 미해석 시 폴백
- 메모: 엔티티 ID는 있으나 현재 시트에 미존재 → 토스트("다른 도면에 있음, 이동?") + 시트 전환 제안.

## viewer.select / fitToView 트리거
- 식별자가 해석된 뒤 뷰어가 실제로 수행하는 시각적 동작(API 계약)을 정의.

### viewer.select(targets)
- 메모: 선택 = 오버레이 레이어에 하이라이트 표시(원본 기하 불변). 다중 타겟 동시 선택 허용 여부, 기존 선택 누적 vs 교체.
- 메모: 하이라이트 스타일(색·외곽선·블링크)은 [[2D뷰어]] 렌더 사양에 위임, 본 문서는 "select 호출 시점·인자"만.

### viewer.fitToView(targets)
- 메모: 강조 대상이 화면 밖일 때 자동 줌·팬. 단일 타겟=센터링, 다중 타겟=바운딩박스 fit.
- ❓ TBD: 자동 이동 정책 — 항상 fit vs 사용자 옵션(과한 화면 점프 방지). ACC 기본 동작 벤치마크 필요.

### 역방향: viewer → chat 컨텍스트 주입
- 메모: 사용자가 뷰어에서 핀/객체 클릭 → `viewer:entityClicked` 발행 → 챗봇 입력창에 컨텍스트 칩("선택: 도면 A-101 / 펌프 P-01") 자동 첨부.
- 메모: 이 칩이 다음 AI 질의의 스코프가 됨(온톨로지 질의 범위 한정). 질의 처리 자체는 [[08_AI-온톨로지연동]] 위임.

### 마크업/이슈 핀과의 관계
- 메모: 동일 select/fitToView 트리거를 이슈 핀·마크업에도 재사용(픽셀 핀 + 엔티티 핀 공존). 중복 구현 금지 → 공통 뷰어 API 재사용.
- 메모: 핀/마크업을 엔티티 ID에도 묶는 "우리의 무기"는 [[엔티티-ID-바인딩]] 소유, 본 문서는 그 ID를 select에 흘리는 경로만.
