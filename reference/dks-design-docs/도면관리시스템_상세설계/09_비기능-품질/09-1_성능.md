## 목적
청주 사업장 [[도면관리 시스템]]의 **뷰어·목록·API 성능 목표와 달성 전략**을 정의.
- 소유 범위 = 렌더 응답성, 목록 스크롤, API 지연/처리량, 캐시/[[CDN]] 전략 **한정**.
- 데이터 모델·스키마는 [[05_데이터-모델]], 보안은 [[09-2_보안]]로 위임(DRY).

## 배경 / 전제
- 흐름 = [[DWG]] → 백엔드 파싱 → 기하 [[JSON]] → 웹 2D 뷰어 렌더 → 마크업/이슈 오버레이. **CAD 편집기 아님**.
- 성능 압박 지점 = (1) 대형 기하 JSON의 초기 로드/렌더, (2) 수백~수천 행 시트/이슈 목록 스크롤, (3) 다중 사용자 오버레이 동시성.
- 1차 범위 = 2D 시트 한정. 3D/Bridge 범위 밖 → 성능 목표에서도 제외.
- 차별점(향후): 핀/마크업을 픽셀이 아닌 "설비 엔티티 ID"에 바인딩 → 엔티티 인덱스 조회 성능이 [[TypeDB]] 연동 Phase 2 핵심 변수(여기선 인터페이스 지점만 명시, 상세는 위임).

## 뷰어 최적화(타일/가상화)
> 채울 것: 기하 JSON을 화면에 그릴 때의 렌더 파이프라인·레벨오브디테일 전략.

### 렌더 백엔드 선택
- 후보: Canvas 2D vs WebGL vs [[PDF.js]](PDF 폴백 경로) — 선택 기준·트레이드오프 표 채울 것 ❓.
- 결정 대기: 기하 JSON 직접 렌더 vs 서버 사전 래스터화(타일) **TBD**.

### 타일링 / 뷰포트 컬링
#### 공간 분할
- 뷰포트 밖 요소 컬링(quadtree/grid 인덱스) — 채울 것.
#### LOD(레벨오브디테일)
- 줌 레벨별 요소 단순화/숨김 규칙 — 채울 것.
#### 사전 타일 캐시
- 줌 단계별 타일 사전 생성 여부(서버 래스터화 채택 시) — 캐시 절과 연계.

### 오버레이 레이어 분리
- 베이스 도면(정적) vs 마크업/이슈 핀(동적) 레이어 분리 → 재렌더 최소화. 채울 것.
- 핀/마크업 좌표계: 픽셀 좌표 + 향후 엔티티 ID 앵커(이중 바인딩) → 렌더 시 변환 비용 메모.

### 점진 로딩
- 기하 JSON 스트리밍/청크 로드, 저해상도 프리뷰 우선 표시 — 채울 것.

## 캐시 / CDN
> 채울 것: 정적 자산·기하 JSON·타일·썸네일의 캐시 계층과 무효화 규칙.

### 캐시 계층
#### 브라우저 캐시
- 기하 JSON / 타일 / 썸네일의 캐시 헤더·ETag·버전 키 — 채울 것.
#### 서버/엣지 캐시
- [[CDN]] 적용 대상(불변 산출물=파싱 결과·타일·사진 썸네일) vs 비대상(동적 오버레이) — 채울 것.
- 청주 사업장 배포 형태(온프레미스/폐쇄망 여부)에 따라 CDN 가용성 **TBD** ❓.

### 캐시 키 / 무효화
- 도면 버전(version) 기반 불변 URL 전략 → 시트비교(버전 오버레이 diff)와 정합. 채울 것.
- 마크업/이슈 변경 시 무효화 범위(오버레이만, 베이스 도면 불변).

### 실시간 동기화 vs 캐시
- 다중 사용자 오버레이 동기화: [[WebSocket]] 푸시 vs 폴링 — 캐시된 베이스 위에 델타만 전송. 채울 것.
- 상세 동시성/충돌 처리는 [[06_협업-실시간]]로 위임, 여기선 캐시 일관성 경계만.
