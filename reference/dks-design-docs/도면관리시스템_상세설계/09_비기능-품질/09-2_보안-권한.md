# 보안 & 권한

> 소유 범위: **RBAC · 도면 원본 보호 · 감사(Audit) 상세**.
> 인증(SSO/세션 발급), 네트워크/인프라 보안, 데이터 암호화 저장은 본 문서 범위 밖 → 각 소유 문서로 위임.

## 1. 목적
- 청주 사업장 [[도면관리시스템]]의 접근통제·원본보호·추적성 설계 기준 정의
- ACC Build의 멤버/역할 권한 모델을 동등 재현하되, 우리 차별점인 **엔티티 단위 권한**(향후) 여지 확보
- (의도 메모) "누가 무엇을 볼·할 수 있는가" + "원본 [[DWG]]는 어떻게 불변 보장되는가" + "사고 시 무엇을 추적하는가" 세 축을 한 문서에서 확정

## 2. 배경 / 전제
- 핵심 스코프 제약: **DWG 원본 비편집**. 흐름 = DWG → 백엔드 파싱 → 기하 JSON → 2D 뷰어 렌더 → 마크업/이슈/이벤트 오버레이
- 따라서 권한 모델의 1차 보호 대상 = ① 원본 파일 ② 파생 기하 JSON ③ 오버레이(마크업·이슈) ④ 메타/감사로그 — 네 자산군의 보호 수준이 다름
- 오버레이는 픽셀 좌표 + (향후) **설비 엔티티 ID** 양쪽에 바인딩 → 엔티티 단위 권한은 Phase 2 ([[TypeDB]] 온톨로지 연동) 이후 확장 항목으로 표기
- 인증 주체·세션 토큰 형식은 [[SSO 인증]] 문서가 소유. 본 문서는 "인증된 주체"를 입력으로 가정

## 3. 역할 / 권한 매트릭스
> (의도) ACC Build 역할 체계를 기준선으로 우리 역할 세트를 확정하고, 자산군 × 액션 매트릭스를 채운다.

### 3.1 역할 정의
- (채울 것) 후보 역할: 프로젝트관리자 / 설계자 / 검토자(이슈 처리) / 협력사(읽기+제한마크업) / 뷰어(읽기전용)
- 각 역할의 책임·대표 사용자 1줄 정의
- ❓ ACC의 "프로젝트 멤버 vs 회사 멤버" 2계층을 청주 단일 사업장에서도 유지할지 → TBD

### 3.2 권한 모델 방식
- (채울 것) RBAC 기본. 역할 → 권한 집합 매핑 테이블 형태
- (채울 것) 권한 입도: 모듈 레벨(시트/이슈/마크업/파일/사진) + 액션 레벨(view/create/edit/delete/export)
- ❓ 폴더/시트별 ABAC식 세분 권한 필요 여부 → 1차는 프로젝트 전역 RBAC, 폴더 단위는 TBD

### 3.3 자산군 × 액션 매트릭스
> (의도) 행=역할, 열=액션, 셀=허용/조건부/거부. 자산군별로 분리 표 작성.

#### 3.3.1 원본 DWG / 파생 기하 JSON
- (채울 것) 업로드·재파싱·다운로드 권한. 원본 다운로드는 관리자/설계자 한정 등
#### 3.3.2 시트(도면) 뷰
- (채울 것) view/version-switch/compare 권한
#### 3.3.3 마크업 오버레이
- (채울 것) 본인 마크업 vs 타인 마크업 edit/delete 구분 (소유권 기반)
#### 3.3.4 이슈
- (채울 것) 생성/상태변경/배정/closed 전이 권한, 핀 이동 권한
#### 3.3.5 파일 [[CDE]] / 사진
- (채울 것) 폴더 접근·업로드·삭제 권한 → 상세는 [[04_파일-CDE-설계]]에 위임, 본 표는 역할 매핑만

### 3.4 권한 평가 흐름
- (채울 것) 요청 → 미들웨어에서 (주체, 액션, 리소스) 평가 → 허용/거부
- (채울 것) 프런트엔드 UI 가드(버튼 숨김)는 보조일 뿐, **백엔드 강제(enforcement)가 진실원천** 명시
- ❓ 권한 캐싱 전략(세션당 권한 스냅샷 vs 매 요청 평가) → TBD

## 4. 도면 원본 보호
> (의도) "원본 불변" 제약을 기술적으로 보장하는 방법 명세. 본 시스템의 신뢰 기반.

### 4.1 불변성(Immutability) 보장
- (채울 것) 업로드된 DWG는 write-once. 수정 흐름 없음 → 신버전 = 새 객체로 추가
- (채울 것) 무결성 검증: 업로드 시 해시(SHA-256 등) 산출·저장, 다운로드/재파싱 시 대조
- ❓ 저장소가 객체 스토리지면 버킷 정책 레벨 WORM 적용 가능 여부 → 인프라 문서와 협의 TBD

### 4.2 파생물 격리
- (채울 것) 기하 JSON·썸네일은 파생물로 분류, 원본과 권한·생명주기 분리
- (채울 것) 파생물 재생성 = 원본 기준 멱등(idempotent), 손상 시 재파싱으로 복구

### 4.3 다운로드 / 반출 통제
- (채울 것) 원본 DWG 다운로드 권한 제한 + 다운로드 이벤트 감사 기록(§5 연동)
- ❓ 워터마크/반출 표식 요구 여부(청주 사업장 보안정책) → TBD

### 4.4 오버레이-원본 분리 원칙
- (채울 것) 마크업/이슈는 별도 레이어 저장 → 원본·기하 JSON 절대 미변경 재확인
- (의도) 이 절은 "왜 우리가 CAD 편집기가 아닌가"의 보안적 근거를 명문화

## 5. 감사 로그 (Audit Log)
> (의도) 무엇을·언제·누가·어떤 결과로 했는지 추적. ACC의 활동 로그 수준 재현 + 원본 반출 추적 강화.

### 5.1 감사 대상 이벤트
- (채울 것) 로그인/권한변경/원본 업로드·다운로드/버전 추가/이슈 상태전이/마크업 삭제/내보내기(export)
- (채울 것) 우선순위: 원본 반출·삭제·권한변경은 **필수 감사**, 단순 view는 선택(볼륨 고려)

### 5.2 로그 레코드 스키마
- (채울 것) 필드: who(주체) / when(타임스탬프) / action / resource(타입+ID) / result(성공·거부) / context(IP·세션)
- ❓ 변경 이벤트에 before/after 스냅샷 포함 범위 → 권한변경·이슈상태는 포함, 마크업은 TBD

### 5.3 저장 / 보존 / 불변성
- (채울 것) append-only 저장. 감사 로그 자체는 수정·삭제 불가 보장
- ❓ 보존 기간(법적/사내 정책) → TBD (청주 사업장 정책 확인 필요)

### 5.4 조회 / 내보내기
- (채울 것) 관리자용 감사 조회 화면 — 공통 데이터테이블(정렬·필터·export) 재사용
- (채울 것) 필터 축: 기간 / 주체 / 액션 / 리소스
- (의도) UI는 [[_ACC-Build-화면분석-재현설계]] 공통 데이터테이블 패턴 위임, 본 절은 필드·필터 정의만

### 5.5 실시간 이벤트 연동
- (채울 것) 이슈/마크업 변경의 실시간 통지는 [[WebSocket]] 채널과 공유 → 전송은 [[09-1_성능-확장성]] 위임, 감사 영속화만 본 문서 소유

## 6. 취약점 대응
> (의도) 뷰어+오버레이 아키텍처 특유의 위협을 식별하고 방어책 매핑. 일반 OWASP 전반은 위임, 본 시스템 고유 위협 집중.

### 6.1 위협 모델 (시스템 고유)
- (채울 것) ① 악성 DWG 업로드(파서 공격) ② 권한 우회 직접 API 호출 ③ 타인 오버레이 무단 조작 ④ 원본 무단 반출 ⑤ 파생 JSON 변조 후 오인 렌더
- (의도) 행=위협, 열=영향/가능성/방어, STRIDE 약식 표

### 6.2 입력/파싱 보안
- (채울 것) 업로드 파일 타입·크기 검증, 파서 샌드박싱/타임아웃, 파싱 실패 격리
- ❓ DWG 파싱 라이브러리 CVE 모니터링 주체·주기 → TBD

### 6.3 웹 클라이언트 보안
- (채울 것) [[PDF.js]]/2D 렌더 경로 XSS 차단, 마크업 텍스트·이슈 코멘트 입력 새니타이즈
- (채울 것) API 권한 강제 재확인(§3.4와 연결) — 프런트 가드 신뢰 금지

### 6.4 의존성 / 패치 관리
- (채울 것) 라이브러리 취약점 스캔·업데이트 운영 절차 (운영 문서와 연계)
- ❓ 자동 스캔 도구·게이트 도입 여부 → TBD

### 6.5 사고 대응
- (채울 것) 권한·원본 관련 보안 이벤트 탐지 시 대응 절차(감사로그 기반 추적 → 격리 → 통지)
