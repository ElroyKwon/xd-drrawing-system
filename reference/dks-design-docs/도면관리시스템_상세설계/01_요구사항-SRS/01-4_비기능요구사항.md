# 비기능 요구사항 (NFR)

## 1. 목적
- 청주 사업장 [[도면관리 시스템]]의 **품질 속성**(성능·보안·가용성·호환성·확장성)을 정량 기준으로 정의.
- 기능 요구사항([[01-3_기능요구사항]])이 "무엇을 하는가"라면, 본 문서는 "얼마나 잘 해야 하는가"를 측정 가능한 목표로 고정.
- 각 NFR은 검수(인수 테스트) 통과 기준으로 직결 → 모호한 "빠르게/안전하게" 금지, 수치·조건 명시.

## 2. 배경 / 전제
- ★스코프 제약: 본 시스템은 **[[CAD]] 편집기가 아니라 "뷰어 + 주석/이벤트 오버레이"**. 흐름 = [[DWG]] → 백엔드 파싱 → 기하 JSON → 웹 2D 뷰어 렌더 → 마크업/이슈/이벤트 오버레이.
- 정밀 CAD 편집(osnap, 좌표 편집, [[SHX]] 폰트 정밀 렌더)은 범위 밖 → 관련 성능 NFR도 **렌더/조회/오버레이**에 한정.
- 1차 범위 = 2D 시트 한정. 3D BIM 뷰어/Bridge 제외.
- 벤치마크 = [[ACC Build]] 체감 성능. 단, 동등 수준 목표이되 사내망 단일 사업장 규모 기준으로 현실화.
- 측정 환경 전제: TBD ❓ (사내망 대역폭, 동시 사용자 수 가정치, 표준 도면 1장 평균 엔티티 수 — [[01-1_시스템개요]]에서 확정 후 인용).

## 3. 성능 (뷰어 로딩 / 조작 응답)

### 3.1 도면 로딩 성능
- 초기 시트 오픈(파싱 완료 JSON 기준)부터 첫 렌더까지 목표 응답시간 표 작성.
- 콜드(최초 파싱) vs 웜(캐시된 기하 JSON) 구분 → 파싱은 비동기 1회·이후 재사용 전제.
- ❓ 기준 도면 크기 등급(소/중/대 = 엔티티 수 기준) 정의 후 등급별 목표 채울 것.

### 3.2 뷰어 조작 응답 (팬/줌/레이어 토글)
- 팬·줌·확대축소 시 프레임 응답 목표(체감 끊김 없는 수준) 기재.
- 레이어/분야 on-off 토글 반영 지연 목표.
- 대용량 시트 렌더링 전략(타일링/LOD/캔버스 vs SVG)은 [[02-1_아키텍처]]에 위임 — 여기선 **결과 응답시간만** 규정.

### 3.3 오버레이(마크업·핀·이슈) 반응성
- 마크업 드로잉 중 입력→화면 반영 지연 목표.
- 이슈 핀 다수(N개) 표시 시 뷰어 응답 저하 한계선.
- 실시간 협업 이벤트([[WebSocket]]) 수신→화면 반영 지연 목표.

### 3.4 데이터 조회/목록 성능
- 시트 목록·이슈 목록 데이터테이블(정렬·필터·페이지네이션) 응답 목표.
- 내보내기(CSV/PDF) 생성 허용 시간.

### 3.5 동시성 목표
- 목표 동시 접속 사용자 수에서 위 응답 목표 유지 조건.
- ❓ 청주 사업장 동시 사용자 가정치 확정 필요.

## 4. 보안 / 권한

### 4.1 인증 (Authentication)
- 로그인 방식(사내 계정 연동 여부, SSO/LDAP) — ❓ 사내 인증 인프라 확인 후 결정.
- 세션 만료·재인증 정책.

### 4.2 인가 (Authorization) / 권한 모델
- 역할 정의(예: 뷰어/마크업작성자/이슈관리자/관리자) 및 모듈·액션별 권한 매트릭스.
- 프로젝트(사업장) 단위 격리 — 다른 프로젝트 도면 접근 차단.
- [[CDE]] 폴더/파일 단위 접근 권한(읽기/쓰기/공유) 정의.

### 4.3 도면 데이터 보호
- 원본 [[DWG]] 무결성: 시스템이 **원본을 수정하지 않음**을 보증(읽기 전용 보관·해시 검증).
- 파싱 산출물(기하 JSON)·마크업·이슈 데이터 저장 시 보호 수준.
- 전송 구간 암호화(사내망 HTTPS 적용 여부) — ❓ 사내 정책 확인.

### 4.4 감사 / 추적성
- 마크업·이슈·이벤트의 작성자·시각 기록(감사 로그) 보존 정책.
- 다운로드/내보내기 이력 추적 여부.

### 4.5 데이터 보존/파기
- 보존 기간·파기 정책 — ❓ 고객사 요구 확인 후.

## 5. 가용성 / 백업

### 5.1 가용성 목표
- 운영 시간대(업무시간 한정 vs 24/7) 및 목표 가동률(%) — ❓ 사업장 운영 형태 확인.
- 계획 정비(점검) 윈도우 정의.

### 5.2 장애 복구
- 목표 복구시간(RTO)·복구시점(RPO) 정의.
- 단일 장애점(SPOF) 식별 및 대응 — 이중화 범위는 [[02-1_아키텍처]]에 위임.

### 5.3 백업
- 백업 대상: 원본 [[DWG]] 보관소·파싱 JSON·마크업/이슈 DB·첨부(사진/파일).
- 백업 주기·보관 세대·복구 검증 절차.

### 5.4 데이터 정합성
- 파싱 JSON과 원본 도면 버전 동기화 보증(원본 갱신 시 재파싱 트리거 정책).
- 오버레이 데이터가 참조하는 도면 버전 추적([[시트 비교]] 버전 관리와 연계).

---

## 6. 호환성 (DWG / DXF / PDF 범위)

### 6.1 입력 포맷 지원 범위
- [[DWG]] 지원 버전 범위(AutoCAD 릴리스별) — ❓ 청주 도면 실제 버전 조사 후 확정.
- [[DXF]] 지원 여부 및 버전.
- [[PDF]] 시트 지원(벡터 PDF vs 래스터 PDF) 및 렌더 경로([[PDF.js]] 활용 여부).
- 미지원 포맷 처리 정책(거부/사전 변환 안내).

### 6.2 파싱 정확도 한계 명시
- 지원 엔티티 유형(라인/폴리라인/호/텍스트/블록 등)과 **미지원/근사 처리** 유형 명시.
- [[SHX]] 폰트·복잡 해치·외부 참조(XREF) 처리 한계 — 범위 밖 항목 명문화(오해 방지).

### 6.3 브라우저 호환성
- 지원 브라우저·버전(사내 표준 브라우저 기준) — ❓ 사내 표준 확인.
- 최소 화면 해상도·태블릿 지원 여부.

### 6.4 산출물 호환
- 내보내기 포맷(주석 포함 PDF, CSV) 호환 기준.

## 7. 확장성

### 7.1 데이터 규모 확장
- 프로젝트/시트/이슈/마크업 증가에 따른 성능 유지 한계(목표 상한 규모).
- 대용량 단일 시트(고밀도 엔티티) 대응 한계.

### 7.2 기능 확장 (Phase 2 대비)
- 향후 [[TypeDB]] 온톨로지 연동 시 영향 최소화 구조 요구.
- ★차별점 전제: 핀/마크업/이슈를 픽셀 좌표뿐 아니라 **설비 엔티티 ID**에 바인딩 가능하도록 데이터 모델 확장성 확보 → 상세 모델은 [[02-2_데이터모델]]에 위임.
- AI 채팅/지식질의(Phase 2) 추가 시 뷰어/이슈 모듈 재작성 없이 연동 가능한 인터페이스 여지.

### 7.3 운영 확장
- 사용자·동시성 증가 시 수평 확장 가능 여부(아키텍처 위임).
- 다중 사업장 확장 시 프로젝트 격리 유지.
