# 요구사항 정의서 (SRS) 개요

## 1. 목적
- 청주 사업장 납품용 [[도면관리 시스템]] 요구사항의 **진입점 + 읽는 순서** 정의
- 이 문서는 SRS "표지/인덱스"만 소유 — 실제 요구사항 본문은 하위 문서가 소유
- 독자: 설계팀·개발팀·고객 검수자가 어느 문서부터 어떤 순서로 읽을지 1초 판단

## 2. 배경/전제
- 1차 목표 = [[ACC]] Build 제품 동등 재현 (벤치마크: [[_ACC-Build-화면분석-재현설계]] 37화면)
- ★핵심 스코프 제약 (모든 하위 요구사항이 전제):
  - [[DWG]] 원본 **편집 금지** — DWG → 백엔드 파싱 → 기하 JSON → 웹 2D 뷰어 → 마크업/이슈/이벤트 **오버레이**
  - 정밀 CAD 편집(osnap·좌표편집·SHX 정밀렌더)은 **범위 밖**
  - 1차 = **2D 시트 한정**, 3D BIM 뷰어/Bridge 범위 밖
- 차별점([[DKS]]): ACC엔 없는 "설비 엔티티 ID 바인딩" — 핀/마크업/이슈를 픽셀이 아닌 [[온톨로지]] 엔티티에도 결속 (Phase 2 [[TypeDB]] 연동, 상세는 위임)
- 이 README는 그 본질 검토를 **요구사항 분류·ID·문서 인덱스**로만 다룸

## 5. 요구사항 ID 체계
> 무엇을 채울지: 추적성(요구사항↔설계↔테스트) 확보용 ID 규칙. 충돌 없는 네이밍 + 모듈 코드 정의.

### 5.1 ID 포맷
- 형식: `{분류}-{모듈}-{일련번호}` (예: `FR-SHEET-001`, `NFR-PERF-003`)
- 분류 코드: FR / NFR / UI / DATA / CONSTRAINT
- 모듈 코드: HOME, SHEET, MARKUP, ISSUE, COMPARE, FILE, PHOTO, COMMON, ONTO(온톨로지)
  - ❓ ONTO 모듈을 별도 둘지, 각 모듈 하위 Phase2 플래그로 둘지 결정 대기

### 5.2 추적성 매트릭스 규칙
> 요구사항 ID → 설계 문서 절 → 테스트 케이스 연결 방식. 어디서 관리할지(별도 매트릭스 파일 vs 각 문서 표) 결정.

### 5.3 우선순위·검수 기준 표기
> MoSCoW(Must/Should/Could/Won't) 또는 P0~P2 중 택1. 고객 검수 합격 기준과 연동.
