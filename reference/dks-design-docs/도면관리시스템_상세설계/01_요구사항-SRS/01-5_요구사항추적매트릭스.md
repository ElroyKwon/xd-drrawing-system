# 요구사항 추적 매트릭스 (RTM)

## ① 목적

- 요구 ID(REQ) ↔ 설계 산출물 ↔ 테스트 케이스(TC)의 **양방향 추적성** 단일 관리.
- 누락(설계·테스트 미연결)·고아(요구 없는 설계) 탐지로 [[ACC]] Build 동등재현 범위의 **빠짐없는 구현** 보장.
- 릴리스 전 **커버리지 게이트** 통과 근거 데이터 제공.

> 이 문서는 "추적 관계"만 소유. 요구 정의 본문은 [[01-1_요구사항명세서-SRS]], 검증 방법은 [[01-4_수용기준]]·[[09_테스트계획]]에 위임(DRY).

## ② 배경 / 전제

- **핵심 스코프 제약**: [[DWG]] 원본 비편집. 흐름 = DWG → 백엔드 파싱 → 기하 JSON → 웹 2D 뷰어 렌더 → 마크업/이슈/이벤트 **오버레이**. CAD 편집기 아님(뷰어 + 주석/이벤트 오버레이).
- 정밀 CAD 편집(osnap, 좌표편집, [[SHX]] 정밀렌더)·3D BIM 뷰어/Bridge = **범위 밖** → RTM에 `OUT` 표기로 명시 추적(범위 이탈 방지).
- 1차 = 2D 시트 한정. Phase 2 = [[TypeDB]] 온톨로지 연동 AI 채팅/엔티티 검색 → RTM에 `P2` 마킹으로 분리.
- 모듈 출처: 홈/시트/마크업/이슈/시트비교/파일([[CDE]])/사진 = [[_ACC-Build-화면분석-재현설계]] 37화면 분석 기준.

## RTM 표 양식

> RTM 1행 = 1개 요구. 모든 REQ는 최소 1개 설계·1개 TC 연결 의무.

### 표준 컬럼 정의
<!-- 채울 것: 아래 컬럼 표를 단일 진실 테이블로 확정. 도구(엑셀/Dataview/CSV) 선정은 TBD. -->

| 컬럼 | 설명 | 예시 |
|---|---|---|
| REQ-ID | 요구 고유 ID (모듈 접두) | `REQ-MKP-012` |
| 요구 요약 | 한 줄 개조식 | 마크업 화살표 도구 |
| 모듈 | 홈/시트/마크업/이슈/비교/파일/사진/공통 | 마크업 |
| 우선순위 | MoSCoW (M/S/C/W) | M |
| 스코프 | IN / OUT / P2 | IN |
| 설계 산출물 | 설계 문서·절 링크 | [[04_뷰어아키텍처#마크업레이어]] |
| TC-ID | 테스트 케이스 ID | `TC-MKP-012-01` |
| 상태 | TODO/구현중/완료/검증완료 | 검증완료 |
| 비고 | 엔티티 바인딩 여부 등 | 엔티티ID 묶음 대상 |

#### REQ-ID 명명 규칙
<!-- 채울 것: 모듈 접두사 코드표. -->
- 형식 `REQ-<MOD>-<3자리>`. MOD = HOM/SHT/MKP/ISS/CMP/FIL/PHO/COM(공통UI).
- 공통 UI(앱셸·데이터테이블·인스펙터·위젯·모달·빈상태·토스트) = `COM` 접두.

#### TC-ID 명명 규칙
<!-- 채울 것: TC ID와 REQ ID 연동 규칙. -->
- 형식 `TC-<MOD>-<REQ3자리>-<2자리시퀀스>`. 한 REQ에 다중 TC 허용.

### 엔티티 바인딩 추적 컬럼 (차별점)
<!-- 채울 것: ACC엔 없는 "설비 엔티티 ID 바인딩" 요구를 별도 추적하는 보조 컬럼. -->
- 핀/마크업/이슈를 픽셀 좌표뿐 아니라 **설비 엔티티 ID**에도 묶는 요구는 `비고`에 `[ENT]` 태그.
- Phase 0 도면 파싱 산출물(분야·도면번호·태그)과의 연결은 [[TypeDB]] 온톨로지 트랙(`P2`)으로 분기 추적.

### 샘플 행 (양식 예시, 실데이터 아님)
<!-- 채울 것: 모듈별 대표 1행씩 시드 후 팀이 확장. -->

| REQ-ID | 요구 요약 | 모듈 | 우선 | 스코프 | 설계 | TC-ID | 상태 | 비고 |
|---|---|---|---|---|---|---|---|---|
| REQ-SHT-001 | 2D 시트 [[PDF.js]] 렌더 | 시트 | M | IN | TBD | TBD | TODO | |
| REQ-MKP-001 | 펜·도형·화살표 오버레이 | 마크업 | M | IN | TBD | TBD | TODO | [ENT] |
| REQ-ISS-001 | 핀 + 상세폼 + 목록·필터 | 이슈 | M | IN | TBD | TBD | TODO | [ENT] |
| REQ-CMP-001 | 버전 오버레이 diff | 비교 | S | IN | TBD | TBD | TODO | |

## 추적 규칙

### 양방향 추적 원칙
<!-- 채울 것: forward(요구→설계→TC) / backward(TC→요구) 양방향 무결성 정의. -->
- **Forward**: 모든 `IN` REQ → ≥1 설계 → ≥1 TC.
- **Backward**: 모든 설계 산출물·TC → 출처 REQ 역추적 가능. 출처 없는 산출물 = 고아 → 검토.

### 변경 전파 규칙
<!-- 채울 것: REQ 변경 시 연결 설계·TC 동반 갱신 절차. -->
- REQ 수정·삭제 시 연결된 설계·TC `상태` 강제 재검토(`완료` → `구현중` 강등).
- 변경 이력은 [[03_결정]]/회귀로그 또는 git diff로 추적(채널 확정 TBD).

### 스코프 이탈 추적 (OUT/P2)
<!-- 채울 것: 범위 밖·Phase2 항목을 삭제 않고 명시 보존하는 이유와 표기. -->
- 폐기된 과한 CAD-editor 가정(osnap·좌표편집·SHX 정밀)은 삭제 대신 `OUT` 행으로 보존 → "왜 안 하는지" 추적.
- AI 채팅·엔티티 검색·영향도 = `P2` 행 → Phase 2 착수 시 활성화.

### 갱신 책임·주기
<!-- 채울 것: RTM 오너, 갱신 트리거(스프린트 종료/PR머지 등). -->
- 오너: TBD(설계 리드 1인 권장). 갱신 트리거: PR 머지 시 해당 REQ 상태 동기화.

## 커버리지 게이트

### 게이트 지표 정의
<!-- 채울 것: 릴리스 차단 기준이 되는 정량 지표 목록. -->
- **설계 커버리지** = (설계 연결된 IN REQ) / (전체 IN REQ). 목표 100%.
- **테스트 커버리지** = (TC 연결된 IN REQ) / (전체 IN REQ). 목표 100%.
- **고아율** = (출처 REQ 없는 설계·TC) / 전체. 목표 0.

### 마일스톤별 게이트
<!-- 채울 것: 설계완료/구현완료/릴리스 단계별 통과 임계치. -->

| 마일스톤 | 설계 커버리지 | 테스트 커버리지 | 비고 |
|---|---|---|---|
| 설계 완료 | 100% (IN) | — | 모든 IN REQ 설계 연결 |
| 구현 완료 | 100% | ≥ 목표(TBD) | M 우선순위 우선 |
| 릴리스 | 100% | 100% (M/S) | C/W는 예외 허용 ❓ |

### 게이트 미통과 처리
<!-- 채울 것: 임계치 미달 시 릴리스 차단/예외승인 절차. -->
- 미통과 항목 = 미연결 REQ 리스트 자동 산출 → 차단. 예외는 명시 승인 기록.

### 자동 검증 방법
<!-- 채울 것: RTM 무결성 자동 점검 수단(Dataview 쿼리/스크립트). 도구 TBD. -->
- Obsidian Dataview 또는 CSV+스크립트로 누락·고아 자동 추출. 채택 도구 ❓.
