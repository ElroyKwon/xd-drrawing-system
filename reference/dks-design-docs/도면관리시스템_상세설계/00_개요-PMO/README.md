# 도면관리 시스템 상세설계 — 마스터 문서 맵 (START HERE)

> **이 폴더 전체의 진입점.** 청주 사업장 납품용 **도면관리 시스템**(Autodesk Construction Cloud [[Build]] 동등 재현)의 대기업급 설계 문서 트리. 13개 섹션 · 92개 문서.
> 벤치마크 원본: `../screenshot/_ACC-Build-화면분석-재현설계.md` (ACC Build 37화면 분석)

---

## 0. 한 장 요약 — 무엇을 만드는가

- **목표**: 청주 현장에 ACC Build처럼 동작하는 **도면관리 시스템** 납품. (AI 채팅/지식질의는 **Phase 2**에서 [[TypeDB]] 온톨로지로 추가)
- **★핵심 전제 (전 문서 공통)**:
  - **DWG 원본을 편집하지 않는다.** 흐름 = [[DWG]] → 백엔드 파싱 → 기하 JSON → 웹 2D 뷰어 렌더 → 그 위에 **마크업·이슈·이벤트 오버레이**
  - CAD 편집기 아님 = **"뷰어 + 주석/이벤트 오버레이"**. 정밀 CAD 편집(osnap·좌표편집·SHX 정밀렌더) 범위 밖
  - 1차 범위 = **2D 시트 한정**. 3D BIM 뷰어 / Bridge 제외 (평가판에도 없음)
- **차별 무기**: 핀·마크업·이슈를 픽셀이 아닌 **설비 엔티티 ID**에도 바인딩 → ACC엔 없는 [[온톨로지]] 검색·영향도·AI채팅 기반 ([[데이터 지식 스튜디오]] Phase 0 파싱 산출물 활용)

## 4. 작업 방식 (이 문서 트리의 운용 규칙)

1. **현 상태 = 스캐폴드**. 각 문서는 frontmatter + 목적 + 상세 목차(H2~H4) + "무엇을 채울지" 의도 메모 + ❓TBD + 관련링크 골격. **팀이 절(節) 단위로 채워나간다.**
2. **소유 주제 원칙(DRY)**: 한 문서는 자기 주제만. 중복은 [[링크]] 위임.
3. **❓/TBD = 미결정 의사결정 지점**. 채울 때 결정 근거를 함께 기록 (의사결정 로그화).
4. **내부 위키링크 정합**: 일부 문서의 `related` 링크는 작성 시 추정된 잠정 이름 → 본 마스터 맵의 실제 경로 기준으로 채울 때 교정.
5. **결정 누적**: 큰 기술결정은 [[12-1_기술스택ADR]]에 ADR로 박제.
