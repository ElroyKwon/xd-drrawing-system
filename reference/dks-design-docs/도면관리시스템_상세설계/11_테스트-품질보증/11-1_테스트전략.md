## 목적

청주 도면관리 시스템의 테스트 전략 단일 진실 위치(SoT). 무엇을, 어느 레벨에서, 어떻게 검증할지 정의.
- 핵심 전제: 이 시스템은 **CAD 편집기가 아니라 "뷰어 + 주석/이벤트 오버레이"** → 정밀 CAD 검증(osnap, 좌표 정밀도, [[SHX]] 폰트 픽셀 일치)은 **테스트 범위 밖**.
- 검증 대상의 진실원천 = [[DWG]] 파싱 산출 **기하 JSON**과 그 위 오버레이(마크업/이슈/이벤트)의 정합성.

## 배경 / 전제

- 흐름: [[DWG]] → 백엔드 파싱 → 기하 JSON → 웹 2D 뷰어 렌더 → 마크업/이슈/이벤트 오버레이. 각 화살표가 테스트 경계.
- 1차 범위 = **2D 시트 한정**. 3D BIM/Bridge 검증 제외.
- 차별점 검증: 핀/마크업/이슈가 픽셀 좌표뿐 아니라 **"설비 엔티티 ID"에 바인딩**됨 → 엔티티 바인딩 정합성은 우리 고유 테스트 항목(Phase 1 후반 도입, [[TypeDB]] 연동 시 확장).
- 픽스처는 **실제 청주 도면 샘플 사용** (mock 금지, 테스트 코드 내부 한정 stub만 허용 — 프로젝트 규칙 §2.3).

> 채울 내용: 테스트 도구 스택(러너/어설션/E2E 프레임워크) 확정표. 현재 후보만 나열, 선정은 ❓.

## 테스트 피라미드

전체 레벨 구성과 비중 원칙. 위로 갈수록 적고 느리고 비싼 테스트.

### 레벨 정의

| 레벨 | 대상 | 도구(후보) | 비중(목표) |
|------|------|-----------|-----------|
| 단위 | 파서 함수, 좌표 변환, 오버레이 좌표 계산, 도메인 로직 | TBD (예: vitest/pytest) | 다수 |
| 통합 | 파싱 파이프라인, API↔DB, 엔티티 바인딩 | TBD | 중간 |
| E2E | 시트 열람→마크업→이슈 생성 사용자 플로우 | TBD (예: Playwright) | 소수 |

> 채울 내용: 각 레벨 정확한 비중 목표·실행 시간 예산. 구체 수치는 도구 선정 후 ❓.

### 레벨별 책임 경계

#### 단위가 소유하는 것
순수 함수: 좌표 변환(모델좌표↔뷰포트), 마크업 도형 직렬화, 이슈 상태 전이 규칙. 외부 의존 없음.

#### 통합이 소유하는 것
파싱 파이프라인 1건(DWG 입력→기하 JSON 출력) 정합성, API 라우트↔[[CDE]] 저장소 왕복, 엔티티 ID 바인딩 영속성.

#### E2E가 소유하는 것
브라우저 실제 렌더 포함 핵심 플로우만. 빈상태/토스트/모달폼 등 공통 UI는 대표 케이스 1~2개.

> 채울 내용: 레벨 간 중복 회피 규칙 — 같은 로직 중복 검증 금지(DRY). 경계 표 1개.

## TDD 적용 범위

어디에 테스트 우선(red→green→refactor)을 강제하고, 어디는 면제하는지.

### TDD 강제 영역 (테스트 먼저)
- 비즈니스/도메인 로직: 이슈 상태 머신, 마크업 권한, 시트 버전 diff 판정, 엔티티 영향도 계산.
- 파싱 산출물 변환 로직(좌표·레이어·태그 매핑).

### TDD 면제 / 사후 테스트 영역
- 2D 뷰어 픽셀 렌더 외형([[PDF.js]]/Canvas 렌더 결과) — 탐색적 개발(CLAUDE.md §4 Frontend-First), 플로우 확정 후 회귀 스냅샷만.
- 단순 UI 바인딩, 스타일.

> 채울 내용: "확정 후 TDD 전환 시점" 게이트 정의. [[11-2_품질-게이트]]로 위임할지 ❓.

### 시트 비교(diff) 테스트 규칙
버전 오버레이 diff는 도메인 로직(TDD 강제). 입력=두 기하 JSON, 출력=추가/삭제/변경 분류. 픽셀 비교 아님.

## 데이터 / 픽스처 (실샘플)

mock 금지 원칙 하에 실제 도면을 픽스처로 운용하는 방식.

### 픽스처 소스
- Phase 0 파싱 산출물(분야·도면번호·태그 포함) 재사용 — 이미 존재.
- 청주 사업장 실제 [[DWG]] 대표 샘플 N종(분야 다양성 확보: 전기/배관/건축 등).

> 채울 내용: 픽스처 도면 목록·선정 기준·저장 위치(레포 내/외부). 개수 N ❓.

### 픽스처 계층

#### 골든 파일(기하 JSON)
파싱 회귀 기준선. DWG→JSON 결과를 커밋된 골든과 비교. 파서 변경 시 의도적 갱신 절차 필요.

#### 오버레이 시드 데이터
이슈/마크업/엔티티 바인딩 샘플 — 통합·E2E 시나리오용 실데이터 시드.

> 채울 내용: 골든 파일 갱신 워크플로우(누가 승인, diff 리뷰 방식). 절차 ❓.

### 민감정보 / 라이선스
실제 도면 포함 → 레포 공개 범위·접근 제어 검토.

## 자동화

CI에서의 실행 방식과 게이트 연동.

### CI 파이프라인 단계
1. 린트/타입체크 → verify: 통과
2. 단위 → verify: 전부 green
3. 통합(DB·파서) → verify: 골든 일치
4. E2E(주요 플로우) → verify: 플로우 통과

> 채울 내용: CI 플랫폼(GitHub Actions 등) 확정·각 단계 실패 시 차단 정책. 플랫폼 ❓.

### 실행 트리거
- PR 시: 단위+통합 필수.
- 머지/야간: E2E 포함 전체.

> 채울 내용: 트리거별 잡 매트릭스. E2E 빈도(매 PR vs 야간) ❓.

### 커버리지 정책
커버리지 수치 게이트 여부·대상. 도메인 로직 우선, 뷰어 렌더 제외.

> TBD: 커버리지 임계값 강제할지/얼마로 할지 — 임의 수치 설정 금지, 팀 합의 후. ❓
