# 테스트/QA 개요

## ① 목적
- 청주 사업장 납품용 [[도면관리시스템]]의 품질을 보증하기 위한 **테스트 문서 인덱스** 제공.
- 이 문서는 "테스트 전략·계획·케이스의 진입점"만 소유. 실제 케이스/시나리오 본문은 하위 문서로 위임([[DRY]] 원칙).

## ② 배경/전제
- 시스템 성격 = **뷰어 + 주석/이벤트 오버레이** (CAD 편집기 아님). 따라서 테스트 초점도 "렌더 정확도 + 오버레이 좌표 정합성 + 이벤트 영속성"에 집중.
- 핵심 흐름 = [[DWG]] → 백엔드 파싱 → 기하 JSON → 웹 2D 뷰어([[PDF.js]] 계열/Canvas) 렌더 → 마크업/이슈/이벤트 오버레이. 각 단계가 별도 테스트 경계.
- 정밀 CAD 편집(osnap 스냅, 좌표 편집, [[SHX]] 폰트 정밀 렌더)은 **범위 밖** → 테스트 대상 아님.
- 1차 범위 = 2D 시트 한정. 3D BIM 뷰어/Bridge는 범위 밖.
- 벤치마크 = [[ACC Build]] 37화면. 동등 재현 여부를 수용 기준의 한 축으로 삼음.
- Phase 2(향후): [[TypeDB]] 온톨로지 연동 시 "엔티티 ID 바인딩" 검색·영향도·AI채팅 테스트가 추가됨 → 본 인덱스에 슬롯만 예약, 상세는 TBD.

### 읽는 순서
- 신규 합류자가 어떤 순서로 읽어야 전체 QA 그림을 잡는지 안내.
- *(채울 내용)* 전략 → 계획 → 레벨별 가이드 → 시나리오 순.

### 작성/네이밍 규약
- 테스트 케이스 ID 체계(예: `TC-<모듈>-<번호>`), 시나리오 ID, 결함 ID 규칙 정의.
- *(채울 내용)* 모듈 약어 표(SHEET/MARKUP/ISSUE/COMPARE/FILE/PHOTO/HOME) 확정 → ❓ 약어 표준 미정.

## 테스트 레벨
- 어떤 레벨로 무엇을 검증하는지 계층 정의. 레벨별 책임 경계를 명확히.

### 단위 테스트 (Unit)
- 대상 = 백엔드 파서, 기하 JSON 변환, 좌표 변환 함수, 오버레이 좌표 계산 로직.
- *(채울 내용)* 파서 입력=샘플 DWG, 기대=기하 JSON 스키마 일치. mock 금지·실제 샘플 도면 사용.
- ❓ 단위 테스트 프레임워크 미정(백엔드 언어 확정 후 결정) — TBD.

### 통합 테스트 (Integration)
- 대상 = DWG→파싱→JSON→뷰어 렌더 파이프라인, REST API 계약, [[WebSocket]] 이벤트 브로드캐스트.
- *(채울 내용)* 파일 업로드→파싱 완료 이벤트→뷰어 로드까지 한 흐름 검증. 표준 response format 준수 확인.

### E2E 테스트 (End-to-End)
- 대상 = 실제 브라우저에서 모듈별 사용자 플로우.
- *(채울 내용)* 모듈별 핵심 시나리오 1줄씩.

#### 모듈별 핵심 플로우
- 홈(위젯 대시보드 로드), 시트(목록→2D뷰어 진입), 마크업(펜/도형/화살표 그리기·저장), 이슈(핀 생성→상세폼→목록 필터), 시트비교(버전 오버레이 diff), 파일([[CDE]] 폴더 탐색), 사진.
- *(채울 내용)* 각 플로우의 성공 판정 조건.

#### 공통 UI 컴포넌트 검증
- 앱셸(모듈 사이드바·프로젝트 스위처), 데이터테이블(정렬·컬럼토글·필터·내보내기), 우측 인스펙터 패널, 위젯카드, 모달폼, 빈상태, 토스트.
- *(채울 내용)* 공통 컴포넌트별 상태(빈/로딩/에러/정상) 매트릭스.

#### 오버레이 정합성 테스트 (★우리 시스템 특수)
- 줌/팬/리사이즈 시 핀·마크업이 도면 위 동일 위치 유지(픽셀 좌표 ↔ 도면 좌표 변환 정확도).
- *(채울 내용)* 회전/스케일 변경 후 좌표 드리프트 허용 오차 정의 → ❓ 허용 오차 수치 미정.

### 비기능 테스트 (Non-functional)
- 대상 = 대용량 도면 렌더 성능, 동시 접속, 접근성.
- *(채울 내용)* 성능 임계값 후보(초기 렌더 시간 등)는 측정 후 확정 → TBD.

### (Phase 2) 온톨로지 바인딩 테스트
- 핀/마크업/이슈가 픽셀이 아닌 **설비 엔티티 ID**에도 묶이는지 검증. [[TypeDB]] 질의→영향도→AI채팅 응답 정확도.
- *(채울 내용)* Phase 2 착수 전까지 슬롯만 예약 — 상세 TBD.

## 커버리지 목표
- 무엇을 "충분히 테스트했다"로 볼지 합의 기준. 레벨별 측정 방식과 목표치.

### 측정 기준
- 단위=코드 커버리지(라인/브랜치), 통합=API 엔드포인트 커버리지, E2E=모듈/플로우 커버리지.
- *(채울 내용)* 각 기준의 측정 도구·산출 방법.

### 목표치
- *(채울 내용)* 레벨별 목표 % 표 — 과장 수치 금지, 팀 합의 후 기입.
- ❓ 구체적 목표 수치(예: 핵심 로직 라인 커버리지 X%) 전부 미정 → TBD.

### 우선순위 (리스크 기반)
- 파서/기하 변환·오버레이 좌표 로직을 최우선 커버리지 대상으로(결함 시 영향 최대).
- *(채울 내용)* 리스크 매트릭스(영향도×발생가능성)로 모듈 우선순위 표.

### 추적성
- 요구사항 ↔ 테스트 케이스 ↔ 결함 연결(추적 매트릭스) 유지 방식.
- *(채울 내용)* RTM 위치·관리 도구 → [[도면관리시스템_상세설계]] 요구사항 문서와 매핑.
