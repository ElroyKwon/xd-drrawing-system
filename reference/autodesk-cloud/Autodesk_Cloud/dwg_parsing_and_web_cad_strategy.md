# 독자적 웹 CAD 도면관리 및 뷰어 시스템 구축을 위한 기술 조사 및 전략 보고서

## 1. DWG 파일의 기술적 본질과 타사 디코딩 실태
[[DWG]] 포맷은 Autodesk의 독점적 설계 자산이며 비공개, 바이너리, 압축 포맷입니다. R13/R14 버전부터 독자적인 비트스트림 압축 방식(Huffman 및 Lempel-Ziv 계열 디코딩 알고리즘)과 암호화가 도입되었으며, 최신 R2018(AC1032) 포맷에 이르기까지 복잡성이 고도화되었습니다. 타사 메이커들이 이 포맷을 파싱하는 실태는 다음과 같습니다.

### ① Open Design Alliance (ODA) 컨소시엄
* ** Drawings SDK (구 Teigha)**: ODA는 Autodesk [[DWG]] 바이너리를 리버스 엔지니어링하여 C++로 완벽 디코딩/인코딩 기능을 구현한 사실상의 유일무이한 상용 표준입니다.
* **inWEB™ 플랫폼**: 최신 웹 표준 흐름에 따라 ODA는 Drawings SDK를 **WebAssembly (WASM)**로 빌드한 **Drawings inWEB™ SDK**와 WebGL 기반 시각화 엔진인 **Visualize inWEB™ SDK**를 제공합니다. 브라우저에서 서버 의존 없이 [[DWG]]를 파싱할 수 있으나, **ODA 상용 유료 멤버십 가입이 필수**이므로 개발자가 개별적으로 오픈소스로 배포하거나 독자적으로 무료 상용화하기에 높은 비용 장벽이 존재합니다.

### ② 오픈소스 파싱 라이브러리 분석
* **GNU LibreDWG**:
    * **동작 원리**: C 언어로 구현되어 [[DWG]] 바이너리 스트림을 바이트 단위로 디코딩하는 오픈소스 라이브러리입니다. C/C++ 바인딩을 통해 WebAssembly 포팅 프로젝트(`libredwg-web`)도 진행되었습니다.
    * **라이선스 제약 (GPLv3)**: 이 라이브러리를 직접 결합하여 소프트웨어나 서비스를 배포(SaaS 포함) 시 **전체 서비스 소스 코드를 GPLv3로 공개해야 하는 법적/비즈니스적 리스크**가 있습니다.
* **ACadSharp (C# .NET)**:
    * **동작 원리**: .NET 환경에서 구동되는 순수 C# 기반의 CAD 파일 파서입니다. 외부 C++ 네이티브 DLL 의존성 없이 [[DWG]] R14(AC1014)부터 2018+(AC1032) 파일의 헤더, 테이블, 블록 및 엔티티를 파싱하고 작성할 수 있습니다.
    * **비즈니스 장점**: 오픈소스(MIT/Apache 계열 등)로 관리되어 상용 서버 백엔드 파이프라인에서 라이선스 전파 걱정 없이 안정적으로 가동할 수 있는 핵심 솔루션입니다.
* **ezdxf (Python)**:
    * **동작 원리**: 파이썬 환경의 고성능 [[DXF]] 판독/제조 라이브러리입니다. 단, **[[DWG]] 포맷은 직접 읽을 수 없습니다**. ezdxf를 사용하기 위해서는 반드시 백엔드 서버에서 `LibreDWG` CLI 도구나 Autodesk `AcCoreConsole`을 프로세스로 호출하여 `DWG → DXF` 변환을 선행 완료해 주어야 합니다.

## 2. 독자적 백엔드 도면 변환 파이프라인 기술 명세
Autodesk [[클라우드]] API(APS)를 전적으로 대체하기 위해서는, 업로드된 [[DWG]]/[[DXF]] 파일을 분석하여 웹 렌더링에 적합한 데이터 구조로 무손실 정제하는 변환 파이프라인을 구축해야 합니다.

### ① 변환 시 데이터 누실(Fidelity Loss) 최소화 방안
* **폰트 매핑 (Font Mapping)**:
    * CAD 도면은 특수 기호와 설계 텍스트를 위해 AutoCAD 전용 SHX 폰트를 주로 사용합니다. 폰트 매핑 데이터가 부재하면 웹 렌더러 상에서 치수선 기호, 한글, 장비 태그가 물음표(`?`) 또는 깨진 사각형으로 표시됩니다.
    * **해결책**: 백엔드 서버에 표준 SHX 파일(`txt.shx`, `romans.shx`, `ghs.shx` 등) 라이브러리를 확보하고, 이를 웹 폰트인 [[WOFF]]/[[WOFF2]] 및 벡터 패스(Vector Path) 데이터로 자동 매핑해주는 폰트 매퍼(Font Mapper)를 변환기 내에 작성해야 합니다.
* **해치(Hatch) 및 선종류(Linetype)의 기하학적 전개**:
    * 도면 내 점선(Dashed), 일점쇄선 등의 패턴 정보는 CAD 테이블 내 정의 데이터로만 존재합니다.
    * **해결책**: 서버 파서 단에서 이러한 메타데이터 정의를 참조하여, 실제 브라우저 Canvas나 WebGL이 직접 그릴 수 있도록 패턴 길이를 분절된 독립 선분(Line Segments)들의 기하 배열로 전개(Explode)하여 [[JSON]] 파일로 출력해야 합니다.
* **블록 참조(INSERT/BLOCK) 구조의 보존**:
    * 하나의 도면에 100개 이상 배치된 밸브나 [[차단기]] 객체는 [[DWG]] 내에서 1개의 블록 정의(BLOCK)와 100개의 삽입점/회전값 참조(INSERT) 구조로 최적화되어 있습니다.
    * **해결책**: 파서 변환 시 이 참조 구조를 해제하지 않고 `BlockDefinitions`와 `Entities.Insert` 정보로 그대로 이원화하여 [[JSON]]에 유지해야 변환된 파일 크기를 수십 배 이상 줄일 수 있고, [[TypeDB]] [[온톨로지]]의 '일대다 관계' 계통망 정의와 손쉽게 1:1 매핑할 수 있습니다.

## 3. 웹 렌더링 기술 분석 및 거대 좌표계 정밀도 해결

### ① 웹 그래픽스 렌더링 기술 비교

| 기술 스택 | 렌더링 성능 (객체 수 기준) | 데이터 구조 제어 및 클릭 반응성 | 최적화 한계 및 비고 |
| :--- | :---: | :---: | :--- |
| **SVG (Scalable Vector Graphics)** | 낮음 (1만 개 미만) | 매우 쉬움 (DOM 이벤트 바인딩) | 객체 수가 많아지면 브라우저 DOM 엔진 병목으로 화면이 멈춤. |
| **HTML5 Canvas 2D** | 보통 (5만 개 미만) | 보통 (좌표 기반 레이캐스팅 필요) | CPU 단일 스레드로 비트맵을 직접 갱신하므로 픽셀 최적화가 필수적임. |
| **WebGL (Three.js 기반)** | 높음 (수십만 개 이상) | 어려움 (Raycasting 및 GPU 최적화 필요) | GPU 하드웨어 가속 및 Instanced Mesh를 사용하여 복잡한 설비도 프레임 드랍 없이 연산. |

### ② GPU WebGL 단정밀도 지터(Jittering) 현상 해결 수학적 모델
* **문제의 본질**: GPU 내부 셰이더(Vertex Shader)는 일반적으로 32비트 Float(단정밀도 부동소수점) 연산을 지원하며, 유효 자릿수는 7자리에 불과합니다. 거대 플랜트나 국가 좌표계(예: $X=350,000.123$, $Y=4,000,000.456$)를 지닌 CAD 데이터를 그대로 GPU 메모리에 올리고 줌인을 가할 경우, 부동소수점 하위 자리 유실로 인해 도면의 선이 격하게 찢어지거나 흔들리는 **Jittering** 현상이 일어납니다.
* **해결 기술: 상대 좌표 변환 (Relative to Center, RTC)**:
    1. 백엔드 파싱 시, 전체 도면 엔티티의 최소/최대 좌표를 비교하여 경계 상자(Bounding Box)의 중심점 좌표 $\mathbf{C} = (X_c, Y_c)$를 계산합니다.
    2. 모든 기하 정점 좌표 $\mathbf{P}_{world} = (X_w, Y_w)$를 중심점 기준의 상대 좌표 $\mathbf{P}_{local}$로 치환합니다.
       $$\mathbf{P}_{local} = \mathbf{P}_{world} - \mathbf{C}$$
    3. Three.js의 3D 공간 상에는 원점 $(0, 0)$ 근처에 로컬 좌표 $\mathbf{P}_{local}$ 데이터를 배치해 렌더링을 지터 없이 부드럽게 수행합니다.
    4. 원본 좌표값 $\mathbf{C}$는 메타데이터에 별도 보관하며, 사용자가 뷰어 상에서 치수 측정을 시도할 때만 임시로 이 중심점 오프셋 값을 역산하여 연산 결과를 화면에 표출합니다.

## 4. 오브젝트 스냅(Osnap) 및 공간 인덱싱 알고리즘

사용자가 치수를 재거나 정확한 특징점을 짚기 위한 Osnap(끝점, 교차점 스냅) 기능을 수만 개의 벡터 객체 상에서 성능 저하 없이 구현하기 위해서는 공간 인덱싱이 핵심입니다.

### ① KD-Tree (k-dimensional tree)를 이용한 점 탐색 ($O(\log N)$)
* **메커니즘**: 도면 내 선들의 양 끝점(Endpoints), 원의 중심점(Center) 등의 기하 정점 리스트를 2차원 공간 상에 분할하여 KD-Tree로 사전 적재합니다.
* **검색 흐름**:
    1. 사용자의 마우스가 웹 화면 상의 픽셀 좌표 $\mathbf{M}$을 지날 때, 뷰어의 현재 줌 배율을 계산하여 월드 좌표 상의 스냅 탐색 반경 $R$을 계산합니다.
    2. KD-Tree의 **최근접 이웃 탐색(Nearest Neighbor Search)** 알고리즘을 수행하여 반경 $R$ 내에 존재하는 가장 가까운 정점 $\mathbf{V}$를 $O(\log N)$ 성능으로 고속 색출합니다.
    3. 탐색된 $\mathbf{V}$의 좌표로 마우스 포인터를 강제 끌어당겨 안착(Snap)시키고 화면에 스냅 마커(초록색 네모 등)를 렌더링합니다.

### ② R-Tree (Rectangle Tree)를 이용한 객체 스냅
* **메커니즘**: 점이 아닌 직선(Line), 호(Arc) 등의 물리적 길이와 바운딩 박스를 갖는 도형 엔티티들을 계층적 바운딩 박스 트리로 구성하여 R-Tree에 보관합니다.
* **검색 흐름**: 사용자의 마우스 커서가 특정 선분 위에 닿았을 때(Osnap Nearest), 선분 영역과의 교차 여부 및 수선의 발 계산을 고속 처리하기 위해 사용합니다.

## 5. 자체 도면관리 시스템 구축 전략 비교 매트릭스

자체 도면관리 플랫폼을 기획 및 구현할 때 실현 가능한 대안들의 상세 기술 비교입니다.

| 구분 | [방안 1] 백엔드 변환(ACadSharp) + Three.js 커스텀 뷰어 | [방안 2] WASM 로컬 변환(LibreDWG Web) + Canvas 2D | [방안 3] ODA Drawings inWEB™ SDK 적용 |
| :--- | :--- | :--- | :--- |
| **백엔드 리소스** | 중간 (서버 파이프라인 프로세스 필요) | 매우 낮음 (서버 연산 불필요) | 중간 (ODA 서버 아키텍처 연동) |
| **초기 비용** | 0원 (오픈소스 자체 개발) | 0원 (오픈소스 자체 개발) | 매우 높음 (ODA 연간 회원비 필요) |
| **라이선스 리스크** | 없음 (MIT/Apache 라이선스) | 있음 (GPLv3 소스 공개 의무 저촉 여부 확인 필요) | 없음 (상용 계약 기반 보장) |
| **구현 난이도** | 높음 (렌더러 및 스냅 엔진 직접 구현) | 매우 높음 (WASM 메모리 관리 및 Canvas 최적화) | 보통 (제공되는 SDK의 자바스크립트 API 호출) |
| **추천 대상** | **DKS와 같이 데이터 주권을 확보하고, AI와 함께 독자 기술 자산을 내재화하려는 비즈니스 모델** | 내부 개발 및 배포용 보안 도면 뷰어 시스템 | 상용 CAD 수준의 편집기(Editor) 기능을 웹에서 직접 구현해야 하는 대형 엔터프라이즈 플랫폼 |

## 5-2. 타사 상용 바이너리 포맷 분석 및 DKS 실용주의 노선 타당성
대량의 도면을 웹 브라우저에서 지연 없이 부드럽게 로딩하기 위해 메이커사들이 사용하는 경량 바이너리 포맷의 기술적 사양과 DKS가 취해야 할 노선 비교입니다.

* **DWG FastView의 OCF 포맷**:
    * **동작**: 서버 단에서 [[DWG]]/[[DXF]] 파일을 독자적인 바이너리 압축 포맷인 **`OCF`** 파일로 일차 변환합니다.
    * **전송 및 렌더링**: 프론트엔드가 이 OCF 파일을 `arraybuffer` 형태로 수신한 후, 브라우저 메모리 상에서 HTML5/[[WebGL]]로 다이렉트 해독하여 렌더링합니다. 텍스트 파싱 부하가 전무하여 로딩 성능이 극도로 우수합니다.
* **ODA inWEB SDK의 .vs / .vsz 포맷**:
    * **동작**: 도면 정보를 전용 시각화 스트림 포맷인 **`.vs`** (Visualize Stream) 또는 압축된 **`.vsz`** 파일로 백엔드에서 미리 컴파일합니다.
    * **전송 및 렌더링**: 브라우저의 **WebAssembly (WASM)** 모듈이 이를 직접 수신해 C++ 데이터베이스(`OdDbDatabase`) 구조로 올리고 WebGL 버텍스 버퍼로 직접 주입합니다.
* **데이터 지식 스튜디오 (DKS)의 실용주의 노선**:
    * **바이너리 포맷 개발의 무용성**: DKS는 범용적인 대안 CAD 패키지나 상용 뷰어 제품 자체가 되는 것이 목표가 아닙니다. 따라서 OCF나 .vs 같은 독자적인 전용 바이너리 압축 포맷을 완전히 새로 연구/개발하는 것은 리소스 낭비입니다.
    * **하이브리드 결합에 집중**: DKS의 본질은 **"5대 기본 도면 관리 기능(시각화, 비교, 스냅, 측정, 마크업)"**을 안정적으로 보장하면서, **"채팅 질의를 통해 관련 설비의 도면 영역으로 빠르게 이동 및 하이라이트"**해주는 융합형 사용자 경험에 있습니다.
    * **역할 분담**: [[TypeDB]]를 활용한 온톨로지 지식 연동과 추론은 백엔드가 [[DXF]] 등에서 변환해 알아서 처리하므로 뷰어 단에서 이를 직접 고민할 필요가 없습니다. 뷰어는 가볍고 구현 난이도가 낮은 **[[JSON]] 및 DXF 변환 기반의 실용적 기하 포맷**을 유지하며, 프론트엔드 이벤트 버스를 통한 하이라이팅 연동에 주력합니다.

## 6. 결론 및 향후 기술 검토 방향
* **ACadSharp 기반 파서 파일 추출 검증**:
    * MIT 라이선스로 비즈니스에 영향이 없는 **ACadSharp**를 백엔드 코어 파서로 선정하고, [[DWG]] 업로드 시 `Entities.Insert`(블록)와 `Entities.Text` 정보가 누락 없이 [[JSON]]으로 추출되는지 구조 신뢰성 평가를 선행해야 합니다.
* **Three.js와 KD-Tree 공간 인덱싱 라이브러리 결합**:
    * 프론트엔드단에서 Three.js 씬을 생성하고, 추출된 JSON 좌표 리스트를 기반으로 `kdbush` 또는 `flatbush` 등 고성능 자바스크립트 공간 인덱싱 패키지를 탑재하여 $O(\log N)$ 자석 스냅(Osnap) 인터랙션 기능의 프레임 유지력을 계측해야 합니다.
