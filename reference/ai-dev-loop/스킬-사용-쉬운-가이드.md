# 스킬 사용 쉬운 가이드

## 결론

우리가 만든 것은 [[인공지능]] 개발을 자동으로 완성해주는 프로그램이 아니라, [[Codex CLI]]나 [[Claude Code]]가 프로젝트를 더 질서 있게 진행하도록 돕는 **작업 절차 스킬 묶음**이다.

핵심은 아래 흐름이다.

```text
프로젝트 확인 → 기능 문서 생성 → 계획 검수 → 구현 → 검증 → 증거 보고
```

이 흐름을 매번 말로 다시 설명하지 않도록 만든 것이 `project-bootstrap`, `feature-docs-scaffold`, `planning-gate`, `development-loop-orchestrator`, `validator-loop`, `evidence-report` 스킬이다.

---

## 1. 우리가 만든 스킬 위치

원본 위치:

```text
G:\내 드라이브\_Obsidian\지식관리\AI개발루프-스킬학습\05-팀공유-스킬패키지\skills\
```

현재 `xd-drawing-system` 프로젝트에 복사된 위치:

```text
D:\_Project\xd-drawing-system\.agents\skills\
D:\_Project\xd-drawing-system\.claude\skills\
```

즉, 새 터미널에서 `D:\_Project\xd-drawing-system`으로 들어가면 이 스킬들을 프로젝트 로컬 기준으로 사용할 수 있다.

---

## 2. 스킬 7개를 쉽게 말하면

| 스킬 | 쉬운 설명 | 언제 쓰나 |
|---|---|---|
| `project-bootstrap` | 프로젝트 시작 전 파일과 규칙이 준비됐는지 확인 | 새 프로젝트 폴더에 처음 들어갔을 때 |
| `feature-docs-scaffold` | 기능 구현 전 7개 설계 문서를 만들게 함 | 새 기능 또는 화면 slice 시작 |
| `planning-gate` | 만들기 전에 계획이 빠지거나 충돌하지 않는지 확인 | 구현 시작 전 |
| `development-loop-orchestrator` | 지금 어느 단계인지 판단하고 다음 스킬을 고름 | 세션이 끊겼거나 여러 터미널을 쓸 때 |
| `validator-loop` | 만들고 난 뒤 진짜 동작하는지 명령과 화면으로 검증 | 완료 보고 전 |
| `evidence-report` | 무엇을 했고 무엇을 검증했는지 짧게 보고 | 세션 종료, 인수인계, 최종 보고 |
| `tag-alarm-review` | 태그, 알람, 히스토리언 자료를 검토 | [[PLC]], [[HMI]], [[xD-HUB]] 자료 검토 시 |

이번 `xd-drawing-system`에서는 `project-bootstrap` → `feature-docs-scaffold` → `planning-gate` → 구현 → `validator-loop` → `evidence-report` 순서로 사용한다.

---

## 3. 새 터미널에서 시작하는 방법

먼저 터미널에서 프로젝트 폴더로 이동한다.

```powershell
cd "D:\_Project\xd-drawing-system"
codex
```

Codex가 열리면 아래 프롬프트를 그대로 붙여 넣는다.

```text
AGENTS.md를 먼저 읽고, README.md, SPEC.md, PLAN.md, CHECKS.md, HUMAN_GATE.md, docs/sessions/NEXT_SESSION.md, reference/README.md 순서로 확인해줘.

이 프로젝트는 xd-drawing-system이고, G:\내 드라이브\_Obsidian\지식관리\AI개발루프-스킬학습 방식대로 기능 하나씩 계획-구현-검증-증거기록 루프로 개발할 거야.

프로젝트 로컬 스킬은 .agents/skills 아래에 있어. 먼저 development-loop-orchestrator 기준으로 현재 단계와 다음에 실행해야 할 스킬을 판단해줘.

아직 코드는 만들지 말고, 첫 기능 후보인 "초기 설정 화면"을 시작하기 전에 reference/acc-screenshots 와 reference/acc-analysis/_ACC-Build-화면분석-재현설계.md 에서 관련 스크린샷과 분석 항목을 찾아서 정리해줘.
```

이 프롬프트의 목적은 바로 구현하게 만드는 것이 아니라, 먼저 프로젝트 구조와 참고자료를 제대로 읽게 만드는 것이다.

---

## 4. 첫 기능 문서를 만들 때 쓰는 프롬프트

프로젝트 점검이 끝나면 바로 planning-gate로 가지 말고, 먼저 원문 루프의 7개 문서를 만든다.

```text
feature-docs-scaffold 기준으로 첫 기능 "초기 설정 화면"의 설계 문서를 만들어줘.

대상은 ACC 스크린샷의 프로젝트 작성 모달, 프로젝트 목록, 프로젝트 템플릿, Project Admin 초기 진입 화면이야.

최소 7개 문서인 docs/PRD.md, docs/TRD.md, docs/UI_Spec.md, docs/Data_Model.md, docs/Task_List.md, docs/Acceptance_Criteria.md, docs/Test_Scenarios.md를 만들거나 갱신해줘.

필요하면 docs/Design_Map.md, docs/User_Flow.md, docs/Planning_Gate_Checklist.md도 만들어줘.

아직 구현은 하지 마.
```

여기서 Codex가 해야 할 일:

- 관련 스크린샷 찾기
- 분석 문서의 해당 항목 요약
- 첫 기능 범위 제안
- 7개 핵심 문서 생성
- 요구사항, 작업, 완료 기준, 테스트 연결
- 구현 전 확인할 점 정리

---

## 5. planning-gate를 통과시킬 때 쓰는 프롬프트

문서가 만들어진 뒤에 아래처럼 말한다.

```text
planning-gate 기준으로 초기 설정 화면 문서들을 검토해줘.

PRD 기능이 Task_List, Acceptance_Criteria, Test_Scenarios에 모두 연결됐는지 확인하고, UI_Spec과 Data_Model도 대조해줘.

정상 PASS인지, 아니면 SLICE-ONLY PASS인지 명확히 구분해서 보고해줘.

FAIL이면 구현하지 말고 고쳐야 할 문서를 알려줘.
```

---

## 6. 구현을 승인할 때 쓰는 프롬프트

계획을 보고 괜찮으면 이렇게 말한다.

```text
계획 승인. 이제 첫 기능 구현을 시작해.

단, reference 폴더는 수정하지 말고, 구현 후 CHECKS.md 기준으로 검증하고 EVIDENCE.md에 결과를 기록해줘.
```

이때부터 코드를 만들거나 수정하게 한다.

---

## 7. 완료 전에 검증시키는 프롬프트

구현이 끝났다고 말하면 바로 믿지 말고 아래처럼 말한다.

```text
validator-loop 기준으로 완료 검증해줘.

CHECKS.md에 있는 명령과 수동 확인 항목을 기준으로 실행한 것, 실패한 것, 아직 못 한 것을 나눠서 EVIDENCE.md에 기록해줘.
```

이 프롬프트의 목적은 "다 만든 것 같다"를 막고, 실제 증거를 남기게 하는 것이다.

---

## 8. 세션을 끝낼 때 쓰는 프롬프트

작업을 멈추거나 다음 사람에게 넘길 때는 이렇게 말한다.

```text
evidence-report 기준으로 이번 세션 종료 보고를 만들어줘.

변경 파일, 구현한 항목, 실행한 검증, 실패 또는 미실행 검증, 남은 리스크, 다음 세션 첫 작업을 짧게 정리해줘.

필요하면 docs/sessions/NEXT_SESSION.md도 업데이트해줘.
```

---

## 9. 내가 계속 지원할 수 있는 방식

사용자가 원하면 나는 계속 다음을 확인해줄 수 있다.

### 8.1 스킬 완성도 점검

확인할 것:

- 스킬 설명이 실제 작업 상황에서 잘 호출될 만큼 명확한지
- `SKILL.md`가 너무 추상적이지 않은지
- 출력 형식이 실제 개발자가 바로 쓸 수 있는지
- `xd-drawing-system`에서 반복 사용했을 때 빠지는 단계가 없는지
- 실패한 검증이 `validator-loop`와 `evidence-report`에 제대로 남는지

쓸 프롬프트:

```text
AI개발루프-스킬학습의 현재 스킬 7개를 xd-drawing-system 실제 개발 흐름 기준으로 점검해줘.

각 스킬별로 좋은 점, 부족한 점, 실제 개발에서 막힐 수 있는 점, 개선안을 정리해줘.
스킬 파일을 바로 수정하지 말고 먼저 제안만 해줘.
```

### 8.2 프로젝트 진행 모니터링

확인할 것:

- `SPEC.md`와 실제 구현이 맞는지
- `PLAN.md` 작업이 완료 표시만 되어 있고 증거가 빠진 것은 아닌지
- `CHECKS.md`가 실제 실행 가능한지
- `EVIDENCE.md`가 빈 주장만 담고 있지 않은지
- `reference/` 원본을 실수로 수정하지 않았는지

쓸 프롬프트:

```text
xd-drawing-system 프로젝트 진행 상태를 모니터링해줘.

SPEC.md, PLAN.md, CHECKS.md, EVIDENCE.md, docs/sessions/NEXT_SESSION.md를 읽고 현재 개발 가능 상태인지, 막힌 점이 있는지, 다음에 무엇을 해야 하는지 정리해줘.
```

### 8.3 실제 기능 개발 지원

확인할 것:

- 해당 기능에 맞는 ACC 스크린샷을 먼저 찾았는지
- UI/UX가 스크린샷 근거와 맞는지
- 기능 범위가 너무 넓어지지 않았는지
- 검증할 수 있는 작은 단위로 끝났는지

쓸 프롬프트:

```text
다음 기능 하나를 개발하려고 해.

먼저 reference/acc-screenshots와 reference/acc-analysis에서 해당 기능의 근거를 찾고, feature-docs-scaffold로 7개 문서를 만든 뒤 planning-gate 기준으로 검수해줘.
```

## 10. 기억할 원칙

1. 스킬은 자동 마법이 아니다. 필요할 때 이름을 직접 말해주면 더 잘 쓴다.
2. 새 기능은 `feature-docs-scaffold`로 7개 문서부터 만든다.
3. 구현 전에 `planning-gate`를 거친다.
4. `SLICE-ONLY PASS`는 정식 PASS가 아니다.
5. 완료 전에 `validator-loop`를 거친다.
6. 세션 종료 전 `evidence-report`를 남긴다.
7. `reference/`는 원본 보관소이고, 작업 문서는 `docs/`에 만든다.
8. `xd-drawing-system`은 한 번에 전체를 만들지 않고 기능 하나씩 만든다.
9. Hook 자동화는 아직 완성된 것이 아니며 별도 스크립트와 검증 증거가 필요하다.

## 11. 가장 짧은 시작 프롬프트

길게 치기 싫으면 이것만 써도 된다.

```text
AGENTS.md와 docs/sessions/NEXT_SESSION.md를 읽고, development-loop-orchestrator 기준으로 xd-drawing-system 현재 단계와 다음 스킬을 판단해줘. 아직 구현하지 말고 다음 행동만 제안해줘.
```
