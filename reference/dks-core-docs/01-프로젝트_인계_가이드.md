## 1. 프로젝트 목적 및 궁극적 비전

**설비 중심의 지식화/온톨로지화 플랫폼 구축**
- 공장, 생산라인, 건물, 신재생발전소 등 **모든 설비 정보를 지식화**(그래프 데이터화)
- 기존의 수많은 도면(DWG, PDF) 및 설계, 운영 문서를 AI 인지 과정(Multi-Agent, OCR)을 거쳐 TypeDB에 구조화.
- 이를 바탕으로 DCIM, BMS, FMS 등의 타 시스템이나 AI 모델이 직관적으로 정보(RAG, 자연어 질의응답)를 조회할 수 있게 하는 통합 지식 허브 구축.

## 2. 현재 달성 상태 및 결과 (Phase 3 완료)

현재 **Phase 3(검증 및 통합 테스트) 단계가 100% 완료**되었으며, 검증된 모듈을 바탕으로 실제 서비스 연동을 위한 **Phase 4 (프로덕션 배포 및 파이프라인 완성) 단계 진입 직전**입니다.

- 전체 코드 내역은 4층 아키텍처(API - Service - Core - DB)로 체계적으로 분리되어 있습니다.
- **테스트 커버리지**: 단기 E2E, 통합, 단위 테스트 244개 (Backend 177 / E2E 67) 성공 확인✅. 
- **TypeDB 기반 지식 그래프 연동 완료**: 도면 내 객체 및 속성, 관계를 처리할 수 있는 스키마 작업(`04-drawings.tql`, `01-entities.tql` 등) 및 DB CRUD 완벽 동작.
- **Frontend 및 백엔드 UI/UX 구현 (HITL 시스템)**: AI 모델이 추출한 엔티티 데이터나 BBounding Box 등을 직접 확인하고 승인/반려(Human-in-the-Loop) 할 수 있는 Svelte UI 완성 (증거 추적, 점수 계산, 일괄 승인).
- **자동화 & 비동기 큐 (Celery, Redis) 인프라 연동** 완료. (인프라는 `docker-compose`로 구현 완료)
- 보안 요건 충족을 위한 TypeQL Injection 강력 방어 적용.

## 3. 개발팀 인계 후 구현해야 할 향후 Task (Phase 4 집중)

가장 중점적으로 진행해야 하는 부분입니다. 현재 컴포넌트나 모듈 각각의 단위는 구현이 되어 있으나, **파이프라인 전체를 `조립`하고 `실서비스`와 연동**하는 작업이 필요합니다. 

### 🚨 P0 - 크리티컬 태스크 (가장 먼저 조립해야 할 핵심 기능)
1. **DWG → DXF 자동 변환 파이프라인 연동**
   - **위치**: `backend/services/dxf_service.py` 내부 `convert_dwg_to_dxf()` 함수 (`NotImplementedError` 상태)
   - **과제**: ODA Converter 등 CLI 도구나 LibreDWG 연동하여 대량의 DWG 파일을 쉽게 읽을 수 있도록 변환.
2. **페이지(장수) 감지 확장 (Model Space 보강)**
   - **위치**: `backend/services/dxf_service.py` 
   - **과제**: 현재 Layout 기반의 Paper Space만 감지. 그러나 한국 건축 도면의 다수는 `Model Space` 상에 도면이 위치함. 따라서 Model Space 내 도면도 페이지별로 자동 분리하는 로직(Viewport 분석 등) 추가.
3. **멀티모달 (이미지+텍스트) AI 전송 프로토콜 구현 (Provider)**
   - **위치**: `backend/services/llm/base_provider.py` 및 각각의 모델 프로바이더(Anthropic, OpenAI, Google) 
   - **과제**: 기존의 `generate_structured` 외에 이미지+텍스트를 함께 전송하는 `generate_with_image` 구현. (GPT/Gemini 동시 전송 위함)
4. **Celery Worker(비동기 Task) 내 실제 LLM API 연결 (Mock 걷어내기)**
   - **위치**: `backend/tasks/analysis_tasks.py` (`run_agent_task()`)
   - **과제**: 현재는 `time.sleep()` 또는 더미(Mock) 결과를 반환하도록 되어 있습니다. 앞서 구현된 ProviderFactory, 교차 검증(`cross_verification.py`), 신뢰도 계산(`confidence_calculator.py`)을 실제 워커에 붙여서 End-to-End가 구동되게 엮어주기.
5. **도면 스키마 자동 로드 (Auto Load)**
   - **위치**: `backend/main.py`
   - **과제**: 현재 `04-drawings.tql`을 API 기반으로 수동 로드해야 함. FastAPI startup 이벤트 등에 TypeDB 접속 시 스키마가 비어있으면 자동 Define 하도록 개선.
6. **한글 파일명 (UTF-8) 업로드 인코딩 문제 해결**
   - **위치**: 파일 업로드 API 단 (Backend/Frontend)
   - **과제**: 한글 파일명 업로드 시 인코딩이 깨지는 현상 수정.

### 🟡 P1 - 중요 태스크 (향후 추가 및 최적화 진행)
- **PDF 도면 → PNG 변환 로직 (pdf2image 등)** 추가
- **OCR 파이프라인 연동**: 도면 텍스트를 고정밀도로 추출할 다중 OCR (PaddleOCR + EasyOCR 앙상블 등) 도커 컨테이너화 및 FastAPI Gateway 연동
- 상용 환경을 위한 **CORS 도메인 제한 설정**.

## 4. 시스템 아키텍처 및 로컬 실행 가이드

### 기술 스택
- **DB/지식 관리**: TypeDB (3.7.3)
- **Backend**: Python 3.11+, FastAPI, Celery, Redis (Broker/ResultBackend), Python TypeDB Driver (3.7.0)
- **Frontend**: Node.js, Svelte 5 (Vite 7), Tailwind CSS
- **시각화 및 도면 렌더링**: Cytoscape.js (DAG), Konva, OpenSeadragon.

### 실행 가이드 (개발용 로컬)
**반드시 64비트 Python 환경을 이용해야 합니다.**

**1. 터미널 1 (인프라 실행 - 최우선 진행)**
```bash
docker-compose up -d typedb redis
```
**2. 터미널 2 (Backend 실행)**
```bash
cd backend
python -m uvicorn main:app --reload --port 8000
```
**3. 터미널 3 (Celery Worker 실행 - 비동기 처리용)**
```bash
cd backend
celery -A core.celery_app worker -l info -P solo
```
**4. 터미널 4 (Frontend Svelte)**
```bash
cd frontend-svelte
npm run dev
# 접속: http://localhost:5173
```

*(참고: 서버 스키마 수동 적용은 CLI/powershell을 통하여 `04-drawings.tql` define API를 호출해 주시면 됩니다. `TODO-NEXT-SESSION.md` 참조 바랍니디.)*
