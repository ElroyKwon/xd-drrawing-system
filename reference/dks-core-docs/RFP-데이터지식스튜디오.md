## 1. 사업 개요

### 1.1 한줄 정의

**산업 설비 도면을 AI가 이해할 수 있는 구조화된 지식으로 변환하고, 이를 외부 시스템에 API로 제공하는 "설비 지식 플랫폼"**

### 1.2 사업 배경

현재 산업 현장(공장, 데이터센터, 빌딩, 발전소 등)의 설비 정보는 **도면(DWG/PDF)과 문서**에 갇혀 있습니다.

```
현재 상태:
┌──────────────┐     사람이 직접      ┌──────────────┐
│  DWG 도면    │ ──── 읽고 해석 ────→ │  엑셀 정리   │
│  PDF 문서    │                      │  수동 입력   │
│  시공 기록   │                      │  기억에 의존 │
└──────────────┘                      └──────────────┘
     ↓ 문제
  • 신규 담당자가 "이 펌프 고장나면 뭐가 멈춰?"를 알려면 수일 소요
  • 설비 간 관계(공급, 제어, 모니터링)가 문서화되어 있지 않음
  • AI가 활용하려 해도 구조화된 데이터가 없음
```

### 1.3 솔루션 비전

```
목표 상태:
┌──────────────┐     AI 자동 추출     ┌──────────────────────────┐
│  DWG 도면    │ ──────────────────→  │  지식 그래프 (TypeDB)    │
│  PDF 문서    │                      │  벡터 DB (PostgreSQL)    │
│  시공 기록   │                      │  구조화된 설비 지식      │
└──────────────┘                      └────────────┬─────────────┘
                                                   │
                                      ┌────────────┴─────────────┐
                                      ▼                          ▼
                              [API 서비스 A]              [API 서비스 B]
                              외부 시스템이               외부에서 자연어
                              직접 쿼리 조회              질문 → 답변 반환
                              (TypeDB + RAG)              (MCP 서버형)
```

### 1.4 우리가 되고 싶은 것

> **"AI를 도입하고 싶은 기업이 설비 데이터 구조화를 우리에게 맡기는 솔루션 회사"**

기업이 AI를 도입하려 할 때 가장 큰 장벽은 **데이터 준비**입니다.
도면과 문서에서 설비 관계를 추출하고, AI가 활용할 수 있는 형태로 가공하는 **중간 과정** — 이것을 우리가 서비스로 제공합니다.

```
고객사의 AI 도입 여정:

  "AI 도입하고 싶어"
       ↓
  "데이터가 없네..." ← 여기서 대부분 멈춤
       ↓
  ┌─────────────────────────────────────────┐
  │  ★ 우리가 맡는 영역 (데이터 지식 스튜디오)  │
  │                                         │
  │  1. 도면/문서 수집                       │
  │  2. AI 기반 설비 추출 + 관계 매핑        │
  │  3. 지식 그래프 구축                     │
  │  4. 전문가 검증 (Human-in-the-Loop)     │
  │  5. API로 제공                           │
  └─────────────────────────────────────────┘
       ↓
  "데이터 준비 완료!"
       ↓
  고객사의 AI 시스템이 우리 API를 호출하여 활용
  (DCIM, BMS, FMS, PMS, 통합SI 등)
```

## 2. 왜 이것을 해야 하는가

### 2.1 도면은 모든 정보의 원천이다

산업 설비의 전 생애주기가 도면에서 시작됩니다:

```
도면 → 견적 → 상세설계 → 샵도면 → 시스템정의
  → 태그생성 → 엔지니어링 래더 → HMI 화면
  → 운영 → 유지보수
```

이 모든 과정의 **원천 데이터가 도면**이지만, 현재는 사람만 읽을 수 있습니다.

### 2.2 질의 가능성이 가치다

설비 지식이 구조화되면 다음 질문에 **즉시** 답할 수 있습니다:

| 질문 | 현재 | 구축 후 |
|------|------|---------|
| "펌프 P-101 고장나면 영향받는 설비는?" | 수일 (경험 의존) | **즉시** (그래프 탐색) |
| "3층 기계실의 모든 밸브 목록은?" | 도면 수동 확인 | **즉시** (공간 쿼리) |
| "냉수 계통 상류에서 하류까지 경로는?" | 불가능 | **즉시** (관계 추적) |
| "이 센서가 모니터링하는 설비는?" | 담당자 기억 | **즉시** (monitors 관계) |

### 2.3 AI 시대의 인프라

```
현재 AI의 한계:
  GPT/Claude에게 "펌프 P-101의 상류 설비가 뭐야?" 물으면?
  → "알 수 없습니다" (데이터가 없으니까)

우리 시스템 구축 후:
  GPT/Claude가 우리 MCP 서버에 질의
  → TypeDB에서 관계 추적
  → "CH-01(냉동기) → P-101(펌프) → AHU-01(공조기)" 반환
  → AI가 정확한 답변 가능
```

### 2.4 시장 기회

| 대상 시장 | 설비 유형 | 고객 예시 |
|-----------|-----------|-----------|
| **데이터센터** | CRAC, UPS, PDU, 냉각탑 | 네이버, 카카오, KT |
| **빌딩 관리 (BMS)** | 공조기, 냉동기, 보일러 | 삼성SDS, LG CNS |
| **공장/제조 (FMS)** | 로봇, 컨베이어, PLC | 현대, 삼성, SK |
| **신재생에너지 (PMS)** | 인버터, 태양광 모듈, ESS | 한화, 두산 |
| **수처리** | 펌프, 슬러지 처리, 여과기 | 한국수자원공사 |

### 2.5 축적되는 경쟁 우위

```
프로젝트 A (HVAC) → 설비 지식 축적
프로젝트 B (수처리) → 새 도메인 지식 추가
프로젝트 C (데이터센터) → 또 다른 도메인 추가
    ↓
프로젝트가 늘수록 → 지식이 풍부해짐 → 추출 정확도 상승 → 경쟁 우위 심화
```

## 3. 데이터 수집 대상과 목적

### 3.1 입력 데이터 유형

| 구분 | 데이터 유형 | 형식 | 포함 정보 | 우선순위 |
|------|------------|------|-----------|:--------:|
| **도면** | 기계설비 도면 | DWG, DXF | 설비 배치, 배관 경로, 계통도 | **P0** |
| **도면** | 전기 단선도 | DWG, PDF | 전력 계통, 변압기, 차단기 | P1 |
| **도면** | 계장 P&ID | DWG, PDF | 센서, 제어밸브, 제어루프 | P1 |
| **문서** | 설계 사양서 | PDF, DOCX | 설비 스펙, 용량, 재질 | P1 |
| **문서** | 시운전 보고서 | PDF | 성능 데이터, 설정값 | P2 |
| **문서** | 유지보수 이력 | Excel, PDF | 고장 기록, 교체 이력 | P2 |
| **데이터** | 태그 리스트 | Excel | HMI 태그명, 설비 매핑 | P1 |
| **데이터** | 설비 대장 | Excel | 제조사, 모델, 설치일 | P1 |

### 3.2 추출 목표

도면과 문서에서 **다음을 자동으로 추출**하여 지식 그래프에 저장합니다:

#### 3.2.1 설비 엔티티 (7가지 타입)

| 타입 | 설명 | 도면에서의 식별 | 예시 |
|------|------|----------------|------|
| **Equipment** | 동력 사용, 공정 수행 설비 | 심볼 + 태그번호 | 펌프 P-101, 냉동기 CH-01 |
| **Valve** | 흐름 차단/조절 장치 | 심볼 + 태그번호 | 게이트밸브 GV-101, 제어밸브 CV-201 |
| **Sensor** | 물리량 측정 장치 | 원형 심볼 | 온도센서 TE-101, 압력센서 PT-201 |
| **Instrument** | 측정값 표시/기록 장치 | 심볼 | 압력계 PI-101, 유량계 FI-201 |
| **Pipe** | 매체 이동 경로 | 라인 + 번호 | 냉수 배관 CW-01, 급수관 DW-01 |
| **Space** | 물리적 공간 | 영역 경계선 | 기계실, 3층, B동 |
| **System** | 논리적 설비 그룹 | 계통도 범위 | 냉수 시스템, 급수 시스템 |

#### 3.2.2 설비 관계 (6가지)

| 관계 | 의미 | 추론 활용 | 예시 |
|------|------|-----------|------|
| **feeds** | A가 B에게 매체 공급 | 상류/하류 추적 | 냉동기 → 펌프 → 공조기 |
| **control-relation** | A가 B를 제어 | 제어 루프 파악 | 온도센서 → 제어밸브 |
| **monitors** | A가 B를 모니터링 | 감시 대상 확인 | 압력센서 → 배관 |
| **spatial-containment** | A 공간에 B 위치 | 위치 기반 검색 | 기계실 ⊃ 펌프 |
| **connected-via** | A와 B가 배관 연결 | 물리적 경로 | 펌프 ─ 배관 ─ 밸브 |
| **system-membership** | A 시스템에 B 소속 | 계통 관리 | 냉수시스템 ⊃ 냉동기 |

#### 3.2.3 설비 속성 (8개 필수)

| 속성 | 설명 | 예시 |
|------|------|------|
| global-id | 시스템 내 고유 식별자 | `uuid-xxx-xxx` |
| tag | 현장 태그번호 | `P-101` |
| name | 설비명 (현장 언어) | `3층 냉수 순환펌프` |
| description | 설명 | `3층 AHU 냉수 공급용` |
| discipline | 분야 | `mechanical` |
| function-category | 기능 분류 | `pump` |
| status | 상태 | `active` |
| lifecycle-phase | 생애주기 | `operation` |

### 3.3 수집→지식화 파이프라인

```
[1단계: 수집]
  DWG/PDF 도면 업로드
      ↓
[2단계: 변환]
  DWG → DXF → PNG (페이지별)
  PDF → PNG (페이지별)
  텍스트 추출 (OCR + DXF 파싱)
      ↓
[3단계: AI 분석]
  Multi-Agent 교차검증 (3개 LLM 독립 분석)
  ┌─────────┐  ┌──────────┐  ┌──────────┐
  │ Agent A  │  │ Agent B  │  │ Agent C  │
  │ (GPT-4)  │  │ (규칙)   │  │ (Gemini) │
  └────┬─────┘  └────┬─────┘  └────┬─────┘
       └──────────────┼──────────────┘
                      ▼
              교차검증 + 신뢰도 산출
      ↓
[4단계: 검증]
  Human-in-the-Loop (전문가 검토 UI)
  승인/수정/거부 워크플로우
      ↓
[5단계: 저장]
  TypeDB (구조화된 관계 지식)
  PostgreSQL + pgvector (문서 벡터, RAG용)
      ↓
[6단계: 제공]
  API 서비스 A: 구조화 쿼리 (TypeDB + RAG)
  API 서비스 B: 자연어 질의응답 (MCP 서버형)
```

### 4.2 API 서비스 A: 구조화 쿼리 서비스

**외부 시스템이 직접 데이터를 조회하는 방식**

```
[외부 클라이언트 → 우리 서버]

1) TypeDB 쿼리 API
   POST /api/v1/query
   {
     "database": "project-alpha",
     "query": "match $e isa equipment, has tag 'P-101'; fetch { ... };"
   }
   → 설비 정보 + 연결된 관계 반환

2) RAG 벡터 검색 API
   POST /api/v1/search/semantic
   {
     "query": "냉수 순환 펌프 관련 문서",
     "top_k": 10
   }
   → 유사도 높은 문서 청크 + 메타데이터 반환

3) 그래프 탐색 API
   GET /api/v1/graph/neighbors?entity=P-101&depth=2
   → P-101 기준 2단계 관계 설비 목록 반환
```

**사용 시나리오**:
- DCIM 시스템이 알람 발생 시 영향 범위를 우리 API로 조회
- BMS가 설비 목록/관계를 우리 API에서 가져와 화면에 표시
- ERP가 유지보수 계획 수립 시 설비 관계도 조회

### 4.3 API 서비스 B: 자연어 질의응답 (MCP 서버형)

**외부에서 자연어로 질문하면 우리가 조회하여 답변을 반환하는 방식**

```
[외부 AI 시스템 → 우리 MCP 서버]

요청:
  POST /api/v1/mcp/query
  {
    "question": "항온항습기의 냉수 공급은 누가 해주지?",
    "project": "datacenter-alpha"
  }

내부 처리:
  1. LLM이 질문 의도 파악 → "feeds 관계 역추적"
  2. TypeQL 자동 생성:
     match
       $target isa equipment, has name "항온항습기";
       $source isa equipment;
       (supplier: $source, consumer: $target) isa feeds;
     fetch { "supplier": $source.tag, "name": $source.name };
  3. TypeDB 실행 → 결과 수집
  4. RAG에서 관련 문서 보강
  5. 구조화된 답변 생성

응답:
  {
    "answer": "CH-01(냉동기) → P-101(펌프) → 항온항습기",
    "evidence": [
      { "source": "3F-HVAC-계통도.dwg", "page": 2, "confidence": 0.95 }
    ],
    "related_sensors": ["TE-101 (공급측)", "TE-102 (환수측)"],
    "graph_path": [
      { "from": "CH-01", "relation": "feeds", "to": "P-101" },
      { "from": "P-101", "relation": "feeds", "to": "THCU-01" }
    ]
  }
```

**MCP (Model Context Protocol) 도구 형태**:

```
우리가 제공하는 MCP 도구 목록:
┌───────────────────────────────────────────────────┐
│  Tool: query_equipment                            │
│  설명: 설비 정보 조회                              │
│  입력: { tag: string, project: string }           │
│  출력: 설비 상세 정보 + 연결 관계                  │
├───────────────────────────────────────────────────┤
│  Tool: trace_upstream                             │
│  설명: 상류 설비 추적                              │
│  입력: { tag: string, depth: number }             │
│  출력: 상류 설비 경로 (feeds 관계 역추적)          │
├───────────────────────────────────────────────────┤
│  Tool: find_by_space                              │
│  설명: 공간 기반 설비 검색                         │
│  입력: { space: string, type?: string }           │
│  출력: 해당 공간의 설비 목록                       │
├───────────────────────────────────────────────────┤
│  Tool: impact_analysis                            │
│  설명: 설비 고장 시 영향도 분석                    │
│  입력: { tag: string, scenario: string }          │
│  출력: 영향받는 설비 + 심각도                      │
├───────────────────────────────────────────────────┤
│  Tool: search_documents                           │
│  설명: RAG 기반 문서 검색                          │
│  입력: { query: string, top_k: number }           │
│  출력: 관련 문서 청크 + 출처                       │
└───────────────────────────────────────────────────┘
```

**사용 시나리오**:
- 고객사의 AI 챗봇이 우리 MCP 서버를 도구로 연결 → 설비 관련 질문에 정확한 답변
- 운영 의사결정 시스템이 "밸브 V-201을 잠그면 영향 범위?"를 자동으로 분석
- 유지보수 AI가 고장 설비의 관련 문서와 이력을 자동 수집
