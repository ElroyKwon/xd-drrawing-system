## 1. 우리가 풀려는 문제

### 1.1 현실

산업 현장(공장, 데이터센터, 빌딩, 발전소)의 **가장 중요한 지식은 도면 안에 있다.**

- 설비가 어디에 있는지
- 무엇이 무엇에 연결되어 있는지
- 냉수가 어디서 어디로 흐르는지
- 전력이 어떤 경로로 공급되는지
- 센서가 어떤 설비를 감시하는지

이 정보는 **P&ID, 단선도, 배관도, 배치도** 등의 도면에 기호와 선으로 표현되어 있다.

### 1.2 문제

하지만 현실에서는:

| 문제 | 상황 |
|------|------|
| **접근 불가** | 도면은 DWG/PDF 파일로 잠겨 있고, AutoCAD를 열어서 한 장씩 봐야 한다 |
| **사람 의존** | "이 밸브 잠그면 어디가 영향받지?" → 경력 20년 차 김 과장만 안다 |
| **지식 소멸** | 김 과장이 퇴직하면, 그 지식은 사라진다 |
| **변경 추적 불가** | 준공 도면과 현재 상태가 다른데, 어디가 다른지 아무도 모른다 |
| **검색 불가** | "3층 기계실의 모든 펌프" → 도면을 다 뒤져봐야 한다 |

### 1.3 우리가 만들고 싶은 것

> **도면에 갇혀 있는 설비 지식을, 누구든(사람이든 AI든) 질문하면 답을 얻을 수 있는 시스템으로 바꾸는 것**

```
[현재]                              [목표]

도면 (DWG/PDF)                     "항온항습기 냉수공급은 누가 해줘?"
  → AutoCAD 열기                    → "CH-01 냉동기 → P-101 펌프 → 항온항습기"
  → 한 장씩 찾기
  → 김 과장한테 물어보기             "CH-01 고장나면 어디까지 영향?"
  → 며칠 걸림                       → "AHU-01~03, FCU 12대, 3~5층 냉방 중단"
                                    → 즉시 응답
```

### 1.4 도면만? 문서도?

도면이 **핵심**이지만, 그게 전부는 아니다.

| 소스                | 포함 정보            |  우선순위   |
| ----------------- | ---------------- | :-----: |
| **도면 (DWG/PDF)**  | 설비 배치, 연결 관계, 흐름 | **1순위** |
| 준공 문서 (PDF)       | 설비 사양, 설치 조건     |   2순위   |
| 운영 매뉴얼 (Word/PPT) | 운전 절차, 점검 방법     |   3순위   |
| 시운전 보고서           | 실제 성능 데이터        |   3순위   |

도면 = 구조화된 지식 (관계, 연결, 위치) → **지식그래프에 적합**
문서 = 비구조화된 맥락 (절차, 사양, 히스토리) → **RAG/벡터DB에 적합**

**궁극적으로는 둘 다 필요하다.** 하지만 도면 지식화가 먼저다.

## 2. 가치 제안 (왜 돈을 내고 쓰는가)

### 2.1 핵심 가치

| # | 가치 | 설명 | 예시 |
|:-:|------|------|------|
| 1 | **지식 보존** | 사람 머릿속 지식을 시스템으로 | 김 과장 퇴직해도 설비 관계 정보 유지 |
| 2 | **리스크 감소** | 설비 고장 시 영향 즉시 파악 | "이 펌프 멈추면 어디까지?" 즉답 |
| 3 | **정보 접근성** | 도면 없이도 설비 정보 조회 | 자연어로 질문, 그래프로 시각화 |
| 4 | **변경 추적** | 신규 정보 vs 과거 정보 분리 관리 | 준공 시점 vs 현재 상태 비교 |

### 2.2 서비스 제공 방식 (미결정)

이 시스템을 **누가** 쓰는가에 따라 제공 방식이 달라진다:

```
방식 A: 데이터 서비스 (B2B API)
┌─────────────────────┐
│  상위 AI 시스템      │  ← 외부 고객의 AI가 우리 API를 호출
│  (DCIM, BMS, FMS)   │
└────────┬────────────┘
         │ TypeQL 쿼리 / Vector Search API
         ▼
┌─────────────────────┐
│  우리 시스템         │  ← 지식 저장소 + 검색 엔진 역할
│  (TypeDB + VectorDB)│
└─────────────────────┘

방식 B: 완성형 서비스 (B2B SaaS)
┌─────────────────────┐
│  사용자 (운영팀)     │  ← 직접 웹 UI에서 질문
│  "3층 펌프 상태?"    │
└────────┬────────────┘
         │ 자연어 질의
         ▼
┌─────────────────────┐
│  우리 시스템         │  ← 질문 이해 + 답변 생성까지 전부
│  (UI + AI + 지식DB) │
└─────────────────────┘
```

## 3. 전략적 갈림길: TypeDB를 유지할 것인가

### 3.1 현재 경로 (TypeDB 중심)

```
도면 (DWG) → DXF 변환 → DXF 파싱 → 텍스트/구조 추출
                                          ↓
                                    AI 분석 (Multi-Agent)
                                          ↓
                                    온톨로지 매핑 (표준 검증)
                                          ↓
                                    TypeDB에 저장 (지식그래프)
                                          ↓
                                    TypeQL로 추론 쿼리
                                          ↓
                                    사용자에게 답변
```

**장점**:
- TypeDB 추론(inference)으로 명시적으로 입력하지 않은 관계도 유추 가능
- 온톨로지 표준이 강제되어 데이터 품질 보장
- 구조화된 지식이므로 정확한 답변 가능

**단점**:
- 파이프라인이 너무 길다 (DWG→DXF→파싱→AI→매핑→검증→저장)
- 각 단계마다 넘어야 할 기술적 산이 있다
- TypeDB 자체가 니치한 기술이라 개발자 확보 어려움
- 여기까지 오는 데 Phase 1~3를 다 썼는데 아직 end-to-end가 안 됨

### 3.2 대안 경로 (AI-Direct)

```
도면 (DWG) → DXF 변환 → DXF + 이미지
                              ↓
                    AI API에 직접 전송 (멀티모달)
                    "이 도면을 분석해서 설비 관계를 JSON으로 줘"
                              ↓
                    AI가 여러 번 분석 (교차검증)
                              ↓
                    결과를 VectorDB에 저장 (RAG)
                              ↓
                    사용자 질문 → RAG 검색 → AI 답변
```

**장점**:
- 파이프라인이 훨씬 짧다
- AI API 성능이 빠르게 좋아지고 있다 (멀티모달, 긴 컨텍스트)
- VectorDB + RAG는 생태계가 성숙해 있다
- 개발 인력 확보가 쉽다

**단점**:
- AI 환각(hallucination) 리스크 — 도면을 잘못 읽을 수 있다
- 구조화된 추론이 불가능 ("A→B→C→D 영향 체인" 같은 것)
- 표준 검증이 약하다 (AI가 알아서 하는 건 불안정)
- 비용이 API 호출량에 비례

### 3.3 하이브리드 경로

```
도면 (DWG) → DXF 변환 → DXF + 이미지
                              ↓
                    AI API 멀티모달 분석
                    "설비 목록, 연결 관계 추출"
                              ↓
                    ┌─────────┴─────────┐
                    ↓                   ↓
              TypeDB (구조화)     VectorDB (비구조화)
              설비 관계, 추론     문서 맥락, 사양 정보
                    ↓                   ↓
                    └─────────┬─────────┘
                              ↓
                    통합 질의 엔진
                    (GraphRAG 자체 구현)
```

**장점**: 양쪽의 장점을 결합
**단점**: 복잡도가 가장 높음. 두 DB를 동기화하고 통합 질의하는 것 자체가 큰 과제.

### 3.4 비교 요약

| 항목 | TypeDB 중심 | AI-Direct | 하이브리드 |
|------|:----------:|:---------:|:----------:|
| 파이프라인 복잡도 | 높음 | **낮음** | 매우 높음 |
| 답변 정확도 (추론) | **높음** | 보통 | 높음 |
| 개발 난이도 | 높음 | **낮음** | 매우 높음 |
| 개발자 확보 | 어려움 | **쉬움** | 어려움 |
| AI 기술 발전 수혜 | 제한적 | **높음** | 높음 |
| 데이터 품질 보장 | **높음** | 낮음 | 높음 |
| MVP까지 시간 | 길다 | **짧다** | 매우 길다 |
| 운영 비용 | 서버 비용 | API 비용 | 둘 다 |

> **핵심 질문**: "TypeDB의 추론(inference) 기능이 이 서비스에 꼭 필요한가?"
>
> - **Yes라면** → TypeDB 중심 or 하이브리드
> - **No라면** → AI-Direct가 훨씬 빠르게 MVP를 만들 수 있다
>
> 추론이 필요한 예: "CH-01 → P-101 → AHU-01 → 3층 냉방" 같은 다단계 영향 체인
> AI만으로도 가능한 예: "이 도면에 있는 펌프 목록 알려줘"

### 4.1 그래서 지금 뭐가 있나 (솔직하게)

| 온톨로지 표준 + 검증 엔진 | 100% 완성 | **경로에 관계없이 재사용 가능** |

| 244개 테스트 | PASS | 코드 품질 보증 |

### 4.2 어떤 경로를 선택하든 살리는 것

| 자산 | 재사용 가능 여부 | 설명 |
|------|:----------------:|------|
| **온톨로지 표준 (스키마)** | 어디서든 사용 가능 | 설비 분류 체계, 관계 정의, 속성 표준 |
| **검증 엔진 (YAML 규칙)** | 어디서든 사용 가능 | AI가 만든 결과를 검증하는 데 활용 |
| **4층 아키텍처 패턴** | 구조적 참고 가능 | API-Service-Core-DB 분리 |
| **LLM Factory 패턴** | 그대로 재사용 | Claude/GPT/Gemini 추상화 |
| **HITL 워크플로우** | 그대로 재사용 | AI 결과 검토/승인 프로세스 |
| 도면 업로드/뷰어 | 그대로 재사용 | 파일 관리 기본 기능 |
| TypeDB 웹 UI | TypeDB 경로에서만 | AI-Direct로 가면 불필요 |

### 4.3 버린 시간이 아닌 이유

Phase 1~3에서 얻은 것:
1. **온톨로지 표준이 정의되었다** — 이건 기술 스택과 무관하게 가치가 있다
2. **AI가 검증 없이 데이터를 만들면 안 된다는 걸 확인했다** — HITL의 필요성 검증
3. **DWG→DXF→파싱→AI의 파이프라인 복잡도를 체감했다** — 대안 경로를 고민할 수 있는 근거
4. **TypeDB의 가능성과 한계를 모두 경험했다** — 합리적 판단이 가능해짐

### 5.3 이관팀에게 전달해야 할 메시지

> "우리는 **산업 현장의 도면을 지식화**하는 시스템을 만들고 있습니다.
>
> 핵심 문제: 도면에 갇힌 설비 관계 정보를 꺼내서, 질문하면 답할 수 있게 만드는 것.
> 핵심 가치: 지식 보존(사람 의존 탈피) + 리스크 감소(영향도 즉시 파악).
>
> Phase 1~3에서 TypeDB 기반으로 부품을 만들었고, 온톨로지 표준과 검증 엔진은 완성했습니다.
> 하지만 end-to-end 파이프라인은 아직 동작하지 않습니다.
>
> 이관팀이 가장 먼저 해야 할 일은 **기술 경로를 결정**하는 것입니다.
> TypeDB를 유지할지, AI-Direct로 전환할지에 따라 이후 작업이 완전히 달라집니다."

## 6. 도면 분석: 실제로 어떻게 하는가

### 6.1 도면의 3가지 종류

| 분류 | 도면 종류 | 포함 정보 | 분석 난이도 |
|------|----------|----------|:----------:|
| **건축** | 배치도, 평면도, 단면도 | 공간, 층, 실(Room), 동선 | 중 |
| **기계** | P&ID, 배관도, 공조 계통도 | 설비, 배관, 밸브, 흐름 방향 | **상** |
| **전기** | 단선도, 결선도, 판넬 스케줄 | 차단기, 계통, 전력 경로 | **상** |

### 6.2 도면 안의 선은 전부 다르다

도면에서 가장 어려운 문제: **선이 뭘 의미하는지 구분하는 것**

```
도면 위의 선들:
  ─────────  실선 (배관, 덕트, 케이블 = 물리적 연결)
  - - - - -  점선 (제어 신호, 미래 설치 예정)
  ──┤├──     기호의 일부 (밸브, 센서, 장비 심볼)
  ──────┐    꺾임선 (경로 변경, 분기)
        │
  ═══════    이중선 (벽, 구조물)
```

| 선의 종류 | 의미 | AI가 구분 가능? |
|----------|------|:--------------:|
| **연결선** (배관/덕트/케이블) | 설비와 설비를 잇는 물리적 경로 | 어려움 — 문맥 필요 |
| **기호선** (심볼 내부) | 밸브, 펌프 등 장비 기호의 일부 | 비교적 가능 — 패턴 매칭 |
| **경계선** (벽, 영역) | 공간 구분 | 비교적 가능 |
| **제어선** (점선, 파선) | 신호 흐름, 제어 관계 | 어려움 — 범례 필요 |
| **주석선** (치수, 인출선) | 크기, 설명 | OCR로 텍스트 추출 |

### 6.3 우리가 구상한 분석 방법 (Multi-Pass AI)

```
[1단계] 저렴하고 빠른 모델로 초벌 분석
┌──────────────────────────────────────────┐
│  도면 이미지 → Gemini Flash / GPT-4o-mini │
│  "이 도면에서 설비 목록, 배관 연결을 추출해줘" │
│  → 초벌 결과 (정확도 60~70%)              │
└──────────────────────────────────────────┘
                    ↓
[2단계] OCR로 텍스트 보강
┌──────────────────────────────────────────┐
│  PaddleOCR / EasyOCR → 도면 내 텍스트 추출 │
│  태그명: "P-101", "AHU-01", "V-201"      │
│  사양: "100A", "2.2kW", "DN50"           │
└──────────────────────────────────────────┘
                    ↓
[3단계] 텍스트 + 이미지를 고급 모델에 재전송
┌──────────────────────────────────────────┐
│  OCR 텍스트 + 도면 이미지 → Claude / GPT-4o │
│  "OCR에서 이런 텍스트가 나왔어. 이걸 참고해서 │
│   다시 분석해줘"                           │
│  → 보강된 결과 (정확도 80~85%)             │
└──────────────────────────────────────────┘
                    ↓
[4단계] 교차검증 (다른 모델로 비교)
┌──────────────────────────────────────────┐
│  모델 A 결과 vs 모델 B 결과 비교           │
│  일치 → 신뢰도 높음                       │
│  불일치 → 3번째 모델로 타이브레이크         │
│  → 최종 결과 (신뢰도 점수 포함)            │
└──────────────────────────────────────────┘
                    ↓
[5단계] 사람 검토 (HITL)
┌──────────────────────────────────────────┐
│  AI 결과를 도면 위에 오버레이로 표시        │
│  운영자가 확인 → 승인/수정/반려            │
└──────────────────────────────────────────┘
```

> **이 파이프라인은 이미 Phase 3에서 코드로 구현됨 (Mock 상태).**
> Phase 4에서 실제 AI API를 연결하면 동작한다.

## 7. 핵심 논쟁: 그래프 DB가 정말 필요한가

### 7.1 질문의 종류에 따라 답이 다르다

| 질문 유형 | 예시 | Vector RAG | GraphRAG | 그래프 DB 직접 탐색 |
|----------|------|:---------:|:--------:|:-----------------:|
| **단순 조회** | "3층 펌프 목록" | **가능** | **가능** | 가능 |
| **사양 질문** | "P-101 모터 용량?" | **가능** | **가능** | 가능 |
| **1단계 관계** | "P-101은 뭐랑 연결돼?" | 부분 가능 | **가능** | **가능** |
| **다단계 영향** | "CH-01 고장 → 어디까지?" | 불가 | 부분 가능 (LLM 의존) | **정확하게 가능** |
| **역추적** | "5층 냉방 안 되면 원인은?" | 불가 | 부분 가능 (LLM 의존) | **정확하게 가능** |
| **What-if** | "밸브 V-201 잠그면?" | 불가 | 부분 가능 (LLM 의존) | **정확하게 가능** |

### 7.2 영향도 분석 — 문서화만으로는 안 되는 이유

밸브 하나를 잠그면 무슨 일이 생기는가?

```
[그래프 DB 방식 — 관계를 따라가며 추론]

V-201 (밸브, 잠금)
  → feeds: P-101 (냉수펌프) — 공급 차단
    → feeds: AHU-01 (공조기) — 냉수 없음
      → spatial-containment: 3F-ZONE-A — 냉방 중단
        → spatial-containment: SERVER-ROOM-01 — 서버실 온도 상승!
          → monitors: TE-301 (온도센서) — 알람 발생

결과: "V-201 잠금 → 3층 서버실 냉방 중단, TE-301 알람 예상"
       이 체인은 6단계를 자동으로 따라간 것
```

```
[Vector RAG 방식 — 유사한 텍스트를 검색]

질문: "V-201 잠그면 어떻게 돼?"
검색: "V-201" 관련 문서 chunk 찾기
결과: "V-201은 냉수 배관의 차단 밸브입니다" (← 여기까지만 나옴)

문제: P-101 → AHU-01 → 3층 → 서버실 체인은
      각각 다른 문서/도면에 흩어져 있어서 한 번의 검색으로 못 찾음
```

**이것이 GraphRAG가 그래프 구조를 활용하는 이유다.**

### 7.3 GraphRAG vs Vector RAG vs 그래프 DB 직접 탐색

> **정정**: GraphRAG는 "그래프 DB에서 BFS/DFS 탐색"하는 것이 아니다.
> 실제로는 **벡터 검색으로 엔티티를 찾고 → 그래프 구조로 관계를 확장하고 → LLM이 답변을 생성**하는 하이브리드 방식이다.

| | Vector RAG만 | GraphRAG (Graph + Vector) | 그래프 DB 직접 탐색 |
|--|:------------:|:------------------------:|:------------------:|
| 엔티티 찾기 | 벡터 유사도 | **벡터 유사도** | 정확한 ID/속성 매칭 |
| 관계 확장 | 없음 | **1~2 hop + 커뮤니티 요약** | **N hop 정확한 탐색** |
| 답변 생성 | LLM (검색된 chunk 기반) | LLM (엔티티+관계+커뮤니티 기반) | 쿼리 결과 → LLM 자연어화 |
| 강점 | 사양, 설명, 절차 질문 | **1~2단계 관계, 요약 질문** | **다단계 경로, 정확한 체인** |
| 약점 | 관계 추론 불가 | 깊은 체인 불확실 (LLM 의존) | 데이터 수동 구조화 필요 |
| 핵심 원리 | 유사도 (cosine similarity) | **벡터 검색 + 그래프 구조 보강** | **그래프 탐색 (traversal)** |

```
GraphRAG의 실제 동작 원리 (Microsoft GraphRAG 기준):

1. 질문 → "CH-01 고장 시 영향?"
2. 질문을 벡터화 → 벡터 DB에서 유사한 엔티티 검색 (embedding similarity)
   → CH-01 엔티티 발견
3. CH-01과 연결된 관계(relationship) 조회 → P-101, AHU-01 등
4. CH-01이 속한 커뮤니티(community) 요약 조회 → "냉수 공급 계통" 요약
5. 관련 원본 텍스트 chunk도 함께 수집
6. 이 모든 컨텍스트를 LLM에 전달 → 답변 생성

→ 1~2단계 관계는 잘 찾지만, 6단계 체인(V-201→P-101→AHU-01→3F→서버실→센서)은
   커뮤니티에 포함되어 있지 않으면 놓칠 수 있다.
→ 그래프 DB 직접 탐색은 경로를 정확하게 따라가므로 놓치지 않는다.
```

### 7.4 그래서 문서화만으로 영향도를 담을 수 있나?

**이론적으로는 가능하다.** 예를 들어:

```json
{
  "entity": "V-201",
  "type": "valve",
  "impact_chain": [
    {"target": "P-101", "effect": "냉수 공급 차단"},
    {"target": "AHU-01", "effect": "냉수 없어 냉방 불가", "via": "P-101"},
    {"target": "3F-ZONE-A", "effect": "냉방 중단", "via": "AHU-01"},
    {"target": "SERVER-ROOM-01", "effect": "온도 상승", "via": "3F-ZONE-A"}
  ]
}
```

**하지만 현실적 문제:**

| 문제 | 설명 |
|------|------|
| **조합 폭발** | 설비 1000개 × 관계 5000개면, 모든 영향 체인을 미리 문서화하는 건 불가능 |
| **변경 시 재생성** | 설비 하나 추가/삭제하면 관련된 모든 영향 체인 문서를 다시 만들어야 함 |
| **방향성** | "밸브를 열면?" vs "밸브를 닫으면?" → 같은 엔티티인데 결과가 다름 |
| **조건부 영향** | "펌프 2대 중 1대만 고장이면 50% 감소, 2대 다 고장이면 완전 중단" |

**결론**: 단순 관계는 문서화 가능, **다단계 영향도 분석은 그래프 탐색이 필수**

### 7.5 각 도면 유형별 영향도 분석

#### 기계 도면 (P&ID, 배관도)
```
밸브 열기/닫기 → 유체 흐름 변경 → 하류 설비 영향
FAN 가동/정지 → 공기 흐름 변경 → 공간 온도/환기 영향
펌프 고장     → 냉/온수 공급 중단 → 냉난방 설비 영향

분석 방식: feeds 관계를 따라 하류 탐색
그래프 DB 필요도: ★★★★★ (가장 높음)
```

#### 전기 도면 (단선도, 판넬)
```
차단기 Open   → 하위 계통 정전 → 연결된 모든 부하 영향
차단기 Close  → 하위 계통 급전
변압기 고장   → 전체 뱅크 정전

분석 방식: 전력 계통 트리를 따라 하위 탐색
그래프 DB 필요도: ★★★★★ (가장 높음, 트리 구조)
```

#### 건축 도면 (평면도, 배치도)
```
공간 내 설비 목록 → 해당 공간 영향받는 설비 파악
층간 관계      → 상하층 영향 (누수, 화재 등)
동선 분석      → 대피 경로, 접근 경로

분석 방식: spatial-containment 관계로 공간 내 설비 목록화
그래프 DB 필요도: ★★★☆☆ (상대적으로 단순)
```

### 7.6 최종 판단 기준

```
"우리 서비스에서 영향도 분석이 핵심 가치인가?"

├── YES → 그래프 DB가 필요하다
│         (TypeDB, Neo4j, 또는 최소 PostgreSQL + 재귀 쿼리)
│
│         단, TypeDB가 유일한 선택지는 아님
│         Neo4j가 더 생태계가 넓고 개발자 확보가 쉬움
│
└── NO  → Vector RAG만으로 충분하다
          (설비 목록, 사양 조회, 문서 검색이 주 용도라면)
```

> **GraphRAG를 직접 만들지 않아도 된다.**
> Microsoft GraphRAG는 MIT 라이선스 오픈소스이고, LightRAG, nano-graphrag 등 경량 대안도 있다.
> RAGFlow에도 GraphRAG 기능이 내장되어 있다 (v0.16+).
> 또는 Neo4j + LangChain 조합으로 유사한 것을 만들 수 있다.
> **핵심은 "그래프 DB가 필요한가"이지, "TypeDB여야 하는가"가 아니다.**

## 8. 대안 스택 평가: RAGFlow + Apache AGE + FastAPI

> 아래 내용은 2026년 3월 기준 최신 공식 문서와 GitHub을 기반으로 조사한 결과.

### 8.1 제안된 조합

| 구분 | 도구 | 역할 |
|------|------|------|
| LLM Engine | 외부 Cloud API (Gemini, GPT 등) | 지능형 분석, 답변 생성 |
| RAG System | **RAGFlow** (오픈소스) | 도면/문서 파싱, 벡터화, 검색 |
| Graph DB | **Apache AGE** (PostgreSQL 확장) | 설비 관계 저장, 영향도 분석 |
| Orchestration | Python FastAPI | RAGFlow ↔ Graph DB 연결 커스텀 로직 |

**매력적인 이유**: 전부 무료(Apache 2.0), PostgreSQL 기반이라 운영 부담 적음

### 8.2 RAGFlow — 실제로 뭘 할 수 있나

**RAGFlow** (GitHub 48k+ stars, InfiniFlow 개발)

**할 수 있는 것:**
- PDF, Word, PPT, Excel, 이미지, 스캔 문서 파싱
- 테이블, 레이아웃, 시각 요소까지 구조화 추출 (Deep Document Understanding)
- LLM 기반 엔티티/관계 추출 → GraphRAG 지원 (v0.9부터, v0.16+에서 개선)
- 벡터 검색 + 전문 검색 하이브리드
- 에이전트 워크플로우, MCP 지원 (2025.08+)
- S3, Google Drive, Notion, Confluence 데이터 동기화
- 관리자 Web UI 대시보드

**할 수 없는 것 (중요!):**
- **DWG/DXF 파일 파싱 미지원** — PDF, Word, 이미지만 가능
- **외부 그래프 DB 연동 없음** — Neo4j, Apache AGE와 직접 연결 불가
- **지식그래프 내보내기 불가** — RAGFlow 내부에만 존재 (Elasticsearch/Infinity에 저장)
- **깊은 다단계 체인 추적은 불확실** — 1~2 hop 관계와 커뮤니티 요약은 가능하지만, 6단계 영향 체인의 정확한 추적은 LLM 추론에 의존하므로 보장되지 않음

```
RAGFlow의 GraphRAG 동작 방식:

  [인덱싱 단계]
  문서 → LLM이 엔티티/관계 추출 → Elasticsearch/Infinity에 저장
       → 엔티티 중복 제거 (deduplication)
       → 커뮤니티 감지 및 요약 생성

  [검색 단계]
  질문 → 벡터 검색으로 관련 엔티티 찾기
       → 해당 엔티티의 관계(relationship) 확장
       → 소속 커뮤니티 요약 참조
       → 관련 원본 text chunk 수집
       → 이 모든 컨텍스트를 LLM에 전달 → 답변 생성

  "벡터 검색 + 그래프 구조 보강"의 하이브리드 방식이다.
  단, 그래프 DB처럼 정확한 N-hop 경로 탐색을 하는 것은 아니므로,
  깊은 영향 체인(V-201→P-101→AHU-01→3층→서버실)은 놓칠 수 있다.

  참고: v0.16+부터 데이터셋 내 다중 파일에 걸친 통합 그래프 구축 지원.
```

### 8.3 Apache AGE — 실제로 뭘 할 수 있나

**Apache AGE** (PostgreSQL 확장, Apache 2.0)

**할 수 있는 것:**
- PostgreSQL 위에서 Cypher 쿼리로 그래프 데이터 CRUD
- 그래프 탐색 (BFS/DFS, 가변 길이 경로)
- 관계형 데이터(테이블)와 그래프 데이터를 하나의 DB에서 동시 사용
- ACID 트랜잭션 (PostgreSQL 상속)
- Azure Database for PostgreSQL에서도 공식 지원

**할 수 없는 것 (TypeDB와의 핵심 차이):**
- **추론 규칙(inference rules) 없음** — 모든 관계를 명시적으로 저장해야 함
- **타입 시스템/서브타이핑 없음** — `pump`가 `equipment`의 하위 타입이라는 걸 DB가 모름
- **온톨로지 수준 스키마 검증 없음** — 데이터 무결성은 애플리케이션 레벨에서 직접 구현

### 8.4 TypeDB vs Apache AGE: 추론의 차이

이것이 **가장 중요한 차이**다.

```
[TypeDB 방식 — 선언적 추론 규칙]

규칙 정의 (한 번만):
  rule transitive-feeds:
    when {
      (supplier: $a, consumer: $b) isa feeds;
      (supplier: $b, consumer: $c) isa feeds;
    } then {
      (supplier: $a, consumer: $c) isa indirect-feeds;
    };

저장된 데이터 (실제 관계 3개만):
  CH-01 --feeds-→ P-101
  P-101 --feeds-→ AHU-01
  AHU-01 --feeds-→ 3F-ZONE

쿼리:
  match (supplier: $ch, consumer: $zone) isa indirect-feeds;
  $ch has tag "CH-01";

결과: CH-01 → 3F-ZONE (자동 추론, 저장하지 않았지만 나옴)
  → 규칙이 쿼리 시점에 자동으로 체인을 따라감
  → 데이터가 바뀌면 추론 결과도 자동으로 바뀜
```

```
[Apache AGE 방식 — 명시적 쿼리]

저장된 데이터 (동일):
  CH-01 --feeds-→ P-101
  P-101 --feeds-→ AHU-01
  AHU-01 --feeds-→ 3F-ZONE

쿼리 (개발자가 직접 작성):
  SELECT * FROM cypher('facility', $$
    MATCH (start {tag: 'CH-01'})-[:FEEDS*1..10]->(downstream)
    RETURN downstream
  $$) AS (result agtype);

결과: P-101, AHU-01, 3F-ZONE (같은 결과)
  → 하지만 "간접 공급"이라는 개념은 DB가 모름
  → 모든 추론 로직을 쿼리 or 애플리케이션에서 직접 구현해야 함
  → 복잡한 조건부 추론(밸브 상태에 따라 경로 변경 등)은 쿼리가 매우 복잡해짐
```

**현실적 비교:**

| 항목 | TypeDB | Apache AGE |
|------|--------|-----------|
| 다단계 경로 탐색 | 규칙으로 자동 | Cypher `*1..N`으로 가능 |
| 조건부 추론 ("밸브 열림 시만") | 규칙에 조건 추가 | 쿼리에 WHERE 조건 직접 작성 |
| 타입 계층 ("pump는 equipment") | 서브타이핑 내장 | 라벨로 흉내, 계층 쿼리 직접 구현 |
| 새로운 추론 패턴 추가 | 규칙 하나 추가 | 쿼리 + 애플리케이션 코드 수정 |
| 개발자 학습 곡선 | **높음** (TypeQL) | **낮음** (SQL + Cypher) |
| 생태계/커뮤니티 | 작음 | **큼** (PostgreSQL 생태계) |
| 운영 안정성 | 보통 | **높음** (PostgreSQL 기반) |

> **솔직한 결론**: 단순 경로 탐색(`*1..N`)만 필요하면 Apache AGE로 충분하다.
> TypeDB의 진짜 가치는 **선언적 추론 규칙** — 하지만 이 가치가 개발/운영 비용을 정당화하는지는 별도 판단 필요.

### 8.5 RAGFlow + Apache AGE 조합의 현실

**핵심 문제: 이 둘은 직접 연결되지 않는다.**

```
[기대한 구조]
RAGFlow ←──연동──→ Apache AGE
  (RAG)               (Graph)

[실제 현실]
RAGFlow (Elasticsearch 내부)     Apache AGE (PostgreSQL 내부)
     │                                  │
     │         연결 없음 ❌              │
     │                                  │
     └──────── FastAPI ────────────────┘
               (직접 만들어야 함)
```

**직접 구현해야 하는 것들:**

| 구현 항목 | 설명 | 난이도 |
|----------|------|:------:|
| RAGFlow → AGE 동기화 | RAGFlow가 추출한 엔티티/관계를 AGE에 저장 | 중 |
| AGE → RAGFlow 피드백 | 그래프 탐색 결과를 RAGFlow 컨텍스트에 주입 | **상** |
| 통합 질의 라우팅 | "이건 벡터 검색" vs "이건 그래프 탐색" 판단 | **상** |
| DWG/DXF 파싱 | RAGFlow가 지원 안 하므로 별도 구현 필요 | 중 |
| 온톨로지 검증 | AGE에는 없으므로 FastAPI에서 직접 구현 | 중 |

### 8.6 4가지 스택 최종 비교

| 항목 | 현재 (TypeDB) | 대안 A (AGE + RAGFlow) | 대안 B (Neo4j + LangChain) | 대안 C (AI-Direct Only) |
|------|:------------:|:---------------------:|:-------------------------:|:---------------------:|
| **그래프 탐색** | 내장 + 추론 | Cypher 쿼리 | Cypher 쿼리 | 불가 |
| **추론 규칙** | **내장** | 직접 구현 | 직접 구현 | AI에 의존 |
| **RAG/벡터 검색** | 별도 구축 필요 | RAGFlow 내장 | LangChain 연동 | VectorDB 직접 |
| **문서 파싱** | 별도 구축 필요 | **RAGFlow 내장** | 별도 구축 | 별도 구축 |
| **DWG 파싱** | 직접 구현 | 직접 구현 | 직접 구현 | 직접 구현 |
| **연동 복잡도** | 중 (단일 DB) | **상** (2개 시스템 연결) | 중 (생태계 성숙) | **하** |
| **개발자 확보** | 어려움 | 보통 | **쉬움** | **쉬움** |
| **라이선스 비용** | 무료 | 무료 | Community 무료 | API 비용만 |
| **운영 복잡도** | 중 | 상 (RAGFlow+PG) | 중 | **하** |
| **MVP 속도** | 느림 | 보통 | **빠름** | **가장 빠름** |

### 8.7 조사 결론

1. **RAGFlow는 문서 파싱 + 벡터 RAG에는 훌륭하고, GraphRAG도 내장되어 있다**
   - GraphRAG 기능(v0.16+)으로 엔티티/관계 추출, 커뮤니티 요약, 벡터+그래프 하이브리드 검색 지원
   - 단, 그래프 DB 수준의 정확한 N-hop 경로 탐색과는 다르며, 깊은 체인은 불확실
   - 외부 그래프 DB와 직접 연결하는 기능은 없음
   - DWG/DXF 파싱은 지원하지 않음

2. **Apache AGE는 충분히 좋은 그래프 DB이지만, TypeDB의 추론을 대체하지는 못한다**
   - 단순 경로 탐색(`MATCH -[:FEEDS*]->`)은 잘 됨
   - 선언적 추론 규칙은 없으므로 복잡한 추론은 애플리케이션에서 구현
   - PostgreSQL 생태계의 안정성은 큰 장점

3. **RAGFlow + AGE 조합은 "레고 블록"이지, "완성품"이 아니다**
   - 둘 사이의 연결(오케스트레이션)을 직접 만들어야 함
   - 이 연결 레이어가 사실상 우리 시스템의 핵심이 됨
   - TypeDB를 쓰든 AGE를 쓰든, **이 오케스트레이션 레이어는 어차피 만들어야 한다**

4. **DWG 파싱은 어떤 조합을 선택해도 직접 구현해야 한다**
   - RAGFlow도, LangChain도, 어떤 RAG 프레임워크도 DWG를 지원하지 않음
   - DWG → DXF 변환 → AI 멀티모달 분석은 피할 수 없는 자체 개발 영역

### 9.1 RAGFlow 한 줄 요약

**오픈소스 RAG 엔진. 문서를 넣으면 파싱 → 벡터화 → 검색 → AI 답변까지 해주는 올인원 플랫폼.**

- GitHub: 48k+ stars (infiniflow/ragflow)
- 라이선스: Apache 2.0 (완전 무료, 상업 사용 가능)
- 최신 버전: v0.20+ (2025.12 기준)

### 9.2 지원 문서 형식과 자동 파싱

**PDF, Word, PPT 자동 파싱 여부: YES** — 전부 자동으로 처리한다.

| 파일 형식 | 자동 파싱 | 파싱 수준 | 비고 |
|----------|:---------:|----------|------|
| **PDF** | **자동** | 레이아웃 + OCR + 테이블 구조 | DeepDoc 엔진 (YOLOv8 기반) |

| **PPTX** | **자동** | 슬라이드별 제목/본문/노트 분리 추출 | 페이지 단위 처리 |
| **XLSX/XLS/CSV** | **자동** | 테이블 구조 인식 (TSR) | 행/열/헤더/셀 병합 인식 |
| **이미지 (JPG/PNG/TIF/GIF)** | **자동** | OCR 텍스트 추출 | 스캔 문서도 가능 |
| **MD/MDX** | **자동** | 마크다운 구조 보존 | |
| **TXT** | **자동** | 순차 청킹 | |
| **HTML** | **자동** | 구조 추출 | |
| **EML** | **자동** | 이메일 파싱 | |
| **JSON** | **자동** | 구조 보존 | |
| **DWG** | **미지원** | - | **우리가 직접 구현해야 함** |
| **DXF** | **미지원** | - | **우리가 직접 구현해야 함** |

### 9.4 DeepDoc 엔진 (핵심 경쟁력)

RAGFlow의 가장 강력한 부분. PDF를 "진짜로" 이해한다.

```
DeepDoc 파이프라인:

PDF 문서 입력
    ↓
[1] DLR (Document Layout Recognition)
    YOLOv8 기반으로 페이지 구조 인식
    → 제목, 단락, 테이블, 이미지, 헤더, 푸터 영역 식별
    ↓
[2] TSR (Table Structure Recognition)
    테이블 내부 구조 분석
    → 행, 열, 헤더, 셀 병합 인식
    → 회전된 테이블 자동 감지 (OCR 신뢰도로 최적 각도 결정)
    ↓
[3] OCR (Optical Character Recognition)
    이미지/스캔 문서에서 텍스트 추출
    ↓
[4] 구조화된 청크 출력
    → 각 청크에 출처(페이지, 위치) 정보 포함
    → 인용(citation) 추적 가능
```

**대안 PDF 파서도 지원:**
| 파서 | 특징 |
|------|------|
| DeepDoc (기본) | OCR + TSR + DLR 풀스택, 시간 오래 걸림 |
| Naive | OCR/TSR/DLR 건너뜀, 텍스트 PDF만 빠르게 |
| MinerU | 오픈소스, FastAPI 원격 API 방식 |
| Docling | IBM 오픈소스, 로컬/원격 모두 가능 |

### 9.9 RAGFlow-Only로 시작하는 MVP 제안

**Phase A: 문서 RAG (즉시 가능)**
1. RAGFlow 설치 (Docker Compose)
2. 준공 문서, 매뉴얼, 보고서 업로드
3. 자동 파싱 + 벡터화
4. 채팅 UI에서 질문/답변 테스트
5. **검증**: "문서 기반 질의응답만으로 고객 가치가 있는가?"

**Phase B: 도면 파이프라인 추가 (2~4주)**
1. DWG → DXF 변환 파이프라인 구현
2. DXF + 이미지 → AI 멀티모달 분석
3. 분석 결과를 RAGFlow에 문서로 저장
4. **검증**: "도면 내용도 질의응답이 되는가?"

**Phase C: 그래프 DB 추가 (필요 시)**
1. 도면에서 추출한 관계를 Apache AGE에 저장
2. 영향도 분석 쿼리 구현
3. RAGFlow + AGE 통합 질의
4. **검증**: "영향도 분석이 실제 고객 가치를 만드는가?"

> **핵심**: Phase A → B → C 순서로 가면, 각 단계마다 "이게 가치가 있는가?" 검증 가능.
> TypeDB 경로는 Phase C를 먼저 만들고 A, B를 뒤에 붙이는 순서였다.
> **순서를 뒤집는 것만으로도 리스크가 크게 줄어든다.**

## 10. 고급 RAG 기법과 적용 전략

> RAGFlow를 기반으로 할 때, 검색 정확도를 높이기 위해 활용할 수 있는 고급 기법들.
> 각 기법의 원리, RAGFlow 내 적용 가능성, 우리 도면 시나리오에서의 효용을 정리.

### 10.1 하이브리드 서치 (Hybrid Search)

**문제**: 설비 태그명 "P-101"과 "P-102"는 의미적으로 거의 같다. 벡터 검색(dense embedding)만 쓰면 둘을 혼동한다.

**원리**: 키워드 검색(sparse/BM25)과 의미 검색(dense embedding)을 결합하는 것.

```
[키워드 검색 (BM25)]              [의미 검색 (Vector)]
"P-101" → 정확히 "P-101" 포함된   "냉수펌프 상태" → 의미적으로
문서만 반환                        관련된 모든 펌프 문서 반환
         ↓                                  ↓
         └──────── 결합 + 점수 가중 ────────┘
                          ↓
                  최종 검색 결과 (정확성 + 관련성)
```

**RAGFlow 적용**:
- RAGFlow는 Elasticsearch/Infinity 백엔드로 **하이브리드 서치를 기본 지원**
- 설정에서 키워드(BM25) 가중치와 벡터 가중치 조절 가능
- 추가 개발 없이 활성화 가능

**우리 시나리오에서의 효용**: **매우 높음**
- 설비 태그(P-101, V-201, CH-01)는 키워드 매칭이 필수 — 의미 검색만으로는 혼동
- 운영 매뉴얼의 절차 설명은 의미 검색이 유리
- 두 방식을 합치면 "정확한 설비 + 관련 맥락" 모두 검색 가능

### 10.2 컨텍스트얼 리트리벌 (Contextual Retrieval)

**문제**: 도면이나 문서를 청크로 자르면 맥락이 사라진다.

```
원본: "3층 기계실 냉수 계통도 — 밸브 V-201을 잠그시오"
청크: "밸브를 잠그시오"  ← 어떤 밸브? 어느 층? 무슨 계통?
```

**원리** (Anthropic 제안, 2024): 각 청크 앞에 문서 전체 맥락을 LLM으로 요약해서 붙여넣는 것.

```
[기존 청크]
"밸브를 잠그시오"

[Contextual Retrieval 적용 후]
"[3층 기계실 냉수 계통도의 운전 절차 중 긴급 정지 항목]
 밸브를 잠그시오"
```

**효과**: Anthropic 발표 기준 검색 실패율 49% 감소, 리랭커 병행 시 67% 감소.

**RAGFlow 적용**:
- RAGFlow의 DDU(Deep Document Understanding)가 문서 레이아웃을 인식하므로, 상위 제목/섹션 정보를 자동으로 유지하는 기본 기능이 있음
- 하지만 Anthropic 방식의 "LLM으로 각 청크에 맥락 요약 추가"는 **별도 커스텀 스크립트** 필요
- RAGFlow의 수집 파이프라인(Ingestion Pipeline)이나 에이전트 워크플로우에서 전처리 단계로 구현 가능

**우리 시나리오에서의 효용**: **매우 높음**
- 도면 텍스트는 특히 맥락 손실이 심함 (태그명만 있고 소속 계통이 없으면 무의미)
- "이 청크는 3층 기계실 냉수 계통도의 P&ID에서 추출됨" 같은 접두사가 검색 정확도를 크게 높임
- 단, 모든 청크에 LLM을 호출하므로 **토큰 비용 증가** (대량 도면 시 고려 필요)

### 10.3 리랭커 (Reranker)

**문제**: 1차 검색 결과에 관련 없는 문서가 섞여 있으면, LLM이 잘못된 정보로 답변을 생성한다.

**원리**: 1차 검색(벡터/키워드)으로 후보 N개를 뽑고, 리랭커 모델이 질문과 각 후보의 **관련성을 다시 평가**하여 순위를 재정렬.

```
1차 검색 결과 (Top 20):
  1. "P-101은 냉수 펌프이다" (관련)
  2. "P-101 유지보수 일정" (부분 관련)
  3. "P-102 사양서" (무관 — 태그 유사해서 딸려옴)
  ...

리랭커 적용 후 (Top 5):
  1. "P-101은 냉수 펌프이다" ← 관련도 0.95
  2. "P-101이 AHU-01에 냉수 공급" ← 관련도 0.89
  3. "P-101 유지보수 일정" ← 관련도 0.72
```

**대표 모델**:
| 모델 | 특징 | 라이선스 |
|------|------|---------|
| **BGE-Reranker-v2-m3** | 다국어 지원, 600M 파라미터, 50~100ms GPU | Apache 2.0 (무료) |
| Jina Reranker v2 | 고성능, 긴 문서 지원 | Apache 2.0 |
| Cohere Rerank | API 기반, 높은 정확도 | 유료 API |

**RAGFlow 적용**:
- RAGFlow는 BGE-Reranker-v2-m3를 **기본 리랭커로 내장**
- 설정에서 활성화 가능, 로컬 GPU에서 구동

**우리 시나리오에서의 효용**: **높음**
- "CH-01 고장 시 영향"이라는 질문에 대해, 단순 언급이 아닌 실제 '연결 관계'가 설명된 청크를 상위로 올림
- 하이브리드 서치 + 리랭커 조합으로 MRR(Mean Reciprocal Rank) 약 +9~10% 개선 (벤치마크 기준)

### 10.4 레이트 청킹 (Late Chunking)

**문제**: 전통적 RAG는 문서를 먼저 자르고(chunk) 각 청크를 따로 임베딩한다. 잘린 순간 맥락이 사라진다.

**원리** (Jina AI 제안): **문서 전체를 먼저 임베딩하고, 그 다음에 자른다.**

```
[전통 방식] 자르고 → 임베딩
  문서 → [청크1] [청크2] [청크3] → 각각 따로 임베딩
  → 청크2의 "그것"이 뭘 가리키는지 임베딩이 모름

[레이트 청킹] 임베딩 → 자르기
  문서 → 전체를 한 번에 토큰화 + 트랜스포머 통과
       → 각 토큰이 전체 문맥을 아는 상태에서 임베딩 생성
       → 그 다음에 청크 경계로 자름
  → 청크2의 "그것"이 가리키는 대상을 임베딩이 알고 있음
```

**요구 사항**: 긴 컨텍스트를 지원하는 임베딩 모델 필요 (예: jina-embeddings-v2, 8192 토큰)

**RAGFlow 적용**:
- RAGFlow의 DDU가 문서 전체 레이아웃을 먼저 파악한 뒤 청킹하므로, **구조적 맥락은 어느 정도 보존**
- 하지만 Jina AI가 제안한 "임베딩 후 청킹"과는 다른 접근 — RAGFlow는 "레이아웃 인식 후 청킹 후 임베딩"
- 진정한 레이트 청킹을 쓰려면 임베딩 모델을 jina-embeddings-v2 등으로 교체하고 커스텀 파이프라인 필요

**우리 시나리오에서의 효용**: **중간**
- 도면에서 추출한 텍스트는 보통 짧은 태그명 + 사양이라 맥락 단절 문제가 적음
- 준공 문서/매뉴얼 같은 긴 문서에서는 효과 있을 수 있음
- 우선순위는 하이브리드 서치, 컨텍스트얼 리트리벌, 리랭커보다 낮음

### 10.5 에이전틱 RAG (Agentic RAG)

**문제**: 복잡한 질문은 한 번의 검색으로 답할 수 없다.

```
질문: "CH-01 냉동기 고장 시 3층에 미치는 영향과 대응 절차를 알려줘"

→ 이건 두 가지를 합쳐야 답변 가능:
  1. CH-01의 영향 범위 (도면/관계 데이터에서)
  2. 대응 절차 (운영 매뉴얼에서)
```

**원리**: AI 에이전트가 **여러 도구를 스스로 선택**하여 검색하고, 결과를 조합해서 답변.

```
에이전트 사고 과정:

1. "CH-01 영향 범위" → [도구: 도면 DB 검색] → 영향 설비 목록
2. "대응 절차" → [도구: 매뉴얼 검색] → 긴급 대응 매뉴얼
3. 두 결과를 합쳐서 최종 답변 생성
```

**플랫폼 선택지**:

| 플랫폼 | 특징 | RAGFlow와의 관계 |
|--------|------|-----------------|
| **RAGFlow 에이전트** (v0.20+) | 내장 에이전트 빌더, MCP 지원 | 자체 내장 |
| **Dify** | 범용 에이전트 워크플로우 빌더, 70k+ GitHub stars | RAGFlow를 MCP로 호출 가능 |
| **LangChain/LangGraph** | 코드 기반, 가장 유연 | RAGFlow API 연동 |

> **Dify와 RAGFlow의 관계**:
> - Dify = 에이전트 오케스트레이션에 강함 (UI, 워크플로우, 다양한 LLM 연동)
> - RAGFlow = 문서 파싱과 검색 품질에 강함 (DDU, 하이브리드 서치)
> - **조합 전략**: RAGFlow가 "지식 엔진" 역할, Dify가 "에이전트 오케스트레이션" 역할
> - RAGFlow가 MCP Server로 동작 가능하므로, Dify 에이전트가 RAGFlow를 도구로 호출 가능

**우리 시나리오에서의 효용**: **매우 높음**
- "도면 DB에서 관계 찾기" + "매뉴얼에서 절차 찾기"를 조합하는 질문이 핵심 유스케이스
- Phase B 이후에 에이전트가 도면 데이터와 문서 데이터를 넘나들며 답변 생성

### 10.6 고급 기법 적용 우선순위

| 우선순위 | 기법 | RAGFlow 내 구현 난이도 | 효과 | 비고 |
|:--------:|------|:---------------------:|:----:|------|
| **1** | 하이브리드 서치 | **낮음** (설정만) | 높음 | 즉시 적용 |
| **2** | 리랭커 | **낮음** (BGE 내장) | 높음 | 즉시 적용 |
| **3** | 컨텍스트얼 리트리벌 | **중간** (커스텀 스크립트) | 매우 높음 | 도면 청크에 특히 효과적 |
| **4** | 에이전틱 RAG | **중간** (RAGFlow 에이전트 or Dify) | 매우 높음 | Phase B 이후 |
| **5** | GraphRAG | **중간** (RAGFlow 내장) | 높음 | 엔티티/관계 추출 |
| **6** | 레이트 청킹 | **높음** (임베딩 모델 교체 필요) | 중간 | 우선순위 낮음 |

### 10.7 최종 아키텍처 조합

```
[Knowledge Studio 아키텍처]

┌─────────────────────────────────────────────────────────────┐
│  에이전트 오케스트레이션 (RAGFlow Agent 또는 Dify)           │
│  - 질문 분석 → 적절한 도구 선택 → 결과 조합 → 답변 생성     │
└──────────┬──────────────────────────┬───────────────────────┘
           │                          │
    ┌──────▼──────┐           ┌───────▼───────┐
    │  RAGFlow    │           │  그래프 DB    │
    │  (지식 엔진)│           │  (관계 엔진)  │
    │             │           │               │
    │ 문서 파싱   │           │ Apache AGE    │
    │ DDU/OCR     │           │ or TypeDB     │
    │ 하이브리드  │           │               │
    │  서치       │           │ 영향도 분석   │
    │ GraphRAG    │           │ 경로 탐색     │
    │ 리랭커      │           │               │
    └──────┬──────┘           └───────┬───────┘
           │                          │
    ┌──────▼──────┐           ┌───────▼───────┐
    │ 문서 저장소  │           │ 도면 파이프라인│
    │             │           │ (자체 구현)   │
    │ PDF, Word   │           │               │
    │ PPT, Excel  │           │ DWG → DXF     │
    │ (자동 파싱) │           │ AI 멀티모달   │
    │             │           │ OCR 보강      │
    │             │           │ 교차검증      │
    └─────────────┘           └───────────────┘
```

> **이 아키텍처에서 RAGFlow가 담당하는 것**: 문서 파싱, 벡터화, 하이브리드 서치, 리랭킹, GraphRAG, 에이전트
> **우리가 직접 만드는 것**: DWG/DXF 파이프라인, 그래프 DB 연동, 오케스트레이션 레이어
> **선택적으로 추가하는 것**: Dify (에이전트 UI), Apache AGE/TypeDB (영향도 분석)
