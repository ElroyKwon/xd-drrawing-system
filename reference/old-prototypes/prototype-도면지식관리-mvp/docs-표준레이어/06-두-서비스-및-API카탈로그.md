# 06. 두 서비스 요약 + 플랫폼 외부 API 카탈로그

> **본 문서의 위상**
> [[05-서비스-정의]]의 플랫폼-서비스 2계층을 **실제 제품 2개 + 외부 API**로 구체화한 중간 설계.
> 상위: 05 플랫폼 아키텍처
> 하위: [[07-서비스1-도면관리-상세설계]] / [[08-서비스2-AI인사이트-상세설계]]

> **작성 목적**
> 영업·투자·파트너에게 "DKS가 뭘 파는가"를 **한 문서로** 설명.
> 동시에 서비스 1·2 상세 설계 문서의 상위 정합성 기준점.

> **한 줄 요약**
> *"DKS 플랫폼 위에 도면 관리와 AI 인사이트 두 서비스가 붙습니다. 외부 로봇·ERP·MES·AI는 플랫폼 API를 통해 맥락을 받아갑니다."*

## §2. 서비스 1 — 도면 관리

### §2.1 한 줄 정체성
> *"도면이 파일이 아니라, 질문 가능한 살아있는 지식이 됩니다."*

### §2.2 기존 EDMS 대비 진짜 가치 — 8대 기능

| # | 기능 | 기존 EDMS | 플랫폼 기반 |
|:-:|------|----------|-------------|
| 1 | 자연어 도면 검색 | 파일명·폴더 | "B동 3층 냉수 배관 평면도 Rev.3" 즉시 |
| 2 | 자동 의미 추출 | 수동 메타 입력 | 업로드 즉시 Phase 0 자동 분석 |
| 3 | 설비↔도면 양방향 | 수동 태그 | "펌프 P-101 나오는 모든 도면" 자동 |
| 4 | 변경 영향도 | 없음 | "이 도면 수정 → 영향 설비·부서" 자동 |
| 5 | 공사 이벤트 재분석 | 수동 갱신 | 새 도면 → 차이 감지 → 변경 통지 |
| 6 | 도곽·표준 자동 변환 | 없음 | 외부 업체 도면 → 사내 표준 자동 |
| 7 | Multi-hop 이력 | 버전 번호만 | 설비 교체·도면 변경 체인 추적 |
| 8 | 온톨로지 연결 주석 | 텍스트 주석 | 주석 = 검색·추론 대상 지식 |

### §2.3 타겟
- 1순위: 공장형 **시설관리 책임자 + 실무자** / 데이터센터형 **통합 인원**
- 2순위: 공장장·운영 총괄 (결재·감사)
- 협력자: 외부 도면 작성 업체 (표준 변환 채널)

### §2.4 플랫폼 의존
- Phase 0 에이전틱 (도면 자동 분석)
- 온톨로지 (양방향 태깅·영향도)
- RAG (자연어 검색)
- 변경 감지 파이프라인

### §2.5 상세 설계 진입점
→ [[07-서비스1-도면관리-상세설계]]

## §3. 서비스 2 — AI 인사이트

### §3.1 한 줄 정체성
> *"숫자가 아니라 답을, 답이 아니라 근거 붙은 지식을 드립니다."*

### §3.2 기존 대시보드·챗봇 대비 진짜 가치 — 8대 기능

| # | 기능 | 기존 도구 | 플랫폼 기반 |
|:-:|------|----------|-------------|
| 1 | 근거 병기 답변 | ChatGPT: 근거 없음 | 답 + 도면 페이지 + 매뉴얼 단락 즉시 링크 |
| 2 | 알람 맥락 답변 | BMS: 코드만 | 알람 → 설비 → 도면 → 영향 30초 |
| 3 | Multi-hop 근본원인 | 대시보드: 숫자 | "이 고온 진짜 원인" 관계 체인 따라 추적 |
| 4 | Dual Engine 보고서 | 수동 작성 | 수치(Python 정확) + 해석(SLM 자연어) |
| 5 | 유사 사례 검색 | 사람 기억 | "과거 5번 발생. 이렇게 해결" 자동 |
| 6 | 협력업체 브리핑 | 사람이 준비 | 방문 전 도면·이력·제약 패키지 자동 |
| 7 | 인수인계 자동 요약 | 수동 인계장 | 당직 이벤트 → 3줄 요약 |
| 8 | 예측 + 설명 | 블랙박스 | "베어링 3개월 내 교체" + 판단 근거 |

### §3.3 타겟
- 1순위: 공장형 **분야 책임자·실무자** / 데이터센터형 **통합 인원**
- 2순위: 공장장·운영 총괄 (월간 리포트)
- 외부 소비자: 협력업체 (브리핑 수신)

### §3.4 플랫폼 의존
- 온톨로지 (영향도·근본원인)
- RAG (근거 검색)
- Dual Engine ([[SLM 인사이트 분석]] 이식)
- 추론 엔진 (Multi-hop)

### §3.5 상세 설계 진입점
→ [[08-서비스2-AI인사이트-상세설계]]

## §4. 두 서비스 시너지 전략

### §4.1 시너지 구조

```
[서비스 1 도면 관리]                [서비스 2 AI 인사이트]
 사용자가 도면을               →    운영·알람 데이터를
 찾고·편집·공유·태깅                 분석·해석·답변
       ↓                                   ↑
       └──── 학습 데이터 축적 ─────────┘
                     │
                     ▼
              [플랫폼 온톨로지]
              (지식 복리 성장)
```

### §4.2 단독 vs 번들 진입 전략

| 진입 유형 | 설명 | 추천 |
|----------|------|:---:|
| **단독 서비스 1** | 도면 관리만 | FMS 좌절 고객 |
| **단독 서비스 2** | AI 인사이트만 | 챗봇 실패 고객 |
| **번들 (추천)** | 둘 다 | PoC 성공 후 자연 업셀 |

### §4.3 번들 락인 효과
- 한 서비스 사용 → 플랫폼 자동 구축 → 두 번째 서비스 추가 비용 적음
- 두 서비스 동시 사용 → 학습 데이터 복리 → 경쟁사 따라 하려면 *플랫폼부터* 만들어야

## §5. 로봇 API 카탈로그

### §5.1 핵심 8 API

| API | 메서드 | 용도 | 응답 예시 |
|-----|:---:|------|---------|
| `/location/resolve` | GET | 좌표 → 의미 위치 | `(12.3, 8.7) → "B동 3라인 컨베이어"` |
| `/object/identify` | POST | 이미지 → 설비 ID | 부품 박스 이미지 → `PART-A-372` |
| `/safety/zone` | GET | 좌표 → 안전 영역 | 금지·회피·고압 구역 |
| `/equipment/{id}/neighbors` | GET | 주변 설비·관계 | 펌프 주변 밸브·배관 |
| `/process/next-step` | POST | 상태 → 다음 공정 | 다음 스테이션 ID + 경로 |
| `/route/plan` | POST | 경로 + 제약 | 금지 구역 반영 최단 경로 |
| `/context/query` | POST | 자연어 컨텍스트 | "이 설비 가동 중?" |
| `/events/subscribe` | STREAM | 실시간 변경 | 설비 상태 변경 푸시 |

### §5.2 사용 맥락
- 휴머노이드·자율 AMR·AGV가 공장·건물 맥락을 이해하기 위해 호출
- SLAM 좌표계와 공장 의미 좌표계의 **의미 변환**이 핵심
- 안전 영역 + 다음 공정 정보로 **자율 행동 의사결정** 지원

### §5.3 표준 포맷 제안
- JSON-LD (의미 그래프 호환)
- MCP 도구화 (외부 AI가 직접 사용 가능)

## §6. ERP API 카탈로그

### §6.1 핵심 7 API

| API | 메서드 | 용도 |
|-----|:---:|------|
| `/asset/{id}/bom` | GET | 설비 BOM 계층 + 도면 근거 |
| `/asset/{id}/lifecycle` | GET | 도입일·감가·교체 예상 |
| `/maintenance/history/{id}` | GET | 정비 이력 + 비용 + 업체 |
| `/vendor/{id}/performance` | GET | 협력업체 성과 누적 |
| `/facility/space/{id}/utilization` | GET | 공간 사용·임차·자산 맵 |
| `/cost/impact` | POST | 변경·교체 TCO 영향 |
| `/compliance/{id}` | GET | 규정 준수 + 근거 문서 |

### §6.2 사용 맥락
- SAP·Oracle·더존·영림원 등 ERP가 DKS 플랫폼에서 **설비 맥락·이력·비용·규정** 데이터 받음
- ERP의 빈 필드(자산 실제 상태·정비 이력 맥락)를 DKS가 채움
- **양방향 가능**: ERP → DKS (계약·비용 주입), DKS → ERP (자산 상태 주입)

### §7.1 핵심 7 API

| API | 메서드 | 용도 |
|-----|:---:|------|
| `/equipment/{id}/capabilities` | GET | 설비 수행 가능 공정 |

- MES의 "왜?" 질문에 DKS가 답변 (근본 원인 Multi-hop)

## §8. MCP 인터페이스 — 외부 AI용

### §8.1 MCP 도구 5종

| Tool | 기능 |
|------|------|
| `dks.query.entity` | 엔티티 속성·관계 조회 |
| `dks.query.natural` | 자연어 질의 |
| `dks.search.document` | 문서·매뉴얼 검색 + 근거 |
| `dks.infer.impact` | 영향도 추론 (Multi-hop) |
| `dks.event.subscribe` | 변경 이벤트 구독 |

### §8.2 사용 시나리오
- 고객사 자체 Claude·GPT·로컬 LLM이 자사 공장·건물에 대해 답할 수 있도록 맥락 공급
- "우리 공장 전문가 AI" 자체 구축하려는 기업에게 핵심 맥락 자원 제공
- *빅테크에 종속되지 않는 자체 AI* 구현의 진입 장벽 해소

### §8.3 가격·접근 정책 *[가설]*
- 기본 요청 수 포함 (월 구독)
- 초과분 토큰 기반 크레딧 과금
- 사이트별 토큰·권한 분리

## §9. "진짜 가치" 5대 판정 기준

두 서비스가 동시에 통과해야 시장 살아남음.

| # | 기준 | 서비스 1 | 서비스 2 | API |
|:-:|------|:---:|:---:|:---:|
| 1 | 기존 도구로 절대 못 함 | ✅ 자연어 검색·양방향 | ✅ 근거 병기·Multi-hop | ✅ 의미 변환 |
| 2 | 30초 가치 경험 | ✅ 검색·즉시 답 | ✅ 알람 맥락 30초 | ✅ API 응답 < 500ms |
| 3 | 근거·출처 병기 | ✅ 도면 직접 링크 | ✅ 페이지·단락 링크 | ✅ 응답에 근거 필드 |
| 4 | 일일 사용 5회+ | 유도 | 유도 | 로봇은 일 수만 |
| 5 | 외부·다른 서비스 연결 | ✅ API 노출 | ✅ API 노출 | ✅ 자체 목적 |

→ 설계 원칙은 5개 모두 통과할 때까지 반복 수정.
