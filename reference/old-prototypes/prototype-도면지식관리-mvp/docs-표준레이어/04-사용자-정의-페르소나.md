# 04. 사용자 정의 페르소나 — 실 데이터 기반 재분석

> **작성 배경**
> [[03-운영자-3대-페르소나]]는 사용자의 짧은 힌트를 제가 *기능 축*(시설/모니터링/평가)으로 추론·증폭한 가설. 2026-04-17 사용자 5문 답변으로 **조직 축이 다름**(분야×직급×현장타입)이 확인되어 신규 작성.

> **결론 한 줄**
> 운영자 = **통합 역할**(한 사람이 운영·도면·알람·보고·평가·히스토리 다 함). 현장 타입 2종(공장형·데이터센터형)으로 조직 구조·외부 의존도 다름. 페르소나 정의 = **5대 실제 불만**의 구조화.

## §0. 이 문서의 데이터 근거 — 실 데이터 vs 가설

| 출처 | 내용 | 상태 |
|------|------|:---:|
| 사용자 5문 답변 (2026-04-17) | 만난 사람·현장·조직·결재·불만 | 실 데이터 |
| 영업·내부 판단 | 시설 운영자·유지보수 책임자·실행자에 가치 | 내부 직관 |
| 실 타겟 인터뷰 | 없음 | ❌ 부재 |
| 본 문서 §3·§4 구조화 | 실 데이터 기반 재구성 | 추론 (근거 표기) |

**원칙**: 추론 부분은 각 섹션에서 *[추론]* 로 표시. 실 데이터는 표기 없음.

## §1. 03의 한계 — 왜 기능 축이 빗나갔나

| 03이 쓴 것 | 실제 현장 |
|-----------|---------|
| A 시설관리자 (도면·공간) | 한 사람 안의 *모자 1* |
| B 모니터링·분석가 (알람·보고서) | 한 사람 안의 *모자 2* |
| C 운영평가·자산 (KPI·업체 평가) | 한 사람 안의 *모자 3* |
| 3인 분리 | **1인 통합** |

### 실제 조직 축
```
분야 (전기·기계·건물·생산설비)  ← 수평
    ×
직급 (공장장·책임자·실무자)       ← 수직
    ×
현장 타입 (공장형·데이터센터형)    ← 범주
```

→ 한 사람 = (특정 분야 + 특정 직급 + 특정 현장 타입)의 교차점. 그 사람이 A·B·C 모자를 모두 쓴다.

## §2. 실 데이터 핵심 요약

| 항목 | 내용 |
|------|------|

## §3. 현장 타입 분리 — 공장형 vs 데이터센터형

### §3.1 공장형 (LS일렉트릭 청주)

**조직 모델**:
```
공장장 (통합 결재 + 생산·유지보수 총괄)
  ├─ 각 건물 유지보수 책임자
  ├─ 전기 책임자
  ├─ 유틸리티·기계 책임자
  ├─ 생산설비 책임자
  └─ 각 책임자 밑 실무자 조직
```

**특징**:
- **분야별 깊은 전문성** — 각 책임자가 자기 분야 장기 전담
- 내부 역량 중심, 외주는 특수 작업만
- 도면·히스토리 관리도 분야별 분리
- *[추론]* 분야 간 정보 전달은 공장장 거쳐 흐를 가능성 → 불만 b (변경 공유 안 됨)의 구조적 원인

### §3.2 데이터센터형 (LG-CNS 제주)

**조직 모델**:
```
운영 총괄
  ├─ 분야 구분 (존재하나 느슨)
  ├─ 통합 유지보수 인원 풀
  │   └─ 분야 순환 점검·기록·확인
  └─ 협력업체 전문가 (이슈 발생 시 호출)
```

**특징**:
- **얕고 넓은 내부 전문성 + 깊은 외부 전문성**
- 통합 인원이 [[CRAH]]·[[UPS]]·전력·냉방 전반 순회
- 낯선 분야·드문 이슈 = 협력업체 호출
- 외부 의존도 매우 높음

### §3.3 타입별 DKS 가치 차이

| 가치 포인트 | 공장형 | 데이터센터형 |
|-----------|--------|-------------|
| 자기 분야 깊은 인사이트 | ★★★ | ★ |
| 낯선 분야 즉답 | ★ | ★★★ |
| 협력업체 정보 공유 게이트웨이 | ★ | ★★★ |
| 도면·알람·영향도 통합 | ★★★ | ★★★ |
| 보고서 자동 초안 | ★★ | ★★★ |

→ **같은 DKS 코어, 두 타입에 다른 모듈 구성**으로 제공 필요.

## §4. 직급·분야·역할 매트릭스

### §4.1 공장장 (통합 결재자)

- 역할: 생산관리 + 유지보수 총괄 + 예산·계약 결재
- 시간: 일일 결재 1~3건, 주간 총괄 회의
- DKS 사용: **직접 사용 낮음**, 보고 받기 형태
- DKS 가치: *월간 요약 + 예외 알림 + 예산 집행 인사이트*
- 결재 근거: ROI 명확성 (불만 c·d·e의 시간 절감 정량화)

### §4.2 분야 책임자 (공장형 핵심)

- 역할: 자기 분야 운영·도면·알람·보고·평가·협력업체 관리 통합
- 시간: 일일 5~50회 조회, 주 보고서 1건, 월 KPI 1건
- DKS 사용: **일일 핵심 사용자**
- DKS 가치:
  - 자기 분야 알람 즉답 (불만 d)
  - 도면 변경 자동 반영·통지 (불만 a·b)
  - 보고서 초안 자동 (불만 c)
  - 협력업체 SLA 추적 (불만 e)

### §4.3 통합 유지보수 인원 (데이터센터형 핵심)

- 역할: 분야 순환 점검·기록·확인 + 이슈 판단 + 협력업체 호출 판단
- 시간: 순회 중 일일 수십 건 조회
- DKS 사용: **가장 빈도 높음 — 낯선 것 즉답 수요**
- DKS 가치:
  - 처음 보는 설비·알람 30초 이해 (불만 d·f)
  - 협력업체 호출 전 브리핑 패키지 자동 생성 (불만 e)
  - 점검 기록 자동 구조화

### §4.4 실무자 (각 책임자 밑)

- 역할: 현장 점검·정비·기록
- 시간: 일일 현장 활동 중심, 모바일 사용 많음
- DKS 사용: **모바일 의존 높음**
- DKS 가치:
  - 현장 즉석 도면·매뉴얼 조회
  - QR·음성 입력
  - 오프라인 캐시

### §4.5 협력업체 전문가 (외부, 데이터센터형 결정적)

- 역할: 호출되어 와서 문제 해결
- 시간: 방문 1~8시간, 연 수십 회
- DKS 사용: **간접 소비자** — 브리핑 받는 쪽
- DKS 가치:
  - 방문 전 현장 맥락 브리핑 자동 수신
  - 관련 도면·이력·제약 사항 일괄 전달
  - 작업 후 보고서 자동 초안 (재방문·분쟁 방지)

→ *[추론]* 협력업체 생태계 자체가 DKS 사용자. B2B2B 구조. 장기적으로 **§4 모델 ③ Context API**의 첫 자연 소비자.

### §4.6 도면 작성자 (겸직 + 외부 업체 양쪽)

**운영자 겸직 측**:
- 공사 후 변경사항 직접 수정·히스토리 관리
- 전담 인력 아니라 책임자·실무자가 짬내서 함
- → 불만 a의 원인: "변경 다 반영 못 함"

**외부 업체 측**:
- 사용자가 실제 만난 사람 (도곽 표준 재작성 요청)
- DKS Phase 0 에이전틱 = 도곽·레이어 자동 정규화 가능
- → *[추론]* 별도 서비스 채널 가능성 (외부 도면 작성 업체용 Phase 0 SaaS)

## §6. 5대 실제 불만 → 페르소나의 일상

각 불만 = 페르소나의 반복 고통 = DKS의 핵심 가치 포인트.

### 불만 a — 도면 최신화 안 됨

| 항목 | 내용 |
|------|------|
| 언제 | 세분화 공사 후 (유지보수·업그레이드) |
| 누가 | 여러 부서 + 협력업체 시공자 |
| 왜 안 됨 | 책임 분산 + 갱신 동기·시간 부재 + 표준 도곽 맞춤 비용 |
| 영향 | 이후 운영자가 *잘못된 도면*으로 판단 → 사고 위험 (불만 d·f로 연쇄) |
| DKS 대응 | 공사 완료 이벤트 → Phase 0 에이전틱 자동 재분석 → HITL 승인 플로우 |

### 불만 b — 변경 공유·알림 없음

| 항목 | 내용 |
|------|------|
| 언제 | 설비·배관·배선 변경 후 |
| 누가 | 변경 모르는 타 분야 책임자·실무자·협력업체 |
| 왜 안 됨 | 통보 채널 미정 + "누구에게 알려야 하는지" 모름 |
| DKS 대응 | 변경 설비 → [[온톨로지]] 1-hop 영향 → 관련 분야 책임자 자동 통지 |

### 불만 c — 보고서 쓰기 싫음

| 항목 | 내용 |
|------|------|
| 언제 | 일·주·월 반복 |
| 누가 | 책임자·실무자 |
| 왜 싫음 | 같은 데이터 매번 수동 정리 + 양식 반복 |
| DKS 대응 | [[SLM 인사이트 분석]] Dual Engine 초안 자동 (수치=Python, 해석=SLM) |

### 불만 d — 알람 많은데 다 모름

| 항목 | 내용 |
|------|------|
| 언제 | 새 알람·드문 알람·야간 당직 |
| 누가 | 실무자·통합 유지보수 인원 |
| 왜 안 됨 | 매뉴얼 분산 + 경험 부족 + 시간 부족 |
| DKS 대응 | 알람 → 설비 → 도면 → 영향 설비 30초 답변 ([[02-기능-경험-요건]] §4 시나리오 1) |

### 불만 e — 협력업체 대응 느림

| 항목 | 내용 |
|------|------|
| 언제 | 외주 호출 시 (데이터센터형 핵심) |
| 왜 느림 | 현장 이해 + 도면 수령 + 설명에 시간 소요 |
| DKS 대응 | 협력업체 방문 전 브리핑 패키지 자동 생성 (도면·이력·제약) + 방문 후 보고서 자동 초안 |

### 불만 f — 챗봇 써도 현장 설비 제대로 답 못함 ★★★

| 항목 | 내용 |
|------|------|
| 언제 | 운영자가 ChatGPT·Claude 등 일반 AI에 질문 |
| 왜 안 됨 | 일반 AI는 **현장 도면·온톨로지·운영 이력 없음** |
| DKS 대응 | **이게 DKS의 존재 이유 그 자체** |
| 시장 의미 | 고객이 이미 일반 챗봇 경험 → 실패 체험 → 공백 자명 |

## §7. 불만 f의 결정적 의미 — 시장 공백 실증

### 일반 영업 상황
> 영업: "AI 맥락 엔진 어쩌고…"
> 고객: "무슨 소리?"

### DKS 영업 상황 (불만 f 활용)
> 영업: "ChatGPT에 우리 공장 펌프 물어보셨죠? 답 제대로 나왔나요?"
> 고객: "아, 그거 엉뚱한 소리만…"
> 영업: "맞습니다. 일반 AI는 *이 공장의 도면·설비·이력*을 모르니까요. DKS는 그 맥락을 AI에게 붙여줍니다."

→ **시장 공백이 말로 증명될 필요 없음**. 고객이 *이미 실패를 경험*함.
→ [[01-의미레이어-회사-전환]] §3 "맥락 엔진" 포지셔닝의 실증 근거.

## §8. FMS 한계 = DKS 진입 메시지

### 사용자 원문
> "FMS 시스템을 도입하고 싶어도 각 공장이나 건물의 요구사항이 달라서 모두 수용하기 어려워."

### 비교

| 항목 | 기존 [[FMS]] | DKS |
|------|-------------|-----|
| 초기 모델링 | **사람이 마스터 데이터 수작업 입력** (6~18개월) | Phase 0 에이전틱이 도면·문서 자동 분석 (4~8주) |
| 현장 맞춤 | 패키지 표준 → 커스터마이징 비용 폭증 | 현장 도면 자체가 모델 → 자동 맞춤 |
| 업데이트 | 수동. 공사 후 입력 누락 일상화 | 공사 이벤트 → 재분석 자동 트리거 |
| 도면 연결 | 파일 첨부 수준 | 도면 의미 = 시스템 본체 |

→ **"FMS 도입 좌절" = DKS의 가장 강력한 시장 진입 레퍼런스**.

## §9. 현장 타입별 MVP 차별화

### §9.1 공장형 MVP 후보

**타겟**: 분야 책임자 1명 + 그 밑 실무자 2~3명
**분야 선택 기준**: 알람 빈도 + 변경 빈도 높은 분야 (보통 전기 또는 기계)
**4주 PoC 구성**:
- 1주차: 해당 분야 도면·매뉴얼 Phase 0 분석
- 2주차: BMS/SCADA 알람 태그 매핑 (샘플 100개)
- 3주차: 웹 UI — 알람→설비→도면→영향 응답
- 4주차: 책임자·실무자 일일 사용 측정 + 보고서 초안 기능 추가

**측정 지표**:
- 일일 쿼리 수 / 사용자
- 응답 만족도 (5점)
- 알람 1차 대응 시간 변화
- 보고서 작성 시간 변화

### §9.2 데이터센터형 MVP 후보

**타겟**: 통합 유지보수 인원 2~3명 + 협력업체 1곳
**핵심 기능 추가**:
- 낯선 분야·설비 30초 즉답 (공장형보다 *얕고 넓게*)
- **협력업체 브리핑 패키지 자동 생성** (공장형 MVP에 없는 차별)
- 방문 후 작업 보고서 자동 초안

**측정 지표**:
- 공장형 4종 동일
- 추가: 협력업체 방문 전 준비 시간 / 방문 후 보고 시간

→ 두 MVP는 **공통 백엔드 + 서로 다른 표현 모듈**로 구현 가능 ([[02-기능-경험-요건]] §8 원칙).

## §10. 독립 서비스 후보 — 불만 기반

| 후보 | 근거 불만 | 타겟 | 진입 저항 | 독립 추천도 |
|------|---------|------|:---:|:---:|
| 알람 맥락 엔진 | d + f | 책임자·실무자·통합 인원 | 낮음 | ★★★ |
| 도면 자동 최신화 + 변경 알림 게이트웨이 | a + b | 시설관리 부서·책임자 | 중 | ★★ |
| 보고서 자동 초안 SaaS | c | 책임자·실무자 전원 | 낮음 | ★★ |
| 협력업체 브리핑·평가 시스템 | e | 공장장·총괄 + 협력업체 | 중 | ★★ (데이터센터형 우선) |
| 외부 도면 업체용 Phase 0 SaaS | 도곽 표준화 요청 | 설계·시공 업체 | 낮음 | ★ (별도 채널) |

→ **1순위 = 알람 맥락 엔진** — 불만 d·f 동시 해결 + 일일 사용 빈도 최고 + 시장 공백 실증.

## §11. 인터뷰 부재 상황의 대안 플랜

사용자 지시: (i)~(v) 모두 활용 가능.

### (i) 사내 영업·운영자 간접 증언 정리

**양식**:
```yaml
증언자: {이름·소속·역할}
일자: YYYY-MM-DD
현장: {청주 / 제주 / 기타}
청취 내용:
  - 고객 발언 (직접 인용 우선)
  - 고객의 업무 흐름
  - 고객의 좌절 / 만족 지점
  - 고객의 지불 의사·결재 구조
검증 필요 사항:
  - {영업 본인도 확신 없는 부분}
```

**수집 대상**: 영업 2~3명 + DKS에 관심 있는 내부 운영 인력 1~2명
**소요**: 각 30분 인터뷰 + 정리 2시간

### (ii) LG-CNS 제주 ACS 프로젝트 산출물 검토

**DKS [[CLAUDE.md]]**: [[ACS-프로젝트]] 300개 문서 + 3개 POC 언급
**검토 초점**:
- 운영자 실제 이슈 로그
- 협력업체 호출 이력
- 알람 패턴 + 대응 이력
- 도면 변경 이력

**소요**: 2~3일 집중 검토 + 요약 1주

### (iii) 청주 도면 분석 결과 역추론

**현재 보유**: Phase 0 에이전틱으로 분석된 청주 68 PDF 결과
**역추론 방법**:
- 각 분야 도면의 복잡도·엔티티 수 → 담당자 업무 부하 추정
- 자주 등장하는 설비·관계 → 운영자 빈번 쿼리 후보
- 도면 간 연결 → 변경 영향 범위 샘플

**산출**: 분야별 페르소나 가설 + 예상 쿼리 Top 20

### (iv) 영업 미팅 끼워넣기 (5분 인터뷰)

**스크립트**:
```
"저희가 도입 검토 중이신데, 실무 관점 의견 5분만…"
Q1: 하루 중 도면·매뉴얼을 찾는 순간은?
Q2: 알람이 떴을 때 처음 30초에 뭘 하시나요?
Q3: 보고서 작성에 주당 몇 시간 쓰세요?
Q4: 협력업체 호출할 때 가장 답답한 순간은?
Q5: ChatGPT 같은 AI에 이런 업무 질문해보신 적?
```

**타이밍**: 기존 영업 미팅 끝에 자연스럽게 삽입
**누적**: 5~10건 쌓이면 페르소나 정의 재검증 가능

### (v) 사내 1시간 워크샵

**참여**: 영업 2 + 운영 2 + 기술 1
**프로토콜**:
- 0~15분: 본 문서 §4 낭독·검토
- 15~45분: "이 사람이 실제로 있다면 어떻게 다를까?" 집단 수정
- 45~60분: 5대 불만 우선순위 투표

**산출**: 04 문서 개정안

## §12. 03과의 차이 비교표 (학습 자산)

| 항목 | 03 (내 추론) | 04 (사용자 데이터 기반) |
|------|-------------|------------------------|
| 페르소나 축 | 기능 (A시설 / B모니터 / C평가) | 분야 × 직급 × 현장 타입 |
| 페르소나 수 | 3인 분리 | 1인 통합 역할 (여러 모자) |
| 현장 | 미구분 | 공장형 / 데이터센터형 2종 |
| 결재·사용자 | 모호 | 공장장 결재 / 책임자·실무자 사용 분리 확인 |
| 협력업체 | 언급 없음 | 데이터센터형 핵심. B2B2B 구조 |
| 도면 작성자 | 언급 없음 | 운영자 겸직 + 외부 업체 양쪽 |
| 불만 정의 | 추론 기반 | 5대 실제 불만 (a~f) |
| MVP 1순위 | B 모듈 (알람+영향도) | **알람 맥락 엔진 (불만 d+f)** |
| 독립 서비스 근거 | 추론 | 불만 기반 |
| FMS 한계 언급 | 없음 | 결정적 진입 메시지 |
| 챗봇 실패 경험 | 없음 | 시장 공백 실증 ★ |

### 교훈
- 기능 축은 *SaaS 제품 관점*, 조직 축은 *실제 인간 관점*. 후자가 우선.
- 불만 기반 정의 > 역할 기반 정의. 불만이 구매 동기.
- "고객이 이미 실패를 경험했는가"가 시장 공백의 가장 강한 증거.
