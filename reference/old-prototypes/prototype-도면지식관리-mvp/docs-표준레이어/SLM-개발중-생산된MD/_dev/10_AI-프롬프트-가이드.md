# GEM AI 프롬프트 가이드

## 목적
배터리 안전 분석 인사이트 생성을 위한 표준 AI 프롬프트 템플릿 관리 및 사용 가이드

---

## 1. 개요

### 1.1 프롬프트 템플릿이란?
Phase 3에서 AI가 Phase 2 검증 결과를 분석하여 엔지니어용 인사이트를 생성할 때 사용하는 **표준 지시사항**입니다.

### 1.2 왜 필요한가?
- **일관성**: 동일한 데이터 → 동일한 형식의 인사이트
- **정확성**: 수치 오류 방지, 추측 금지
- **전문성**: 배터리 안전 도메인 전문 용어 사용
- **재현성**: 누가 실행해도 같은 결과

### 1.3 템플릿 위치
```
config/prompts/
├── thermal_safety.txt            # 열적 안전성 분석
├── spc_balance.txt               # SPC 밸런싱 분석
├── data_integrity.txt            # 센서 무결성 검증
├── system_correlation.txt        # 시스템 상관성 분석
└── comprehensive.txt             # 종합 인사이트
```

## 2. 5가지 표준 프롬프트 템플릿

### 2.1 thermal_safety.txt
**목적**: 온도 변화율(dT/dt) 기반 열폭주 위험 분석

**입력 데이터 형식**:
```json
{
  "analysis_period": {
    "start": "2025-12-13T00:00:00Z",
    "end": "2025-12-13T23:59:59Z"
  },
  "battery_profile": "NMC_LG",
  "threshold": {
    "critical": 1.5,
    "unit": "°C/min"
  },
  "detections": [
    {
      "timestamp": "2025-12-13T10:00:20Z",
      "rack_id": "RACK_05",
      "t_avg": 25.3,
      "dtdt": 12.5,
      "is_thermal_risk": true,
      "threshold_ratio": 8.3
    }
  ],
  "statistics": {
    "total_events": 3,
    "affected_racks": ["RACK_05"],
    "max_dtdt": 51.0
  }
}
```

**출력 형식 예시**:
```markdown
## 1. 요약
Rack-05에서 12.5°C/min의 급격한 온도 상승이 감지되었습니다 (임계값: 1.5°C/min, 8.3배 초과)

## 2. 발생 정보
### 발생 시각
- 2025-12-13T10:00:20Z

### 영향 범위
- 영향 받은 Rack: RACK_05
- 총 발생 건수: 3건

## 3. 측정값
### 최고 온도 변화율
- 측정값: 51.0°C/min
- 임계값: 1.5°C/min
- 초과 비율: 34.0배

### 해당 시점 온도
- 25.3°C

## 4. 위험도 평가
⚠️ **열폭주 위험** - 즉각적인 조치가 필요합니다.

## 5. 가능한 원인
1. **내부 단락 (ISC, Internal Short Circuit)**
   - 셀 내부 분리막 손상
   - 덴드라이트 성장
...

## 6. 권장 조치
### 🔴 긴급 (Immediate)
1. 해당 Rack (RACK_05) 즉시 전기적 격리
2. 소화 설비 대기 태세
...
```

**핵심 제약사항**:
- 수치는 JSON에서 그대로 인용 (반올림 금지)
- "~일 가능성", "~로 추정" 같은 추측 표현 금지
- 전문 용어는 한글 번역 병기 (예: ISC (내부 단락))

### 2.2 spc_balance.txt
**목적**: Z-Score 기반 셀 밸런싱 상태 분석

**입력 데이터 형식**:
```json
{
  "analysis_period": {
    "start": "2025-12-13T00:00:00Z",
    "end": "2025-12-13T23:59:59Z"
  },
  "battery_profile": "NMC_LG",
  "threshold": {
    "critical": 3.0,
    "unit": "σ (sigma)"
  },
  "detections": [
    {
      "rack_id": "RACK_12",
      "v_avg": 3.580,
      "z_score": -3.2,
      "is_outlier": true,
      "group_mean": 3.705,
      "group_std": 0.039
    }
  ],
  "statistics": {
    "total_racks": 160,
    "outlier_count": 5,
    "affected_racks": ["RACK_12", "RACK_45", "RACK_78"],
    "min_z_score": -3.8,
    "max_z_score": 3.5
  }
}
```

**출력 형식 예시**:
```markdown
## 1. 요약
Rack-12는 이상치(Z-Score: -3.2σ)로, 그룹 평균 대비 3.4% 낮은 전압을 나타냅니다

## 3. 측정값
### Rack-12 상세
- 평균 전압: 3.580V
- Z-Score: -3.2σ
- 그룹 평균: 3.705V
- 그룹 표준편차: 0.039V
- 절대 편차: 0.125V (3.4%)

## 4. 위험도 평가
⚠️ **셀 불균형 감지** - 밸런싱 점검이 필요합니다.

## 5. 가능한 원인
1. **셀 노화 불균형**
2. **BMS 밸런싱 실패**
3. **제조 편차**
4. **온도 불균형**
...
```

**핵심 제약사항**:
- Z-Score는 소수점 1자리, 전압은 소수점 3자리
- absolute_deviation과 deviation_percent 반드시 계산
- z_score가 음수면 "낮은", 양수면 "높은"

### 2.3 data_integrity.txt
**목적**: 센서 고착, 범위 이탈, NULL 값 감지 (센서 문제 vs 배터리 문제 구분)

**입력 데이터 형식**:
```json
{
  "analysis_period": {
    "start": "2025-12-13T00:00:00Z",
    "end": "2025-12-13T23:59:59Z"
  },
  "battery_profile": "NMC_LG",
  "detections": {
    "sensor_frozen": [
      {
        "rack_id": "RACK_99",
        "affected_sensors": ["temperature", "voltage"],
        "frozen_duration_minutes": 120,
        "last_value": {
          "temperature": 25.5,
          "voltage": 3.70
        },
        "std_dev": 0.0001
      }
    ],
    "out_of_range": [
      {
        "rack_id": "RACK_100",
        "sensor": "voltage",
        "value": 5.2,
        "valid_range": [2.5, 4.3],
        "timestamp": "2025-12-13T14:30:00Z"
      }
    ],
    "null_values": [
      {
        "rack_id": "RACK_101",
        "null_count": 50,
        "total_count": 1000,
        "null_percentage": 5.0
      }
    ]
  },
  "statistics": {
    "total_racks": 160,
    "valid_data_percentage": 92.0,
    "sensor_frozen_count": 2,
    "out_of_range_count": 1,
    "null_value_racks": 1
  }
}
```

**출력 형식 예시**:
```markdown
## 1. 요약
총 4건의 데이터 무결성 문제가 감지되었습니다. 유효 데이터: 92.0%

## 3. 센서 고착
### Rack-99
**영향 받은 센서**: temperature, voltage

**증상**:
- 고착 지속 시간: 120분
- 고정된 값:
  - 온도: 25.5°C
  - 전압: 3.70V
- 표준편차: 0.0001 (거의 0에 가까움)

**판정**:
⚠️ **센서 고착** - 실제 배터리 문제가 아닌 센서/통신 장애입니다.

## 6. 중요 구분 사항
### ⚠️ 센서 문제 vs 배터리 문제

**센서 문제 징후**:
- 값이 완전히 고정됨 (표준편차 ≈ 0)
- 물리적으로 불가능한 값
- 갑작스러운 NULL 발생

**배터리 실제 문제 징후**:
- 값이 변하지만 비정상적 (예: 온도 상승)
- 값의 범위는 유효하지만 패턴 이상
- 다른 검증 기법(dT/dt, Z-Score)에서도 감지
...
```

**핵심 제약사항**:
- 센서 문제와 배터리 문제를 명확히 구분
- NULL 비율에 따른 3단계 판정 (< 5%, 5~10%, ≥ 10%)
- 오경보 방지가 최우선

### 2.4 system_correlation.txt
**목적**: Rack 전압과 메인 DC 버스 전압 상관관계 분석 (전기적 연결 문제 진단)

**입력 데이터 형식**:
```json
{
  "analysis_period": {
    "start": "2025-12-13T00:00:00Z",
    "end": "2025-12-13T23:59:59Z"
  },
  "battery_profile": "NMC_LG",
  "detections": [
    {
      "rack_id": "RACK_08",
      "issue_type": "voltage_tracking_failure",
      "correlation_coefficient": 0.15,
      "threshold": 0.90,
      "system_behavior": "discharging",
      "time_window": {
        "start": "2025-12-13T12:00:00Z",
        "end": "2025-12-13T13:00:00Z"
      },
      "measurements": {
        "main_dc_voltage_start": 800.0,
        "main_dc_voltage_end": 700.0,
        "main_dc_change": -100.0,
        "rack_voltage_start": 798.0,
        "rack_voltage_end": 798.5,
        "rack_voltage_change": 0.5
      }
    }
  ],
  "statistics": {
    "total_racks": 160,
    "normal_tracking": 155,
    "abnormal_tracking": 5,
    "affected_racks": ["RACK_08", "RACK_12", "RACK_34"]
  }
}
```

**출력 형식 예시**:
```markdown
## 1. 요약
Rack-08의 전압이 시스템 DC 버스를 추종하지 않습니다 (상관계수: 0.15, 임계값: 0.90)

## 4. 위험도 평가
⚠️ **전기적 단선 의심** - Rack이 시스템에서 격리된 상태일 가능성이 높습니다.

**증상 해석**:
- 메인 DC: -100.0V 변화 → 시스템은 정상적으로 방전 중
- Rack: 0.5V 변화 → 거의 변화 없음 (단선 의심)

## 5. 가능한 원인
1. **퓨즈 단선 (Fuse Blown)** - 가장 흔한 원인
2. **컨택터 개방 (Contactor Open)**
3. **차단기 트립 (Circuit Breaker Trip)**
4. **케이블 단선 (Cable Break)**

## 7. 진단 흐름도
```
Step 1: 육안 점검
├─ 퓨즈 끊어짐? → YES → 퓨즈 교체
├─ 컨택터 열림? → YES → Step 2
└─ 차단기 트립? → YES → 원인 확인 후 리셋
...
```
...
```

**핵심 제약사항**:
- 상관계수 소수점 2자리
- 변화량 백분율 포함, 소수점 1자리
- 진단 흐름도 반드시 포함
- 열폭주와 명확히 구분

## 4. 사용자 Custom Prompt

### 4.1 Custom Prompt란?
엔지니어가 특정 분석에 대해 **추가 지시사항**을 제공하는 경우

**예시**:
```
"RACK_05의 온도 상승이 주변 Rack에 미치는 영향을 중점적으로 분석해주세요."
```

### 4.2 Custom Prompt 처리 방식
1. **표준 프롬프트 유지**: 기본 프롬프트 템플릿은 그대로 사용
2. **추가 지시사항**: Custom prompt를 "추가 분석 요구사항" 섹션에 append
3. **기록 보존**: `results/{RUN_ID}/custom_prompts.yaml`에 기록

### 4.3 custom_prompts.yaml 예시
```yaml
- timestamp: "2025-12-13T17:00:00Z"
  phase: "phase3"
  analysis_type: "thermal_safety"
  prompt: "RACK_05의 온도 상승이 주변 Rack에 미치는 영향을 중점적으로 분석해주세요."
  applied_to: "results/20251213_153045_NMC_LG/phase3_insights/insights_thermal.md"

- timestamp: "2025-12-13T18:30:00Z"
  phase: "phase3"
  analysis_type: "comprehensive"
  prompt: "위험 Rack 교체 시 시스템 전체 용량 감소량을 계산해주세요."
  applied_to: "results/20251213_153045_NMC_LG/phase3_insights/insights_comprehensive.md"
```

### 4.4 Custom Prompt 주입 예시 (Python)
```python
# 표준 프롬프트 로드
with open("config/prompts/thermal_safety.txt", "r", encoding="utf-8") as f:
    base_prompt = f.read()

# Custom prompt 추가
custom_instruction = """
# 추가 분석 요구사항
사용자가 다음과 같은 추가 분석을 요청했습니다:
- RACK_05의 온도 상승이 주변 Rack에 미치는 영향을 중점적으로 분석해주세요.

위 요구사항을 반영하여 인사이트를 생성하되, 기본 OUTPUT FORMAT은 그대로 유지하세요.
"""

final_prompt = base_prompt + "\n\n" + custom_instruction

# AI 실행
response = ollama.chat(
    model="qwen2.5:32b",
    messages=[
        {"role": "system", "content": final_prompt},
        {"role": "user", "content": json.dumps(detection_data, ensure_ascii=False)}
    ],
    options={"temperature": 0.1}
)
```

## 5. 일관성 검증 방법

### 5.1 일관성 테스트 스크립트
`tests/test_prompt_consistency.py`:
```python
import ollama
import json

def test_consistency(prompt_path, test_data_path, num_runs=3):
    """동일 입력으로 3회 실행 → 결과가 동일한지 확인"""

    with open(prompt_path, "r", encoding="utf-8") as f:
        prompt = f.read()

    with open(test_data_path, "r", encoding="utf-8") as f:
        data = json.load(f)

    results = []
    for i in range(num_runs):
        response = ollama.chat(
            model="qwen2.5:32b",
            messages=[
                {"role": "system", "content": prompt},
                {"role": "user", "content": json.dumps(data, ensure_ascii=False)}
            ],
            options={"temperature": 0.1}
        )
        results.append(response['message']['content'])

    # 3회 결과 비교
    if results[0] == results[1] == results[2]:
        print("✅ 일관성 테스트 통과")
    else:
        print("❌ 일관성 테스트 실패")
        print("Run 1과 Run 2의 차이:")
        print(diff(results[0], results[1]))

if __name__ == "__main__":
    test_consistency(
        "config/prompts/thermal_safety.txt",
        "tests/data/thermal_test.json",
        num_runs=3
    )
```

## 9. 모델 선정 근거

### 9.1 왜 Qwen2.5를 선택했는가?

**결론**: ESS(배터리) 특화 모델은 **없습니다**. 범용 LLM + 프롬프트 엔지니어링이 최선입니다.

#### Ollama 제공 모델 현황 (2025년 기준)
Ollama는 다음과 같은 범용 모델만 제공합니다:

- **Llama 3** (Meta) - 영어 강함
- **Mistral** (Mistral AI) - 유럽어 강함
- **Gemma** (Google) - 일반 대화 강함

**ESS 특화 모델은 존재하지 않습니다!**

#### Qwen2.5 선택 이유

| 평가 항목 | Qwen2.5 | Llama 3 | Mistral | Gemma |
|----------|---------|---------|---------|-------|
| **한국어 성능** | ⭐⭐⭐⭐⭐ | ⭐⭐☆☆☆ | ⭐☆☆☆☆ | ⭐⭐☆☆☆ |
| **기술 문서 이해** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐☆ | ⭐⭐⭐☆☆ | ⭐⭐⭐☆☆ |
| **수치 정확성** | ⭐⭐⭐⭐☆ | ⭐⭐⭐⭐☆ | ⭐⭐⭐☆☆ | ⭐⭐⭐☆☆ |
| **프롬프트 준수** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐☆ | ⭐⭐⭐☆☆ | ⭐⭐⭐☆☆ |
| **속도 (14B 기준)** | 150 tokens/s | 120 tokens/s | 180 tokens/s | 140 tokens/s |
| **메모리 (32B 기준)** | 20GB | 22GB | - | - |

**최종 선택: Qwen2.5** ✅
- **한국어 인사이트 생성**이 필수이므로 Qwen2.5가 압도적으로 유리

### 9.2 ESS 특화 모델이 없는데 어떻게 하나?

**답변**: **프롬프트 엔지니어링**으로 해결합니다!

#### 우리의 전략
```
범용 LLM (Qwen2.5)
    +
상세한 프롬프트 템플릿 (config/prompts/*.txt)
    +
구조화된 입력 데이터 (JSON)
    +
낮은 Temperature (0.1)
    =
배터리 전문가 수준의 인사이트 생성 ✅
```

#### 프롬프트 템플릿의 역할

```markdown
- ESS 냉각 시스템 설계

ANALYSIS OBJECTIVE
온도 변화율(dT/dt)을 분석하여 열폭주 위험을 진단합니다.
...
```

→ Qwen2.5는 이 프롬프트를 보고 **배터리 전문가처럼 행동**합니다.

## 10. 양자화 레벨 비교 (Q8 vs Q5 vs Q4)

### 10.1 양자화(Quantization)란?

**양자화**: 모델의 가중치를 낮은 비트로 압축하여 메모리와 연산량을 줄이는 기술

```
FP16 (원본, 16-bit)
  ↓ 양자화
Q8 (8-bit) → 절반 크기, 품질 99.9%
  ↓
Q5 (5-bit) → 1/3 크기, 품질 98.5%
  ↓
Q4 (4-bit) → 1/4 크기, 품질 95%
```

## 11. 환경별 모델 구성 관리

### 11.1 개발/운영 환경 자동 전환

GEM 프로젝트는 `.env` 파일의 `ENVIRONMENT` 변수로 환경을 자동 감지합니다.

#### **환경 설정 (.env)**

```bash
# 개발 환경
ENVIRONMENT=development

# 운영 환경
ENVIRONMENT=production
```

#### **환경별 모델 자동 로드**

```python
# main.py에서 자동 전환
def load_ai_model_config():
    env = os.getenv('ENVIRONMENT', 'development')

    if env == 'production':
        # 운영: qwen2.5:3b + qwen3-coder:32b-q5_k_m
        return production_config
    else:
        # 개발: qwen2.5:3b + qwen2.5:14b
        return development_config
```
