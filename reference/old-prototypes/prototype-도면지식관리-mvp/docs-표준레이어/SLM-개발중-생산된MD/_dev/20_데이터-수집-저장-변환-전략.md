## 1. 데이터 수집 경로

### 1.1 경로 개요

GEM은 두 가지 데이터 수집 경로를 지원합니다:

```
경로 1: API 실시간 수집
  ↓
data/source/api/{date}/{channel}_realtime.csv

경로 2: API 일괄 수집
  ↓
data/source/api/{date}/{channel}_batch.csv

경로 3: CSV 업로드 (XD-HUB 포맷)
  ↓
results/{RUN_ID}/raw/original.csv
```

### 1.2 데이터 형식 비교

#### API CSV 형식 (표준)
```csv
time,BSC_CH_01.BANK_01.RACK_01.MOD_AVG_TEMP,BSC_CH_01.BANK_01.RACK_01.CELL_AVG_V,...
2025-12-26 00:00:00,25.3,3.65,...
2025-12-26 00:05:00,25.4,3.66,...
```
- **특징**: Wide 포맷, 태그별 컬럼 (810개)
- **맵 파일**: `data/map/{site}/{channel}_tags.txt` 순서와 동일
- **레코드 수**: 하루 288개 (5분 간격 기준)

#### XD-HUB CSV 형식 (업로드용)
```csv
time,rack_name,module_name,voltage_avg,voltage_min,voltage_max,temp_avg,temp_min,temp_max,current,soc
2025-12-26 00:00:00,RACK_001,MODULE_01,3.65,3.60,3.70,25.3,22.5,28.1,150.2,85.5
```
- **특징**: Long 포맷, 집계값 컬럼
- **변환 필요**: API 형식으로 전처리 후 Parquet 변환

### 2.2 파일명 규칙

## 3. Parquet 변환 시점

### 3.1 모드별 변환 전략

#### 모드 1: 실시간 수집 (Realtime)

```
5분마다 API 호출
  ↓
CSV append (data/source/api/2025-12-26/bsc_ch_01_realtime.csv)
  - 00:05 → 1 row
  - 00:10 → 2 rows
  - ...
  - 23:55 → 288 rows (완료)
  ↓
하루 종료 후 자동 처리:
  1. 무결성 검증 (누락/중복 체크)
  2. 검증 PASS → Parquet 변환
  3. 검증 FAIL → 알림 + 재시도
```

**Parquet 변환 시점:**
- ✅ **하루 종료 후 (23:59 이후) 자동 변환**
- 변환 위치: `data/source/api/{date}/{channel}_realtime.parquet`

#### 모드 2: 일괄 수집 (Batch)

```
API 일괄 호출 (00:00 ~ 23:59 전체)
  ↓
CSV 저장 (data/source/api/2025-12-26/bsc_ch_01_batch.csv)
  - 288 rows 한번에 저장
  ↓
즉시 처리:
  1. 무결성 검증
  2. 검증 PASS → 즉시 Parquet 변환
  3. 검증 FAIL → 재수집 제안
```

**Parquet 변환 시점:**
- ✅ **CSV 저장 직후 즉시 변환**
- 변환 위치: `data/source/api/{date}/{channel}_batch.parquet`

#### 모드 3: CSV 업로드 (Upload)

```
사용자 업로드 CSV (XD-HUB 포맷)
  ↓
Phase 1 실행:
  1. XD-HUB → API 형식 변환
  2. Parquet 변환
  3. Phase 2~4 진행
```

**Parquet 변환 시점:**
- ✅ **Phase 1 실행 시 변환**
- 변환 위치: `results/{RUN_ID}/phase1/battery_data.parquet`

### 3.2 Parquet 저장 위치 구분

| 데이터 소스 | Parquet 위치 | 용도 |

## 4. 데이터 무결성 검증

### 4.1 검증 항목

```json
{
  "date": "2025-12-26",
  "channel": "bsc_ch_01",
  "mode": "realtime",
  "expected_records": 288,
  "actual_records": 288,
  "missing_timestamps": [],
  "duplicate_timestamps": [],
  "time_gaps": [],
  "validation_status": "PASS",
  "validated_at": "2025-12-26 23:59:30"
}
```

### 4.2 검증 로직

```python
def validate_integrity(csv_path: Path, date: str, interval_minutes: int = 5):
    """
    데이터 무결성 검증

    Returns:
        {
            'status': 'PASS' | 'FAIL',
            'expected_records': 288,
            'actual_records': 287,
            'missing_timestamps': ['2025-12-26 14:35:00'],
            'duplicate_timestamps': [],
            'time_gaps': [
                {
                    'start': '2025-12-26 14:30:00',
                    'end': '2025-12-26 14:40:00',
                    'gap_minutes': 10
                }
            ]
        }
    """
    # 1. 예상 타임스탬프 생성
    expected_times = []
    start = datetime.strptime(f"{date} 00:00:00", '%Y-%m-%d %H:%M:%S')
    for i in range(24 * 60 // interval_minutes):
        ts = start + timedelta(minutes=i * interval_minutes)
        expected_times.append(ts.strftime('%Y-%m-%d %H:%M:%S'))

    # 2. 실제 CSV 타임스탬프 로드
    df = pl.read_csv(csv_path)
    actual_times = df['time'].to_list()

    # 3. 누락 확인
    missing = list(set(expected_times) - set(actual_times))

    # 4. 중복 확인
    from collections import Counter
    duplicates = [t for t, count in Counter(actual_times).items() if count > 1]

    # 5. 시간 갭 확인 (10분 이상)
    time_gaps = []
    for i in range(len(actual_times) - 1):
        t1 = datetime.strptime(actual_times[i], '%Y-%m-%d %H:%M:%S')
        t2 = datetime.strptime(actual_times[i+1], '%Y-%m-%d %H:%M:%S')
        gap_minutes = (t2 - t1).seconds // 60

        if gap_minutes > interval_minutes * 2:  # 2배 이상 차이
            time_gaps.append({
                'start': actual_times[i],
                'end': actual_times[i+1],
                'gap_minutes': gap_minutes
            })

    return {
        'status': 'PASS' if not (missing or duplicates or time_gaps) else 'FAIL',
        'expected_records': len(expected_times),
        'actual_records': len(actual_times),
        'missing_timestamps': sorted(missing),
        'duplicate_timestamps': duplicates,
        'time_gaps': time_gaps
    }
```

### 4.3 검증 실패 시 처리

## 5. 실시간 수집 실패 처리

### 5.1 실패 요청 저장

```json
{
  "date": "2025-12-26",
  "channel": "bsc_ch_01",
  "failed_requests": [
    {
      "timestamp": "2025-12-26 14:35:00",

      "reason": "API timeout",
      "retry_count": 0,
      "status": "pending"

    {
      "timestamp": "2025-12-26 16:20:00",
      "failed_at": "2025-12-26 16:20:03",
      "reason": "Connection error",
      "retry_count": 2,
      "status": "success"
    }
  ]
```

### 5.2 재시도 전략

```python
class RealtimeCollector:
    def collect_loop(self):
        """5분마다 실행되는 메인 루프"""
        while True:
            current_time = datetime.now()

            # 1. 정규 수집
            try:
                data = self.fetch_api_data(current_time)
                self.save_to_csv(data, current_time)
                logger.info(f"✅ {current_time}: 수집 성공")

            except Exception as e:
                # 2. 실패 시 실패 요청 저장
                logger.error(f"❌ {current_time}: 수집 실패 - {e}")
                self.save_failed_request(current_time, str(e))

            # 3. 매 시간 정각마다 실패 요청 재시도
            if current_time.minute == 0:  # 00분
                self.retry_failed_requests()

            # 4. 하루 종료 시 처리
            if current_time.hour == 23 and current_time.minute == 55:
                self.end_of_day_process()

            sleep(300)  # 5분 대기

    def retry_failed_requests(self):
        """실패한 요청들을 재시도"""
        failed_data = self.load_failed_requests()

        for req in failed_data['failed_requests']:
            if req['status'] == 'pending':
                try:
                    # 실패한 시점의 데이터 재요청
                    timestamp = datetime.strptime(req['timestamp'], '%Y-%m-%d %H:%M:%S')
                    data = self.fetch_api_data(timestamp)

                    # CSV에 시간 순서대로 삽입
                    self.insert_to_csv(data, timestamp)

                    # 성공 처리
                    req['status'] = 'success'
                    req['retry_count'] += 1
                    logger.info(f"✅ 재시도 성공: {req['timestamp']}")

                except Exception as e:
                    # 재시도 실패
                    req['retry_count'] += 1
                    logger.warning(f"❌ 재시도 실패 ({req['retry_count']}회): {req['timestamp']}")

                    # 5회 이상 실패 시 포기
                    if req['retry_count'] >= 5:
                        req['status'] = 'failed'
                        logger.error(f"❌ 재시도 포기: {req['timestamp']}")

        # 상태 저장
        self.save_failed_requests(failed_data)

    def insert_to_csv(self, data: dict, timestamp: datetime):
        """재시도 데이터를 시간 순서에 맞게 삽입"""
        csv_path = f"data/source/api/{self.date}/{self.channel}_realtime.csv"

        # CSV 로드
        df = pl.read_csv(csv_path)

        # 새 행 추가
        new_row = pl.DataFrame([{
            'time': timestamp.strftime('%Y-%m-%d %H:%M:%S'),
            **data
        }])

        # 병합 후 시간 정렬
        df = pl.concat([df, new_row]).sort('time')

        # 저장
        df.write_csv(csv_path)
```

### 5.3 재시도 스케줄

| 시점 | 동작 |
|------|------|
| 5분마다 | 정규 수집 (실패 시 `.failed_requests.json`에 저장) |
| 매시 정각 (00분) | 실패 요청 재시도 (최대 5회) |
| 하루 종료 (23:55) | 최종 재시도 + 무결성 검증 |

**장점:**
- ✅ 일시적 API 장애 복구 가능
- ✅ 시간 순서 보장 (insert + sort)
- ✅ 부하 분산 (매시 정각에만 재시도)
- ✅ 추적 가능 (실패 이력 기록)

```python
def run_phase1_internal(input_source, mode, progress_callback=None):
    """
    Phase 1 내부 실행 (통합)

    Args:
        input_source:
            - mode='api_realtime': data/source/api/{date}/{channel}_realtime.parquet
            - mode='api_batch': data/source/api/{date}/{channel}_batch.parquet
            - mode='upload': 사용자 업로드 CSV 파일 경로
        mode: 'api_realtime' | 'api_batch' | 'upload'
        progress_callback: 진행 상황 콜백

    Returns:
        {
            'status': 'success' | 'error',
            'parquet_path': str,
            'row_count': int,
            'mode': str,
            ...
        }
    """
    manager = ResultManager()  # RUN_ID 자동 생성
    manager.initialize()

    if mode in ['api_realtime', 'api_batch']:
        # API 수집 Parquet → Phase 1 Parquet 복사
        source_parquet = Path(input_source)
        dest_parquet = manager.phase1_dir / "battery_data.parquet"

        shutil.copy2(source_parquet, dest_parquet)

        # 메타데이터
        df = pl.read_parquet(source_parquet)
        row_count = len(df)

        metadata = {
            'source_type': mode,
            'source_file': str(source_parquet),
            'row_count': row_count,
            'schema_version': '3.0',
        }

    elif mode == 'upload':
        # XD-HUB CSV → API 형식 변환 → Parquet
        df = convert_xdhub_to_api_format(input_source)

        dest_parquet = manager.phase1_dir / "battery_data.parquet"
        df.write_parquet(
            dest_parquet,
            compression='zstd',
            compression_level=3
        )

        row_count = len(df)

        metadata = {
            'source_type': 'upload',
            'source_file': str(input_source),
            'row_count': row_count,
            'schema_version': '3.0',
        }

    manager.save_phase1_result(str(dest_parquet), metadata)

    return {
        'status': 'success',
        'parquet_path': str(dest_parquet),
        'row_count': row_count,
        'mode': mode,
        'run_id': manager.run_id
    }
```

### 9.5 Phase 6: 프롬프트 명령어 시스템 ✅

**구현 완료:** 2025-12-29

#### 왜 웹 UI가 아닌 프롬프트인가?

**문제점:**
- API 데이터는 서버에 저장됨
- 브라우저는 서버 파일에 직접 접근 불가
- 웹 UI로 "파일 선택"은 구조적으로 불가능

**해결책:**
- SLM 기반 프롬프트 명령어 시스템
- 서버에서 실행되므로 파일 시스템 직접 접근 가능
- 자연어로 간편하게 명령

#### 지원 명령어

1. **날짜 기반 분석**
   ```
   "2025-12-29 데이터 분석해줘"
   "어제 데이터 분석해줘"
   ```

2. **무결성 확인**
   ```
   "2025-12-29 데이터 품질 확인해줘"
   ```

3. **실패 요청 현황**
   ```
   "실패 요청 현황 보여줘"
   ```

4. **도움말**
   ```
   "명령어"
   ```

#### 구현 파일

- `GEM/modules/prompt/api_data_commands.py` - 명령어 핸들러 (신규)
- `GEM/config/ai_prompts.yaml` - 프롬프트 설정 (수정)
- `_dev/21_프롬프트-명령어-시스템.md` - 상세 문서
