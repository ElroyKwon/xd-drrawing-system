## 🎯 프로젝트 목표

제주 BESS 배터리 시스템(160개 Rack, 3.4억 건 데이터)에 대한 **자율 안전 진단 플랫폼** 구축

---

## 💡 핵심 컨셉: Dual Engine Architecture

### 문제 인식
- 기존: 모든 태그 분석 시도 → 분석 기법 파편화 및 AI 환각 문제
- 해결: 배터리 집중 + 수학 우선 + AI 차순

### 듀얼 엔진 구조

```
┌──────────────────┐         ┌──────────────────┐
│   🧠 Brain       │         │   💪 Muscle      │
│   (SLM)          │ ──────▶ │   (Python)       │
│                  │  Code   │                  │
│ • 의도 파악      │         │ • 3.4억 건 로드  │
│ • 계획 수립      │         │ • 병렬 연산      │
│ • Python 번역    │         │ • Pandas 처리    │
│ • 리포트 요약    │         │ • Parquet 저장   │
└──────────────────┘         └──────────────────┘
```

**핵심 차별점:**
- SLM은 직접 계산하지 않음 (토큰 한계 극복)
- Python은 해석하지 않음 (정확성 보장)

## 🔬 4대 핵심 분석 알고리즘

### 1. 열적 안전성 (Thermal Safety)
- **수식:** dT/dt (온도 변화율)
- **임계값:** > 2.0°C/min
- **의미:** 급격한 온도 상승 = 내부 단락 전조

### 2. 셀 밸런싱 (SPC - Statistical Process Control)
- **수식:** Z-Score = (X - μ) / σ
- **임계값:** > 3.0σ
- **의미:** 평균 대비 이탈 = 셀 노화/불균형

### 3. 내부 단락 (ISC - Internal Short Circuit)
- **조건:** 전압 하락(-3σ) AND 온도 상승(+3σ) 동시 발생
- **의미:** 물리적 내부 단락의 강력한 증거

### 4. 데이터 무결성 (Integrity Check)
- **조건:** Rolling Variance = 0 (30분간)
- **의미:** 센서 고착(Freezing), 통신 장애

## 🎭 주요 사용자 시나리오

### 현장 엔지니어
1. 매일 오전 출근 시 전날 데이터 자동 분석 결과 확인
2. "Rack-12 온도 급상승" 알림 수신
3. 단일 뷰어에서 해당 Rack 상세 차트 확인
4. 조치 사항(교체/점검) 결정

### 개발자
1. Dev Console에서 CSV 기반 로컬 테스트
2. 각 Phase별 단계 실행 및 로그 확인
3. 알고리즘 파라미터 튜닝
4. 결과 뷰어 즉시 미리보기

## 🚫 제약 조건

### 환경 제약
- **No Docker:** Windows Server 네이티브 실행
- **폐쇄망:** 외부 API 호출 불가
- **On-Premise:** 클라우드 사용 금지

### 기술 제약
- **No DB:** 파일 시스템 기반
- **SLM 크기:** 20b 모델 (로컬 실행 가능 범위)
- **브라우저:** IE11 제외 (모던 브라우저만)
