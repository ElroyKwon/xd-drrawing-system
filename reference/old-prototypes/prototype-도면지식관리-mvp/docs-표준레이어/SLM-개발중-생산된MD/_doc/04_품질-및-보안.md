## 🔄 Retry 전략

### Exponential Backoff + Jitter

```python
def retry_with_backoff(max_attempts=3):
    for attempt in range(max_attempts):
        try:
            return execute()
        except RecoverableError:
            # Base delay: 2^attempt (1s, 2s, 4s)
            base_delay = 2 ** attempt

            # Jitter: 0~500ms 랜덤 추가 (Thundering Herd 방지)
            jitter = random.uniform(0, 0.5)

            delay = base_delay + jitter
            time.sleep(delay)

    raise FinalFailure()
```

**적용 대상:**
- XD-HUB API 네트워크 요청
- Ollama SLM 호출
- 파일 시스템 쓰기 (일시적 Lock)

## ✅ 데이터 검증 (Validation)

### 1. 입력 검증 (Pydantic)

```python
class RackDataSchema(BaseModel):
    timestamp: datetime
    v_max: float = Field(ge=2.0, le=5.0)     # 리튬이온 범위
    v_min: float = Field(ge=2.0, le=5.0)
    t_max: float = Field(ge=-40.0, le=100.0) # 동작 온도
    t_min: float = Field(ge=-40.0, le=100.0)
    i_rack: float

    @validator('v_min')
    def check_voltage_logic(cls, v, values):
        if 'v_max' in values and v > values['v_max']:
            raise ValueError('v_min > v_max (논리 오류)')
        return v
```

### 2. 물리 검증 (PhysicsValidator)

| 항목 | 임계값 | 근거 | 조치 |
|------|--------|------|------|
| **dT/dt** | > 10.0°C/min | 제조사 스펙 (열폭주 속도) | Sensor Error 마킹 |
| **Z-Score** | > 20.0σ | 통계적 불가능 (6σ Rule) | Data Noise 제거 |
| **Voltage** | > 5.0V or < 0.0V | 센서 측정 범위 | Out of Range |

**검증 실패 처리:**
```
validation_failed = True
warnings = ["dT/dt exceeds physical limit"]

# SLM 입력에서 제외 (sensor_error 섹션으로 분리)
# 리포트에는 "센서 점검 필요" 항목으로 포함
```

### Path Traversal 방어

**공격 시나리오:**
```
?source=../../etc/passwd
?source=../../../windows/system32
```

**방어 로직:**
```javascript
function sanitizeDataPath(rawPath) {
    // 허용 패턴: /data/날짜_ID
    const pattern = /^\/data\/\d{8}_[A-Z0-9]+$/;

    if (!pattern.test(rawPath)) {
        throw new SecurityException("Invalid Path");
    }

    // 추가 차단
    if (rawPath.includes("..") || rawPath.includes("~")) {
        throw new SecurityException("Path Traversal Attempt");
    }

    return rawPath;
}
```

## 📊 성능 모니터링

### SLA (Service Level Agreement)

| Phase | 목표 시간 | 측정 방법 | 초과 시 조치 |
|-------|----------|----------|-------------|
| Phase 1 | < 3초 | `perf_counter` | 경고 로그 |
| Phase 2 | < 10초 | `@measure_time` | 경고 로그 |
| Phase 3 | < 60초 | Timeout 강제 | Fallback |
| Phase 4 | < 5초 | 파일 쓰기 시간 | - |
| **Total** | **< 15초** | End-to-End | **SLA 초과 플래그** |

### 성능 저하 원인 분석

**Phase 2 초과 시:**
- CPU 코어 수 확인
- 병렬 Worker 수 조정
- 데이터 크기 확인 (예상보다 많은 Row)

**Phase 3 초과 시:**
- Ollama 서버 리소스 확인
- 모델 크기 재검토 (27b → 20b)
- Context 토큰 수 줄이기
