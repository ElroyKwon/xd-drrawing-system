## §1. 서비스 개요

### §1.1 해결 불만 매핑

| 불만 | 해결 방식 |
|------|----------|
| c 보고서 쓰기 싫음 | Dual Engine 자동 초안 |
| d 알람 많은데 다 모름 | 알람→설비→도면→영향 30초 답변 |
| e 협력업체 대응 느림 | 방문 전 브리핑 자동 + 방문 후 보고 초안 |
| f 챗봇 맥락 부족 | 근거 병기·Multi-hop·온톨로지 기반 답 |

### §1.2 핵심 차별 1문장
> 일반 챗봇 = 맥락 없이 짐작. 본 서비스 = **현장 도면·온톨로지·이력을 근거로 답변하고 출처를 병기**.

### §1.3 [[SLM 인사이트 분석]]과의 관계
- Dual Engine 사상 이식: **수치는 Python, 해석은 SLM**
- 4대 엔진(Thermal·SPC·Integrity·Correlation) → 운영·알람 영역으로 재적용
- AI 환각 구조적 차단

## §2. 핵심 사용자 여정 (8 시나리오)

### 시나리오 1 — 낯선 알람 즉답 (신○○ 야간 당직)
- 상황: `CHW-T-HI-007` 알람 발생. 처음 봄.
- 기존: 매뉴얼 검색 → 선임 호출 (15~30분)
- 신규: 알람 패널 "AI 물어보기" → 30초 답:
  ```
  알람: CHW-T-HI-007 = 냉수 펌프 1호기 출구 고온
  위치: B1 기계실, 도면 M-001 p.3
  영향 가능 설비: AHU-3F-01/02/03
  1차 점검: 응축기 청결·냉각수 유량
  출처: 매뉴얼 §4.2.1 [원본]
  ```

### 시나리오 2 — 근본원인 추적 (박○○ 책임자)
- 상황: AHU-3F-01 성능 저하. 원인 파악 필요.
- 기존: 경험·직감 + 도면 조합
- 신규: "AHU-3F-01 성능 저하 근본원인" → Multi-hop 그래프 + 상위 후보 3개 + 각 후보의 근거

### 시나리오 3 — 유사 사례 검색 (김○○ 실무자)
- 상황: 오늘의 이상 패턴, 어디서 본 것 같은데?
- 기존: 기억·인계장 뒤지기
- 신규: 유사도 검색 → 과거 사례 5건 + 각 해결 방법

### 시나리오 4 — 일일 보고서 자동 초안 (박○○ 책임자)
- 상황: 매일 17시 보고서. 같은 양식 반복.
- 기존: 1~2시간 수동 작성
- 신규: "오늘 보고서" 버튼 → Dual Engine 초안 → 확인·수정만

### 시나리오 5 — 협력업체 브리핑 (정○○ 시설관리)
- 상황: 냉동기 2호기 정비 협력업체 호출. 사전 정보 준비.
- 기존: 도면·이력 수동 패키지 (1~2시간)
- 신규: "협력업체 브리핑 생성" → 도면·이력·제약·연관 설비 PDF 자동 (30초)

### 시나리오 6 — 인수인계 요약 (신○○→최○○)
- 상황: 야간 당직 교대.
- 기존: 인계장 작성·읽기
- 신규: 당직 이벤트 요약 3줄 + 주의 설비 5개 자동

### 시나리오 7 — 월간 KPI 리포트 (윤○○ 공장장)
- 상황: 월말 운영 평가.
- 기존: 부서별 취합·엑셀
- 신규: Dual Engine 월간 보고서 + KPI 자동 계산 + 전월 비교 해설

### 시나리오 8 — "왜?" 반복 질의 (최○○ 신참)
- 상황: "이 펌프 왜 여기 있어요?" → "왜 이 온도에서 알람 나요?" → …
- 기존: 선임 붙잡기
- 신규: 자연어 연속 질의 + 근거 즉시

## §4. 데이터 모델

### §4.1 주요 엔티티

```typescript
interface AlarmEvent {
  id: string;
  alarm_code: string;          // "CHW-T-HI-007"
  alarm_name: string;
  equipment_tag: string;       // 매핑 설비
  severity: "info" | "warning" | "critical";
  triggered_at: string;
  resolved_at?: string;
  value?: number;
  threshold?: number;
  source_system: "bms" | "scada" | "manual";
}

interface InsightQuery {
  id: string;
  user: string;
  query_text: string;
  query_type: "alarm" | "natural" | "root_cause" | "similar" | "report";
  context: {
    alarm_id?: string;
    equipment_tag?: string;
    time_range?: { start: string; end: string };
  };
  created_at: string;
}

interface InsightAnswer {
  id: string;
  query_id: string;
  answer_text: string;         // 자연어 답
  structured: {                // 구조화 필드
    equipment?: Entity;
    location?: string;
    impact_equipment?: Entity[];
    first_actions?: string[];
    root_causes?: { cause: string; confidence: number }[];
  };
  evidence: Evidence[];        // 근거 병기 ★
  dual_engine: {
    python_output?: Record<string, any>;  // 수치 계산 결과
    slm_interpretation?: string;           // SLM 해석
  };
  generated_at: string;
  latency_ms: number;
  model_used: string;
}

interface Evidence {
  type: "drawing_page" | "manual_section" | "similar_case" | "ontology_path";
  reference: string;           // "M-001 p.3" 또는 매뉴얼 §
  snippet?: string;            // 해당 부분 미리보기
  link: string;                // 서비스 1·문서로 이동 URL
  relevance_score: number;
}

interface Report {
  id: string;
  report_type: "daily" | "weekly" | "monthly" | "incident" | "vendor";
  period: { start: string; end: string };
  author: "auto_dual_engine" | string;
  sections: ReportSection[];
  status: "draft" | "reviewed" | "finalized";
  generated_at: string;
}

interface ReportSection {
  title: string;
  python_data: Record<string, any>;   // 수치
  slm_narrative: string;              // 해석
  chart_refs?: string[];
  evidence_refs?: string[];
}

interface SimilarCase {
  id: string;
  alarm_code?: string;
  description: string;
  resolution: string;
  occurred_at: string;
  similarity_score: number;
}
```

### §4.2 Dual Engine 파이프라인

```python
def answer_alarm(alarm_event):
    # 1. Python Engine — 정확한 맥락 수집
    equipment = ontology.get_entity(alarm_event.equipment_tag)
    impact = ontology.multi_hop(equipment.id, relation="affects", hops=2)
    history = db.query_similar_alarms(alarm_event.alarm_code, limit=5)
    manual_refs = rag.search(f"{alarm_event.alarm_name} 대응")
    
    # 2. 수치·통계 계산 (Python)
    stats = {
        "similar_count": len(history),
        "avg_resolution_time": mean(h.resolution_duration for h in history),
        "severity_trend": calculate_trend(recent_alarms),
    }
    
    # 3. SLM Engine — 자연어 해석
    prompt = build_prompt(
        alarm=alarm_event,
        equipment=equipment,
        impact=impact,
        history=history,
        manual=manual_refs,
        stats=stats,
    )
    narrative = slm.generate(prompt)  # 근거·출처 포함 강제
    
    # 4. 응답 조립 + 근거 구조화
    return InsightAnswer(
        answer_text=narrative,
        structured={...},
        evidence=[
            Evidence("drawing_page", equipment.source_drawings[0].ref),
            Evidence("manual_section", manual_refs[0].location),
            ...
        ],
        dual_engine={"python_output": stats, "slm_interpretation": narrative},
    )
```

- **우(SLM)**: textarea. 사용자 편집 가능. 변경 시 `status: draft → reviewed`.
- **확정 버튼**: `status: finalized`. 이후 수정 시 새 revision 생성.

**모킹 전략** (4주 MVP):
- 고정 JSON 응답 1건 (`mocks/report-daily-20260417.json`)
- 재생성 버튼 → 3~5초 지연 후 동일 응답 반환 (체감용)

### §15.11 시연 스크립트 — 3단 버전

#### 5분 (핵심 가치)
1. `/insight/alarms` 진입 → critical 3건 표시 (10초)
2. 첫 알람 `CHW-T-HI-007` → `[AI 물어보기]` 클릭 (5초)
3. AnswerCard 4섹션 스트리밍 (30초)
4. evidence의 "도면 M-001 p.3 [열기]" 클릭 → 서비스 1 도면 자동 이동 (15초)
5. 도면 위에서 CH-001 태그 하이라이트 확인 → 브레드크럼으로 인사이트 복귀 (15초)
6. **포인트**: 근거 없는 챗봇 대비 차별 — "답 + 근거 + 자연스러운 도면 이동"

#### 10분 (+ 시나리오 2 근본원인)
- 5분 시나리오 + AnswerCard 하단 FollowUp 클릭 → `/insight/root-cause/ahu-3f-01`
- Cytoscape 그래프 + 원인 후보 3개 + 각 신뢰도·근거
- 2순위 후보의 "도면으로" 클릭 → 서비스 1 이동

#### 20분 (+ 시나리오 4 보고서 + 서비스 1 크로스오버)
- `/insight/reports` → 오늘의 일일 보고서 초안
- 좌측 수치 + 우측 SLM 해석 편집 가능 시연
- "확정" 버튼 → `draft → finalized` 상태 전환
- 공용 쉘 시연: GlobalSearchBar 탭 전환으로 서비스 1↔2 자연 이동
- **포인트**: Dual Engine 철학 ("수치는 Python, 해석은 SLM") + 근거 병기
