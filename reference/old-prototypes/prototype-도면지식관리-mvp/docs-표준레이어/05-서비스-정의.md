# 05. 서비스 정의 v2 — 플랫폼(맥락 엔진) + 응용 서비스 2계층 아키텍처

> **사용자 결정 (2026-04-17)**
> 후보 B 기준 코어 플랫폼 구축 + 그 위에 여러 서비스 API로 붙임.
> 응용 서비스에 후보 A(운영자 맥락 엔진) + 후보 C(FMS 대체)·로봇 게이트웨이 등 추가.
> 맥락 추론 근거 데이터: **지금은 도면·준공문서**, 이후 **로깅·알람 데이터**로 확장.

> **v1과의 차이**
> v1: 기능 모듈 M1~M5 평면 나열 (서비스 = 기능 번들)
> v2: **플랫폼 1개 + 응용 서비스 N개** 2계층 아키텍처
> 결과: M1~M5는 v2의 응용 서비스 내부 기능으로 재배치. v1은 `-v1-기능모듈식-보관.md`로 아카이브.

> **한 문장 아키텍처**
> DKS = **공장·건물의 의미를 이해하는 맥락 엔진 플랫폼** (후보 B)
> + 그 위에 API로 붙는 **운영자 서비스·FMS 대체 서비스·로봇 게이트웨이** (A·C·미래)

## §2. 코어 플랫폼 — 맥락 엔진 (후보 B)

### §2.1 정체성

> "**현장의 모든 AI가 공장·건물의 의미를 이해하게 만드는 맥락 게이트웨이**"

### §2.2 구성 3요소

| 구성 | 기술 | 역할 |
|------|------|------|
| 구조화 지식 | [[TypeDB]] | 엔티티·관계·추론 규칙. "펌프 X는 냉동기 Y의 상류" 같은 의미 그래프 |
| 비정형 지식 | [[RAGflow]] | 문서·매뉴얼 벡터화. 자연어 질의 |
| 맥락 추론 | [[SLM 인사이트 분석]] Dual Engine + LLM 오케스트레이션 | 수치(Python) + 해석(SLM) + 자연어 응답 |

### §2.3 데이터 소스 진화 로드맵 ★

사용자 명시 방향:

```
Phase 1 (지금~근시)
  입력: 도면 (DWG/PDF) + 준공 문서
  출력: 설비 엔티티·관계·속성·매뉴얼 벡터
  산출 서비스: S1·S2·S4 운영 가능
  진행 상태: Phase 0 에이전틱 (청주 68 PDF 처리 중)

Phase 2 (1~2년)
  입력 추가: 운영 로깅 데이터 + 알람 이력
  출력 추가: 알람 패턴·이상 탐지·RCA 정보
  산출 서비스: S1 고도화 + S3 본격 가동

Phase 3 (3~5년)
  입력 추가: 실시간 BMS/SCADA/MES + IoT 센서
  출력 추가: 실시간 상태·예측 정비·자율 추론
  산출 서비스: S4 로봇 게이트웨이 본격
```

### §2.4 플랫폼 노출 인터페이스

| 인터페이스 | 용도 | 소비자 |
|-----------|------|--------|
| **REST API** | 표준 쿼리 (엔티티·관계·영향도) | 응용 서비스 S1~S5 |
| **MCP 서버** | 외부 AI 맥락 주입 | 고객사 자체 AI / Claude / GPT |
| **GraphQL** | 복합 쿼리 (관계 그래프 탐색) | 개발자 통합 시 |
| **Webhook** | 변경·이벤트 구독 | 외부 시스템 연동 |
| **SDK** | 로봇 제조사·파트너 | 장기 생태계 |

→ **모든 응용 서비스와 외부 시스템은 동일 API를 통과**. 플랫폼 단일 진실의 원천.

### §2.5 플랫폼 자체 기능 (서비스 무관)

- 도면 자동 분석 (Phase 0 에이전틱 파이프라인)
- 온톨로지 자동 생성 (Phase 1 9단계 프로세스)
- RAG 자동 수집·인덱싱
- 변경 이벤트 감지·재분석
- 추론 엔진 (multi-hop·Dual Engine)
- 인증·권한·감사 로그
- 사이트 격리 (멀티 테넌시)

## §3. 응용 서비스 계층

### §3.1 서비스 S1 — 운영자 맥락 엔진 (후보 A)

> "**공장·건물 운영자가 매일 쓰는 도면·알람·보고서 맥락 엔진**"

| 항목        | 내용                                                                                                                     |
| --------- | ---------------------------------------------------------------------------------------------------------------------- |
| 타겟        | 공장형 분야 책임자·실무자 / 데이터센터형 통합 인원                                                                                          |
| 해결 불만     | [[04-사용자-정의-페르소나]] §6 a·b·c·d·f                                                                                        |
| 핵심 기능     | ① 알람 → 설비 → 도면 → 영향 30초 답변 (v1의 M1) ② 보고서 자동 초안 Dual Engine (v1의 M3) ③ 자연어 도면·매뉴얼 질의 ④ 변경 알림 (v1의 M2 중 b) ⑤ 인수인계 자동 요약 |
| 플랫폼 의존    | 온톨로지·RAG·추론 엔진 전부                                                                                                      |
| 데이터 소스 의존 | Phase 1 충분. Phase 2 시 알람 이력 추가 시 고도화                                                                                   |
| 독립 판매     | ✅ 가능                                                                                                                   |
| MVP 시점    | **4주 PoC — 1순위**                                                                                                       |
| 영업 멘트     | *"ChatGPT가 못 답한 그 알람·설비 질문, 우리가 답합니다."*                                                                                |

### §3.2 서비스 S2 — 현장 맞춤 서비스 (후보 C, FMS 대체)

> "**FMS가 못 한 현장 맞춤을 AI로 가능하게**"

| 항목        | 내용                                                                                                     |
| --------- | ------------------------------------------------------------------------------------------------------ |
| 타겟        | 공장장·시설관리 총괄 (FMS 도입 좌절 고객)                                                                             |
| 해결 불만     | a (도면 최신화), b (변경 공유), FMS 커스터마이징 한계                                                                   |
| 핵심 기능     | ① 공사 이벤트 → Phase 0 자동 재분석 (v1의 M2 핵심) ② 마스터 데이터 자동 생성 (도면 기반) ③ 자산·공간·설비 대장 자동 구조화 ④ 현장별 맞춤 온톨로지 자동 진화 |
| 플랫폼 의존    | 온톨로지·Phase 0 파이프라인·변경 감지                                                                               |
| 데이터 소스 의존 | Phase 1 충분                                                                                             |
| 독립 판매     | ⚠️ 부분 — S1과 번들 시 가치 배가                                                                                 |
| MVP 시점    | S1 검증 후 +8~12주                                                                                         |
| 영업 멘트     | *"FMS 도입하려다 현장 맞춤에 막히셨죠? AI가 도면을 읽어 자동 모델링합니다."*                                                       |

### §3.3 서비스 S3 — 협력업체 게이트웨이

> "**협력업체 부르기 전 브리핑, 부른 후 평가 자동**"

| 항목 | 내용 |
|------|------|
| 타겟 | 데이터센터형 핵심 / 공장형 부가 (공장장·운영 총괄) |
| 해결 불만 | e (협력업체 대응 느림) |
| 핵심 기능 | ① 호출 시 현장 맥락·도면·이력 자동 패키지 ② 방문 후 보고서 자동 초안 (협력업체 작성 부담 경감) ③ SLA·품질·응답 시간 누적 평가 ④ 협력업체용 제한 API (현장 맥락 사전 제공) |
| 플랫폼 의존 | 온톨로지·추론·Context API |
| 데이터 소스 의존 | Phase 1 시작, Phase 2 로깅 쌓이면 강화 |
| 독립 판매 | ✅ 가능 (C-level 결재) |
| MVP 시점 | S1 + Phase 2 데이터 누적 6개월 후 |
| 영업 멘트 | *"협력업체가 방문 전 이미 현장을 이해하면, 대응 시간이 반으로."* |

### §3.4 서비스 S4 — 로봇 게이트웨이 (미래)

> "**로봇·외부 AI가 공장 의미를 MCP로 즉시 이해**"

| 항목 | 내용 |
|------|------|
| 타겟 | 로봇 도입 공장·물류 센터 |
| 핵심 기능 | ① 위치 의미 변환 (로봇 SLAM ↔ 도면 좌표) ② 설비·객체 식별 변환 ③ 안전 영역 쿼리 ④ 공정 흐름 조회 ⑤ MCP 서버 형태 제공 |
| 플랫폼 의존 | 전체 + 실시간 데이터 (Phase 3) |
| 데이터 소스 의존 | Phase 2~3 필수 |
| 독립 판매 | 장기 |
| MVP 시점 | **2028~2030 목표** (Phase 2 검증 후) |
| 영업 멘트 | *"로봇이 공장을 이해하는 첫날부터, 우리 API가 그 다리입니다."* |

### §3.5 서비스 S5 — 외부 도면 업체용 Phase 0 SaaS [부가]

| 항목     | 내용                               |
| ------ | -------------------------------- |
| 타겟     | 설계·시공 업체 (DKS 직접 고객 아닌 협력자)      |
| 핵심 기능  | 도곽 표준 자동 변환 + 의미 추출 + 표준 YAML 산출 |
| 플랫폼 의존 | Phase 0 파이프라인만                   |
| 독립 판매  | ✅ 별도 채널 (B2B2B)                  |
| *[가설]* | 주력 아님. 주력 서비스 검증 후 검토            |

## §4. 서비스 × 데이터 소스 의존 매트릭스

어느 서비스가 어느 데이터 Phase에서 본격 작동하는가.

| 서비스         | Phase 1 (도면+문서) | Phase 2 (+ 로깅·알람) | Phase 3 (+ 실시간) |
| ----------- | :-------------: | :---------------: | :-------------: |
| S1 운영자 맥락   |     ★ 가동 가능     |      ★★ 고도화       |     ★★★ 실시간     |
| S2 현장 맞춤    |    ★★★ 완전 가동    |      ★★★ 유지       |     ★★★ 유지      |
| S3 협력업체     |    ★ 기본 브리핑     |     ★★ 평가 누적      |   ★★★ 실시간 상황    |
| S4 로봇 게이트웨이 |        —        |       ★ 제한        |     ★★★ 본격      |
| S5 외부 도면    |    ★★★ 완전 가동    |      ★★★ 유지       |     ★★★ 유지      |

→ **Phase 1만으로도 S1·S2·S5는 즉시 사업 가능**. S3·S4는 시간 필요.

## §5. 고객 여정 — 플랫폼 관점

### §5.1 단계별 경험

```
[1] 첫 접촉 (0~2주)
    - 영업: "ChatGPT가 못 답한 질문 + FMS 도입 좌절"
    - 플랫폼 가치 설명: "현장 맥락 엔진"
    - 서비스 선택: "어느 서비스(S1/S2/S3) 부터?"
    ↓
[2] 진단 (2주)
    - 샘플 도면 10~30장 Phase 0 분석
    - 운영자 4인 인터뷰 (5대 불만 검증)
    - 산출: 플랫폼 구축 견적 + 첫 서비스 PoC 범위
    ↓
[3] PoC (4주)
    - 플랫폼 일부 + 단일 서비스 (권장 S1 단일 분야)
    - 지표: 응답 시간·만족도·일일 쿼리
    ↓
[4] 플랫폼 구축 (3~6개월)
    - 전 도면 Phase 0 분석
    - 온톨로지·RAG 구축
    - API·MCP 활성화
    ↓
[5] 서비스 단계 추가
    - 시점 A: S1 가동 (플랫폼 구축과 병행)
    - 시점 B: S2 추가 (공사 이벤트 발생 시)
    - 시점 C: Phase 2 데이터 누적 후 S3 추가
    - 시점 D: 로봇 도입 시 S4 활성화
    ↓
[6] 확장
    - 다사이트 확장 (플랫폼 재사용)
    - 외부 AI 통합 (MCP 공유)
    - Context API 개방 (생태계)
```

### §5.2 여정의 의미
- **플랫폼 한 번 구축 = 영구 자산**. 이후 서비스 추가는 *증분 비용*.
- 고객이 처음부터 모든 서비스 안 사도 됨. 단계 확장 자연스러움.
- 사이트 확장 시 각 사이트의 도면·문서만 추가 분석. 플랫폼 기술은 재사용.

## §6. 판매·가격 모델

### §6.1 2단계 과금

```
① 플랫폼 계약 (구축 + 유지)
   ├─ 초기 구축비 (일시불): 플랫폼 설치 + 사이트 도면 분석 + 온톨로지 구축
   └─ 월 플랫폼 구독: 플랫폼 운영 + 업데이트 + 인프라

② 서비스 추가 과금 (사용 서비스별)
   ├─ 서비스당 월 추가 구독 (S1/S2/S3/…)
   └─ 쿼리 크레딧 (사용량 종량)
```

### §6.2 가격 레인지 *[전면 가설 — PoC 후 확정]*

| 항목 | 소규모 | 중규모 | 대규모 |
|------|--------|--------|--------|
| 플랫폼 구축비 | 3천만~1억 | 1억~3억 | 3억+ |
| 플랫폼 월 구독 | 200만~500만 | 500만~1천 | 협의 |
| 서비스 월 구독 (서비스당) | 300만~1천만 | 1천~2천만 | 협의 |
| 쿼리 크레딧 | 포함 10k/월 + 50원~500원/초과 | 포함 50k/월 | 포함 200k/월 |

### §6.3 이 모델의 장점

- **플랫폼 락인**: 고객이 플랫폼에 온톨로지를 쌓을수록 이탈 비용 증가
- **확장 수익**: 서비스 추가·쿼리 증가·사이트 확장 모두 수익 증가 경로
- **로봇 대비**: Phase 3에서 쿼리 폭증 시 플랫폼 자연 수익 성장
- **B2B2B**: 외부 AI가 Context API 호출해도 쿼리 과금 가능

## §7. 서비스 경계 — We Are / We Are Not

### We Are (Platform Level)
- 현장의 의미 엔진 (Context Engine)
- 도면·문서 자동 분석 (Phase 0)
- 온톨로지·RAG·추론 통합
- 외부 AI용 MCP·Context API

### We Are (Application Level)
- 운영자 옆 AI (S1)
- FMS 수준 현장 맞춤 자동 구축 (S2)
- 협력업체 브리핑·평가 (S3)
- 로봇 의미 게이트웨이 (S4)

### We Are Not
- [[BMS]]/[[SCADA]] 대체 X — 위에 얹는 맥락 레이어
- [[FMS]] 패키지 대체 X — FMS가 하는 워크오더·자산대장은 기존 유지. 우리는 *맥락 지식*
- [[CMMS]]/[[EAM]] 대체 X — 정비 계획은 기존
- 3D 디지털 트윈 X — [[NVIDIA Omniverse]] 영역
- 대시보드 X — BMS·DCIM 영역
- 제어 시스템 X — 읽기 전용

→ 핵심 메시지: *"기존 시스템 교체 없이, 그 위에 맥락을 더합니다."*

## §8. 경쟁 위치 — 플랫폼 vs 서비스

### §8.1 두 레벨 경쟁 분리

```
플랫폼 레벨 경쟁                  서비스 레벨 경쟁
(의미 엔진)                       (업무 서비스)

NVIDIA Omniverse (수동 트윈)       BMS/SCADA 알람 (맥락 없음)
Siemens Industrial Ops X           FMS (수동 마스터)
AWS Manufacturing                   CMMS/EAM (이력 관리)
ISA-95/ISO 15926 (무거운 표준)      일반 챗봇 (맥락 없음)
                                    
                                    
DKS 차별 (플랫폼 레벨):           DKS 차별 (서비스 레벨):
"도면 자동 의미 추출"              "플랫폼 재사용 + 현장 맞춤"
```

### §8.2 단일 축 집중

**플랫폼 차별 단일 축 = Phase 0 에이전틱 (도면 자동 의미 추출)**.
이 한 점에 모든 자원 집중. 여기가 무너지면 빅테크에 흡수됨.

### §8.3 서비스 레벨의 차별
- 각 서비스는 "플랫폼이 이미 가진 의미"를 활용하는 업무 앱
- 개별 서비스는 경쟁이 많지만, **플랫폼이 쌓이는 한 서비스 경쟁은 양산 가능**
- ServiceNow·Salesforce 전략 동일: 플랫폼 락인, 앱 확장

  - Phase 2 데이터 활용

  기간: 3개월

[Step 6] Phase 3 실시간 + S4 로봇 게이트웨이
  - BMS/SCADA 실시간 통합

  - 로봇 MCP SDK
  기간: 2028~ 중장기

## §10. 플랫폼-서비스 아키텍처의 결정적 장점

| 장점 | 내용 |
|------|------|
| **고객 선택권** | 모든 서비스 강제 아님. 필요한 것부터 구매 |
| **재사용** | 플랫폼 한 번, 서비스 다수. 개발 효율 |
| **확장성** | 새 서비스 추가 = 플랫폼 재사용. 고객사 요구별 커스텀 가능 |
| **락인 강도** | 플랫폼 온톨로지 누적 = 이탈 비용 지수 증가 |
| **생태계 가능** | Context API 개방 = 파트너·로봇 제조사가 붙을 수 있음 |
| **미래 대비** | Phase 2·3 데이터 확장 시 플랫폼에 올려 서비스 자동 업그레이드 |
| **투자 논리 강화** | 플랫폼 = 복리 자산. VC·M&A 밸류에이션 상승 |
| **영업 단일화** | "우리는 플랫폼이다. 위에 서비스가 붙는다" 한 메시지 |

## §11. 서비스 성공 지표

### §11.1 플랫폼 지표
| 지표 | 목표 |
|------|------|
| 사이트당 엔티티 수 | 1,000+ (도면 완료 시) |
| 온톨로지 추론 규칙 수 | 100+ |
| API 일일 쿼리 처리 | 10,000+ (Phase 2 후) |
| RAG 검색 정확도 | 85%+ |
| Context API 호출 외부 AI 수 | 3+ (장기) |

### §11.2 서비스 지표 (S1 기준, [[02-기능-경험-요건]] §9)
1. 30초 이내 답
2. 근거 병기 만족
3. 매뉴얼 없이 첫 사용
4. 환경(모바일·오프라인·소음) 견딤
5. 일 5회 자발 사용

### §11.3 사업 지표
- 플랫폼 계약 수 (사이트)
- 사이트당 서비스 평균 개수
- 확장률 (초기 1 서비스 → 3 서비스)
- 쿼리 월간 성장률
- Context API 외부 생태계 규모
