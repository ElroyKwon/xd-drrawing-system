## 2. 평가 프레임워크: 3축

### 축 1: 커버리지 (Coverage) — 얼마나 많이 찾았나

| 지표 | 측정 방법 |
|------|----------|
| 태그 합집합 대비 발견율 | 세 결과(144018, 152722, 루트)의 태그 합집합 → 각 결과의 포함률 |
| 공간 발견 수 | 도면에 존재하는 실제 공간 vs 추출된 공간 |
| 페이지 커버리지 | 분석 대상 페이지 수 / 전체 도면 페이지 수 |
| 범례 심볼 커버리지 | 추출된 심볼 수 / 도면 내 실제 심볼 수 |

**실행 방법 (Step 1 — 자동화)**:
```
1. 각 산출물에서 tag 목록 추출 (YAML 파싱)
2. 합집합 매트릭스 생성
3. 각 결과별 누락 태그 식별
4. 결과를 eval_coverage.yaml로 저장
```

### 축 2: 정확성 (Accuracy) — 얼마나 맞았나

| 지표 | 측정 방법 |
|------|----------|
| V3.2 필수 속성 채움률 | 8+2 필수 필드 중 실제 값이 있는 비율 |
| Enum 값 정합성 | discipline, function_category, status 등이 V3.2 표준 Enum에 포함되는지 |
| 태그 일관성 | 같은 설비를 다른 이름으로 추출한 건 (P01 vs PIT 등 중복/혼동) |
| 거짓양성률 | 설비/공간이 아닌 것을 추출한 비율 (도면 제목, 날짜, 약어 등) |
| 속성값 정확도 | 샘플 10건의 속성을 도면 원본과 대조 |

**실행 방법 (Step 2 — 반자동)**:
```
1. V3.2 enums.yaml 로드 → 각 엔티티의 Enum 값 검증 (자동)
2. 필수 속성 채움률 계산 (자동)
3. 태그 중복/혼동 패턴 탐지 (자동)
4. 거짓양성 후보 식별 → 수동 확인 (반자동)
5. 핵심 엔티티 10건 심층 비교 (수동)
```

### 축 3: 비용 효율 (Cost Efficiency) — 얼마에 얻었나

| 지표 | 산식 |
|------|------|
| 엔티티당 비용 | 총 비용 ÷ 유효 엔티티 수 |
| 정확 엔티티당 비용 | 총 비용 ÷ (엔티티 수 × 정확률) |
| 토큰 효율 | 총 토큰 ÷ 유효 엔티티 수 |
| API 호출 효율 | 총 호출 수 ÷ 유효 엔티티 수 |

**데이터 소스**: 각 날짜 폴더의 `cost_summary.json`, `quality_report.json`

## 3. 실행 순서

### Step 1. 태그 통합 비교표 생성 (자동화 — 코드 작성)

**입력**:
- `20260408_144018/entity_list_data.yaml` (또는 entity_details/)
- `20260408_152722/entity_details/` (94개)
- 루트 `entity_list_data.yaml` + `entity_details/` (49+53)

**출력**: `eval_tag_matrix.yaml`
```yaml
# 예시 구조
- tag: D1
  in_144018: true
  in_152722: true
  in_crossval: true
  match: full
- tag: TRENCH-001
  in_144018: false
  in_152722: true
  in_crossval: true
  match: partial
```

### Step 2. 스키마 준수 검증 (자동화 — 코드 작성)

**입력**: 각 산출물의 entity_detail YAML 파일들
**검증 기준**: `docs-표준-typedb/schema_yaml_v3/enums.yaml`, `entities.yaml`, `axioms.yaml`

**출력**: `eval_schema_compliance.yaml`
```yaml
# 예시 구조
- source: crossval
  total_entities: 49
  required_field_fill_rate: 0.92
  enum_compliance_rate: 0.87
  issues:
    - tag: P01
      issue: "function_category 'BOUNDARY' → V3.2 Enum에 없음"
```

### Step 3. 샘플 심층 비교 (수동 — 10건)

대상 선정 기준:
1. 세 결과 모두에 존재하는 엔티티 3건
2. 한쪽에만 존재하는 엔티티 3건
3. 공간(space) 2건
4. 거짓양성 의심 2건

비교 항목:
- tag, name, description 일치 여부
- discipline, function_category 값 차이
- specs 세부 속성 차이
- confidence 값 비교
- source(출처 페이지) 일치 여부

### Step 4. 평가 보고서 작성

**출력**: `EVAL-REPORT.md`

구조:
```
1. 요약 (어느 쪽이 더 나은가, 한 줄 결론)
2. 커버리지 비교 (합집합 대비 발견율)
3. 정확성 비교 (스키마 준수, Enum, 거짓양성)
4. 비용 효율 비교 (엔티티당 비용)
5. 발견된 문제점 (다음 파이프라인 개선 입력)
6. 권장 사항 (어떤 접근을 기본으로 할지)
```

## 4. 주의 사항

### 비교 시 고려할 점

- **에이전틱 152722 (94개)** 와 **교차검증 (49+53=102개)** 는 수량이 비슷하지만 분류 기준이 다를 수 있음
  - 에이전틱: entity만 추출 (space 별도 분류 안 했을 수 있음)
  - 교차검증: entity + space 분리 추출
  - → 태그 비교 시 entity/space 구분 없이 합산 후 비교

- **교차검증의 관계(relation) 0건**은 파이프라인 한계인지 의도적 제외인지 확인 필요

- **에이전틱 144018의 24개**가 유독 적은 이유 분석 필요
  - 풀 파이프라인인데 왜 152722(94개)보다 적은가?
  - 검증 과정에서 거짓양성을 제거한 결과인가?

- **거짓양성 판단 기준**: 건축도면에서 "설비"의 범위를 어디까지로 볼 것인가
  - 문/창문(D1, W1)은 설비인가 건축요소인가?
  - 도면 제목/날짜/약어는 명백한 거짓양성
  - 건축 구조물(TRENCH, PIT, SUMPPIT)은 설비? 공간?

- [ ] `20260408_152722`의 비용 데이터 확인 (cost_summary.json 누락)

- [ ] 분석 대상 도면 전체 페이지 수 확인 (68페이지? 122페이지?)
