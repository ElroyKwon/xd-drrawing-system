# 01 — Entity Roadmap: 엔티티 타입 누적 확장 순서

## 순서 결정 기준

1. **기하 단순**: polygon 몇 점인가 (실=다각형, 설비=박스 1개, 슬라브=다각형, 벽=선분)
2. **Vision 검출 쉬움**: 이름·위치가 명시된가 (실명 라벨 → 실 검출 쉬움)
3. **데이터 풍부**: 도면에 항목 수 많은가
4. **3D 완성도 기여**: 이것만 있어도 건물이 읽히는가

---

## 확장 순서

### v1 — 실 (room) ★ 1차 MVP 타겟

- **기하**: polygon 4~8점 (grid-anchored)
- **이름**: 직접 텍스트 라벨 ("회의실", "복도")
- **수**: 시트당 10~22개 (풍부)
- **3D 기여**: 건물 내부 구조 즉시 읽힘
- **검출 난이도**: 낮음 (Gemini 앵커율 p060 100%)
- **검수 기준**: 실 개수 육안 일치 + polygon 이 grid 안에 들어감

### v2 — 설비 (equipment)

- **기하**: box (footprint + height)
- **이름**: 설비 태그 (CH-001, VCB-001 등) — `data/doc-entity-links.json` 8개 이미 매핑됨
- **수**: 시트당 3~10개
- **3D 기여**: "여기가 기계실" 식별
- **검출 난이도**: 중 (기호+텍스트 조합)
- **검수 기준**: 태그 매칭 + 위치가 실 안에 들어가는지

### v3 — 층 외곽 (slab footprint)

- **기하**: 닫힌 polygon 1개 (층당)
- **수**: 층당 1
- **3D 기여**: 층 경계 명확화, 파라펫 표시
- **검출 난이도**: 중 (외벽 따라가기)
- **검수 기준**: polygon 이 모든 실·설비를 포함

### v4 — 벽 · 코어

- **기하**: 선분 path (벽), polygon (코어=계단/EV/PS)
- **수**: 시트당 10~20 (벽), 2~5 (코어)
- **3D 기여**: 물리적 구분, 수직 관통 시각화
- **검출 난이도**: 높음 (벽 선분은 치수·보조선과 구분 어려움)
- **의존**: Step 2 의 grid 앵커율 90%+ 전제

### v5 — 단면 · 입면 정보

- **기하**: 층고·슬라브 두께·파라펫 높이·외벽 마감
- **수**: 시트당 소량, 값으로만
- **3D 기여**: 기본값(2700/250mm) 을 실측으로 교체, 외벽 두께 반영
- **검출 난이도**: 높음 (단면도 읽기 전용 프롬프트 필요)
- **전제**: 실제 주단면(A-A') PDF 가 확보되어야 (arch_p064 는 계단실 전용)

### v6 — 문 · 창 · 가구 · 조명 · 콘센트 (장식 수준)

- 범위 밖. BIM 수준이므로 별도 요구사항 확보 후 판단.

---

## 누적 규칙

1. 각 버전은 **기존 씬 파괴 금지**. `scene/{type}.js` 를 추가만 함.
2. 검수 통과 전 다음 버전 진입 금지.
3. 진입 시 이전 `review-{type}.md` 서명 확인.
4. 특정 타입에서 막히면 skip 가능 — 다음 타입으로 이동, 이슈 로그.
