## 1. 화면 개요

![일일 보고서 목록 · 상태 뱃지](../screenshots/il-reports-list.png)

이 화면은 **날짜별 일일 보고서 4건**을 상태 뱃지·리비전·섹션 수와 함께 단일 세로 리스트로 보여줍니다. 사용자가 "어제 무슨 일이 있었는지" 또는 "이번 주 보고서의 확정 상태"를 한눈에 훑어보고, 원하는 보고서를 클릭해 상세·편집 화면으로 진입하게 하는 **얕고 빠른 진입점**입니다.

리스트에 들어가는 정보는 의도적으로 최소화되어 있습니다. 차트도, 요약 문구도, 썸네일도 없습니다. 대신 **상태(초안·검토됨·확정) → 리비전 번호 → 날짜 → 섹션 수**의 네 요소만으로 "읽을 준비가 되어 있는지"를 판단하게 합니다. 보고서라는 아티팩트가 워크플로우의 상태를 가진 문서라는 점을 전면에 내세운 UX입니다.

## 3. UX 상세 설명

### 3.1 카드 디자인 (단일 행 구조)

- 카드는 **흰 배경 + 얇은 회색 테두리 + 라운드**. 호버 시 테두리 색이 진해지며 클릭 가능함을 알려줍니다
- 내부는 **좌(콘텐츠) + 우(›화살표)** 의 단순 2구역 flex 레이아웃. 단일 세로 방향 리스트에서 "이 카드가 클릭 가능한 링크"임을 우측 화살표로 반복 강조합니다
- 콘텐츠 영역 첫 줄은 **상태 뱃지 + Rev 표시**. 둘째 줄은 **"YYYY년 M월 D일 일일 보고서"** 형식의 제목. 셋째 줄은 **"N개 섹션"** 메타
- 카드 한 장에 들어가는 정보는 네 개뿐 — 과도한 정보로 스캔 속도를 떨어뜨리지 않기 위한 의도적 절제

### 3.2 상태 뱃지 3종

| 상태 | 라벨 | 배경·글자 |
|---|---|---|
| `draft` | 초안 | 회색 배경 (`bg-slate-100 text-slate-600`) |
| `reviewed` | 검토됨 | 호박색 (`bg-amber-100 text-amber-700`) |
| `finalized` | 확정 | 에메랄드 (`bg-emerald-100 text-emerald-700`) |

상태 색상은 **신호등 은유**를 따릅니다. 회색(중립)→호박색(진행 중)→초록(완료)으로 색만으로 워크플로우의 위치를 인지합니다. 확정된 보고서와 초안을 시각적으로 뚜렷이 구별해, 외부 공유·상부 보고 가능한 것이 어느 것인지 즉시 판단하게 합니다.

### 3.3 Zustand 영속 저장소와 하이드레이션

이 페이지는 `useReportsStore()` 훅을 통해 **localStorage 기반 영속 상태**를 구독합니다. 새로고침해도 이전에 편집·확정한 보고서 상태가 유지됩니다.

- 서버 렌더 직후(`hydrated === false`)에는 "로딩…"이라는 텍스트만 표시
- 클라이언트 마운트 시 `loadFromStorage()`가 실행되면 `hydrated`가 `true`가 되고 카드 리스트가 그려짐
- SSR/CSR 간 hydration mismatch를 피하기 위한 의도적인 2-phase 렌더 패턴

### 3.4 날짜 포맷

- `period.start` (ISO 8601) → `toLocaleDateString("ko-KR", { year: "numeric", month: "long", day: "numeric" })`로 변환
- "2026년 4월 20일" 같은 한국어 자연어 형식. 숫자 `2026-04-20`보다 **사람이 읽기 좋은 형태** 우선

### 3.5 제목이 아닌 부제로 "AI 생성 + 담당자 편집" 강조

헤더 블록의 부제는 단순 설명이 아니라 **운영 모델의 공지**입니다. "이 보고서는 AI가 완전 자동 생성하지 않는다, 담당자가 편집한다"는 사실을 리스트 진입 단계부터 알려줌으로써 상세 화면의 편집 가능한 textarea에 대한 정신적 준비를 만듭니다.

## 4. 이 UX가 만드는 효과

| UX 결정 | 사용 경험에서의 변화 |
|---|---|
| 카드당 정보 4개로 절제 | 제목·리비전·상태·섹션 수만 노출해 스캔 속도 극대화 |
| 신호등 색상 뱃지 | 상태를 읽지 않고 색만으로 인지 → 확정 보고서와 초안을 즉시 구별 |
| 우측 › 화살표 반복 | 카드가 "링크"임을 텍스트 설명 없이 시각 기호로만 소통 |
| 부제 "AI 생성 + 담당자 편집" | 워크플로우의 역할 분담을 리스트 단계에서 선공지 |
| 한국어 자연어 날짜 | ISO 형식 대신 "2026년 4월 20일" — 내부 도구보다 문서 같은 톤 |
| Zustand + localStorage | 새로고침해도 편집 상태 유지 → 부담 없이 탐색하고 돌아올 수 있음 |
| hydrated 플래그로 2-phase 렌더 | SSR/CSR 불일치 회피 — 첫 페인트 때 잘못된 상태가 깜빡이지 않음 |
| 차트·그래프·썸네일 제로 | 상세에 가야 얻을 수 있는 정보를 리스트에 미리 보여주지 않음 → 상세 진입 동기 유지 |

## 5. 사용자 동작 흐름

| # | 액션 | 결과 | UX 의도 |
|---|---|---|---|
| 1 | 사이드 네비에서 "일일 보고서" 메뉴 진입 | 목록 페이지 렌더, "로딩…" 잠시 표시 후 카드 4장 등장 | 하이드레이션 지연을 사용자에게 숨기지 않음 |
| 2 | 상단 카드(최신) 스캔 — 초안(회색) | "오늘치는 아직 검토 안 됨" 즉시 인지 | 상태 뱃지 색으로 첫 판단 |
| 3 | 확정(에메랄드) 카드 스캔 | "이건 외부로 공유 가능" 즉시 인지 | 색 구별로 필터링 없이 상태 판별 |
| 4 | 카드 전체를 클릭 | `/insight/reports/[id]`로 이동, 상세·편집 화면 진입 | 카드 전체가 링크 히트 영역 (작은 버튼 클릭 스트레스 제거) |
| 5 | 상세에서 편집 후 뒤로 가기 | 목록에 돌아오면 상태 뱃지가 `reviewed`로 갱신되어 있음 | localStorage 영속 덕분에 워크플로우 진행이 시각적으로 누적 |
| 6 | 새 보고서가 `POST /api/reports/generate`로 만들어짐 (다른 경로) | 페이지 새로고침 시 초기 시드에 없는 신규 건이 앞에 붙어 노출 | 기존 편집 상태 보호 + 신규 건 병합 |

## 6. 데이터·API 의존성

### 원천 데이터 (빌드타임 import)
- `data/insight/reports/report-daily-20260420.json`
- `data/insight/reports/report-daily-20260419.json`
- `data/insight/reports/report-daily-20260418.json`
- `data/insight/reports/report-daily-20260417.json`

각 JSON은 `Report` 인터페이스(id · report_type · period · status · revision · sections[])를 따릅니다. `sections`는 자동 계산된 `python_data`(수치 dict)와 LLM이 생성한 `slm_narrative`(문장)로 구성됩니다.

### 상태 저장소
- `src/lib/insight/use-reports-store.ts` (Zustand 없이 `useState + localStorage` 수동 구현)
- Storage key: `dks-reports-v1`
- 저장 시점: `updateSection`, `finalizeReport` 액션 호출 시 자동 `saveToStorage`
- 병합 로직: localStorage에 없는 초기 시드 건은 병합 시 **앞쪽에** 추가됨 (최신순 유지)

### 참조하는 컴포넌트
- `next/link` (Link) — 카드 클릭 시 상세 페이지 이동
- 외부 UI 라이브러리 없음 — Tailwind 유틸리티 클래스만 사용

### 실제 LLM 호출 여부
**없음**. 이 화면은 순수 리스트 렌더링만 수행합니다. `slm_narrative` 문장은 빌드타임에 시드 JSON으로 포함되어 있어, 재생성이 필요하면 상세 화면의 "재생성" 버튼에서 `POST /api/reports/generate`가 호출됩니다.

## 7. 이 화면이 기여하는 서비스 측면

| DKS 서비스 측면 | 이 화면이 맡는 역할 |
|---|---|
| **서비스 2 (AI 인사이트) — 보고서 산출물** | 가장 가시적인 AI 결과물인 일일 보고서의 진입점 역할 |
| **운영 워크플로우 가시화** | 초안→검토→확정의 3단 상태를 시각 코드(색 뱃지)로 노출해 운영 리듬을 UI에서 재현 |
| **AI + 담당자 협업 모델** | 부제 "AI 생성 + 담당자 편집"과 리비전 표시로 **AI가 혼자 쓰지 않는다**는 운영 철학 전달 |
| **아카이브 가치 축적** | 날짜별 보고서가 쌓이면서 "우리 센터의 운영 역사"가 되는 시각적 증거 — 목업은 4일치만 있으나 구조는 무제한 확장 |
| **낮은 진입 장벽** | 복잡한 필터·검색 없이도 "최근 며칠치"만 빠르게 훑을 수 있는 가장 빠른 경로 |

**이 화면이 해결하지 않는 것**: 필터링(상태별·기간별), 검색, 여러 보고서 간 비교·diff, CSV·PDF 다운로드, 다른 사용자와의 공유 링크 — 모두 상위 워크플로우 화면의 영역으로 둡니다.

## 8. 의견 수렴 포인트

### 스스로 본 보완 포인트

- **필터·검색 없음**: 4건만 있을 때는 괜찮지만 30일치가 쌓이면 상태별·기간별 필터 칩이 필수. 현재는 세로 스크롤밖에 없음
- **다운로드·공유 버튼 부재**: "확정"된 보고서를 외부로 내보내는 기능이 없음. PDF·DOCX 내보내기, 공유 링크 복사가 보고서의 마지막 단계로 필요
- **알람·이벤트와의 연결 고리 없음**: 리스트 카드에서 해당 일자에 어떤 CRITICAL 알람이 있었는지 미니 인디케이터(점 3개 등)가 있으면 중요도 판단이 빨라질 것
- **신규 보고서 생성 버튼 없음**: 현재는 API를 직접 호출해야 신규 생성. 목록 상단에 "오늘 보고서 생성" 버튼이 있어야 자연스러움
- **섹션 수 외의 품질 지표 없음**: "4개 섹션"만으로는 보고서 풍부도를 판단하기 어려움. "차트 N개 포함", "확인된 알람 M건 참조" 같은 보조 지표 고려
- **모바일 레이아웃**: 카드가 풀폭이라 모바일에서도 그대로 작동하나, `›` 화살표가 우측에 있어 오른손 엄지 거리에서 약간 멀 수 있음
- **정렬 옵션**: 항상 `initialReports` 배열 순서(최신순)만 지원. 오래된 것부터, 상태별 정렬 옵션 고려
