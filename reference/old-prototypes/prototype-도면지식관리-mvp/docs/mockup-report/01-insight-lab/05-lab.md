# 화면 · Insight Lab 화면 1 · 통계×온톨로지 실험실

**경로**: `/insight/lab`
**소속 트랙**: Insight Lab (1-트랙)
**화면 분류**: 실시간 분석·대화형 질의 중심

---

## 1. 화면 개요

![Insight Lab · 통계×온톨로지 Bundle 조립 상태](../screenshots/il-lab-bundle.png)

이 화면은 **가상 통계 이벤트 리스트 → 온톨로지와 결합한 Bundle(묶음) → Gemini 대화**로 이어지는 한 호흡의 흐름을 하나의 화면에서 시연합니다. 사용자는 좌측에서 실시간 이상 징후 이벤트를 클릭하고, 중앙에서 그 이벤트가 엔티티·관계·근거 도면과 어떻게 결합되었는지 확인하며, 우측에서 Bundle 컨텍스트를 근거로 Gemini에게 질의합니다.

전통적인 대시보드는 수치 차트와 알람 리스트만 나열하지만, Insight Lab은 **"통계 이상치가 나왔다"는 사실 자체보다 그 수치가 어떤 설비·어떤 상위 계통·어떤 도면과 맞물려 있는지를 결합한 의미 번들**을 보여줍니다. 이 번들이 그대로 LLM 프롬프트에 주입되기 때문에, 챗 응답은 일반론이 아니라 "이 엔티티·이 스펙·이 관계에 근거한" 답변이 됩니다. 좌측에서 이벤트를 바꾸면 중앙과 우측이 동시에 교체되는 3-way 연동이 핵심입니다.

## 3. UX 상세 설명

### 3.1 좌측 — 이벤트 카드 디자인

- 카드 상단에 **SEVERITY 뱃지**(CRITICAL·WARNING·NORMAL)와 **카테고리 태그**(열·통계·무결성)가 나란히 표시됩니다. 시선이 먼저 "얼마나 급한가 · 어떤 종류 이상인가"에 닿게 하기 위해서입니다
- 우상단에 타임스탬프를 `MM/DD HH:mm` 포맷으로 축약해, 이벤트가 **언제 발생했는지** 빠르게 인지하게 합니다
- 본문은 **대상 라벨 → 메트릭=값·단위 / 임계값 → narrative_hint**의 세 줄로 쌓입니다. 메트릭과 값은 모노스페이스로 렌더해, 수치 이상성을 시각적으로 강조합니다
- `narrative_hint`는 2줄 line-clamp로 잘려서 "이 이벤트가 왜 찍혔는가"에 대한 힌트만 노출합니다 — 상세는 중앙 Bundle 뷰에서 보도록 유도
- 선택된 카드는 `bg-slate-100`으로 배경색이 바뀌어, 중앙·우측과의 **연결**을 시각적으로 유지합니다

### 3.2 중앙 — Bundle 뷰의 5단 수직 스택

Bundle 뷰는 통계-엔티티-관계-도면-요약 순으로 **의미의 깊이가 점진적으로 증가**하는 구조입니다. 독자가 스크롤을 내리면서 단순 수치에서 출발해 엔티티 정의, 계통 관계, 근거 도면, 최종적으로는 LLM에 주입되는 요약 문장까지 내려오도록 설계되어 있습니다.

| 섹션 | 렌더 | 의도 |
|---|---|---|
| 헤더 | SEVERITY 뱃지 + event-id + 로컬 타임스탬프 + (옵션) `온톨로지 미매칭` 배지 | 수치 이상성과 온톨로지 매칭 상태를 한 줄로 압축 |
| 통계 이벤트 | 4-column `StatBox` 그리드 (대상·메트릭·측정값·임계값) + `💡 narrative_hint` | 수치 카드를 쪼개서 나열 → 원본 통계의 "무엇이 얼마나" 인식 |
| 온톨로지 엔티티 | 큰 카드 하나. tag·name·discipline·type + 주요 스펙 8개 (2열 그리드) | 그 설비가 YAML 온톨로지 내에서 어떻게 정의되어 있는지 |
| 관계 확장 | 최대 2홉, 행마다 1-HOP/2-HOP 뱃지 + `from → relation → to` | 엔티티를 **계통 네트워크의 한 노드**로 인식하게 함 |
| 근거 도면 | 2-column 카드 그리드. 클릭 시 라이트박스로 PDF 페이지 이미지 | 도면을 **바로** 볼 수 있다는 감각 제공 |
| Semantic Summary | 검은 배경 + 흰 모노 텍스트 | 이 모든 정보가 LLM 프롬프트로 어떻게 직렬화되는지 투명하게 공개 |

### 3.3 중앙 — 온톨로지 미매칭(no_match) 분기

이벤트의 `target_tag`가 온톨로지에서 매칭되지 않으면 엔티티 카드 자리에 **호박색 경고 박스**가 표시됩니다:

> "현재 온톨로지는 R-Center 기계·전기 설비(379건)만 커버합니다. 배터리(RACK), 인버터, BMS 통신 채널 등은 별도 YAML 확장이 필요합니다. — 목업은 한계를 숨기지 않습니다."

이 박스의 톤은 의도적으로 **방어적이지 않고 투명합니다**. "모든 것을 알 수 있다"는 허위 약속을 하지 않고, 현재 온톨로지가 무엇을 커버하고 무엇을 못 하는지를 이해관계자에게 직접 드러냅니다.

### 3.4 중앙 — 관계 확장 행 (RelatedRow)

- 좌측에 `1-HOP`(에메랄드) 또는 `2-HOP`(회색) 뱃지
- 그다음 `from → [relation] → to` 형식. `direction`이 `outgoing`이면 루트 태그가 왼쪽, `incoming`이면 오른쪽
- `relation` 이름은 옅은 회색 배경 chip (`supplies_chw`, `feeds` 등)
- 행 끝에 `via` 힌트가 있으면 표시 (예: 2홉 경유 노드)

### 3.5 중앙 — 근거 도면 카드 + 라이트박스

- 카드는 파일 아이콘 + `doc_id · p.N` (모노스페이스) + reason 한 줄
- 카드 클릭 시 `ImageLightbox`가 열리며 `page_NNN.png`를 로드
- 라이트박스 상단에 **"전체 뷰어로 열기 ↗"** 링크 — 도면지식관리 본 서비스(`/drawings/[id]?page=N`)로 새 탭 열기
- 이미지가 없는 페이지는 `onError` 시 dashed 박스로 fallback — "샘플 이미지가 없습니다 · p.3·4·5·10·22·40만 커버" 문구로 목업 한계를 그대로 노출
- ESC 키 · 배경 클릭으로 닫기 지원

### 3.6 우측 — Bundle Chat

- 상단에 `Bundle Chat` 레이블 + 현재 엔티티 태그 안내 ("VCB-001 근거로 답변합니다")
- 대화가 없으면 **추천 질문 3개** 표시:
  - "이 이상의 물리적 원인 후보는?"
  - "어떤 스펙·관계를 추가로 확인해야 할까?"
  - "관련 도면은 어디에서 보지?"
- 추천 질문은 클릭 한 번에 전송 — 사용자가 "무엇을 물어야 할지"를 고민하지 않게 함
- no_match 이벤트일 때는 추천 질문이 `disabled + 40% opacity`로 표시되고, 아래에 **"일반론만 가능"** 안내가 나타남
- 사용자 말풍선은 검은 배경 + 우측 정렬, AI 말풍선은 흰 카드 + AI 아바타 + 좌측 정렬
- AI 응답은 **SSE 스트리밍** (`data: {token: "..."}`)으로 실시간 누적. 스트리밍 중에는 커서 `animate-pulse`로 타이핑 감각 부여
- Markdown 렌더링 (ReactMarkdown + remarkGfm) — 표·리스트·코드블록 전부 지원
- 입력창은 2행 textarea, Enter 전송·Shift+Enter 줄바꿈. 전송 버튼은 검은 배경 원형 아이콘

### 3.7 Bundle 전환 시 대화 초기화

새 이벤트를 선택하면 `useEffect`에서 `bundle.event.id` 변화를 감지해 `messages`와 `input`을 리셋합니다. 이전 Bundle에 대한 대화가 새 Bundle 컨텍스트에 섞이지 않도록 **명시적으로 끊어 줍니다**.

## 4. 이 UX가 만드는 효과

| UX 결정 | 사용 경험에서의 변화 |
|---|---|
| 좌·중·우 3분할 한 화면 | 이벤트 선택 → Bundle 조립 → 질의의 흐름을 페이지 이동 없이 한 호흡으로 완결 |
| Bundle 뷰를 5단 수직 스택으로 | 통계(얕음) → 엔티티 → 관계 → 도면 → 프롬프트(깊음) 순으로 의미를 내려가면서 읽게 유도 |
| 관계 확장을 최대 2홉까지만 | 온톨로지 전체 그래프를 열람하는 게 아니라 **이 이벤트에 충분한** 이웃만 선별해 인지 부하 관리 |
| Semantic Summary를 검은 배경으로 노출 | LLM 프롬프트가 무엇인지를 숨기지 않고 투명하게 공개 → 이해관계자가 AI 응답의 근거를 신뢰 가능 |
| no_match 경고 박스 | "AI가 모든 것을 답한다"는 허위 약속 대신 온톨로지 커버리지 한계를 직접 드러내 신뢰 손상 방지 |
| 추천 질문 3개 (고정 프리셋) | 첫 질의 시 "무엇을 물어야 하지"의 빈 백지 공포 제거 |
| 챗 응답 SSE 스트리밍 | 완성된 답변 블록을 기다리는 대신, 토큰 단위로 흘러나오는 느낌 → "실시간으로 사고한다"는 감각 |
| Bundle 전환 시 대화 초기화 | 이전 맥락과의 오염 방지 · 새 이벤트에 대한 **깨끗한** 추론 보장 |
| 근거 도면 라이트박스 | 챗 응답이 참조한 도면을 한 클릭에 확대해 보고, 새 탭으로 본 서비스까지 즉시 이동 |

## 6. 데이터·API 의존성

### 원천 데이터
- `data/insight/mock-events.json` — 통계 이벤트 시드 (CRITICAL·WARNING·NORMAL 혼합, 카테고리 3종)
- YAML 온톨로지 (`@/lib/ontology/loader`의 `EntityDetail`) — R-Center 기계·전기 설비 379건
- `data/insight/root-cause/*.json` — 엔티티별 관계 그래프(부분) + 근거 도면 매핑
- `public/drawings/page_NNN.png` — 6장 (p.3·4·5·10·22·40) 샘플 이미지

### API 엔드포인트
- `GET /api/insight/lab/bundles` — 이벤트 리스트 전체 반환 (`events: EventSummary[]`)
- `GET /api/insight/lab/bundles?id={event_id}` — 특정 이벤트 ID로 Bundle 조립 (`bundle: InsightBundle`)
- `POST /api/insight/lab/chat/stream` — Bundle을 컨텍스트로 Gemini 스트리밍 응답 (SSE)

### 참조하는 lib/컴포넌트
- `@/lib/insight-mockup/types` — `EventSummary`, `InsightBundle`, `BundleRelated`, `BundleEvidence`, `BundleStatus`
- `@/lib/ontology/loader` — `EntityDetail` 타입 + YAML 로딩
- `@/app/(s2)/insight/lab/EventList` — 좌측 카드 리스트
- `@/app/(s2)/insight/lab/BundleView` — 중앙 5단 스택
- `@/app/(s2)/insight/lab/BundleChat` — 우측 챗
- `@/app/(s2)/insight/lab/ImageLightbox` — 근거 도면 라이트박스

### 실제 LLM 호출 여부
**있음** (조건부). `/api/insight/lab/chat/stream` 엔드포인트가 Gemini API를 호출합니다. Bundle의 `semantic_summary`가 프롬프트 시스템 메시지에 주입되어, 응답이 해당 엔티티·스펙·관계에 기반하도록 제약됩니다. API 키가 없거나 호출이 실패하면 모의 응답(mock-responses)으로 fallback합니다.

## 7. 이 화면이 기여하는 서비스 측면

## 8. 의견 수렴 포인트

### 스스로 본 보완 포인트

- **샘플 이미지 6장 한계**: 온톨로지 YAML의 `source_page`는 원본 PDF 기준이라 라이트박스에서 fallback dashed 박스가 자주 노출됨. 본 시연 전에 추가 페이지 이미지 확보 필요
- **관계 확장 2홉 고정**: 시각적 복잡도 때문에 2홉으로 잘랐지만 전기 계통처럼 깊은 의존성에서는 3~4홉이 필요할 수 있음 — 사용자가 hop 레벨을 토글할 수 있어야 하나
- **추천 질문 3개 고정**: Bundle 카테고리(열·통계·무결성)별로 다른 질문을 제공하는 게 맞음. 현재는 모든 Bundle에 같은 3개
- **no_match 이벤트에서도 LLM 호출은 가능**: 다만 응답 품질은 일반론 수준. "디폴트로 차단"이 맞는지 "주의 문구 + 허용"이 맞는지 운영 정책 결정 필요
- **이벤트 필터링 없음**: 좌측 리스트가 단순 세로 나열. 카테고리·SEVERITY·설비 그룹별 필터가 필요할 수 있음
- **Semantic Summary 노출 여부**: 프롬프트를 그대로 공개하는 것이 투명성 측면에서는 좋으나, 운영 시연에서는 "내부 세부사항 과노출" 지적 가능. 접고 펼치기 토글 고려
- **Bundle 조립 속도**: 현재 API는 mock JSON 읽기라 100ms 이내지만, 실 온톨로지 그래프 쿼리 + 도면 매칭이 결합되면 로딩 표시(스켈레톤) 보강 필요
