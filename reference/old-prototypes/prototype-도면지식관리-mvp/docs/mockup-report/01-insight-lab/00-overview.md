# 트랙 · Insight Lab 개요

**경로 루트**: `/insight/*`
**소속 트랙**: Insight Lab (1-트랙) · DKS 서비스 2 계열
**포함 화면 수**: 8개

---

## 1. 트랙 소개

Insight Lab 트랙은 **운전 중 발생하는 설비 이상 신호를 AI가 해석하고, 도면·엔티티·과거 이력을 근거로 삼아 일상 보고까지 연결하는** 일련의 화면 묶음입니다. 화면 하나로 모든 것을 보여주는 대신, **"알람이 뜬다 → 이 알람이 왜 났는지 안다 → 자유롭게 더 물어본다 → 근본원인을 시각적으로 확인한다 → 그 결과가 일일 보고서에 자동으로 쌓인다"** 라는 하나의 업무 흐름을 8개 화면으로 쪼개 배치했습니다.

트랙의 핵심 가치는 다음 세 축입니다.

1. **AI 기반 설비 이상 분석** — 원시 알람을 사람이 읽을 수 있는 "왜·어디서·무엇을 할지"로 변환
2. **통계 × 온톨로지 기반 Bundle 질의** — 숫자로만 보이는 이벤트를 도면·엔티티 관계와 결합해 대화로 탐색
3. **일일 보고서 자동화** — 하루 동안의 알람·질의·근본원인 결과를 편집 가능한 보고서로 자동 수합

기존 도면 뷰어(서비스 1)가 "찾아본다"에 최적화되어 있다면, Insight Lab은 **"신호가 들어온다 → 해석하고 기록한다"**의 능동적 흐름을 담당합니다.

## 3. 트랙이 해결하는 서비스 측면

| DKS 서비스 측면 | 이 트랙이 맡는 역할 |
|---|---|
| **AI 기반 설비 이상 분석** (서비스 2 핵심) | 허브·알람 목록에서 시작해 답변 상세·근본원인 그래프로 이어지는 분석 파이프라인 |
| **통계 × 온톨로지 결합 질의** | 랩(`/insight/lab`) 화면에서 통계 이벤트를 온톨로지 관계와 묶은 *Bundle*로 AI와 대화 |
| **자연어 기반 운전 질의** | 챗(`/insight/chat`)은 알람과 독립된 자유 질문 창구 — "지난 주 3F 소방 점검 결과는?" 식 |
| **일일 보고서 자동화** | 하루치 알람·답변·원인 결과가 보고서 초안으로 자동 수합, 사람이 편집·확정 |
| **도면 지식과의 연계** | 답변 상세·근본원인에서 참조 도면이 역링크로 나타나 서비스 1로 자연 전이 |
| **운전 맥락의 기록화** | 기록 없이 잊히던 "왜 그랬더라" 지식을 보고서로 고정 |

**이 트랙이 해결하지 않는 것**: 도면 자체의 탐색·편집·주석. 그 영역은 서비스 1(`src/app/page.tsx`) 및 Design Lab 트랙이 맡습니다.

## 4. 화면 간 이동·관계

### 4.1 주요 흐름 (전형적인 업무 시나리오)

```
  [허브 /insight]
        │
        ├── 최근 알람 섹션 클릭 ──▶ [알람 목록 /insight/alarms]
        │                                  │
        │                                  └─ 알람 클릭 ──▶ [답변 상세 /insight/answers/:id]
        │                                                        │
        │                                                        ├─ "근본원인 보기" ──▶ [근본원인 /insight/root-cause/:id]
        │                                                        └─ "참조 도면" ──▶ (서비스 1 도면뷰어로 전이)
        │
        ├── 자주 묻는 질문 섹션 ──▶ [AI 자유 질의 /insight/chat]
        │
        ├── 일일 보고서 섹션 ──▶ [보고서 목록 /insight/reports]
        │                                  │
        │                                  └─ 항목 클릭 ──▶ [보고서 상세 /insight/reports/:id]
        │
        └── (별도 진입) ──▶ [Insight Lab /insight/lab]
                                   │
                                   └─ Bundle 대화 결과가 보고서 섹션 시드가 됨
```

### 4.2 허브 동심원 구조

허브(`/insight`)는 **세 개의 동심원**을 담습니다.

- **최근 알람** (실시간성 높음) — 새 신호가 있는지 즉시 인지
- **자주 묻는 질문** (익숙함) — 오늘 처음 쓰는 사람도 첫 클릭을 고르기 쉽게
- **일일 보고서** (회고성) — 어제까지의 결과를 회고

허브는 진입 시 부담을 낮추기 위해 **카드 3개 초과를 보여주지 않습니다**. "전체보기 →" 링크로 상세 목록(알람 목록·챗·보고서 목록)으로 나뉩니다.

### 4.3 Lab과 다른 화면의 관계

`/insight/lab`은 나머지 7개와 **느슨하게 연결된 별도 시연 화면**입니다. 통계 이벤트를 온톨로지와 합쳐 AI에게 맥락을 주는 기법 자체를 시연하는 목적이라, 허브에서 직접 링크되지 않고 별도 진입합니다. 단 Lab에서 생성된 Bundle 요약은 **보고서 섹션의 시드로 재사용**되도록 설계돼 있어, 장기적으로 보고서 흐름과 합류합니다.

## 5. 트랙 공통 UX 원칙

### 5.1 좌우 분할 + ChatPanel 재사용 패턴

답변 상세·챗·랩·근본원인 화면 모두 **공통의 챗 패널(ChatPanel) 컴포넌트**를 재사용합니다. 화면마다 좌측 콘텐츠(알람 정보 / 통계 이벤트 / 그래프)는 다르지만 **우측 챗 영역의 외형·조작법은 동일**합니다. 사용자가 한 번 챗 조작을 익히면 모든 하위 화면에서 동일하게 쓸 수 있습니다.

### 5.2 AI 응답은 항상 "근거"를 달고 온다

AI의 모든 답변에는 참조된 도면·엔티티·이전 알람이 **카드나 칩 형태로 함께 표시**됩니다. "AI가 이렇게 말했다"가 아니라 "이 근거들로 이렇게 말했다"가 기본 표현 방식입니다. 이는 운전 현장에서 AI 답변을 **검증 가능한 의견**으로 다루기 위한 약속입니다.

### 5.3 Severity 색 체계

`critical = 로즈(rose-500)` / `warning = 앰버(amber-400)` / `info = 스카이(sky-400)` 는 모든 화면에서 동일하게 적용됩니다. 허브의 점 인디케이터, 알람 목록의 좌측 바, 보고서 배지까지 모두 같은 매핑을 씁니다.

### 5.4 라이트 톤 + 슬레이트 베이스

전체 배경은 `bg-slate-50`, 카드 표면은 `bg-white`, 텍스트는 `text-slate-700~900`. 다크 모드를 피하고 **문서·보고서에 가까운 정적인 라이트 톤**을 고수합니다. 설비 운전 현장의 조명 환경(밝은 현장·주간 근무 중심)과 **보고서 인쇄 친화성**을 동시에 고려한 선택입니다.

### 5.5 "목록 → 상세 → 관련"의 3-depth

8개 화면은 모두 `목록(리스트) → 개별 상세 → 관련된 다른 화면으로 전이` 패턴을 따릅니다. 사용자는 최대 3-클릭 안에 자신이 있는 깊이를 파악하게 되고, 뒤로가기로 즉시 복귀할 수 있습니다.

### 5.6 실제 LLM 호출과 목업의 혼재

Lab·챗 일부는 실 Gemini API에 연결되고(`src/lib/insight/llm-client.ts`), 나머지는 시드 응답입니다(`mock-responses.ts`). 이 구조는 **데모 중 안정성**을 위한 타협이며, 본 서비스 연결 시에는 단일 채널로 통합됩니다.

## 6. 의견 수렴 포인트 (트랙 전체 관점)

### 스스로 본 보완 포인트

- **허브의 "최근 알람 3건"이 충분한가**: 심각도가 높은 critical이 5건 이상 들어올 때도 3건으로 자르면 놓칠 수 있음. 상한을 severity 기반으로 조정할지, 또는 critical만 전부 노출할지 논의 필요
- **랩과 나머지의 분리감**: Lab 화면이 나머지 7개와 시각적으로 너무 달라 "같은 트랙인지" 인지가 어렵다는 내부 피드백. 상단 네비에 트랙 뱃지 도입 검토
- **보고서 자동화의 원천 경로**: 현재 초안은 시드 JSON. 실 운영에서는 "어디서 어떤 주기로" 자동 집계할지가 미결
- **근본원인 그래프의 인터랙션 깊이**: 현재는 클릭해서 노드를 확장·축소하는 정도. 현장 사용자가 "왜 이게 원인인지 더"를 원할 때 어떻게 drill-down 시킬지
- **자유 질의(챗) vs Bundle(랩)의 경계**: 두 화면 모두 "AI와 대화"지만 목적이 다름. 사용자가 "어느 화면을 열어야 하는지" 첫 판단에서 헤매지 않을지 검증 필요
- **모바일·태블릿 대응**: 현재 좌우 분할이 1100px 이하에서 무너짐. 현장 테블릿 사용을 가정하면 시트(drawer) 전환 필요
