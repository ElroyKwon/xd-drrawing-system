# 비교 매트릭스 — Insight Lab · Design Lab

**읽는 시간**: 10~15분
**목적**: 두 트랙(Insight Lab 8화면 · Design Lab 3화면)이 각각 어떤 정체성을 가지고, 어떤 UX 축에서 같고 다른지, 서비스 측면에서 어디가 중첩되고 어디가 고유한지를 한 문서에서 대조합니다. 개별 화면 문서를 읽은 후 **"이제 둘을 같이 보고 판단해야 한다"** 단계의 지도로 사용하십시오.

> 이 문서는 **결론을 내리지 않습니다**. 합침·분리 양쪽 모두 합리적인 선택지로 제시하고, 결정 근거는 이해관계자 의견 수렴(§5) 이후에 별도 회의·FRS에서 확정합니다.

## 1. 두 트랙의 정체성 비교 (1줄 요약)

| 축 | Insight Lab (8화면) | Design Lab (3화면) |
|---|---|---|
| **주역** | AI가 해석한 **이상 신호·보고서** | 사용자가 탐색하는 **도면 자산** |
| **대화의 역할** | 답변 생성의 주역 — 알람·보고서 맥락에서 추론 엔진 | 탐색의 보조(화면 1) 또는 질의의 주역(화면 2) — 두 대안 제시 |
| **진입 목적** | *"이 이상이 왜 발생했나"* · *"오늘 뭐가 일어났나"* | *"뭐가 있나 훑자"* · *"이 질문에 답해줘"* |
| **데이터 의존성** | 알람 시드 · 보고서 시드 · 온톨로지 YAML · 근본원인 그래프 JSON · mock 또는 Gemini 응답 | `documents.json` 26건 + 도면-엔티티 링크 + 목업 메타 시드 — **실 LLM 호출 없음** |
| **레이아웃 특성** | 좌우 2분할(58:42 또는 62:38)을 6화면이 재사용, 허브·리스트·랩만 다른 구조 | 3화면 모두 **서로 다른 레이아웃**: 허브(카드 2개), 카드 그리드(3분할·우측 듀얼), 챗 워크스페이스(3분할·중앙 고정) |
| **트랙 정체성 색** | 라이트 슬레이트 베이스 + 심각도 3색(rose·amber·sky) | 라이트 슬레이트 베이스 + violet/fuchsia · cyan/blue 그라디언트 페어 |
| **실 LLM 연결** | 있음 (`/api/chat/stream`, `/insight/lab/chat/stream`, `/api/insights/*/stream` — mock 폴백) | 없음 — 시나리오 JSON + setTimeout |
| **프로덕션 영향** | 프로덕션 서비스 2의 **핵심 산출물 후보**로 이어짐 | 프로덕션 서비스 1(도면 뷰어)의 **UX 개선 검토용** — 채택 여부 미결 |

## 2. UX 축별 비교표

행은 공통 UX 축, 열은 두 트랙. 각 셀은 해당 트랙의 구체적 선택이며, "비고" 열은 두 선택이 **서로 다른 이유 또는 교훈**을 적습니다.

| UX 축 | Insight Lab | Design Lab | 비고 |
|---|---|---|---|
| **정보 밀도** | 좌측 풍부(답변+근거+구조화+후속) / 우측 대화 스트림 — 한 화면에 읽을 게 많음 | 화면 1은 썸네일 카드 1~3열(훑기 최적) / 화면 2는 중앙 대화만 집중 | Insight는 "정제된 보고서를 읽힌다"는 태도, Design은 "의도별로 밀도를 조절"하는 태도 |
| **조작감** | 주로 **읽고 클릭**(알람→드로어→답변→근거→후속) · 편집은 보고서 화면 한정 | 화면 1은 **훑고 토글**(공종·카드·탭·모드) · 화면 2는 **대화로 조작**(재생→누적→클릭으로 좌측 전환) | Insight는 일방향 업무 흐름, Design은 양방향·탐색적 조작 |
| **스트리밍 연출** | SSE 토큰 스트리밍(실 API) + 깜빡이 캐럿이 전 화면 표준 — "AI가 생각 중"의 살아있는 감각 | 시나리오 재생 `searching → reading → answering` 3단(setTimeout) — "에이전트가 찾고 읽고 답한다"의 단계적 체감 | 같은 "스트리밍 감각"을 목표로 하지만 **실 SSE vs 연출 애니메이션**으로 수단이 다름 |
| **탭 구조** | 답변 상세의 좌측 5-부 수직 스택(QueryHeader·AnswerText·Evidence·Structured·FollowUp), 랩의 5단 수직 스택 — **탭이 아닌 수직 누적** | 화면 1 인스펙터의 **가로 6탭**(미리보기·목적·태그·대화요약·주석·온톨로지) — 한 번에 하나에 집중 | Insight는 "정보 계층을 시간으로 전개", Design은 "정보 덩어리를 공간으로 쪼갬" |
| **사이드 패널 용도** | 좌측: 구조화 콘텐츠 주인공 / 우측: 챗 (ChatPanel 재사용) — 역할 고정 | 좌측: 진입 모드에 따라 **형태를 바꿈**(리스트↔뷰어) / 우측: 누적 스택(수동 추가 아님, 답변이 push) | Insight는 패널 역할을 고정해 학습 용이 / Design은 맥락에 따라 패널이 변형되어 유연 |
| **상태 관리** | localStorage(보고서 편집) + SSE 스트리밍 상태 + AbortController cleanup — **실 서비스 수준의 상태 조율** | 클라이언트 useState 중심 · 시나리오 배열 순회 · URL 쿼리(`?source=library&id=`)로 딥링크 — **목업 수준의 얕은 상태** | Design Lab은 의도적으로 상태를 가볍게 유지(실험 트랙이므로) |
| **근거·출처 표시** | "AI는 근거를 달고 온다" 원칙 — evidence 카드 + 관련도% + 도면 열기 링크 | 화면 2의 참조 도면 카드 · 추출 엔티티 chip 자동 누적 | 두 트랙 모두 **답변의 출처 투명성**을 공통 UX 원칙으로 가져감 |
| **서비스 1 연결** | 답변 evidence · 근본원인 candidate evidence · 랩의 라이트박스에서 **외부 링크**로 `/drawings/[id]?page=N` 점프 | 카드 그리드 자체가 서비스 1 개선 후보 · 화면 2 우측 참조 도면은 내부 좌측 전환 | Insight는 "서비스 1을 **인용**", Design은 "서비스 1을 **대체·확장**" 검토 |
| **빈 상태 디자인** | 챗 EmptyState에 추천 질문 5개, 드로어 선택 전 그리드만, 근거 카드 없음일 때 조용히 생략 | 허브는 빈 상태가 본문, 화면 1은 초기 선택 없음 상태, 화면 2는 큰 💬 + 예시 프롬프트 3개 | 두 트랙 모두 **"뭘 해야 하지 공포"** 완화에 공들임 |
| **모바일 대응** | 모든 화면 스스로 인정: "좁은 화면에서 2분할 붕괴 — 반응형 미검증" | 허브만 `lg:grid-cols-2 → 1` 자동 fallback, 나머지 2화면은 1100px/1600px 데스크탑 전제 | **공통 숙제** — 양 트랙 모두 모바일·태블릿 대응이 의견 수렴 이후의 과제 |
| **실패·에러 UX** | "부분 수신" 상태 보존, mock 폴백 자동 전환, 토스트로 재생성·확정 피드백 | 대부분 실패가 나올 여지 없음(로컬 시드만 다룸). 화면 2는 "목업 환경 안내" 고정 문구 | Insight는 **실 서비스 근접** 수준의 방어, Design은 **시연 안정성** 중심 |
| **접근성** | Severity 색 + 텍스트 배지 병기, 키보드 포커스 button 요소 사용, 스크린리더용 live region은 미흡 | 컬러만으로 화면 1·2 구분하는 부분 있음(허브 배지), 숫자 뱃지로 최소 보완 | 두 트랙 모두 접근성은 "차후 보강" 단계 |

## 3. 서비스 측면 중첩/고유

두 트랙이 **같은 서비스 측면을 다루는 부분**과 **각자만 다루는 부분**을 나누어 정리합니다. 중첩은 "합치면 시너지가 나는가", 고유는 "섞으면 훼손되는가"를 판단할 재료입니다.

### 3.1 중첩 — AI 질의 (Insight/chat vs Design/chat)

두 트랙 모두 **"AI에게 자유롭게 물어보는 화면"**이 있습니다. 겉보기에는 비슷하지만 역할과 구현이 다릅니다.

| 측면 | [Insight · `/insight/chat`](./01-insight-lab/04-chat.md) | [Design · `/design-lab/chat`](./02-design-lab/03-chat.md) |
|---|---|---|
| **레이아웃** | 단일 중앙 컬럼(`max-w-3xl`) · 전형적 챗봇 | 3분할 워크스페이스 · 중앙 고정 챗 + 좌(모드별 패널) + 우(누적 스택) |
| **AI 호출** | 실제 SSE (`/api/chat/stream`) — Gemini 또는 mock 폴백 | 시나리오 재생(setTimeout) — 실 LLM 없음 |
| **답변 연출** | SSE 토큰 타자기 + 깜빡이 캐럿 | 3단 연출: searching → reading → answering |
| **답변 부산물** | 메시지 아래 근거 카드 + 후속 질문 칩 | 답변에 엔티티 chip + 우측 스택에 참조 도면 자동 push |
| **세션 유지** | 대화 이력 비영속 · "새 대화" 버튼으로 초기화 | 우측 스택이 대화 세션 동안 누적 · 초기화 UI 부재 |
| **빈 상태** | 추천 질문 5개 버튼(공종 다양화) | 예시 프롬프트 3개 pill(클릭 불가, 시각 힌트용) · ▶ 재생 버튼 |
| **철학** | "AI가 자유 질문을 받는 창구" — 단발·플랫 | "AI가 조사·독해·답변을 거쳐 지식을 누적하는 작업 공간" |

**어떻게 다른가**: Insight는 **답변 자체의 품질과 근거 표시**에 집중한 실제 대화형 챗. Design은 **대화가 만드는 주변 효과**(참조 누적·모드 전환)에 집중한 연출적 워크스페이스. 둘은 경쟁 관계가 아니라 **"같은 기술(AI 질의)로 무엇을 달성할 것인가"**의 두 시각입니다.

### 3.2 고유 1: 온톨로지 Bundle (Insight Lab 전용)

[Insight Lab 화면 5 — `/insight/lab`](./01-insight-lab/05-lab.md)의 핵심. 통계 이벤트(수치 이상치) × 온톨로지 엔티티(YAML 379건) × 2홉 관계 × 근거 도면 × Semantic Summary(LLM 프롬프트)를 **하나의 Bundle**로 직렬화한 뒤, 우측 챗에 컨텍스트로 주입합니다.

- Design Lab에는 **유사 개념이 없습니다** — 카드 그리드의 "온톨로지 탭"은 단일 도면 중심의 작은 SVG 그래프로 범위가 훨씬 좁음
- Bundle은 "숫자만 보이는 이상이 어떤 설비·계통·도면과 맞물려 있는지"의 **의미 결합**이 핵심
- no_match 분기(온톨로지 커버리지 밖)를 호박색 경고로 투명 공개 — Design Lab에는 이런 "한계 고백" UI가 없음

### 3.3 고유 2: 카드 탐색 + 자연어 필터 (Design Lab 전용)

[Design Lab 화면 1 — `/design-lab/cards`](./02-design-lab/02-cards.md)의 핵심. 썸네일 중심 카드 그리드 + 공종 필터 + 우측 6탭 인스펙터 + 우측 듀얼 모드("대화로 찾기"로 전환 시 자연어 → 실시간 필터).

- Insight Lab에는 **카드 그리드가 없습니다** — 알람 목록은 세로 리스트, 보고서 목록도 세로 카드 리스트. 공간적 훑기 경험 없음
- 자연어 필터 애니메이션(제외 카드 투명·축소, stagger) — 대화가 *실제로 결과를 바꾼다*는 인지를 시각화
- 인스펙터의 "온톨로지 탭"은 도면을 **엔티티 네트워크의 한 노드**로 보게 함 — Bundle과는 **스케일과 맥락이 다름**

### 3.4 고유 3: 자동 보고서 생성·재생성 (Insight Lab 전용)

[보고서 목록](./01-insight-lab/06-reports-list.md) + [보고서 상세·편집](./01-insight-lab/07-reports-detail.md)의 핵심. 4건 시드 + draft→reviewed→finalized 상태기계 + Python 수치 / LLM 서사 **협업 편집 모델** + 재생성 API + 보고서 전체를 contextSummary로 주입받는 챗.

- Design Lab에는 **산출물 개념이 없습니다** — Design은 "조회·질의" 영역이고, 결과물을 문서로 쌓지 않음
- 상태기계 + 리비전 + 확정 취소 UI(미결) — 운영 워크플로우의 UI 재현
- 챗 패널이 보고서 전체를 읽은 상태로 질의 응답 — "섹션 번호로 직접 참조" 가능

### 3.5 고유 4: 근본원인 그래프 (Insight Lab 전용)

[Insight Lab 화면 8 — `/insight/root-cause/[id]`](./01-insight-lab/08-root-cause.md). Cytoscape 기반 force-directed 그래프 + 신뢰도 바 + 근거 칩.

- Design Lab 화면 1의 "온톨로지 탭" SVG와 목적이 완전히 다름: 후자는 **정적 관계 표시**, 전자는 **인과 추론 시각화 + 후보 랭킹**
- AI 확신 수준(78%·42%·31%)을 숨기지 않고 바로 드러냄 — Design Lab에는 이런 "확신도" 표시 개념 없음

## 4. 통합 시나리오와 분리 유지 근거

두 트랙을 합칠 것인지, 분리된 채 유지할 것인지는 **현재 단계에서 결정하지 않습니다**. 양쪽의 가능성을 명시적으로 적어 이해관계자가 판단할 수 있게 합니다.

### 4.1 합친다면: 어떤 구조가 될 수 있는가

가능한 통합 형태는 **"공통 좌측 네비 + 트랙별 중앙·우측"** 구조입니다.

```
┌─ 통합 SideNav ──┬──────────────────────────────────────────────┐
│                 │                                              │
│  [홈]            │   (Insight 또는 Design 화면이 여기에)         │
│  ──── 도면 ────  │                                              │
│  · 도면 카드     │   선택된 화면에 따라 레이아웃이 완전히 다르게 │
│  · 도면 챗       │   렌더됨 — 좌측 네비만 공통                   │
│  ──── AI ────   │                                              │
│  · 인사이트 홈    │                                              │
│  · 알람          │                                              │
│  · 챗            │                                              │
│  · Lab           │                                              │
│  · 근본원인       │                                              │
│  ──── 보고서 ── │                                              │
│  · 일일 보고서    │                                              │
└─────────────────┴──────────────────────────────────────────────┘
```

이 구조의 장점:
- **사용자 접근성**: "AI에게 묻고 싶다"와 "도면을 찾고 싶다"가 같은 네비에서 선택 가능
- **단일 브랜드·톤**: 라이트 슬레이트 베이스가 이미 공통이므로 시각적 통합 비용은 낮음
- **서비스 1·2 경계 해소**: DKS의 비전인 "도면 지식 × AI 인사이트"가 네비 구조로 체화

단점:
- **네비 복잡도**: 11개 화면이 한 네비에 나열되면 정보 과부하 — 그룹 헤더 필수
- **심적 모델 충돌**: Design의 "목업 전용 실험 트랙" 배지와 Insight의 "실 SSE 연결" 프로덕션 수준이 한 네비에 섞이면 품질 기대가 균질해지지 않음
- **레이아웃 축 다양성**: 11화면이 각자 다른 레이아웃이라 "뭘 기대해야 할지" 예측성 떨어짐

### 4.2 분리 유지 근거

반대로 두 트랙을 끝까지 분리하는 것도 합리적입니다.

- **UX 원형이 다르다**: Insight는 "읽고 대응하는 문서형 UX"(라이트 톤·수직 스택·보고서 인쇄 친화), Design은 "훑고 조작하는 탐색형 UX"(그라디언트·카드·연출 애니메이션). 섞으면 **둘 다 흐려질 위험**
- **트랙 성숙도가 다르다**: Insight는 실 LLM 연결·mock 폴백·localStorage 영속 등 프로덕션에 가까운 수준. Design은 목업 전용. 같은 네비에 놓이면 사용자가 **기능 완성도를 오인**
- **의사결정 리듬이 다르다**: Insight는 "그대로 프로덕션 진입" 후보, Design은 "채택 여부를 결정하기 위한 실험". 같은 트랙에 두면 결정 타이밍이 뒤엉킴
- **격리의 안전성**: Design Lab은 `/design-lab/*` 네임스페이스 + 보존 6항목 회피로 **프로덕션에 부작용 0**. 통합 시 이 격리가 깨짐

### 4.3 제3안: 부분 채택·합성

극단 양자택일이 아닌 **선택적 합성**도 열려 있습니다.

- Design 화면 1의 카드 그리드 + 인스펙터 패턴을 서비스 1 도면 뷰어에 **이식**
- Design 화면 2의 "참조 도면 자동 누적" 구조를 Insight 챗 패널에 **통합**
- Insight Lab의 Bundle 개념을 Design Lab 화면 1의 온톨로지 탭에 **확장 적용**

이 경우 두 트랙의 고유 강점을 각자 살리면서 서로의 약점을 보완합니다. 의견 수렴 이후 FRS 단계에서 **어느 요소를 어디로 이식할지**를 합의하게 됩니다.

## 5. 의견 수렴 포인트

이 비교 문서가 다루는 결정은 **다음 세 가지**입니다:

1. **두 트랙을 합칠 것인가, 분리 유지할 것인가, 부분 합성할 것인가**
2. **중첩 영역(AI 질의)에서 어느 쪽 모델을 채택할 것인가** — Insight의 단일 컬럼 실 SSE 챗 vs Design의 3분할 워크스페이스
3. **고유 요소 중 어떤 것을 우선 개발할 것인가** — Bundle · 카드 그리드 · 보고서 · 근본원인 4가지의 우선순위

### 스스로 본 보완 포인트

- **비교 매트릭스의 축 선정 주관성**: §2의 축 11개는 작성자가 고른 것. 이해관계자가 "이 축은 왜 없느냐"고 지적할 수 있음 — 의견 수렴 중 추가·재배치 가능
- **중첩/고유 구분의 회색지대**: Design 화면 1의 "대화로 찾기" 모드는 엄밀히는 **검색 보조용 자연어 필터**라 Insight의 AI 질의와 종류가 다른데, §3.1에서는 둘을 "대화로 묶어 비교"하는 방식을 택함 — 이 프레이밍이 적절한지 검토 필요
- **통합 구조 스케치의 구체성 부족**: §4.1의 트리 구조는 개념적. 실제 네비 그룹 라벨·순서·접힘 상태 등 구체 설계는 의견 수렴 이후 별도 문서
- **"부분 합성"의 조합 경우의 수**: §4.3은 3개 예시만 제시. 실제로는 11화면 × 각 요소별 이식 가능성이 조합적으로 많음 — 전체 매트릭스는 FRS 단계에서 정리
