# 트랙 · Design Lab 개요

**경로 루트**: `/design-lab/*`
**소속 트랙**: Design Lab (2-트랙) · 목업 전용 (프로덕션 영향 없음)
**포함 화면 수**: 3개 (허브 + 2개 대안)

---

## 1. 트랙 소개

Design Lab 트랙은 **도면 탐색의 UX 대안 두 가지를 병렬로 놓고 이해관계자가 비교 검토할 수 있도록** 만든 목업 전용 묶음입니다. 기존 프로덕션(`src/app/page.tsx` 루트의 도면 뷰어)은 그대로 두고, Design Lab 하위 경로(`/design-lab/*`)에서만 전혀 다른 두 개의 화면을 나란히 보여줍니다.

두 화면의 핵심 차이는 **"도면 탐색의 주역이 무엇인가"** 입니다.

- **화면 1 (카드)**: 도면이 주역. 대화는 검색을 돕는 보조 수단
- **화면 2 (챗)**: 대화가 주역. 도면은 대화가 꺼내주는 참조 결과

어느 쪽이 사용자에게 맞는지, 혹은 둘을 합쳐 단일 화면으로 가야 하는지 — 이 트랙의 존재 목적은 **그 결정을 위한 시각적 재료 제공**입니다. 프로덕션에는 아직 어느 쪽도 반영되지 않았으며, 양자택일이 아니라 "부분 채택·합성"도 선택지로 열려 있습니다.

## 3. 트랙이 해결하는 서비스 측면

이 트랙은 **새로운 기능**을 제안하지 않습니다. 대신 **같은 데이터(도면 26건 + 엔티티 링크 + 주석)를 두 가지 서로 다른 UX 모델로 조직한** 실험입니다.

| 서비스 측면 | 이 트랙의 개입 |
|---|---|
| **도면 탐색의 UX 대안 병렬 비교** | 같은 데이터·같은 화면 크기·같은 목업 메타로 두 모델을 나란히 놓아, **서로의 장단점이 시각적으로 대비되도록** 구성 |
| **카드 중심 UX (화면 1)** | 썸네일·도면번호·공종색을 동시에 인지하는 **훑어보기 경험**. 대화는 필터 보조 역할 |
| **대화 중심 UX (화면 2)** | 검색창을 **대화로 대체**했을 때의 조작 감각. 도면은 대화가 자동으로 꺼내 누적 |
| **진입 시점의 사용자 의도 분기** | "뭐가 있나 보고 싶다" vs "이 질문에 답해줘" — 같은 목적 공간이라도 **진입 시 의도가 다르면 전혀 다른 UX가 필요한가**를 가설 검증 |
| **프로덕션 간섭 없이 디자인 탐색** | `/design-lab/*` 네임스페이스로 격리, 프로덕션 경로(s1/s2/s3) 및 보존 6항목에 영향 없음 |

**이 트랙이 해결하지 않는 것**: AI 이상 분석, 알람, 보고서. 그 영역은 Insight Lab 트랙이 맡습니다. Design Lab은 **도면 탐색의 UI 형식 문제만** 다룹니다.

## 4. 화면 간 이동·관계

### 4.1 허브가 중심축

```
   [/design-lab]  (허브)
        │
        ├── "화면 1" 카드 클릭 ──▶ [/design-lab/cards]
        │                                  │
        │                                  └─ 상단 LabNav "화면 2" 링크 ──▶ [/design-lab/chat]
        │
        ├── "화면 2" 카드 클릭 ──▶ [/design-lab/chat]
        │                                  │
        │                                  └─ 상단 LabNav "화면 1" 링크 ──▶ [/design-lab/cards]
        │
        └── LabNav "[← 프로덕션 복귀]" ──▶ (프로덕션 서비스 1/2로 탈출)
```

허브는 **두 화면을 동시에 카드 2장으로 소개**하고, 하단에 "두 화면의 목적 구분" 비교 표 (주역 · 대화의 역할 · 진입 목적 3축)를 둡니다. 이 표는 이해관계자가 두 화면 진입 전에 **차이를 먼저 알고 들어가도록** 하는 장치입니다.

### 4.2 두 화면 사이는 자유 왕복

화면 1과 화면 2 사이에는 **계층 관계가 없습니다**. LabNav(상단 56px 고정)에 두 화면 링크가 나란히 있어 어디서든 즉시 전환 가능합니다. "어느 쪽이 먼저 열려야 하는지"를 강제하지 않는 것이 의도입니다 — 비교 실험이므로.

### 4.3 두 화면의 목적 구분

| 축 | 화면 1 (카드) | 화면 2 (챗) |
|---|---|---|
| 주역 | 카드(도면) | 대화 |
| 대화의 역할 | 검색 보조 (필터) | 주 작업 (질의·요약) |
| 진입 목적 | "뭐가 있나 훑어보기" | "이 질문 답해줘" |
| 레이아웃 모델 | 좌 네비 + 중 그리드 + 우 인스펙터/챗 토글 | 좌 문맥(뷰어·리스트) + 중 챗 고정 + 우 참조 누적 |
| 정보 밀도 | 한 화면에 다수 카드 노출 | 한 화면에 한 대화 스레드 집중 |

## 5. 트랙 공통 UX 원칙

### 5.1 Claude Code 스킬카드 감성 + Genspark CLO 감성의 결합

화면 1의 카드 디자인은 **Claude Code 스킬카드 감성**(얇은 테두리·썸네일 위 그라디언트 오버레이·공종색 하단 라인)을, 화면 2의 챗 워크스페이스는 **Genspark CLO 감성**(중앙 챗 고정·3분할·우측 자동 누적 카드)을 참고했습니다. 두 감성을 한 트랙 안에서 대비시켜, 같은 도면 지식 자산이라도 **프레임에 따라 얼마나 다른 경험이 되는지**를 드러냅니다.

### 5.2 라이트 톤 고정

`bg-slate-50` 베이스 + `bg-white` 카드 표면 + 보라·청록 악센트(`from-violet-500 to-fuchsia-500`, `from-cyan-500 to-blue-500`). 다크 모드를 제공하지 않습니다 — **시각 비교의 일관성**을 위해 톤을 한 축으로 고정했습니다.

### 5.3 보존 6항목 회피

Design Lab은 **프로덕션의 보존 6항목**(PdfViewer · AnnotationLayer · AnnotationPopover · AnnotationList · EntityToDocs · useAnnotationsStore)에 전혀 의존하지 않습니다. 별도 목업 컴포넌트(`src/components/design-lab/*`)와 목업 메타(`src/lib/design-lab/mock-meta.ts`)만 사용합니다. 이 격리 덕에 Design Lab이 어떻게 바뀌든 프로덕션은 영향을 받지 않습니다.

### 5.4 실 LLM 호출 없음

두 화면 모두 **실제 LLM 호출은 하지 않습니다**. 화면 1은 미리 정의된 4개 프롬프트 시나리오, 화면 2는 시드 응답을 600~1200ms 지연 후 반환합니다. 이는 **데모 안정성** + **이해관계자 관심을 UX에 묶기 위한 의도** — LLM 품질 논쟁으로 UX 비교가 묻히지 않도록.

### 5.5 같은 데이터·같은 메타

두 화면이 쓰는 도면은 `data/drawings.json` 26건 동일, 엔티티 링크·주석 시드도 동일합니다. 목업 메타(목적 한 단락·대화요약·온톨로지 관계)는 `src/lib/design-lab/mock-meta.ts` 단일 소스. **데이터 차이가 아닌 UX 차이만**으로 두 화면이 대비됩니다.

### 5.6 "프로덕션 복귀" 탈출구 상시 노출

LabNav 우측에 **[← 프로덕션 복귀]** 링크를 항상 노출해, Design Lab이 실험 트랙임을 인지하고 쉽게 나갈 수 있게 합니다. 이는 이해관계자가 "이거 실제 서비스인 줄 알았다" 혼란을 막기 위한 장치입니다.

## 6. 의견 수렴 포인트 (트랙 전체 관점)

### 스스로 본 보완 포인트

- **양자택일 프레이밍 위험**: 허브가 "두 화면 중 하나"처럼 보이기 쉬움. 실제로는 **둘을 합성**하거나 **부분만 채택**할 수도 있음. 허브 카피 보강 필요
- **화면 2 쪽으로 쏠릴 가능성**: AI 챗이 2026년 현재 유행이라 이해관계자가 "당연히 2가 맞지"라고 예단할 수 있음. 화면 1의 즉각적 인지 장점(썸네일 훑기)이 충분히 전달되는지 점검 필요
- **모바일 시나리오 미검증**: 두 화면 모두 데스크탑 3분할 전제. 현장 태블릿·모바일에서의 변형이 미설계
- **카드 26건은 충분한가**: 실 서비스에서는 수백~수천 건. 대규모 도면 풀에서 각 UX가 어떻게 스케일하는지는 목업 범위 밖
- **온톨로지·대화요약의 시드 의존**: 목업 메타가 8건(doc-001~008)에만 있음. 나머지는 fallback 처리. 실 데이터 연결 시 온톨로지 관계 자동 추출이 필요
- **보존 6항목과의 최종 통합 경로**: Design Lab 일부를 프로덕션에 이식할 경우 보존 6항목과의 상호작용(주석 핀 좌표·역추적 로직) 합성 방식이 미결
- **이해관계자별 선호 예측**: 검색을 자주 쓰는 사용자 vs 탐색을 자주 쓰는 사용자의 의견이 갈릴 수 있음 — 의견 수렴 시 **응답자의 업무 성격 라벨링**이 필요할 수도
