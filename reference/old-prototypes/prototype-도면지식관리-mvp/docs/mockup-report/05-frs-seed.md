# FRS 시드 — 기능 요구사항 명세 초안

**용도**: 의견 수렴(각 화면 §8 + [03-comparison-matrix.md](./03-comparison-matrix.md) §5)이 안정화된 **이후** 실제 개발 착수용 FRS(Functional Requirement Specification) 문서의 씨앗. 본 파일은 **각 화면의 §5(사용자 동작 흐름) 및 §7(이 화면이 기여하는 서비스 측면)을 기계적으로 기능 문장으로 치환**한 결과이며, 의견 수렴 이후 수정·삭제·병합·추가를 거쳐 확정됩니다.

> 이 문서는 의견 수렴 안정화 후 채워집니다. 현재 버전은 **기계적 1차 초안**이며, §5의 이해관계자 의견 기록란에 합의 내용이 누적되는 대로 재편집됩니다.

---

## 1. FRS 부여 규칙

### 1.1 ID 포맷

`FRS-{트랙코드}-{화면ID}-{일련번호}`

| 트랙코드 | 의미 |
|---|---|
| `IL` | Insight Lab |
| `DL` | Design Lab |

| 화면ID | 트랙 | 화면명 |
|---|---|---|
| `01` | IL | 인사이트 홈 (허브) |
| `02` | IL | 알람 목록 |
| `03` | IL | AI 답변 상세 (알람 분석 + 멀티턴 대화) |
| `04` | IL | AI 자유 질의 (챗) |
| `05` | IL | Insight Lab (통계×온톨로지) |
| `06` | IL | 일일 보고서 목록 |
| `07` | IL | 일일 보고서 상세·편집 |
| `08` | IL | 근본원인 분석 그래프 |
| `01` | DL | Design Lab 허브 |
| `02` | DL | 카드형 도면 리스트 |
| `03` | DL | 챗 워크스페이스 |

**일련번호**는 3자리 0-padded (`001`~`999`). 기능 추가·삭제 시에도 **기존 번호는 재사용하지 않습니다** (중간 결번 허용).

### 1.2 예시

- `FRS-IL-04-001` — Insight Lab 화면 4(AI 자유 질의)의 첫 번째 기능
- `FRS-DL-02-003` — Design Lab 화면 2(카드형 도면 리스트)의 세 번째 기능
- `FRS-NFR-001` — 비기능 요구사항 첫 번째

### 1.3 우선순위 표기

각 FRS 항목 끝에 우선순위 라벨을 붙입니다.

| 라벨 | 의미 |
|---|---|
| `MVP` | 최초 프로덕션 릴리즈에 반드시 포함 |
| `P1` | MVP 직후 다음 반복(2~4주)에 포함 |
| `P2` | 장기 로드맵 (Phase 2 이상) |

## 2. 화면별 기능 요구사항

### 2.1 `IL-01` — 인사이트 홈 (허브)

근거: [01-hub.md §5](./01-insight-lab/01-hub.md)

- `FRS-IL-01-001 · MVP` — 이 화면은 진입 시 최근 알람 상위 3건을 심각도 색점·이름·설비 태그와 함께 세로 리스트로 표시할 수 있어야 한다.
- `FRS-IL-01-002 · MVP` — 이 화면은 최근 알람 카드 헤더의 "전체보기 →" 링크를 통해 `/insight/alarms`로 이동할 수 있어야 한다.
- `FRS-IL-01-003 · MVP` — 이 화면은 자주 묻는 질문 상위 5건을 텍스트 리스트로 표시할 수 있어야 한다.
- `FRS-IL-01-004 · P1` — 이 화면은 자주 묻는 질문 항목 클릭 시 `/insight/chat?q={text}` 프리필 링크로 챗 화면을 열 수 있어야 한다.
- `FRS-IL-01-005 · MVP` — 이 화면은 가장 최신 일일 보고서 1건을 날짜·섹션 수·상태와 함께 카드 전체를 링크화하여 표시할 수 있어야 한다.
- `FRS-IL-01-006 · MVP` — 이 화면은 일일 보고서 카드 헤더의 "전체보기 →" 링크를 통해 `/insight/reports`로 이동할 수 있어야 한다.
- `FRS-IL-01-007 · P1` — 이 화면은 알람 건수가 0건일 때 "오늘 알람 없음, 정상 운전 중" 같은 긍정 빈 상태 메시지를 표시할 수 있어야 한다.
- `FRS-IL-01-008 · P2` — 이 화면은 최신 일일 보고서를 정적 import가 아닌 런타임 fetch로 가져와 날짜가 바뀌어도 자동 갱신할 수 있어야 한다.

### 2.2 `IL-02` — 알람 목록

근거: [02-alarms.md §5](./01-insight-lab/02-alarms.md)

- `FRS-IL-02-001 · MVP` — 이 화면은 진입 시 심각도 카운트(위급·경고·총 건수)를 상단 헤더에 표시할 수 있어야 한다.
- `FRS-IL-02-002 · MVP` — 이 화면은 전체 알람을 `triggered_at` 내림차순으로 세로 리스트로 표시할 수 있어야 한다.
- `FRS-IL-02-003 · MVP` — 이 화면은 각 알람 행에 심각도 색 도트, 배지, 알람 이름, 설비 태그(monospace), 알람 코드, 발생 시각을 표시할 수 있어야 한다.
- `FRS-IL-02-004 · MVP` — 이 화면은 알람 행 클릭 시 우측 드로어를 슬라이드 인으로 열고 상세 정보(설비·현재값·임계값·출처·시각·설명)를 표시할 수 있어야 한다.
- `FRS-IL-02-005 · MVP` — 이 화면은 드로어에서 "AI에게 물어보기" 버튼 클릭 시 `/insight/answers/{query_id}?alarm_id={id}`로 이동할 수 있어야 한다.
- `FRS-IL-02-006 · MVP` — 이 화면은 드로어 백드롭 클릭 또는 ✕ 버튼으로 드로어를 닫을 수 있어야 한다.
- `FRS-IL-02-007 · P1` — 이 화면은 심각도·설비·기간·검색어별 필터 UI를 제공할 수 있어야 한다.
- `FRS-IL-02-008 · P1` — 이 화면은 드로어에 해당 알람에 대한 과거 AI 질의 이력을 표시할 수 있어야 한다.
- `FRS-IL-02-009 · P2` — 이 화면은 WebSocket 또는 polling으로 알람 건수와 리스트를 실시간으로 갱신할 수 있어야 한다.

### 2.3 `IL-03` — AI 답변 상세 (알람 분석 + 멀티턴 대화)

근거: [03-answers-detail.md §5](./01-insight-lab/03-answers-detail.md)

- `FRS-IL-03-001 · MVP` — 이 화면은 진입 즉시 `POST /api/insights/ask/stream`을 호출해 좌측 AnswerCard에 SSE 토큰 스트리밍으로 답변을 표시할 수 있어야 한다.
- `FRS-IL-03-002 · MVP` — 이 화면은 스트리밍 진행 상태(분석 중·완료·부분 수신)를 QueryHeader 우상단에 시각적으로 표시할 수 있어야 한다.
- `FRS-IL-03-003 · MVP` — 이 화면은 답변 본문을 ReactMarkdown + remarkGfm으로 렌더하고 스트리밍 중 깜빡이 캐럿을 표시할 수 있어야 한다.
- `FRS-IL-03-004 · MVP` — 이 화면은 답변 수신 완료 시 근거 도면·문서 카드(`EvidenceList`), 구조화 분석 패널(`StructuredPanel`), 후속 질문 칩(`FollowUpSuggestions`)을 순서대로 표시할 수 있어야 한다.
- `FRS-IL-03-005 · MVP` — 이 화면은 근거 카드의 "도면 열기 →" 클릭 시 `/drawings/{doc_id}?page=N`으로 이동할 수 있어야 한다.
- `FRS-IL-03-006 · MVP` — 이 화면은 좌측 후속 질문 칩 클릭 시 우측 ChatPanel로 자동 전송하여 새 대화 턴을 시작할 수 있어야 한다.
- `FRS-IL-03-007 · MVP` — 이 화면은 우측 ChatPanel에서 사용자 입력(Enter 전송·Shift+Enter 줄바꿈)을 받아 `POST /api/insights/chat/stream`으로 멀티턴 대화를 수행할 수 있어야 한다.
- `FRS-IL-03-008 · MVP` — 이 화면은 페이지 이탈 시 `AbortController`로 진행 중인 스트리밍을 정리할 수 있어야 한다.
- `FRS-IL-03-009 · P1` — 이 화면은 좌측 분석 요약을 `contextSummary`로 우측 ChatPanel에 자동 주입하여 대화가 분석 내용을 참조할 수 있어야 한다.
- `FRS-IL-03-010 · P2` — 이 화면은 좁은 화면(모바일·태블릿)에서 좌/우 분할을 탭 전환(분석|대화)으로 전환할 수 있어야 한다.

### 2.4 `IL-04` — AI 자유 질의 (챗)

근거: [04-chat.md §5](./01-insight-lab/04-chat.md)

- `FRS-IL-04-001 · MVP` — 이 화면은 빈 상태에서 AI 로고, 타이틀, 부제와 함께 추천 질문 5개를 버튼으로 표시할 수 있어야 한다.
- `FRS-IL-04-002 · MVP` — 이 화면은 추천 질문 버튼 클릭 시 즉시 해당 질문으로 `send()`를 실행하고 빈 상태를 대체할 수 있어야 한다.
- `FRS-IL-04-003 · MVP` — 이 화면은 `POST /api/chat/stream` 호출로 SSE 토큰을 수신하여 AI 버블에 타자기 효과로 누적 표시할 수 있어야 한다.
- `FRS-IL-04-004 · MVP` — 이 화면은 AI 답변에 동반된 근거 도면·문서 카드(썸네일·제목·snippet·관련도·도면 열기 링크)와 후속 질문 칩을 AI 버블 아래에 표시할 수 있어야 한다.
- `FRS-IL-04-005 · MVP` — 이 화면은 후속 질문 칩 클릭 시 새로운 user 메시지로 자동 전송할 수 있어야 한다.
- `FRS-IL-04-006 · MVP` — 이 화면은 하단 textarea에 사용자 입력(Enter 전송·Shift+Enter 줄바꿈)을 받아 새 대화 턴을 시작할 수 있어야 한다.
- `FRS-IL-04-007 · MVP` — 이 화면은 메시지가 1건 이상일 때 헤더 우측에 "새 대화" 버튼을 표시하고 클릭 시 전체 이력을 초기화할 수 있어야 한다.
- `FRS-IL-04-008 · P1` — 이 화면은 이전 턴의 history를 서버로 전달하여 멀티턴 맥락이 유지되는 답변을 생성할 수 있어야 한다.
- `FRS-IL-04-009 · P1` — 이 화면은 대화 이력을 localStorage 또는 서버 persistence로 저장하여 새로고침 시 복원할 수 있어야 한다.
- `FRS-IL-04-010 · P1` — 이 화면은 스트리밍 중 취소(stop) 버튼을 제공하여 사용자가 응답 생성을 중단할 수 있어야 한다.
- `FRS-IL-04-011 · P2` — 이 화면은 네트워크 오류 시 재시도 버튼과 상세 에러 메시지를 표시할 수 있어야 한다.

### 2.5 `IL-05` — Insight Lab (통계×온톨로지 Bundle)

근거: [05-lab.md §5](./01-insight-lab/05-lab.md)

- `FRS-IL-05-001 · MVP` — 이 화면은 좌측에 통계 이벤트 리스트(SEVERITY·카테고리·타임스탬프·대상·메트릭·narrative_hint)를 세로 카드로 표시할 수 있어야 한다.
- `FRS-IL-05-002 · MVP` — 이 화면은 이벤트 카드 클릭 시 `GET /api/insight/lab/bundles?id={event_id}`를 호출해 중앙 Bundle 뷰를 갱신할 수 있어야 한다.
- `FRS-IL-05-003 · MVP` — 이 화면은 중앙 Bundle 뷰에 헤더·통계 이벤트·온톨로지 엔티티·관계 확장(최대 2홉)·근거 도면·Semantic Summary 5개 섹션을 수직 스택으로 표시할 수 있어야 한다.
- `FRS-IL-05-004 · MVP` — 이 화면은 온톨로지 미매칭 이벤트(`no_match`) 선택 시 엔티티 카드 자리에 호박색 경고 박스로 커버리지 한계를 표시할 수 있어야 한다.
- `FRS-IL-05-005 · MVP` — 이 화면은 근거 도면 카드 클릭 시 라이트박스로 PDF 페이지 이미지를 확대하고 ESC·백드롭 클릭으로 닫을 수 있어야 한다.
- `FRS-IL-05-006 · MVP` — 이 화면은 라이트박스에서 "전체 뷰어로 열기 ↗" 링크로 새 탭에 `/drawings/[doc_id]?page=N`을 열 수 있어야 한다.
- `FRS-IL-05-007 · MVP` — 이 화면은 우측에 Bundle 컨텍스트 안내와 함께 추천 질문 3개를 표시하고 클릭 시 `POST /api/insight/lab/chat/stream`으로 전송할 수 있어야 한다.
- `FRS-IL-05-008 · MVP` — 이 화면은 `no_match` 이벤트일 때 추천 질문을 disabled 처리하고 "일반론만 가능" 안내를 표시할 수 있어야 한다.
- `FRS-IL-05-009 · MVP` — 이 화면은 좌측에서 새 이벤트를 선택하면 우측 대화 이력(`messages`·`input`)을 초기화할 수 있어야 한다.
- `FRS-IL-05-010 · P1` — 이 화면은 Bundle 카테고리(열·통계·무결성)별로 다른 추천 질문 세트를 제공할 수 있어야 한다.
- `FRS-IL-05-011 · P1` — 이 화면은 관계 확장 hop 레벨(2홉·3홉)을 사용자가 토글할 수 있어야 한다.
- `FRS-IL-05-012 · P2` — 이 화면은 좌측 이벤트 리스트에 카테고리·SEVERITY·설비 그룹별 필터를 제공할 수 있어야 한다.

### 2.6 `IL-06` — 일일 보고서 목록

근거: [06-reports-list.md §5](./01-insight-lab/06-reports-list.md)

- `FRS-IL-06-006 · P1` — 이 화면은 상단에 "오늘 보고서 생성" 버튼을 제공해 `POST /api/reports/generate`를 호출할 수 있어야 한다.
- `FRS-IL-06-007 · P1` — 이 화면은 확정된 보고서의 PDF·DOCX 내보내기 및 공유 링크 복사 기능을 제공할 수 있어야 한다.
- `FRS-IL-06-008 · P2` — 이 화면은 카드에 해당 일자 CRITICAL 알람 수 미니 인디케이터를 표시할 수 있어야 한다.

### 2.7 `IL-07` — 일일 보고서 상세·편집

근거: [07-reports-detail.md §5](./01-insight-lab/07-reports-detail.md)

- `FRS-IL-07-001 · MVP` — 이 화면은 헤더에 "← 목록" 링크, 보고서 제목·리비전, 상태 뱃지, 재생성·확정 버튼을 표시할 수 있어야 한다.
- `FRS-IL-07-002 · MVP` — 이 화면은 각 섹션을 2-column 그리드(좌: 수치 데이터 `DataTable` · 우: AI 해석 `textarea`)로 표시할 수 있어야 한다.
- `FRS-IL-07-003 · MVP` — 이 화면은 textarea `onBlur` 시 `updateSection`을 호출하여 상태를 `draft`→`reviewed`로 자동 전환할 수 있어야 한다.

## 5. 의견 수렴 포인트

이 문서는 **의견 수렴이 안정화된 후** 개발 착수용으로 채워집니다. 현재 내용은 각 화면 문서의 §5(사용자 동작 흐름)와 §8(보완 포인트) 제안을 기계적으로 FRS 문장으로 치환한 **1차 초안**이며, 다음 단계가 끝나야 확정됩니다:

1. 각 화면 문서의 §8 "이해관계자 의견 기록란"에 의견이 누적됨
2. [03-comparison-matrix.md](./03-comparison-matrix.md) §5의 세 가지 결정(합침/분리 · 중첩 영역 채택 · 고유 요소 우선순위)이 합의됨
3. 위 결정에 따라 본 문서의 §2·§3 항목을 수정·삭제·병합·추가하고 우선순위 라벨을 재확정
4. §4의 "향후 추가 항목"을 구체 FRS로 승격

### 스스로 본 보완 포인트 (초안 단계)

- **MVP·P1·P2 분류의 임시성**: 현재 우선순위는 작성자 판단이며, 실 운영 시 유저 스토리·ROI 분석을 거쳐 재확정 필요
- **FRS 문장의 구체성 수준**: 일부 항목은 UI 동작을 직접 기술(레이아웃·색·너비)하고 있어 **기능 요구보다 설계 결정**에 가까움 — 실 FRS에서는 이를 "기능"과 "설계 가이드"로 분리 권고
- **검증 가능성**: 각 FRS 항목에 수락 기준(Acceptance Criteria)이 미포함 — 실 FRS 승격 시 "어떻게 테스트할 것인가"를 명시 필요
- **의존성 그래프**: FRS 간 선후 관계(예: IL-07-007이 IL-07-002를 전제)가 본 문서에 기록되지 않음 — 개발 착수 시 별도 의존성 표 필요
- **비기능 FRS의 정량 기준**: §3의 성능 수치(p50·p95)는 작성자 추정. 실제 SLA는 인프라·DB 설계 후 재설정
- **§4 한계 항목의 규모**: 18개 항목 모두 각각 여러 FRS로 쪼개져야 하므로, 실 개발 범위 산정 시 **본 목업 §2의 11화면보다 §4 항목이 훨씬 큰 작업**
