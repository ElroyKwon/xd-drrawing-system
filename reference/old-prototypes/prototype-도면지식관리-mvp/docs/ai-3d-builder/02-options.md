# 02. 실행 가능한 아키텍처 3개

01에서 본 근거에 맞춰 현실적 선택지 세 개. **A, B, C 중 사용자와 합의 후 03으로 진행**.

---

## Option A — "Pure LLM" (사용자의 원래 발상)

```
사용자 → [도면 이미지 20장 순차 업로드]
         ↓
     Claude Messages API (누적 컨텍스트)
         ↓
     [누적 Three.js HTML 출력]
         ↓
     브라우저 렌더
```

### 장점
- 파이프라인 단순. 구현 1~2주
- AI만 붙이면 끝. 외부 도구 의존 없음
- "AI가 모두 한다" 스토리가 강함

### 단점 (근거 있음)
- **[01 §1.1]** Claude Vision 도면 치수 정확도 **40%**. 20장 누적하면 오류가 기하급수적으로 쌓임
- **[01 §1.1]** 다중 뷰 간 치수 오배정이 Claude의 **가장 약한 실패 모드** — 사용자 시나리오를 정면으로 건드림
- **[01 §1.3]** 학계에서도 해결되지 않은 문제를 LLM에게만 맡김
- **[01 §1.6]** 20턴 좌표 일관성은 보증된 바 없음

### 권장 여부
❌ **비권장**. 단, **Gemini Pro로 교체**하면 40% → 80%로 개선되어 재평가 여지 있음.

### 만약 그래도 A로 간다면
- **모델: Claude가 아니라 Gemini Pro/2.5** (2배 정확)
- 용도: 기술 시연 / 마케팅 데모
- 15장이 아닌 **3~5장 단순 건물**부터 (입력 복잡도 ↓)
- 품질 평가는 "형태가 인식되는가" 수준, 정밀 치수 검증은 포기

## Option B — "하이브리드: AI 추출 + 결정론 조립" ⭐ 권장

```
사용자 → [도면 이미지 20장]
         ↓
     [단계 1] Vision LLM이 도면에서 "구조화 데이터"만 추출
              - 예: {type: wall, grid: A1-A5, length: 12000mm, height: null}
              - JSON 출력, 치수·좌표·식별자만
         ↓
     [단계 2] 사람 검수 UI — 누락/오류 수정 (30초/도면)
         ↓
     [단계 3] 결정론 코드가 JSON → Three.js 변환
              - 사람이 짠 템플릿. AI 코드 생성 없음
              - 좌표 계산·뷰 정합은 수학적으로 정확
         ↓
     브라우저 렌더
```

### 장점 (근거 있음)
- **[01 §1.3]** 2D→3D 복원의 어려움은 추출이지 조립이 아님. 이 분할이 올바름
- **[01 §1.4]** 상용 도구들의 표준 워크플로 ("AI 60~80% + 사람")와 일치
- LLM 정확도 한계를 **사람 검수**가 보완
- Three.js 코드 품질은 **결정론**이라 재현·디버그 쉬움
- Vision 단계만 Gemini Pro 등 전문 모델 교체 가능
- 실패해도 "어느 단계에서 실패했는지" 명확

### 단점
- 구현 복잡. 2~4주. 검수 UI가 핵심 자산
- "AI만으로"라는 마케팅 스토리는 약화

### 권장 여부
✅ **권장**. 근거 기반으로 유일하게 성공 확률 높은 경로.

### B의 실제 파이프라인 초안

| 단계 | 담당 | 산출 |
|---|---|---|
| 1 | Vision LLM (Gemini Pro + Claude 교차 검증) | `elements.json` — 벽·문·장비·그리드 라벨·치수 |
| 2 | 검수 UI (React, `/ai-drawing-lab/review`) | 수정된 `elements.json` |
| 3 | 결정론 변환기 (TypeScript) | `scene.ts` → Three.js 메시 배열 |
| 4 | 뷰어 (React + Three.js) | 3D 씬 |

**핵심**: 다중 뷰 정합(평면/입면/단면의 같은 객체 매핑)을 **사람이 검수 UI에서 명시적으로 연결**. LLM에게 맡기지 않음.

## Option C — "기존 상용 도구 활용"

```
사용자 → [도면 PDF/이미지]
         ↓
     외부 API (Plans2BIM / WiseBIM / archBIM 등)
         ↓
     [IFC 또는 Revit 파일 수신]
         ↓
     IFC.js로 브라우저에서 3D 렌더
         ↓
     (선택) Claude로 의미 레이어 추가
            — 객체 이름, 설명, 설비 연결
```

### 장점
- **[01 §1.4]** 이미 검증된 정확도 (DBAL-YOLO 주장 98.8%)
- 평면도만이면 잘 동작
- IFC 표준 사용 — 상호운용성 높음

### 단점
- 외부 의존 — 서비스 중단·정책 변경 리스크
- 대부분 **평면도만** 지원. 다중 뷰는 여전히 미해결
- API 비용·제한 미공개 (직접 문의 필요)
- "우리 AI로 만들었다"가 아님

### 권장 여부
⚠️ **조건부 권장**. 평면도 단일 입력이고 빠른 MVP가 필요하면 최선. 다중 뷰 요구는 못 품음.

## 비교 요약

| 축 | A (Pure LLM) | **B (하이브리드)** | C (외부 API) |
|---|---|---|---|
| 성공 확률 | 낮음 | **중~높음** | 중 (단일 뷰) |
| 구현 기간 | 1~2주 | 2~4주 | 1주 (통합만) |
| 다중 뷰 지원 | 이론상 가능 | **가능** | 대부분 불가 |
| 외부 의존 | Claude/Gemini | Vision LLM만 | 상용 도구 |
| 정확도 한계 돌파법 | 없음 | 사람 검수 | 도구 벤더 책임 |
| 재현성·디버그 | 낮음 | **높음** | 중 |
| 마케팅 스토리 | "AI가 모두" | "AI + 전문가" | "AI 통합" |

## 결정이 필요한 것들

### Q1. 어느 Option으로 가는가?
- A: 비권장이지만 사용자가 원하면 가능
- **B: 근거 기반 권장**
- C: 평면도 단일이면 최선, 다중 뷰면 탈락

### Q2. Vision LLM은 Claude인가 Gemini인가?
근거상 Gemini Pro가 엔지니어링 도면에서 2배 정확. 하지만:
- 사용자 메시지는 "Claude API" 지정
- 이유: 특정 API 계약? 내부 정책? 선호?
- → **사용자에게 확인**. 계약상 제약이면 Claude 유지 + B 옵션에서 사람 검수 비중 ↑

### Q3. 검수 UI의 시간 비용
- B 옵션은 도면당 30초~2분 사람 검수 가정
- 20장 × 1분 = **20분/건물**
- 이게 허용 범위인가?

### Q4. 다중 뷰가 정말 필수인가?
- 평면도 15장(= 층별)만이면 C 옵션 가능
- 평면+단면+입면 혼합이면 B 옵션
- → **실제 입력 도면의 뷰 구성 샘플이 필요**
