## 시나리오 A — "라면만 있던 국에 건더기가 생긴다" (핵심)

### A-1 (0:00~0:20) 오프닝

> "기존 SLM은 통계 결과 JSON을 바로 LLM에 던져서, 답변이 일반론으로 흘렀습니다. 같은 통계 값에 **설비 온톨로지**를 먼저 끼워 넣으면 어떻게 달라지는지 보여드리겠습니다."

화면: 왼쪽 이벤트 리스트 스크롤, CRITICAL 배지 몇 개가 빨간색으로 보이는 걸 훑어 보임.

> **도면 미리보기 방식**: 근거 도면 카드의 "미리보기" 버튼을 누르면 `public/drawings/page_NNN.png`(PDF 추출본)이 **라이트박스 모달**로 렌더된다. 모달 우상단 "전체 뷰어로 열기"는 기존 `/drawings/[id]?page=N` 라우트로 새 탭 이동. 샘플 이미지는 p.3·4·5·10·22·40 6장이라 그 외 페이지는 모달 안에 "샘플 이미지 없음" 안내가 뜬다 — 한계 시연용 포인트.

### A-2 (0:20~1:00) 첫 이벤트 선택

**클릭**: `evt-001 · VCB 1101 절연저항 Z=-3.2σ · CRITICAL`

중앙 Bundle 뷰가 채워짐:

- **통계 카드**: `z_score = -3.2σ (threshold 3.0σ)` / 시각 / severity 배지
- **엔티티 카드**: `1101 — SHV 수전 VCB 패널 · 전기`
  - 스펙: `vcb_rating: 3P 7.2kV 630A 520MVA`, `protection_relay_model: GIPAM-2000FI`
- **2홉 관계**:
  - `BLK-MOF ─feeds→ 1101` (incoming, 1홉)
  - `1101 ─feeds→ TR-001` (outgoing, 1홉)
  - `1101 ─connects_to→ 1102` (outgoing, 1홉)
  - (2홉 일부 표시)
- **근거 도면**: `dwg-elec-001 · p.5` 썸네일

> "이 한 화면에 통계 수치, 해당 설비의 실제 스펙, 상하류 계통, 그리고 도면 근거까지 한 뿌리에서 묶였습니다. **이 번들 자체가 곧 LLM 프롬프트**가 됩니다."

### A-3 (1:00~2:00) 챗 1 — 원인 후보

**오른쪽 챗에 입력**: `이 이상의 물리적 원인 후보는 뭘까?`

기대 응답 예 (Gemini):
> "Z-Score -3.2σ는 그룹 평균 대비 전압이 현저히 낮음을 의미합니다. **1101의 VCB 520MVA 차단기 접점 열화** 또는 **상류 BLK-MOF 측 전압 공급 불안정**이 후보이며, 보호 계전기 GIPAM-2000FI의 50/51 동작 이력 확인이 우선입니다. 확인 권장."

포인트:
- 답변에 **실제 태그 번호·스펙이 그대로** 들어옴 (기존 SLM의 "X, Y, Z 변수명 노출" 문제 회피)
- 추측은 "확인 권장" 명시

### A-4 (2:00~2:40) 챗 2 — 도면 점프

**입력**: `관련 도면 어디에서 보지?`

응답: `dwg-elec-001 5페이지`에 1101 VCB 패널이 배치돼 있다. 증거 카드 클릭 → 기존 `PdfViewer`로 해당 페이지 점프. (뒤로 가기로 복귀)

## 시나리오 B — "한계도 숨기지 않는다" (정직성 파트)

### B-1 (2:40~3:10) no-match 이벤트

**클릭**: `evt-015 · BESS RACK_05 dT/dt 4.8°C/min` (의도적 no-match — 배터리 태그는 YAML에 없음)

중앙 Bundle 뷰:
- 통계 카드는 정상 렌더
- 엔티티 카드 자리에 **"온톨로지에 `RACK_05` 엔티티가 없습니다 — 이 카테고리는 현재 R-Center 기계/전기 설비만 커버합니다"** 경고 박스
- 챗 비활성 또는 "근거 부족" 상태

> "현재 YAML 온톨로지는 R-Center 기계·전기만 커버합니다. **배터리 도면이 들어오려면 YAML을 확장**해야 한다는 점을 이 카드 하나로 분명히 보여줍니다. 목업은 한계를 숨기지 않습니다."

### B-2 (3:10~3:40) 분야 전환

**클릭**: `evt-008 · 23C-IAC 냉각팬 진동 RMS 과다 · WARNING` (기계 쪽)

동일 흐름이 **기계 설비**에서도 작동하는 걸 보여줌. 온톨로지 확장이 분야 추가만으로 커버된다는 걸 시사.

## 시나리오 C — "그래서 뭐가 다른가" 마무리 (3:40~4:30)

세 가지를 말로 정리:

1. **답변이 구체적** — 태그·스펙·관계·페이지까지 들어감
2. **추적 가능** — 답변 근거가 Bundle에 고스란히 남아 사후 검증 가능
3. **확장이 단선적** — 새 분야는 YAML 디렉토리 하나 추가하면 로더가 자동 인식

> "이 구조를 실제 SLM Phase 2/Phase 3 사이에 끼워 넣으면, 기존 문서(`phase3_insight_improvement_plan.md`)에서 지적된 채널 누락·고정 텍스트 문제를 상당 부분 설계 단계에서 해소할 수 있습니다."

## 예상 FAQ (회의 자리에서 나올 만한 것)

**Q1. 이거 실제로 통계 돌리나요?**
> 아니요. 이벤트 JSON은 수작업으로 15~20건 만들었습니다. 통계 엔진 연결은 별도 트랙입니다.

**Q2. Gemini 대신 우리 사내 LLM 써도 되나요?**
> 네. `src/app/api/insight/lab/chat/stream/route.ts`의 `streamGemini` 함수만 교체하면 됩니다.

**Q3. 온톨로지 관계는 누가 썼나요?**
> 기존 YAML의 `relations[]` 필드에 이미 있습니다. 사람이 작성했거나 agentic 분석이 뽑은 것으로, 이 목업은 그대로 읽기만 합니다.

**Q4. 배터리 시스템에 똑같이 적용 가능한가요?**
> 온톨로지 YAML을 배터리 구조(RACK·BMS·Inverter)로 작성하고 디렉토리 하나 추가하면, 로더·검색·Bundle 로직은 수정 없이 작동해야 합니다 — 가설이며 실증 필요.

**Q5. no-match가 너무 많이 뜨면 어떡하나요?**
> 이벤트 카테고리별로 target 태그 정의 단계에서 커버리지를 합의하고, 이후 통계 메트릭 정의는 온톨로지 엔티티 기준으로 역설계하는 방식이 낫습니다.
