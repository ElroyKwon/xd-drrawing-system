# 03 — 통계 × 온톨로지 × 문서 RAG 통합 설계

> Phase 1(01-design.md)은 "통계 + 온톨로지"까지만 결합한다. 이 문서는 **문서 RAG**까지 올려 "AI가 절차·이력·스펙 문서를 근거로 답하게 하는" Phase 2 아키텍처를 정리한다.
> 회의 시연용 목업 단계에서 저비용으로 흉내낼 수 있는 경로는 §8에 따로 뒀다.

---

## 0. 한 장 요약

```
기존:    [통계 JSON] ─────────────────────────────────► [AI 해석]   (일반론)
Phase 1: [통계] + [온톨로지] ────────────────────────► [AI 해석]   (스펙·관계)   ← 현재 목업
Phase 2: [통계] + [온톨로지] + [문서 RAG] ───────────► [AI 해석]   (절차·이력·인용)
                                                       │
                                                       └─► 답변에 [1][2] 인용,
                                                           원본 문서로 역추적 가능
```

세 축이 서로 보완:

| 축 | 답하는 질문 |
|---|---|
| 통계 | "무슨 값이 얼마나 이상한가" |
| 온톨로지 | "그 설비가 무엇이며 어디에 붙어 있는가" |
| 문서 RAG | "어떻게 처리해야 하고, 전에는 어떻게 처리했는가" |

단독으로는 모두 부족:

- 통계만 → "z-score -3.2σ"만 알고 "1101"이라는 설비도, 매뉴얼 §4.3도 모름
- 통계+온톨로지(Phase 1) → "1101 VCB 패널에서 발생"까지 오지만 "그러면 뭐?"에서 LLM 일반 지식에 의존
- 통계+온톨로지+RAG(Phase 2) → "매뉴얼 §4.3 절연저항 점검 절차 + 3개월 전 유사 사례 이력 [2]" 까지 답변

## 3. 문서 RAG 파이프라인 상세

### 3.1 인덱싱 (오프라인, 배치)

```
  ┌───────────────────────┐       ┌─────────────────────────┐     ┌─────────────────┐
  │ 원본 저장소            │       │ 전처리·청킹·임베딩       │     │ 인덱스 스토어    │
  │                       │       │                         │     │                 │
  │ ├─ /dwg/*.pdf         │       │ 1. 텍스트 추출          │ ───►│ [BM25 sparse]   │
  │ ├─ /매뉴얼/*.pdf      │ ─────►│    · PDF 파서           │     │  term → doc_id  │
  │ ├─ /시방서/*.docx     │       │    · OCR (스캔)         │     │                 │
  │ ├─ /점검기록/*.csv    │       │                         │     │ [Dense vector]  │
  │ ├─ /알람이력.json     │       │ 2. 청크 분할            │ ───►│  cosine sim     │
  │ └─ /주석 (localSt.)   │       │    · 도면 : 페이지+블록 │     │  (Chroma/Qdrant)│
  │                       │       │    · 매뉴얼: § 헤딩     │     │                 │
  └───────────────────────┘       │    · 이력  : 레코드      │     │ [Metadata DB]   │
                                  │      500~1000토큰,      │ ───►│  doc_id, page,  │
                                  │      50 overlap         │     │  discipline,    │
                                  │                         │     │  entity_tags[], │
                                  │ 3. 메타데이터 부착      │     │  updated_at     │
                                  │    (가장 중요)          │     │                 │
                                  └─────────────────────────┘     └─────────────────┘
```

**메타데이터가 핵심**: 온톨로지 태그를 청크에 미리 붙여두면 Phase 1의 Bundle에서 나온 연관 태그로 **후보 좁히기**를 먼저 할 수 있다. 순수 임베딩 검색보다 정확도·비용 모두 우월.

### 3.2 쿼리 시점 (온라인, 이벤트 관측)

```
                         [이벤트 발생]
                              │
                              ▼
             ┌──────────────────────────────────┐
             │  Step 1 — 쿼리 조립              │
             │                                   │
             │  "VCB 1101 절연저항 z_score      │
             │   -3.2σ 이상, 상류 BLK-MOF,      │
             │   하류 TR-001, 22.9kV 패널"       │
             └──────────────┬───────────────────┘
                            │
             ┌──────────────┴──────────────┐
             ▼                             ▼
   ┌──────────────────┐          ┌────────────────────┐
   │  Step 2a — BM25  │          │  Step 2b — Dense    │
   │  (태그·용어 정확)│          │  (의미 유사성)       │
   │                  │          │                    │
   │  → top-20 cand.  │          │  → top-20 cand.    │
   └────────┬─────────┘          └──────────┬─────────┘
            │                                │
            └───────────────┬────────────────┘
                            ▼
             ┌──────────────────────────────────┐
             │  Step 3 — 메타데이터 필터        │
             │                                   │
             │  · discipline: ELECTRICAL         │
             │  · entity_tags ∩ {1101, BLK-MOF, │
             │     TR-001} ≠ ∅                  │
             │  · updated_at ≥ 2024-01 (최신성) │
             └──────────────┬───────────────────┘
                            ▼
             ┌──────────────────────────────────┐
             │  Step 4 — Cross-encoder 리랭크   │
             │  (query context ↔ chunk text)    │
             │                                   │
             │  → top-k (k = 5~8)                │
             └──────────────┬───────────────────┘
                            ▼
               citations[] → Bundle v2에 부착
```

**왜 하이브리드인가**:

```
          BM25 단독                         Dense 단독
          ─────────                         ─────────
   ◎ "1101" 태그 완벽 히트             △ 태그 숫자 약함 (토크나이저)
   ◎ "GIPAM-2000FI" 모델명 정확         ◎ "절연저항 저하 증상" 의미 검색
   × "비슷한 이상 사례"는 약함          × 정확한 절 번호 못 찾음
   × 동의어 처리 안 됨                   ◎ "퍼지 ↔ 절연 파괴" 유사

   하이브리드 + Cross-encoder 리랭크 = 두 장점 결합 + 노이즈 제거
```

## 5. BundleV2 스키마

```ts
interface Citation {
  id: string;                         // "cite-001"
  doc_id: string;
  doc_type: 'drawing' | 'manual' | 'spec'
          | 'alarm_history' | 'inspection' | 'annotation';
  locator: string;                    // "p.5" | "§4.3" | "2025-11-14T09:20"
  title: string;
  snippet: string;                    // 인용 원문 200~400자
  relevance: number;                  // 리랭커 점수 (0~1)
  retrieval: {
    method: 'bm25' | 'dense' | 'hybrid';
    matched_terms?: string[];
    matched_entities?: string[];      // 메타 필터에서 걸린 태그
  };
}

interface InsightBundleV2 extends InsightBundle {
  citations: Citation[];              // RAG 결과 (top-k)
  retrieval_meta: {
    query: string;                    // 조립된 쿼리
    candidate_count: number;          // BM25+Dense 합산 후보 수
    reranked_count: number;           // 최종 top-k
    elapsed_ms: number;
  };
}
```

## 6. 인용 근거 체인 (감사 가능성)

```
             [LLM 답변 본문]
                  │
                  ▼
   "…절연저항이 기준 이하면 매뉴얼 §4.3에 따라
    1차 접점 분해·청소를 먼저 수행하세요 [1].
    2025-11 유사 사례에선 CT 접점 탄화가 원인
    이었으므로 함께 점검 권장합니다 [2]."
                  │
        ┌─────────┴─────────┐
        │                   │
        ▼                   ▼
    [1] cite-001        [2] cite-002
        doc-manual-003      hist-alarm-2025-11-14
        §4.3                2025-11-14
        │                   │
        ▼                   ▼
    청크 원문            이력 레코드
    (800자)              (JSON)
        │                   │
        ▼                   ▼
    원본 PDF            원본 JSON
    /매뉴얼/            /알람/
     VCB-점검.pdf        history-2025.json
     (p.12)
```

→ **답변 → 인용 → 청크 → 원본** 4단계 역추적. 사후 검증·감사 가능성이 RAG가 가져오는 실질적 가치.

## 8. 목업 단계에서 "RAG 흉내" (Phase 1.5 저비용 경로)

실제 벡터 DB 없이 회의 자리에서 "RAG 결합 답변"을 시연하려면:

```
Phase 1 (현재)                 Phase 1.5 (목업 확장)
─────────────                 ─────────────────────

 Insight Bundle v1              Insight Bundle v1.5
 ┌─────────────┐                ┌─────────────┐
 │ event       │                │ event       │
 │ entity      │                │ entity      │
 │ related     │      ──►       │ related     │
 │ evidence    │                │ evidence    │
 │ summary     │                │ summary     │
 └─────────────┘                │ citations[] │ ◄── NEW (수작업 매핑)
                                │  mock 3~5건 │
                                └─────────────┘
```

**구현 순서**:

```
1. data/insight/mock-citations.json 준비
   이벤트별로 3~5건씩 수작업 작성:
   {
     "evt-001": [
       { id, doc_type: "manual", locator: "§4.3",
         title: "VCB 절연저항 점검 절차",
         snippet: "…", retrieval_method: "mock" },
       ...
     ]
   }

2. bundle-builder.ts 확장
   buildBundle(event) 내부에서 citations도 함께 조립

3. BundleView.tsx에 "문서 근거" 섹션 추가
   · doc_type별 컬러 배지 (manual/history/spec/annotation)

   "[1] …", "[2] …" 형식.
   LLM이 자연스럽게 [1][2] 인용 표기하도록 지시.

5. UI에 한계 배너 표시
   "현재 문서 근거는 수작업 목업. 실제 구축은
    하이브리드 RAG 파이프라인 필요."
```

## 10. 한계·리스크

### 10.1 기술적

| 리스크 | 완화책 |
|---|---|
| **인용 환각** — LLM이 존재하지 않는 citation ID를 지어냄 | 후처리 단계에서 답변 내 [N] 추출 → citations[]와 대조 → 미존재 시 답변 reject/재생성 |
| **청킹 경계 손실** — 긴 절차가 청크 경계에서 끊김 | Parent-Document retriever: 청크로 매칭하고 상위 부모 문서 전체 제공 |
| **인덱스 신선도** — 매뉴얼 개정 누락 | `updated_at` 메타 강제. 오래된 청크는 rerank 시 감점 |
| **OCR 품질** — 스캔 도면 오타·누락 | 수동 교정 루프, OCR 신뢰도 점수를 메타에 저장 |
| **임베딩 모델 선택** — 한국어 + 전문용어 | BGE-M3, e5-multilingual, kure-v1 등 후보 비교 평가 필수 |
| **청크-쿼리 mismatch** — 긴 질문 → 짧은 청크 | Hypothetical Document Embeddings (HyDE) 또는 MultiQueryRetriever |

### 10.2 조직적

| 리스크 | 완화책 |
|---|---|
| 매뉴얼·시방이 **부서별 파편화** | 초기 프로젝트의 절반은 수집·표준화. 담당자 지정 |
| **권한 문제** (문서별 열람 제한) | 쿼리 시점에 사용자 역할 필터 주입. `allowed_roles` 메타 |
| **업데이트 주체 불명확** | 소스 문서별 책임자 매핑표 선행 |
| **점검 기록 부재** | 이력이 없으면 Phase 2의 핵심 가치 사라짐. 기록 프로세스 선결 |

### 10.3 평가

```
  retrieval 품질 (독립 측정)        e2e 답변 품질 (사람 평가)
  ──────────────────────           ─────────────────────────

  · Recall@k                         · 골든 Q&A 세트 50~100건
  · MRR (Mean Reciprocal Rank)       · 정답·인용 정확성
  · nDCG                             · 환각률
  · RAGAS faithfulness               · 실사용자 만족도
```
