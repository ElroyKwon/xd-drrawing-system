# 06 · 기본과 혁신 — 4 Layer 계층 재정리

> **이 문서의 역할**: 01~05가 "기능 블록 / 도면 패턴 / 기술 옵션 / 결정 / 비판"을 **평면적으로** 나열했다면, 이 문서는 "정보 검색 → 도면 정보 제공 → 기본 → 혁신"의 **수직 계층**으로 재조립. 04의 옵션 A/B/C 세트가 왜 그 형태로 나왔는지의 근거를 마지막에 같이 정리.
>
> **작성 동기**: 지금까지의 문서 구조는 "무엇이 기본이고 무엇이 혁신인가"의 경계가 흐렸다. 결과적으로 의사결정에서 **"있어야 할 것"과 "차별화"가 뒤섞여** 우선순위를 정하기 어려웠다. 4 Layer로 분리하면 "구멍부터 메우고, 그 다음 혁신 하나를 고른다"는 순서가 분명해진다.

---

## 0. 4 Layer 전체 지도

```
Layer A : 정보 검색 사이트의 보편 기본          [도메인 무관]
  │
  │  "검색·결과·출처·피드백"의 최소 뼈대
  ▼
Layer B : "도면 정보 제공"의 의미                [도메인 특성]
  │
  │  도면이라는 객체의 5가지 정보 층위
  ▼
Layer C : 도면 정보 제공의 기본 요건             [A × B 교차]
  │
  │  "도면 검색 SaaS"라 부를 수 있는 최소 조건
  ▼
Layer D : 도면 정보 제공의 혁신 방향             [C 위에 더하는 것]

     기본이 해결하지 못한 페인을 푸는 5가지 경로
```

**읽는 법**: 위층은 아래층의 **전제**다. Layer D를 아무리 잘해도 Layer C의 구멍이 있으면 사용자는 떠난다. 반대로 Layer C까지만 충족된 제품은 "있어도 되고 없어도 되는 것"이 된다. 이 프로젝트의 위치는 "C는 거의 도달, D 중 무엇을 고를지"의 단계.

## Layer A — 정보 검색 사이트의 보편 기본

### A.1 정의

도메인과 무관하게 "정보를 찾아주고 보여주는 사이트"라고 부르려면 반드시 있어야 하는 최소 요소.

### A.2 최소 8요건

| # | 요건 | 최소 기준 | 없으면 어떤 문제? |
|---|---|---|---|
| 1 | 입력창 | 키워드 수용 | 쿼리할 방법 없음 |
| 2 | 결과 리스트 | 0~N건 열거 | 무엇을 찾았는지 모름 |
| 3 | 개별 상세 | 클릭→상세 화면 | 결과가 의미 없음 |
| 4 | 필터·정렬 | 1축 이상 | 많아지면 사용 불가 |
| 5 | 출처 표시 | 원본 링크/식별자 | 신뢰 불가 |
| 6 | 체감 지연 | 500ms 이하 기본 피드백 | 끊긴 느낌, 재쿼리 증가 |
| 7 | 빈 상태 | "결과 없음" 명확한 안내 | 오류와 구분 불가 |
| 8 | 오류 상태 | 실패 메시지 + 다음 행동 | 사용자 이탈 |

### A.3 한 단계 위 — "정보 제공" 사이트로서의 기본

위 8요건이 "검색이 된다"라면, 다음은 "정보를 **제공**한다"의 기본.

| # | 요건 | 의미 |
|---|---|---|
| 9 | 정답 우선 노출 | 가장 관련 높은 것이 상단 — re-rank |
| 10 | 매칭 근거 | 왜 이게 결과인지 (하이라이트·스니펫) |
| 11 | 다음 행동 연결 | 결과를 가지고 무엇을 할지 (공유·저장·티켓·깊은탐색) |

이 11개가 **"정보 검색 사이트"로 자임하려면 반드시 있어야 하는 것**. 01 문서의 B1~B6은 이 중 1~11을 **엔터프라이즈 규모로 확장한 것**이다. 즉 01은 "스케일 업 판본"이고 여기 A.3이 "최소 판본".

### A.4 평가 단일 지표

**Effort-to-Answer (ETA)**: 사용자가 질문을 떠올린 시점부터 쓸 수 있는 답을 손에 쥘 때까지의 비용. 초·클릭·인지 부하로 측정. 11요건은 모두 ETA를 낮추기 위한 장치로 해석 가능.

### A.5 현 프로젝트 매핑

| 요건 | `(s1)` 홈 | 상태 |
|---|---|:---:|
| 1 입력창 | `SearchBar` | ✅ |
| 2 결과 리스트 | `DrawingsList` | ✅ |
| 3 개별 상세 | `PdfViewer` | ✅ |
| 4 필터·정렬 | 학제·종류 칩 | △ (정렬 없음) |
| 5 출처 표시 | `doc_id` 노출 | △ (링크 아님) |
| 6 체감 지연 | 즉시 | ✅ |
| 7 빈 상태 | "좌측에서 선택" 카피 | ✅ |
| 8 오류 상태 | 미구현 | ❌ |
| 9 정답 우선 노출 | Fuse 기본 score | △ |
| 10 매칭 근거 | 스니펫만, 하이라이트 없음 | △ |
| 11 다음 행동 연결 | 없음 | ❌ |

**진단**: Layer A는 거의 충족. 실제 이탈 유발 확률은 8(오류)과 11(다음 행동) 두 개. Layer A 차원에서의 개선은 고비용 아님.

## Layer B — "도면 정보 제공"의 의미

### B.1 도면이라는 정보 객체의 특성

도면은 일반 문서와 정보 구조가 근본적으로 다르다. 다음 **5가지 정보 층위**를 겹쳐 담는다.

| 층위 | 예 | 일반 문서와 차이 |
|---|---|---|
| L1. 시각적 | 선·기호·색상·레이아웃 | 텍스트 문서는 시각이 부차적 |
| L2. 공간적 | "A가 B 왼쪽" = "A가 B 상류" | 공간 관계 자체가 의미 |
| L3. 관계적 | 두 심볼을 잇는 선 = "연결" | 명시적 관계 엣지가 존재 |
| L4. 계층적 | 시스템도 → 계통도 → 상세도 → 부품도 | 다중 해상도 문서군 |
| L5. 버전적 | Rev 0 → Rev 1 → Rev 2 | 시간축 + 변경 사유 |

### B.2 따라서 "도면 정보 제공"이 의미하는 것

도메인으로서 "도면을 다룬다"는 건 다음 5가지를 **모두 또는 부분적으로** 커버한다는 뜻:

| # | 무엇을 제공하는가 | 예 |
|---|---|---|
| P1. 파일 접근 | 도면 파일을 찾고 열기 | 기본 중의 기본 |
| P2. 객체 속성 | 도면 속 개별 객체의 사양·메타 | "CH-001은 냉동기, 200RT, 2020년식" |
| P3. 관계 해설 | 객체 간 연결·종속 | "CH-001 → CT-01 → VFD-007" |
| P4. 도면 간 연관 | 메인 ↔ 상세, 변경 ↔ 영향 | 콜아웃 하이퍼링크, 관련 도면 리스트 |
| P5. 이력·변경 | Rev 추적, 변경 사유 | Revision History, 변경 관리 |

**포지셔닝**: 제품이 P1~P5 중 어디까지 책임지느냐가 그 제품의 영역을 정의한다. Autodesk ACC는 P1+P4+P5가 강하고 P2·P3은 Revit 계통에 위임. Bluebeam은 P1+P2(일부)+P5. 현 `(s1)`은 P1+P2(메타만)+P4(EntityToDocs).

### B.3 왜 이 구분이 중요한가

사용자가 "도면을 검색한다"고 할 때 실제로 묻는 건 P1이 아니라 **P2~P4인 경우가 더 많다**. "도면 어디 있어요?"는 "도면을 보고 싶어서"가 아니라 **"도면에 있는 그 장비 정보가 필요해서"**라는 경우.

이 구분을 못 하면 Layer C는 P1만 풀고 Layer D로 뛰어버리게 된다. 실제로 이전 프로토타입이 그랬다 — P1·P2의 섬세함 없이 "AI+KG+챗"이라는 Layer D 스택을 쌓았고, 페르소나는 "기본이 약한데 껍데기가 화려하다"는 반응을 보였을 가능성이 크다.

## Layer C — 도면 정보 제공의 기본 요건

### C.1 정의

Layer A(일반 검색 기본) × Layer B(도면 정보 5층위)의 교집합. **"도면 정보 SaaS"라 부를 수 있는 최소 조건**.

### C.2 7요건

| # | 요건 | 무엇 | Layer A·B 연결 |
|---|---|---|---|
| C1 | 도면 검색·리스트 | 제목·번호·종류로 찾기 | A.1~A.4 + B.P1 |
| C2 | 도면 뷰어 | 확대·이동·페이지, 렌더 속도 충분 | A.3 개별 상세 + B.L1 |
| C3 | 도면 메타 | 번호·Rev·작성일·작성자 | A.5 출처 + B.P5 |
| C4 | 도면 내 텍스트 검색 | 뷰어 안에서 단어 찾기 | A.1 + B.L1 |
| C5 | 도면 간 이동 | 관련/참조 도면 리스트 or 콜아웃 링크 | A.11 다음 행동 + B.P4 |
| C6 | 객체 메타(최소) | 태그→문서 역링크 수준 | A.3 + B.P2 |
| C7 | 원본 접근 | 다운로드·공유 | A.5 + B.P1 |

### C.3 품질 기준 (C에서 이미 측정해야 함)

| 축 | 기준 |
|---|---|
| 렌더 속도 | 페이지 전환 후 1초 이내 가독 |
| 검색 정확도 | 정확한 도면번호 쿼리 시 Top-1 |
| 공간 인지성 | 리스트 클릭 → 뷰어 표시까지 2클릭 이내 |
| 인쇄·외부 공유 | 최소한 다운로드 링크 |

### C.4 현 프로젝트 매핑

| 요건 | `(s1)` | 상태 | 비고 |
|---|---|:---:|---|
| C1 도면 검색·리스트 | SearchBar + DrawingsList + 필터 칩 | ✅ | 한글 학제 매칭 버그(07-quirks §2.1)는 소급 해결 가능 |
| C2 도면 뷰어 | PdfViewer + zoom-pan-pinch | ✅ | 실 PDF 확보 시 pickMode 재정비 필요 |
| C3 도면 메타 | `type`, `discipline`, `doc_id`, `drawing_ref` | △ | drawing_number·revision·location 필드 **타입만 있고 값 미채움** |
| C4 도면 내 텍스트 검색 | ❌ | ❌ | PDF.js find controller 미장착 |
| C5 도면 간 이동 | EntityToDocs (태그 역링크) | △ | 콜아웃 하이퍼링크 없음 |
| C6 객체 메타(최소) | doc-entity-links.json (1-hop) | △ | 객체 속성은 없고 문서 연결만 |
| C7 원본 접근 | 없음 | ❌ | 다운로드 버튼 없음 |

**진단**: Layer C는 **"충족 5 / 약함 4 / 결손 2"**. C4(도면 내 텍스트 검색)와 C7(원본 접근)은 명백한 결손. C3·C5·C6은 "있다고 말할 수 있지만 제품으로서는 부족한" 수준.

**중요한 함의**: Layer D(혁신)를 논하기 전에 **Layer C의 결손·약함을 먼저 보강**하는 것이 이전 프로토타입의 실패를 반복하지 않는 길. 혁신 기능을 끼워넣는 것보다, 부족한 기본을 메우는 것이 페르소나 채택률을 더 올릴 수도 있다.

## Layer D — 도면 정보 제공의 혁신 방향

### D.1 "혁신"의 정의

> **기본(Layer C)이 해결하지 못한 사용자 페인을 푸는 것.**

새롭거나 화려한 것이 아니라, **기본이 닿지 못한 지점에 가치를 만드는 것**. 이 정의는 05 비판 평가의 결론과 일관된다 — 기술 자체의 새로움이 아닌, 사용자 문제의 해결이 혁신의 기준.

### D.2 Layer C가 해결하지 못하는 5가지 페인

| 페인 | 증상 | Layer C로는 왜 못 푸나 |
|---|---|---|
| F1. 파일을 찾았지만 **객체**를 모른다 | "도면은 열었는데 이 밸브 뭐지?" | C는 "파일" 단위 제공 |
| F2. 시각↔텍스트 언어의 간극 | 검색은 텍스트, 도면은 그림 | C는 텍스트 검색만 |
| F3. 관계가 보이지 않는다 | "이거 내리면 뭐가 죽나?"를 도면으로 못 품 | C는 조인 없는 리스트까지만 |
| F4. 한 객체의 **스토리**가 흩어져 있다 | 사양·매뉴얼·이력·알람이 다른 시스템에 | C는 도면+메타만 |
| F5. 도면은 **사무실 도구**로만 작동 | 현장에선 안 열린다 | C는 PC·키보드 전제 |

### D.3 이 5페인에 대응하는 혁신 방향 5가지

| # | 혁신 방향 | 푸는 페인 | 업계 선례 | 우리가 더할 수 있는 것 |
|---|---|---|---|---|
| I1 | **객체-중심 정보 제공** | F1 | Forge Property Panel, Revit 객체 | 수동 온톨로지 기반 경량 버전 |
| I2 | **양방향 검색↔뷰어** | F2 | Bluebeam VisualSearch | 검색어→하이라이트 + 영향 범위 색상화 |
| I3 | **관계 탐색 (Graph Traversal)** | F3 | Neo4j Bloom, yFiles | 1-hop → 2-hop 확장, 우측 패널 서사화 |
| I4 | **객체 Asset Timeline + RAG** | F4 | Glean, ACC Asset (부분) | 한 객체에 도면·매뉴얼·알람·주석 통합 답변 |
| I5 | **현장 맥락 UI (포지셔닝 혁신)** | F5 | 부분적·산발적 | 한글·모바일·오프라인·QR·경량 라이선스 |

### D.4 혁신 방향의 "자기 방어력"

05 비판 평가는 "기술 혁신은 낮음, 포지셔닝 혁신은 높음"이라 결론냈다. 이를 I1~I5에 대입하면:

| 방향 | 기술 새로움 | 시장 차별화 | 실패 시 대체재 | 방어력 |
|---|:---:|:---:|---|:---:|
| I1 객체-중심 | 낮음 | 낮음 (ACC 존재) | ACC | 낮음 |
| I2 양방향 검색 | 낮음 | 중간 (Bluebeam 부분) | Bluebeam | 중간 |
| I3 관계 탐색 | 중간 | 중간 (제너럴 툴만) | 없음 | 중상 |
| I4 Asset Timeline+RAG | 높음 (2024+) | 중상 | 없음 | 중상 |
| I5 포지셔닝 (한글·현장) | 낮음 | **높음** | 없음 | **높음** |

**결론**: 하나만 골라야 한다면 **I5 또는 I3+I5 조합**. I5가 단독 혁신이라기보다, I1~I4를 **한국 OT 현장 경량 대안**이라는 그릇 안에 담을 때 가장 방어 가능한 서사가 된다.

### D.5 04에서 논의된 "4패턴"의 재해석

04의 4패턴(Click-to-Search / Search-to-Highlight / 영향도 2-hop / HUD)은 D.3의 I1·I2·I3·I5에 매핑된다. 즉 **이미 혁신 방향 5개 중 4개를 시도하는 설계**다. 빠진 건 I4(RAG Asset Timeline)인데 이는 Insight Lab `(s2)/insight/lab` 트랙이 이미 다룬다.

**함의**: 04의 4패턴 선택이 우연이 아니라 **페인 F1·F2·F3·F5를 노린 선택**으로 해석된다. 단 05에서 지적한 대로 "기술 자체가 새로워서"가 아니라 "한국 OT 현장 페인 풀이용"이라는 프레이밍으로 제시해야 방어력이 선다.

## 7. 04 옵션 A/B/C가 왜 그렇게 나왔는가 — 근거 재조사

04는 D1·D2·D3 세 축에 각 3개 옵션(A/B/C)을 제시했다. 그 구조가 임의적으로 보였는데, Layer 프레임으로 재해석하면 각 축과 옵션의 근거가 분명해진다.

### 7.1 D1. 구현 깊이 — "목업의 설득 방식" 축

| 왜 이게 축인가 | 05의 비판 평가에서 **"목업이 시연에서 매끈할수록 의사결정자의 과신 리스크가 커진다"**를 지적. 그렇다면 "얼마나 진짜처럼 만드냐"는 설득 vs 리스크 트레이드오프의 핵심 축. |
|---|---|

| 옵션 | 본질 | Layer 기준 근거 |
|---|---|---|
| A. 동작 프로토타입 | 진짜 상호작용 | Layer D의 I1~I3을 **체감으로 검증**. 단 "보이지 않는 비용" 과신 리스크 최대 |
| B. 정적 모형 | 스크린샷 | Layer D를 **개념으로만 검증**. 리스크 최소, 설득력 최소 |
| C. 시나리오 인터랙션 | 1분 경로만 진짜 | Layer D를 **스토리로 검증**. 설득과 리스크의 중간 지점 |

**수렴**: C가 "설득/리스크 비"가 가장 좋다. A는 시연 비용 대비 리스크 노출, B는 "그림일 뿐"이라는 반응 위험.

### 7.2 D2. 데이터 소스 — "페르소나 공감" 축

| 왜 이게 축인가 | Layer B의 P2~P4는 **도메인 객체의 구체성**이 생명. 시연 대상이 공감할 수 있는 객체·태그가 아니면 Layer D 혁신의 체감이 없다. |
|---|---|

| 옵션 | 본질 | Layer 기준 근거 |
|---|---|---|
| A. 기존 재활용 | 8건 실 도면 | Layer C 기본과 일관, 현장성 유지 / 임팩트 약할 수도 |
| B. Gemini 예시 | 교과서적 P-101, CH-001 | Layer D 시연 용이, 현장과 괴리 |
| C. 하이브리드 | 기존 도면 + 새 태그·관계 | 두 장점 절충, 준비 비용 중간 |

**수렴**: C가 "Layer C의 일관성을 유지한 채 Layer D 시연 임팩트를 확보"하는 균형점.

### 7.3 D3. 개념 범위 — "서사 vs 카탈로그" 축

| 왜 이게 축인가 | 05는 "의사결정자는 스토리로 기억한다"를 지적. Layer D 혁신 방향 중 여러 개를 어떻게 묶느냐가 **설득의 구조**를 결정. |
|---|---|

| 옵션 | 본질 | Layer 기준 근거 |
|---|---|---|
| A. 4패턴 전부 | I1·I2·I3·I5 카탈로그형 | Layer D의 범위 최대, 인지 부하 큼 |
| B. 집중 2개 | I1+I3만 | 핵심 가치만, 나머지 "없어서 놓친 것" |
| C. 시나리오 연쇄 | 1분 안에 I1→I2→I3→I5 순차 | 카탈로그가 아닌 이야기. **기억에 남는 구조** |

**수렴**: C가 카탈로그의 포괄성과 이야기의 기억력을 동시 확보. 단 "보여준 것 외에는 확장 가능성을 말로 설명"해야 함.

### 7.4 세 축의 상호의존

D1=C → 시나리오 하나를 골라야 함 → D3=C가 자연스러움 → 시나리오에 맞는 데이터가 필요 → D2=C가 수반됨. 즉 **C/C/C는 개별 선택이 아니라 하나의 설계 철학의 세 얼굴**. "시나리오 중심, 경량, 페르소나 공감, 기존 일관"이라는 하나의 선택.

다른 조합(예: A/B/A)은 "총력 카탈로그 데모" 방향이지만 이전 프로토타입이 이미 시도했고 실패했다.

## 8. 우선순위 재제안 — 4 Layer 프레임으로

05에서 "옵션 A vs 옵션 Z" 갈래를 제시했는데, 이를 Layer 프레임으로 다시 쓰면 의사결정이 더 명확해진다.

### 8.1 세 갈래 재제안

| 갈래 | 내용 | 4 Layer 해석 |
|---|---|---|
| **갈래 ①** | **Layer C 결손 보강 먼저** | C4(도면 내 텍스트 검색), C7(원본 접근), C3/C5/C6 약함 해소. **혁신 없이 기본 완성**. 이전 프로토타입 실패가 "기본이 약한데 혁신을 쌓았다"였다면 이 갈래가 그 반성. |
| **갈래 ②** | **Layer D 혁신 1개 프로토타입** | 04 권장안 = C/C/C (시나리오 연쇄 프로토타입). I1·I2·I3·I5를 한 시나리오에 녹이는 목업. |
| **갈래 ③** | **Z — W4-T2 + 페르소나 재검증** | 신규 트랙 없이 기존 4주 MVP 마무리 + 이전 페르소나 2~3인 재검증. 우리가 이미 한 것의 효과를 먼저 측정. |

### 8.2 갈래별 기대 답

| 갈래 | 답하는 질문 | 실패 위험 |
|---|---|---|
| ① | "Layer C 완성만으로 지불 의사가 생기나?" | 기본만으론 여전히 2.3/10 반복 가능 |
| ② | "Layer D 혁신 1개가 결정적 차이를 만드나?" | 혁신 자체는 기존 제품 기능과 유사 → 포지셔닝 서사가 없으면 지난 실패 재현 |
| ③ | "우리가 이미 한 개선이 효과가 있었나?" | 정보는 가장 확실 / 새로운 설득 자산 없음 |

### 8.3 권장

**① + ③** 또는 **② + ③** 중 하나. ① 단독은 매력 부족, ② 단독은 기본 구멍 과신 리스크. ③는 어느 경우든 병행 가치가 있다.

가장 큰 함의: **"D를 하기 전에 C를 먼저 보라"**가 이번 재정리의 핵심 메시지. 04의 C/C/C를 택하더라도 Layer C의 C4/C7 결손을 같은 트랙에서 함께 메우는 것이 제품 건전성을 지키는 길이다.

## 9. 이 문서의 최종 결론

1. **"정보 검색 사이트"의 기본**(Layer A 11요건)은 현 `(s1)`이 거의 충족. 오류 상태와 "다음 행동"이 약점.
2. **"도면 정보 제공"이란**(Layer B) 단순 파일 제공이 아니라 P1~P5 5층위의 정보를 제공한다는 뜻. 도면은 파일이 아니라 객체·관계·이력을 포함한 복합 정보 객체.
3. **"도면 정보 제공의 기본"**(Layer C 7요건)은 현 `(s1)`이 **"충족 5 / 약함 4 / 결손 2"**. 혁신을 논하기 전에 메워야 할 구멍이 남아 있다.
4. **"도면 정보 제공의 혁신"**(Layer D)은 "기본이 해결 못한 페인 5개" → "혁신 방향 5개" 매핑으로 정의. **가장 방어 가능한 혁신은 기술이 아니라 포지셔닝(I5: 한글·현장·경량)**.
5. **04의 옵션 A/B/C**는 임의 나열이 아니라 **"시나리오 중심·경량·페르소나 공감·기존 일관"이라는 하나의 설계 철학의 세 얼굴**(D1/D2/D3 = C/C/C). 상호의존적으로 수렴하는 구조.
6. **다음 의사결정**은 "① Layer C 보강 / ② Layer D 혁신 1개 시나리오 / ③ 페르소나 재검증" 세 갈래에서 **①+③ 또는 ②+③ 조합** 중 택.
