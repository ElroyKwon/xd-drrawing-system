### V-2. 실 현장 파일럿 (청주 OR 여수 1개소)

- **범위**: 도면 50~100건 실제 업로드 + 감독자 2~3명 1주일 사용

- **블로커**: 도면 저작권·기밀 동의 필요

### V-3. 접근성(a11y) 감사

## 2. 시스템 기반 (System Foundation)

### S-1. PWA 오프라인 + IndexedDB (편의 L)
- **배경**: StarNote·현장 앱 조사 **모두에서 1순위**로 지목된 편의. 현장 Wi-Fi 음영 광범위
- **범위**:
  - Service Worker (Workbox 기반)
  - IndexedDB에 PDF 바이너리 + 주석 데이터 저장
  - 동기화 큐 (offline 작업 → online 복귀 시 자동 플러시)
  - 캐시 무효화 (도면 버전 갱신 시)
  - "한 번 본 것은 캐시" 보수적 캐시 정책 (Procore 패턴)
  - 충돌 해결 규칙 (서버 우선 / 클라이언트 우선 / merge)
- **예상 기간**: 1~2주 (단독 스프린트)
- **우선순위**: **🔴 현장 투입 전제**
- **의존성**: 독립 트랙 가능. 단 S-2 백엔드 API가 있어야 동기화 완성
- **선별표 위치**: 13개 후보 중 L

### S-2. 백엔드 최소 기반 (DB · API · 파일 저장)
- **배경**: 현재는 정적 JSON. 다중 사용자·업로드·권한 불가능
- **범위**:
  - DB 스키마 (drawings, annotations, doc_entity_links, favorites, users, orgs, audit_log)
  - REST 또는 GraphQL API
  - 파일 스토리지 (S3 또는 NCP Object Storage — 한국 오픈소스 활용 시)
  - 업로드 엔드포인트 + PDF 미리보기 생성 파이프라인
  - API 인증 (JWT 또는 세션)
- **예상 기간**: 3~4주
- **우선순위**: 🟠 파일럿(V-2) 전 필요
- **의존성**: S-3(인증) 선행 권장

### S-3. 인증·권한·조직
- **배경**: 감독관/작업자/설계자 3자 분리 필요. Procore "Personal vs Published" 레이어 원칙
- **범위**:
  - 사용자 로그인 (SSO 옵션: Microsoft/Google)
  - 역할 기반 권한 (RBAC): Admin / Editor / Viewer / Field
  - 조직·프로젝트 구분
  - 주석 레이어 분리 (개인 draft vs 팀 published)
- **예상 기간**: 2~3주
- **우선순위**: 🟠 S-2와 함께

### S-4. 실시간 협업 (Optional — Phase 2)
- **배경**: Bluebeam Studio Session 모델. 실시간 공유 마크업
- **범위**: Yjs 또는 Liveblocks 기반 CRDT. 주석 실시간 동기, presence 아바타
- **예상 기간**: 2~3주
- **우선순위**: 🟢 검증 후 판단

```
    "좋다" → V-2 파일럿 준비 (S-2/S-3 최소 + D-1 도면)
    "애매" → 목업 보완 → 재인터뷰
    "부정" → Pivot 논의
```

**트랙 B — 기술 완성 우선**
```
S-1 PWA 오프라인 (1~2주)
  → S-2/S-3 백엔드 + 인증 (4~5주)
    → V-2 파일럿 (2~4주)
```

**트랙 C — 제품화 이관 우선**
```
P-1 FRS + P-2 아키텍처 + P-3~5 ERD/API/NFR (2~3주 병행)
  → 외주/내부 개발팀 RFP
    → V-1은 계약 조건 검증용 병행
```

### 7-3. 비교

| 트랙 | 기간(1차 결과) | 비용 | 리스크 | 산출물 |
|:-:|---|---|---|---|
| **A** | 2~3주 | 매우 낮음 (인터뷰 비용) | 낮음 | go/no-go 데이터 |
| **B** | 7~10주 | 중간 (개발 리소스) | 중간 (V-1 없이 구현 → 방향 잘못 가능) | 현장 투입 가능 파일럿 |
| **C** | 2~3주 | 낮음 (내부 작성) | 높음 (V-1 없이 확정된 사양 = 재작업 가능) | 개발 계약 자료 |

**권장**: **A → (결과 positive 시) B → (파일럿 성공 시) C**

## 8. 회의 안건 템플릿

팀별 회의 시 사용할 기본 의제.

### 8-1. 프로젝트팀 내부 회의
1. 13번 문서로 목업 시연 (10분 또는 20분 코스)
2. 14번 문서 §7로 다음 트랙 선택 (A/B/C)
3. 선택된 트랙의 top 3 이슈 담당자·일정 확정
4. V-1 페르소나 2~3인 후보 선정 (해도 되는가? 누가 좋은가?)

### 8-2. 도메인 전문가 인터뷰 (V-1 실시 전)
1. 13번 목차 A (머리말)로 "이건 상용 코드가 아니다" 프레이밍
2. 20분 코스 시연 (13번 §P-2)
3. 4가지 질문 (13번 마지막 메시지)
4. 현장 맥락 (14번 D-1, D-2)에 대한 동의/수정 청취

### 8-3. 이해관계자·경영진 보고
1. 전체 구조 한 장 (13번 §B)
2. 14개 UX 원칙 맵 (13번 §N)
3. 7가지 트랙 선택지 (14번 §7)
4. 예산 견적 (트랙 B 선택 시 대략 7~10주 · 트랙 A는 2~3주)

### 8-4. 외부 개발팀 RFP (트랙 C 선택 시)
1. FRS(P-1) + 아키텍처(P-2) + ERD(P-3) + API(P-4) + NFR(P-5) 패키지 제공
2. 기간·비용·팀 구성 요청
3. V-1 결과를 검증 조건으로 첨부
