## 페르소나 1 — 김기계 (기계설비 엔지니어, 현장 경력 18년)

> "냉동기, 펌프, AHU 다 내가 담당입니다. 손에 오일 묻힌 채로 폰 꺼내서 확인해야 할 때가 많아요."

### 긍정 평가

1. **대시보드의 즐겨찾기** — CH-001, PCHWP-001 등 매일 보는 설비들이 첫 화면에 뜬다. 기존 CMMS에서 메뉴 3번 들어가야 찾던 걸 한 번에 해결.
2. **큰 글씨 설정** (`large: true`) — 장갑 끼고 현장에서 봐도 읽힌다. 글씨 크기 배려가 실제로 체감된다.
3. **다가오는 정비 섹션** — next_due 기준 이번 달 정비가 목록으로 나온다. 아침 점호 전 체크할 수 있다.
4. **영향 체인(ImpactChain)** — "냉동기 끄면 뭐가 멈추나"를 수평 화살표 흐름으로 바로 보여준다. 신입 교육 때 설명 도구로도 쓸 수 있다.
5. **설비 상세 → 정비이력 탭** — 전임자가 뭘 했는지 기록이 남는다면 인수인계 때 최고다.

### 객관적 지적

1. **즐겨찾기가 코드에 하드코딩** — `FAVORITES = ["CH-001", ...]` 이 `p1-mechanical/page.tsx` 에 고정. 실사용에서는 사람마다 담당 설비가 다르다.
2. **정비 날짜 필터가 고정값** — `"2026-04-15" ~ "2026-04-30"` 하드코딩. 오늘 날짜 기준 동적 계산이 아니므로 이 날짜가 지나면 항상 빈 목록.
3. **위치 기반 검색 없음** — "3층 기계실 펌프 전부"처럼 장소로 먼저 찾는 패턴이 현장에서 많다. 현재는 태그/이름 검색만 가능.
4. **정비이력 담당자 연락처 없음** — `engineer` 가 이름 문자열만. 퇴사자나 위탁업체 직원이면 연락처 추적 불가.
5. **ImpactChain 수평 스크롤** — depth 4~5 되면 화면이 옆으로 길어진다. 모바일/태블릿에서 사용하기 어렵다.

### 비관적 시각

1. **AI 답변만 믿다가 큰일 난다** — Chat에서 "CH-001 점검 주기"를 물으면 mock 로그를 읽어준다. 실제 제조사 매뉴얼과 다를 수 있는데 신뢰도 90%로 표시되면 그냥 믿는다.
2. **위키 탭이 항상 "HITL 검수 대기"면 쓸모없다** — 대부분의 설비 위키가 미생성이면 그 탭은 클릭할 이유가 없다. 없는 것만 못하다.
3. **다가오는 정비 클릭 → 정보 보기만 됨** — "정비 시작", "작업 기록" 버튼이 없다. 결국 별도 종이 양식 들고 간다.
4. **영향 분석이 역방향 확인에 불편** — "CH-001 점검하려면 뭘 먼저 꺼야 하나(upstream)"는 버튼 하나 더 눌러야 한다. 현장에서는 역방향 확인이 더 급하다.
5. **오프라인 지원 없음** — 지하 기계실, 배관 피트는 WiFi가 안 잡힌다. 이 시스템은 완전 무용지물이다.

### 불안전성 지적

1. **정비 공문 초안이 권한 검증 없이 생성** — WhatIfControls에서 클릭 몇 번으로 정비통보 공문이 나온다. 협력업체 직원이 버튼 누르고 "시설안전팀" 명의 공문 출력 가능.
2. **비상발전기 가동 준비 문구 무조건 삽입** — 소규모 밸브 교체 작업에도 "비상발전기 가동 준비" 지시가 공문에 자동 포함. 과잉 대응 지시 혼선.
3. **CRITICAL 설비 접근 이력 없음** — CRITICAL 설비의 영향도 분석을 누가 언제 조회했는지 감사 로그가 없다.

## 페르소나 2 — 이전기 (전기 엔지니어, BESS/ESS 전문, 경력 12년)

> "PMS, BMS, 인버터 계통도가 머릿속에 다 들어 있어야 합니다. 데이터가 틀리면 차단기 잘못 건드려요."

### 긍정 평가

1. **구조화된 답변 모드** (`answerMode: "structured"`) — 질의 결과를 상류/하류 테이블로 정리. 기존 엑셀 작업을 대체할 수 있는 수준.
2. **그래프 뷰(ImpactGraph, React Flow)** — 전기 계통은 트리가 아니라 망이다. 그래프로 봐야 분기·루프 구조가 보인다.
3. **Confidence Meter** — AI 답이 몇 % 신뢰인지 숫자로 표시. 83%면 내가 한번 더 확인하겠다는 판단을 스스로 할 수 있다.
4. **Evidence 패널** — 출처 문서 + 페이지 번호가 함께 표시. "어느 도면에서 나온 정보냐"가 전기 쪽에선 필수.
5. **페르소나 간 스위처** — 기계 담당자에게 "이 전기 데이터 봐봐" 할 때 URL 공유로 바로 이동 가능.

### 객관적 지적

1. **실데이터 80건은 너무 적다** — BESS PJT 기준 차단기, 릴레이, CT/PT, 버스바, 케이블 다 넣으면 수백~수천 태그. 지금은 샘플 수준이라 실용성 검증이 안 된다.
2. **도면 연동이 미니 썸네일 수준** — DrawingMiniView는 특정 페이지 이미지만. 단선도, 결선도, 선번장 페이지 네비게이션이 없다.
3. **`evidence_page: null` 관계가 많다** — 관계의 도면 근거가 null이면 "어느 도면에 이 연결이 있냐"를 추적할 수 없다.
4. **ImpactGraph 자동 레이아웃 없음** — 같은 depth 노드 간 엣지가 있으면(크로스 엣지) 그래프가 시각적으로 혼란해진다. 수동 배치다.
5. **방향 개념 혼재** — 전기의 upstream은 전원 측, downstream은 부하 측이다. 기계적 upstream과 혼재되어 있어 혼동 유발.

### 비관적 시각

1. **BESS 안전 규정에서 AI 답변 위험** — BMS 관련 "셀 밸런싱 이상 시 대응 절차"를 여기서 물어보고 따르면 화재 날 수 있다. mock 스니펫 기반 응답.
2. **전기 페르소나에 정비이력 없음** (`showMaintenance: false`) — 차단기 투입/개방 이력, 절연 측정 기록이 없으면 반쪽짜리.
3. **분야 필터가 ELECTRICAL만** — 전기-기계 인터록(펌프 기동 신호 → 전기 패널)은 필터 치면 보이지 않는다. 실제 고장은 경계에서 난다.
4. **신뢰도 92%가 무엇을 의미하는지 모름** — 어떤 관계 데이터가 AI 추론이고 어떤 게 도면 직접 확인인지 구분 없음. 92%는 너무 높다.
5. **대형 노드 수에서 ReactFlow 성능** — 200노드↑ 시 브라우저 렌더링이 느려지는지 검증이 없다.

### 불안전성 지적

1. **계통 영향도를 잘못 읽고 차단기 조작 시** — CRITICAL 전기 설비 차단 영향도가 틀렸는데 그대로 조작하면 대규모 정전. "이 데이터로 실제 조작하지 마시오" 경고 없음.
2. **접지 계통 누락 가능성** — 접지선, 등전위 본딩이 relations에 포함됐는지 불명. 누락 시 영향 분석에서 접지 루프 문제 미반영.
3. **HitlFlag 없는 데이터를 정상으로 오인** — 검수 완료와 미검수가 동일하게 뱃지 없음으로 표시됨.

## 페르소나 3 — 박소방 (소방안전관리자 1급, 건물 담당 8년)

> "소방청 감사에서 도면이랑 실물이 다르면 즉시 과태료입니다. 정확도가 생명이에요."

### 긍정 평가

1. **평면도 오버레이(ImpactFloorPlan)** — 소방 설비는 위치가 전부다. 스프링클러, 감지기가 어느 구역에 있는지 평면도에서 바로 보이는 건 획기적이다.
2. **Citation 답변 모드** — "NFPA 기준", "시방서 몇 페이지"를 출처와 함께 보여준다. 감사 대응 때 근거 추적이 가능하다.
3. **FloorPlan 존(zone) 색상 구분** — 방화구획별 색상이 다르면 어느 구역이 영향받는지 즉시 파악 가능.
4. **DocSnippet의 `drawing_ref` 필드** — 어느 도면 번호에서 나온 스니펫인지 연결됨. 도면 출처 추적 기반이 잡혀 있다.
5. **SVG 기반 평면도 인터랙션** — 마우스 오버 시 존 강조. 웹 표준 SVG라 별도 플러그인 없이 동작한다.

### 객관적 지적

1. **소방 mock 데이터 20건은 너무 적다** — 중규모 건물 기준 감지기만 수백 개. 스프링클러 헤드, 소화전, 방화셔터, 제연설비까지 합치면 수천 태그.
2. **FloorPlan 좌표가 mock** — `polygon`, `x`, `y` 좌표가 실제 CAD/BIM이 아닌 가상 수치. 실물 평면도 연동 방식이 핵심 미해결 과제.
3. **법정 점검 주기 필드 없음** — 소화기 6개월, 자탐설비 연1회 등 법정 주기가 Entity에 속성으로 없다. 점검 스케줄 관리로 확장 불가.
4. **설비 상태가 자유 텍스트** — 표준 상태 코드 없어 감사 시 상태 통계 집계 불가.
5. **소방 페르소나에 정비이력 미표시** (`showMaintenance: false`) — 소방 점검 이력은 법적 기록 의무가 있다. 소방 페르소나에서 안 보이는 건 설계 오류.

### 비관적 시각

1. **AI가 소방 법규를 틀리게 말하면?** — Citation 모드에서 화재안전기준 개정 이후 구버전 기준을 답하면 법적 책임 문제. "이 답변은 참고용이며 법적 효력 없음" 경고 없음.
2. **평면도 영향 분석이 전기적 연동 미반영** — 감지기 → 수신반 → 방화셔터 전기 연동 경로까지 포함하는지 불분명. 단순 그래프 관계만이면 반쪽짜리.
3. **도면-실물 불일치 보고 채널 없음** — 이 시스템은 도면 정보를 사실로 표시한다. 현장이 다를 때 이견 제기할 방법이 없다.
4. **소방청 표준 시스템 연동 없음** — 소방청 소방안전원, FAS(화재경보시스템)와 연동이 없다. 독립 지식 저장소에 그친다.
5. **HitlFlag 처리 담당자 및 기한 없음** — "검수 필요" 뱃지만 있고 누가 언제까지 검수하는지 워크플로가 없다. 영원히 amber 뱃지로 남는다.

### 불안전성 지적

1. **검수 미완료 소방 설비 정보가 모든 사용자에게 표시** — HitlFlag는 있지만 접근 제한이 없다. 잘못된 소화기 위치 정보를 믿고 화재 대응 시 치명적.
2. **평면도 존 데이터 출처 불명** — 방화구획 폴리곤이 어느 허가 도면 기준인지 명시 없음. 리모델링 후 업데이트 안 되면 영향 분석 결과 완전히 틀림.
3. **비상 대응 시 이 화면을 사용하면 안 된다는 경고 없음** — 실제 화재 상황에서 앱 평면도 보고 진압 경로 결정하면 안 된다. 오프라인 비상 대피도가 항상 우선이어야 한다.

## 페르소나 4 — 최안전 (시설안전팀장, 산업안전기사, 경력 20년)

> "전 분야를 다 봐야 합니다. 사고 나면 내가 책임지는 자리예요."

### 긍정 평가

1. **What-If 시뮬레이션** — 복수 설비 동시 정지 가정 + 전체 영향 범위 계산. 정기 보수 공사 계획 수립 시 진짜 필요한 기능이다.
2. **분야별 집계 (byDiscipline)** — 전기 몇 건, 기계 몇 건, 소방 몇 건 즉시 집계. 팀장 입장에서 리스크 레벨 가시화.
3. **정비 공문 초안 자동 생성** — 공문 초안 작성 시간 단축. 초안 기반 수정 후 사용하는 방식으로 실용적.
4. **Composite 답변 모드 + 엑셀 내보내기** — 전 분야 통합 조회 결과를 xlsx로 내려받아 보고서에 첨부 가능.
5. **페르소나 간 스위처** — 각 담당자 화면을 빠르게 이동하며 다른 시각으로 동일 데이터를 볼 수 있다. 보고 준비에 유용.

### 객관적 지적

1. **What-If 대상이 CRITICAL/HIGH만** — 연쇄 영향이 MEDIUM/LOW 설비를 통해 전파되는 경우가 있다. 선택 범위 제한.
2. **공문 초안의 일정이 하드코딩** — "야간 22:00 ~ 익일 06:00" 고정. 실제 정비 일정 입력 필드가 없다.
3. **Composite 답변 신뢰도 78%** — 팀장 보고용 데이터인데 22% 불확실성. 출처 중 어느 게 맞고 틀린지 드러나지 않는다.
4. **criticality 기준 불명확** — CRITICAL/HIGH/MEDIUM/LOW가 어떤 기준으로 할당됐는지 화면에 없다. 보고 때 설명 불가.
5. **권한 관리 없음** — 팀장, 엔지니어, 협력업체가 동일 화면. 공문 초안 생성을 협력업체가 할 수 없어야 한다.

### 비관적 시각

1. **법적 효력 없음** — AI 추출 + LLM-Wiki 기반 정보는 감리 서명, 준공도면 근거 없이는 법적 효력 없음. SSOT로 운영 시 문제.
2. **What-If 결과가 틀렸을 때 책임 소재** — 시뮬레이션 보고 정비 승인했는데 실제 영향 범위가 달랐다면 면책 조항이 없다.
3. **복합 재해 시나리오 없음** — 화재+정전 동시 발생, 지진 시 연쇄 트립 등 복합 시나리오 불가. 안전팀이 진짜 필요한 건 이것.
4. **이력 관리 버전 없음** — 지난달 영향도 분석 결과와 이번 달 결과를 비교할 수 없다. 정기감사에서 "지난번 분석 결과"가 불가능.
5. **타 시스템 연동 없음** — CMMS, ERP, BIM이 있어도 이 시스템은 단독. 정보 사일로 하나 더 추가되는 꼴.

### 불안전성 지적

1. **결재 라인 없는 공문 생성** — "담당: 시설안전팀" 한 줄로 끝난다. 팀장 승인, 관리자 서명 없이 공문 텍스트가 나온다. 실제로 출력되면 문서 위변조와 다름없다.
2. **비상발전기 가동 준비 자동 삽입** — 이 문구는 발전기 담당자 현장 확인 후 승인이 필요하다. 자동 삽입 공문이 무단 조작 지시가 될 수 있다.
3. **CRITICAL 설비 취약점 지도 노출** — "어떤 설비를 끄면 어떤 시스템이 멈추나"가 누구나 조회 가능. 물리 보안 공격의 레시피가 될 수 있다.

## 페르소나 5 — 정감리 (건설 감리원, 준공 검수 전문, 경력 15년)

> "준공도면과 시공 실물이 맞는지 확인하는 게 내 일입니다. 데이터 출처와 갱신 날짜가 핵심이에요."

### 긍정 평가

1. **`last_updated` 날짜 표시** — 데이터가 언제 갱신됐는지 카드에 나온다. 오래된 정보를 신뢰하지 않을 수 있는 단서.
2. **`source_pages` 필드** — 어느 도면 페이지에서 추출했는지 배열로 관리. 원본 추적 기반이 있다.
3. **ConfidenceMeter 색상 구분** — 80%↑ 녹색, 60~80% 노란색, 60%↓ 빨간색. 한눈에 데이터 품질 파악.
4. **HitlFlag 경고** — AI가 불확실하다고 표시한 데이터에 경고 뱃지. 감리 입장에서 검수 우선순위 파악에 유용.
5. **Relation의 `note` 필드** — 관계에 비고 메모 가능. "준공 시 변경됨" 같은 메모 남길 수 있다.

### 객관적 지적

1. **준공도면 버전 관리 없음** — As-Built, Rev A/B/C 구분 없음. `last_updated`만으론 어느 도면 버전에서 나온 건지 모른다.
2. **`confidence` 값 산출 근거 불명** — 0.9, 0.81, 0.65 수치가 어떻게 계산됐는지 UI에 설명 없음. 감리 입장에서 "이 92%의 근거가 뭐냐"고 물으면 답이 없다.
3. **이견 제기 필드 없음** — 실물이 도면과 다를 때 "반론/현장 확인 결과" 메모를 남길 수 없다.
4. **다중 도면 출처 충돌 시 기준 없음** — 같은 설비가 전기 도면과 기계 도면에 다르게 나와 있으면? 어느 게 우선인지 판단 기준 없음.
5. **도면 파일 직접 접근 불가** — DrawingMiniView는 썸네일만. 실제 PDF/DWG 파일 링크 없어 감리 업무에서 사용 불가.

### 비관적 시각

1. **AI 추출 데이터를 준공 기록으로 쓰면 안 된다** — LLM이 도면 텍스트에서 추출한 태그/사양이 실제 제출 도서와 다를 수 있다. SSOT로 운영 시 위험.
2. **설비 추가/삭제/변경 이력 없음** — 시공 중 설계 변경 반영 여부 추적 불가. 준공 시점에 데이터가 최신인지 보장할 방법 없음.

### 불안전성 지적

1. **`hitl_flags` 빈 배열 = 거짓 안심** — HitlFlag 컴포넌트는 `entity.hitl_flags.length > 0` 일 때만 경고. 즉 미검수 데이터도 경고 없이 "정상"처럼 보임. 이는 심각한 거짓 신뢰를 유발한다.

## 페르소나 6 — 홍보안 (IT 보안 담당자, 정보보안기사)

> "시스템에서 데이터가 유출되거나 조작되면 제가 책임집니다."

### 긍정 평가

1. **Phase 0 prototype이라 외부 API 호출 없음** — mock 데이터만 사용하므로 현재는 실제 민감 데이터 외부 유출 없음.

### 객관적 지적

1. **인증/권한 관리 전혀 없음** — 누구나 URL 알면 모든 화면 접근. `/p4-safety/impact?root=CRITICAL-DEVICE` 도 열린다.
2. **URL에 설비 태그 노출** — `?root=CH-001`, `/entity/CH-001` 형태로 주소창에 직접 노출. 북마크 공유만으로 내부 정보 유출 가능.
3. **공문 초안 텍스트가 무제한 복사 가능** — 권한 없는 사람이 정비 공문 텍스트를 copy-paste해서 외부 유출 가능.
4. **설비 관계 데이터가 보안 취약점 지도** — "어떤 설비를 끄면 어떤 설비가 멈추나"가 그대로 노출. 물리 보안 공격 정보로 활용 가능.

### 비관적 시각

1. **실데이터 연결 시 보안 0점** — 암호화, 접근 로그, 세션 관리, RBAC 중 하나도 없다. 그대로 운영하면 심각한 보안 사고.

### 불안전성 지적

1. **`?root=` 파라미터 변조 가능** — URL에서 root 값 변조 시 다른 설비 영향도 조회. 서버 사이드 검증 없음.
2. **LLM 연동 시 프롬프트 인젝션 위험** — 실제 LLM 연동 시 Chat 입력창에 악의적 프롬프트 삽입으로 내부 데이터 유출 가능. 현재 mock이라 해당 없지만 구조적 취약점.

## 전문가 전원 합의 — 실현 가치 상위 기능

**현장에서 진짜 쓸 것 같은 기능 순위** (6인 합산):

1. **영향도 분석 (ImpactChain/Graph)** — 설비 정지 전 영향 확인. 현재 엑셀이나 구두로 하던 걸 자동화. ★★★★★
2. **자연어 질의 (Chat)** — "CH-001 정비 이력" 한 줄 입력으로 정보 확인. 검색창 들어가는 것보다 빠름. ★★★★
3. **What-If 시뮬레이션** — 정기 보수 계획 때 복수 설비 동시 정지 시뮬레이션. ★★★★
4. **설비 상세 EntityCard** — 속성/관계/정비/위키 탭이 한 화면. ★★★★
5. **대시보드 즐겨찾기** — 자주 보는 설비 즉시 접근. (개인화 필요) ★★★

**현장에서 쓰지 않을 것 같은 기능**:
- 평면도 오버레이 — 실제 도면 좌표 연동 전까지는 시각적 데모 수준
- 위키 탭 — 대부분 "HITL 검수 대기" 상태라면 의미 없음
