# 01. 도면지식관리 시스템 — 본질적 가치 검토

> 작성일: 2026-04-17  
> 목적: DWG 뷰어·AI 기능을 제외한 상태에서 이 시스템이 무엇인지, 어떤 가치를 만들 수 있는지 근본적으로 재검토  
> 전제: 사용자가 DWG → PDF(고해상도)로 변환하여 업로드. 시스템은 PDF 장 단위로 관리.

---

## 1. 지금 이 시스템은 무엇인가?

### 현재 상태를 정직하게 정리하면

| 기능 | 실제 수준 |
|------|-----------|
| 도면 목록 보기 | JSON 기반 정적 리스트 |
| 도면 상세 보기 | 사전 생성된 PNG 이미지 표시 |
| 원본 다운로드 | `<a download>` — 브라우저 기본 다운로드 |
| 설비 검색 | Fuse.js 퍼지 검색 (태그, 이름, 설명) |
| 설비 관계 | 정적 JSON 기반 지식 그래프 |
| 영향도 분석 | 정적 관계 데이터 기반 시뮬레이션 |

**솔직한 결론: 도면을 "관리"하는 시스템이 아니라, 도면에서 이미 추출된 "메타데이터"를 조회하는 시스템이다.**

---

## 2. "도면 관리"라는 단어의 해체

사용자가 말하는 "도면 관리"를 세분화하면:

### 2.1 도면의 라이프사이클

```
[생산] → [제출] → [접수/등록] → [검토] → [배포] → [열람] → [개정] → [폐기]
```

| 단계 | 현재 지원? | 필요한 기능 |
|------|:---:|------------|
| 생산 | ✕ | CAD 도구의 영역. 이 시스템이 할 일 아님 |
| 제출 | ✕ | 업로드 기능 없음 (파일 수신 체계) |
| 접수/등록 | ✕ | 문서번호 채번, 메타데이터 입력, 분류 |
| 검토 | ✕ | 승인 워크플로우, 코멘트, 반려 |
| 배포 | △ | 다운로드만 가능. 배포 대상 지정/이력 없음 |
| 열람 | △ | PNG 미리보기만. PDF 인라인 뷰 없음 |
| 개정 | ✕ | 버전 관리(Rev), 변경 이력 없음 |
| 폐기 | ✕ | 폐기 프로세스 없음 |

**8단계 중 2단계만 부분 지원. 이것은 "도면 관리 시스템"이라 부르기 어렵다.**

### 2.2 그럼 누구를 위한 시스템인가?

| 사용자 유형 | 필요한 것 | 현재 시스템이 주는 것 |
|------------|-----------|---------------------|
| **도면 작성자** (설계사) | CAD 도구, 버전관리, 협업, 체크인/체크아웃 | **없음** |
| **도면 등록자** (문서관리자) | 접수, 채번, 분류, 배포 워크플로우 | **없음** |
| **도면 열람자** (현장 엔지니어) | 빠른 검색, 열람, 다운로드, 관련 정보 연결 | **이것** |
| **의사결정자** (팀장/관리자) | 현황 파악, 영향 분석, 보고서 | **부분 지원** |

**이 시스템의 진짜 사용자는 "도면 열람자" — 도면을 만들지 않고, 도면에 담긴 정보를 소비하는 사람이다.**

---

## 3. "쉽게 찾는다"는 것의 실체

### 3.1 현재: 무엇을 기준으로 찾을 수 있나?

| 검색 기준 | 지원? | 방식 |
|-----------|:---:|------|
| 설비 태그 (예: CH-001) | O | Fuse.js |
| 설비 이름 | O | Fuse.js |
| 설비 설명/기능 | O | Fuse.js |
| 학제(discipline) | O | 페르소나 필터 |
| 도면 제목 | O | 도면 리스트 |
| **도면 번호** | **✕** | 필드 자체가 없음 |
| **Revision** | **✕** | 필드 자체가 없음 |
| **위치/층** | **△** | 엔티티에만 있고 도면에는 없음 |
| **날짜/기간** | **△** | last_updated만 |
| **업체명** | **✕** | 제출 업체 정보 없음 |
| **공종** | **△** | discipline만 (세부 공종 없음) |
| **도면 내 텍스트** | **✕** | OCR/풀텍스트 검색 없음 |

### 3.2 현장에서 실제로 찾는 방식

현장 엔지니어가 도면을 찾을 때 머릿속에 있는 것:

> "3층 소방 평면도 최신판 어딨지?"  
> → 위치(3층) + 공종(소방) + 도면종류(평면도) + 버전(최신)

> "냉동기 배관도 Rev.3 이전 버전하고 비교해야 하는데"  
> → 설비(냉동기) + 도면종류(배관도) + 특정 Revision

> "삼성ENG가 제출한 전기 도면 전부"  
> → 업체명 + 학제

> "지난 달 접수한 도면 중에 아직 승인 안 된 것"  
> → 기간 + 상태(승인/미승인)

**현재 시스템은 이 4가지 시나리오 중 어느 것도 완전히 지원하지 못한다.**

---

## 4. 이것만으로 가치가 있는가?

### 4.1 현재 기능만으로 가치가 있으려면

"설비 태그로 검색 → 관련 도면 보기 → 다운로드"  
이 흐름만으로 가치를 만들려면, 그 전제가 충족되어야 한다:

- 도면이 **이미** 시스템에 올라와 있어야 한다 (누가 올리나?)
- 도면과 설비가 **이미** 연결되어 있어야 한다 (누가 연결하나?)
- 검색 결과가 **최신이고 정확**해야 한다 (누가 갱신하나?)

**세 가지 모두 "사람이 수작업으로" 하는 것이 현재 구조다. 이 수작업의 비용이 이 시스템이 주는 편의보다 크면 → 사용되지 않는다.**

### 4.2 기존 시스템과의 비교

| 시스템 | 도면 관리 범위 | DKS와의 차이 |
|--------|---------------|-------------|
| **EDMS** (전자문서관리) | 접수→배포→개정 전 라이프사이클 | DKS에 없는 것이 EDMS의 핵심 |
| **EPMS** (설계관리) | 프로젝트 진행률, 도면 제출/승인 | 프로젝트 관점. DKS는 운영 관점 |
| **CMMS** (정비관리) | 정비이력, 예방정비 스케줄 | 도면은 참조만. DKS와 보완 관계 |
| **BAS/BEMS** | 실시간 제어/에너지 | 도면은 배경. DKS와 목적 다름 |
| **FMS** (시설관리) | 공간, 자산, 계약 | 도면은 하위 기능 중 하나 |
| **DCIM** (데이터센터) | 랙/전력/냉방 실시간 | 도면은 레퍼런스 |
| **통합SI** | 위 전부를 하나로 | 도면 기능은 EDMS 모듈이 담당 |

**DKS가 다른 시스템에 없는 것: "설비 지식 그래프 기반 관계 탐색 + 영향도 분석"**
**DKS에 없는 것: 도면 라이프사이클의 거의 전부**

---

## 5. 그래서 이 시스템은 무엇이어야 하는가?

### 선택지 A: "도면 관리 시스템"을 목표로 하고 EDMS 기능을 구축

→ 업로드, 채번, 분류, 승인, 버전관리, 배포, 검색...  
→ 수년간의 개발. 이미 시장에 성숙한 솔루션 다수.  
→ **비추천. 레드오션에 뒤늦게 진입하는 셈.**

### 선택지 B: "도면 관리"라는 이름을 버리고, 실제 강점에 집중

현재 DKS의 진짜 강점은:

1. **설비 관계 시각화** — 이 설비가 죽으면 어디까지 영향 받나?
2. **분야 간 정보 연결** — 전기 설비와 기계 설비의 관계를 한 화면에서
3. **자연어 질의** (Phase 1) — "냉동기 1호기에 연결된 펌프가 뭐야?"
4. **현장 엔지니어 맞춤 UX** — 4개 페르소나

이것은 **"도면 관리"가 아니라 "설비 지식 탐색 엔진"**이다.

### 선택지 C: PDF를 진짜로 잘 관리하는 것 + 설비 지식 엔진 (혼합)

PDF 업로드를 받되, "도면 관리의 최소 기능"만 구현하고, 설비 지식 엔진의 강점을 살리는 방향.

---

## 6. "PDF 장 단위 관리"에서 진짜 필요한 기능

DWG 뷰어와 AI를 제외하고, 사람이 PDF를 업로드해서 관리하는 최소 기능:

### 6.1 등록 (Registration)

```
사용자가 PDF 업로드
→ 시스템이 페이지 수 자동 감지
→ 각 페이지를 고해상도 이미지로 자동 변환 (서버 사이드)
→ 사용자가 각 페이지에 메타데이터 태깅:
   - 도면 번호 (Drawing No.)
   - 도면 제목
   - Revision 번호
   - 공종 (학제)
   - 도면 종류 (평면도/단면도/상세도/계통도/SLD 등)
   - 제출 업체
→ 원본 PDF + DWG도 함께 첨부 가능
```

### 6.2 분류 체계 (Classification)

현장에서 통용되는 도면 분류:

```
Level 1: 학제 (MECHANICAL / ELECTRICAL / FIRE / ARCHITECTURAL / ...)
Level 2: 도면 종류
  - 배치도 (Layout)
  - 평면도 (Plan)
  - 단면도 (Section)
  - 상세도 (Detail)
  - 계통도 (System Diagram)
  - 단선도 (Single Line Diagram)
  - P&ID (Piping & Instrumentation)
  - 배관도 (Piping)
  - 결선도 (Wiring)
Level 3: 위치/구역
  - 층별 (B1F, 1F, 2F, 3F, ...)
  - 구역별 (기계실, 전기실, 사무실, ...)
Level 4: Revision
  - Rev.0, Rev.1, Rev.2, ...
```

### 6.3 검색 (Search)

| 검색 방식 | 설명 |
|-----------|------|
| **필터 조합** | 학제 + 종류 + 위치 + 업체 조합 필터 |
| **도면 번호 검색** | 정확히 일치 또는 부분 일치 |
| **키워드 검색** | 제목, 설명 텍스트 검색 |
| **설비 역추적** | 설비 태그 → 이 설비가 나오는 도면 목록 |
| **최신 Revision** | 같은 도면 번호의 최신 Rev만 보기 |

### 6.4 열람 (Viewing)

- PDF 인라인 뷰어 (브라우저 내장 or pdf.js)
- 페이지 단위 고해상도 이미지 뷰어 (zoom/pan)
- 좌우 분할뷰: 목록 | 뷰어

### 6.5 이력 (History)

- 언제, 누가, 어떤 도면을 올렸는지
- Revision 변경 이력
- 다운로드 이력 (누가 언제 받았나)

### 6.6 연결 (Linking)

- 도면 ↔ 설비 태그 연결 (이것이 DKS의 진짜 차별점)
- 도면 ↔ 정비이력 연결
- 도면 ↔ 도면 참조 관계

## 7. 단독 서비스의 가치 판단

### 가치가 있는 경우

- **중소규모 건물/플랜트**: EDMS를 도입할 예산이 없는 조직
- **운영 단계**: 이미 설계가 끝나고 As-Built 도면이 확정된 상태에서 운영·유지보수 목적
- **통합 필요 없음**: 독립적으로 도면만 관리하면 되는 환경

### 가치가 부족한 경우

- **대규모 EPC 프로젝트**: EPMS/EDMS가 필수. 이 시스템으로 대체 불가
- **실시간 제어 필요**: BAS/DCIM이 이미 도면 참조 기능 포함
- **이미 통합SI 운영 중**: 별도 시스템 = 데이터 이중 관리 = 거부감

### DKS만의 차별화 가능 포인트

```
기존 EDMS: 도면을 "파일"로 관리 (폴더 구조, 메타데이터, 워크플로우)
    → 도면 안에 무엇이 있는지는 모른다

DKS: 도면을 "지식"으로 관리 (설비 관계, 영향도, 자연어 탐색)
    → 도면 안의 설비가 서로 어떻게 연결되는지 안다
```

**이 차이가 DKS의 존재 이유다. 단순 EDMS 기능만 만들면 가치 없다.**

---

## 8. 제안: 포지셔닝 재정의

### AS-IS (현재)
> "도면지식관리 시스템" — 도면 관리도 아니고, 지식 관리도 아직 미완성

### TO-BE (제안)
> **"설비 지식 탐색 엔진 + 도면 최소 관리"**
> - 핵심: 설비 간 관계 탐색, 영향도 분석, 자연어 질의
> - 보조: PDF 도면 등록/분류/검색/다운로드 (EDMS 경량 버전)
> - 연결고리: 도면 ↔ 설비 태그 양방향 추적

### 한 줄 가치 제안
> "도면 PDF를 올리면, 그 안의 설비 관계를 탐색하고 영향도를 분석할 수 있는 시스템"
