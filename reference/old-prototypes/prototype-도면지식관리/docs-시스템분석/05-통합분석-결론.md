## 1. 8인의 목소리가 수렴하는 곳

8명의 페르소나가 말하는 것이 제각각으로 보이지만, 추출하면 **세 가지 공통점**이 나온다:

### 공통점 1: "지식 그래프"에 감동한 사람은 없다

| 페르소나 | 지식 그래프에 대한 반응 |
|----------|----------------------|
| 김현수 (현장) | "내 머릿속이 더 정확하다" |
| 박정민 (팀장) | "위키+주석이면 충분하다" |
| 이수진 (문서) | "내 업무와 무관하다" |
| 최동훈 (IT) | "연동 안 되면 의미 없다" |
| 윤태호 (소방) | "법적 효력 없다" |
| 장민수 (중소FM) | "과잉 기능이다" |

**기술 주도(technology-push)의 함정:** "우리가 만들 수 있는 것"에서 출발했지, "그들이 돈을 내고 사는 것"에서 출발하지 않았다.

### 공통점 2: 모두가 원하는 것은 "더 단순한 것"

```
김현수: 검색만 되면 된다
박정민: 도면 위 메모만 되면 된다
이수진: 접수 자동화만 되면 된다
최동훈: 기존 시스템에 붙으면 된다
```

아무도 "설비 관계 그래프를 자동으로 만들어주는 시스템"을 달라고 하지 않았다. 이것은 **만드는 사람의 비전**이지 **쓰는 사람의 요구**가 아니다.

### 공통점 3: 지불 의사의 벽

| 페르소나 | 지불 의사 |
|----------|-----------|
| 김현수 | 0원 |
| 박정민 | 조건부 (퇴직자 지식 보존 명분, 3개월 무료 파일럿) |
| 이수진 | 회사 구매 (AI 90%+ 조건) |
| 최동훈 | PoC조차 4가지 전제 충족 필요 |
| 정서연 | 투자 패스 |
| 한경호 | 인수 가치 없음 |
| 윤태호 | 0원 |
| 장민수 | 무료여도 의문 |

**8명 중 자발적으로 돈을 낼 사람이 0명이다.**

## 2. 냉정한 진단

### 현재 DKS의 문제는 기술이 아니다

| 흔한 오해 | 실제 문제 |
|-----------|-----------|
| "AI 정확도가 높아지면 팔린다" | 정확도 이전에 **누가 왜 사는지** 모른다 |
| "기능이 부족해서 안 쓴다" | 기능 전에 **워크플로우에 안 끼어든다** |
| "프로토타입이라 그렇다" | 프로토타입 이전에 **문제-해결 적합성(PSF)**이 없다 |
| "시장 교육이 필요하다" | "시장 교육이 필요하다"는 PMF가 없다는 완곡어법이다 |

### Product-Market Fit 이전 단계

```
[Problem-Solution Fit]  →  [Product-Market Fit]  →  [Scale]
        ?                         ✕                    ✕
    여기조차 불확실              여기 없음            논외
```

- **Problem은 존재한다:** 설비 지식이 파편화되어 있고, 경험자 퇴직 시 소실된다.
- **Solution이 맞는지 모른다:** "자동 온톨로지 구축"이 이 문제의 답인지 검증 안 됨.
- **Market이 돈을 내는지 모른다:** 8명 중 지불 의사 0명.

## 3. 페르소나 분석이 뒤집어 놓은 것

### 이전 방향 (03 문서의 결론)
> "AI 도면 분석 엔진(경로 B)이 유일한 답이다. 실험으로 AI 정확도를 검증하라."

### 8인 분석 후 수정된 시각

**경로 B는 여전히 가장 높은 천장을 가지고 있지만, 바닥이 없다.**

AI가 완벽하게 작동해도:
- 김현수(현장)는 자기 머릿속을 더 신뢰한다
- 최동훈(IT)은 연동/보안 4가지를 먼저 요구한다
- 정서연(VC)은 유료 고객 1곳을 먼저 요구한다
- 한경호(경쟁사)는 "우리가 6개월이면 만든다"고 한다

**기술이 작동하는 것과 시장이 사는 것은 완전히 다른 문제다.**

## 4. 그래서 무엇을 해야 하는가

### 세 가지 레이어를 분리하라

```
Layer 0: 즉시 고통 해결 (진입점)     ← 여기서 돈을 받고 신뢰를 산다
Layer 1: 지식 축적 (차별점)          ← 여기서 해자를 만든다
Layer 2: AI 자동화 (비전)            ← 여기서 스케일한다
```

### Layer 0: "즉시 고통 해결" — 진입점

8인의 페르소나가 공통으로 인정한 고통:

| 고통 | 빈도 | 강도 |
|------|------|------|
| **도면을 못 찾는다** (김현수, 박정민) | 매일 | 중 |
| **도면 접수/분류가 수작업이다** (이수진) | 매일 50건 | 높음 |
| **퇴직자 지식이 사라진다** (박정민) | 연 1-2건 but 치명적 | 매우 높음 |
| **준공 도면과 현실이 다르다** (김현수, 윤태호) | 항상 | 높음 |

이 중 **가장 빈도가 높고 즉시 측정 가능한 고통**: "도면을 못 찾는다" + "접수/분류 수작업"

**최소 진입 제품:**
```
PDF 업로드 → 페이지 분리 → 기본 메타데이터 (AI 보조 or 수동)
→ 검색 (도면번호, 공종, 위치, 제목)
→ 열람 (PDF 인라인 뷰어)
→ 다운로드
→ 도면 위 주석/메모 (경험 기록)
```

이것은 "경량 EDMS"지만, 목적이 다르다:
- 경량 EDMS 자체가 제품이 아니다
- **Layer 1으로 가기 위한 진입점**이다
- 사용자가 도면을 올리고 메타데이터를 태깅하는 과정에서 **학습 데이터가 쌓인다**

### Layer 1: "지식 축적" — 차별점

사용자가 Layer 0를 쓰면서 자연스럽게 만들어지는 것:

```
도면 업로드 시: 도면 ↔ 설비 태그 연결 (수동 태깅 or AI 보조)
주석 입력 시: "이 밸브는 실제로 바이패스됨" → 현장 지식 축적
검색 패턴: 어떤 도면을 자주 찾는지 → 중요도 학습
설비 태그: 같은 도면에 나오는 설비들 → 관계 추론 가능
```

**사용자가 의식하지 못하는 사이에 온톨로지가 구축된다.**

박정민(팀장)이 원하는 "도면 위 포스트잇" = Layer 0의 주석 기능
김현수(현장)가 원하는 "도면 검색 앱" = Layer 0의 검색 기능
이수진(문서)이 원하는 "접수 자동화" = Layer 0의 업로드+분류

**모두의 즉시 고통을 해결하면서 지식이 쌓이는 구조.**

### Layer 2: "AI 자동화" — 비전

Layer 0/1에서 축적된 데이터가 임계점을 넘으면:

```
축적된 [도면 이미지 + 수동 태깅 메타데이터] = AI 학습 데이터
→ AI 모델이 새 도면의 메타데이터를 자동 추출
→ 설비 관계를 자동 추론
→ 영향도 분석 자동화
```

**이때 비로소 "경로 B(AI 엔진)"가 실현된다.** 그런데 처음부터 AI로 시작하는 것이 아니라, **데이터가 쌓인 후에 AI가 온다.**

## 5. 이 접근이 이전 분석과 다른 점

| 이전 (03 문서) | 수정 후 (이 문서) |
|----------------|-------------------|
| AI가 먼저. 정확도 검증이 첫 번째 | 사용자 고통 해결이 먼저. AI는 나중에 |
| "경로 A vs B vs C" 중 하나 선택 | 세 레이어를 **순서대로** 쌓는 것 |
| 기술에서 출발 | 고통에서 출발 |
| "세상에 없는 제품"을 만들자 | "당장 쓰는 제품"을 만들고, 쌓여서 "없던 것"이 되자 |

## 6. 위험 요소 — 여전히 남아있는 것

### 해소된 위험

| 위험 | 해소 방법 |
|------|-----------|
| AI 정확도가 첫날부터 필요 | Layer 0는 AI 없이 시작 |
| 기술 위험이 사업 위험 | Layer 0에서 PMF 먼저 검증 |

### 해소되지 않은 위험

| 위험 | 심각도 | 대응 |
|------|--------|------|
| **"NAS+엑셀로 충분하다" (행동 관성)** | 높음 | Layer 0가 엑셀보다 압도적으로 나은 한 가지를 만들어야 한다 |
| **낮은 ARPU** | 높음 | 진입은 저가/무료, Layer 2에서 과금 전환 |
| **1인 개발로 Layer 0~2 전부 감당** | 매우 높음 | Layer 0 MVP를 극도로 좁게 잡아야 한다 |
| **데이터 축적에 시간이 걸린다** | 중 | 초기 고객 1-2곳에 밀착해서 빠르게 축적 |
| **경쟁사가 같은 기능 출시** | 중 | Layer 1 데이터가 해자. 시간이 곧 방어 |

### 가장 치명적인 미해결 질문

> **"NAS+엑셀보다 이게 나은 한 가지가 뭔가?"**

이 질문에 명확한 답이 없으면 Layer 0도 실패한다. 후보:

1. **모바일 검색**: NAS는 현장(지하 기계실)에서 접근 불가. 웹앱은 된다.
2. **도면 위 주석**: 엑셀에는 도면 위에 메모를 남기는 기능이 없다.
3. **설비→도면 역추적**: 엑셀로 "CH-001이 나오는 도면 전부"를 찾으려면 전수 검색.

이 세 가지 중 현장에서 가장 강렬한 것을 찾아야 한다. **이것이 다음 사용자 인터뷰의 질문이다.**

## 7. 최종 판단 요약

```
┌─────────────────────────────────────────────────┐
│                                                   │
│   "세상에 없는 AI 엔진"을 먼저 만들려 하지 마라.  │
│   "당장 쓰는 도구"를 먼저 만들고,                  │
│   쓰는 과정에서 데이터가 쌓이게 하고,              │
│   데이터가 임계점을 넘으면 AI가 가치를 만든다.     │
│                                                   │
│   Layer 0 (도구) → Layer 1 (데이터) → Layer 2 (AI) │
│                                                   │
└─────────────────────────────────────────────────┘
```

### 즉시 실행 항목

| # | 행동 | 목적 | 소요 |
|---|------|------|------|
| 1 | 사용자 3-5명에게 질문: "NAS+엑셀 대신 이걸 쓸 한 가지 이유가 뭐가 되어야 하나?" | 진입점 확정 | 1주 |
| 2 | Layer 0 MVP 범위 확정 (업로드+검색+열람+주석, 극도로 좁게) | 개발 범위 | 1일 |
| 3 | 병행: 실 도면 PDF 10장으로 AI 추출 실험 (Layer 2 가능성 사전 검증) | 기술 위험 축소 | 2일 |
| 4 | 파일럿 고객 1곳 확보 시도 | PMF 검증 시작 | 2-4주 |
