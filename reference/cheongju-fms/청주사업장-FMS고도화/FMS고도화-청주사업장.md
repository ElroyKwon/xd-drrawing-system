# FMS 고도화 — 청주사업장

> **성격**: 데이터 지식 스튜디오의 **첫 번째 고객 적용 사례**
> **원본**: `청주사업장FMS고도화.pdf` / `.pptx`
> **핵심**: 지식 스튜디오의 기능을 가져다 쓰되, 지식 스튜디오 전체 방향은 건드리지 않는다

---

## 1. 고객 요구사항 (PDF 기준)

### 1.1 제안 배경

현재 설비 운영에 필요한 핵심 정보는 도면(DWG/PDF)에 분산.
실제 활용 시 [[AutoCAD]] 확인 또는 숙련 인력 의존 불가피.

| 문제 | 현재 상황 |
|------|----------|
| 설비 정보 검색 비효율 | 도면을 일일이 열어서 확인 |
| 장애 영향 분석 어려움 | 숙련 인력 경험에 의존 |
| 운영 지식 인적 의존 및 소실 | 퇴직 시 지식 단절 |

### 1.2 요구 기능 — 사용자 관점 (운영팀)

| 기능 | 설명 |
|------|------|
| **도면 기반 정보 조회** | 도면 내 설비 정보 조회, 설비 태그 기반 위치 및 정보 확인 |
| **설비 검색** | 설비 태그 기반 검색 (예: P-101), 설비 및 계통 정보 조회 |

### 1.3 요구 기능 — 엔지니어 관점 (관리자)

| 기능 | 설명 |
|------|------|
| **AI 도면 분석 및 태그 자동화** | 도면(DWG/PDF) 기반 설비 정보 분석, 태그 및 주요 객체 추출, 설비 간 기본 연결 관계 도출 |
| **설비 정보 구조화 (지식화)** | 도면 내 설비 및 연결 관계 구조화, 설비 간 관계 기반 정보 조회 지원 |

### 1.4 기대 효과 (고객 제시)

- 도면 기반 설비 정보 접근성 향상
- 장애 대응 속도 및 운영 안정성 개선
- 도면 탐색 및 분석 업무 효율화
- 인력 의존 최소화 및 기술 자산화

## 2. 요구사항 해석: 어디까지 해야 하나

### 2.1 요구사항 범위 정리

PDF의 요구사항을 분해하면 **4개 기능 블록**:

```
[블록 1] 도면 등록/관리
  → DWG/PDF 파일 업로드, 열람, 분류

[블록 2] AI 도면 분석
  → 도면에서 설비 태그 자동 추출 (P-101, AHU-01 등)
  → 주요 객체(펌프, 밸브, 공조기 등) 인식
  → 설비 간 기본 연결 관계 도출 (A가 B에 공급)

[블록 3] 설비 정보 구조화
  → 추출된 설비/관계를 DB에 구조화 저장
  → 설비 태그 ↔ 도면 위치 연결

[블록 4] 검색/조회 UI
  → 태그명으로 설비 검색 (P-101 → 위치, 사양, 연결 설비)
  → 계통별 설비 목록 조회
  → 도면 뷰어에서 설비 위치 확인
```

### 2.2 요구하지 않는 것 (범위 밖)

| 기능 | FMS 고도화 요구 여부 | 지식 스튜디오 해당 Phase |
|------|:-------------------:|:----------------------:|
| 다단계 영향도 분석 (6 hop 체인) | ❌ 요구 안 함 | Phase C |
| 그래프 DB 추론 규칙 | ❌ 요구 안 함 | Phase C |
| 자연어 질의응답 (ChatBot) | ❌ 명시 안 함 | Phase A 확장 |
| 문서 RAG (준공문서 검색) | ❌ 명시 안 함 | Phase A |
| 변경 추적 (준공 vs 현재) | ❌ 명시 안 함 | 고도화 |
| What-if 시뮬레이션 | ❌ 요구 안 함 | Phase C |

> **핵심**: 고객은 **"도면 → 설비 추출 → 검색"**을 요구.
> 영향도 분석, 자연어 ChatBot, 그래프 DB 추론은 요구하지 않음.

## 3. 지식 스튜디오에서 가져올 것

### 3.1 매핑: FMS 요구 기능 ↔ 지식 스튜디오 기술

| FMS 요구 기능 | 지식 스튜디오 기술 | Phase | 가져오는 수준 |
|:---:|------|:---:|------|
| 블록 1 | **도면 업로드/뷰어** | PoC | 70% 완성된 모듈 재사용 |
| 블록 2 | **Multi-Pass AI 파이프라인** | B | DWG→DXF→AI 분석 흐름 |
| 블록 2 | **OCR 보강 (PaddleOCR/EasyOCR)** | B | 태그명, 사양 텍스트 추출 |
| 블록 2 | **LLM Factory 패턴** | PoC | Claude/GPT/Gemini 추상화 |
| 블록 3 | **온톨로지 표준 (스키마)** | PoC | 설비 분류 체계, 관계 정의 |
| 블록 3 | **검증 엔진 (YAML 규칙)** | PoC | AI 추출 결과 품질 검증 |
| 블록 4 | 신규 구현 | — | FMS UI에 통합 |

### 3.2 가져오지 않는 것

| 지식 스튜디오 기술 | 이유 |
|------|------|
| TypeDB / 그래프 DB | 다단계 추론 요구 없음 |
| TypeQL 쿼리 엔진 | 그래프 DB 미사용 |
| GraphRAG / VectorDB RAG | 자연어 질의 요구 없음 |
| RAGFlow 플랫폼 | 문서 RAG 요구 없음 |
| 복잡한 HITL 워크플로우 | 간소화된 엔지니어 확인으로 대체 |
| Apache AGE / Neo4j | 영향도 분석 요구 없음 |

## 4. 구현 방법

### 4.1 기술 스택 (FMS 고도화 전용)

| 구분 | 기술 | 역할 |
|------|------|------|
| 도면 변환 | [[ODA File Converter]] | DWG → DXF 변환 |
| 도면 파싱 | [[ezdxf]] + 이미지 렌더링 | Layout 분리, 텍스트/구조 추출 |
| AI 분석 | Gemini / GPT / Claude API | 멀티모달 도면 분석, 설비 추출 |
| OCR | [[PaddleOCR]] / [[EasyOCR]] | 도면 내 텍스트 추출 (태그명, 사양) |
| 검증 | 온톨로지 검증 엔진 (재사용) | AI 추출 결과 품질 검증 |
| 구조화 DB | PostgreSQL (또는 SQLite) | 설비/관계 테이블 저장 |
| Backend | [[FastAPI]] | API 오케스트레이션 |
| Frontend | FMS 내장 또는 Svelte | 검색/조회 UI |

### 4.2 파이프라인

```
[입력]
도면 파일 (DWG / PDF)
    │
    ▼
[1] 도면 변환 및 전처리
    DWG → ODA Converter → DXF
    DXF → ezdxf → Layout 분리 → 페이지별 이미지 생성
    PDF → pdf2image → 페이지별 이미지 생성
    │
    ▼
[2] AI 멀티모달 분석
    이미지 → Gemini Flash (초벌, 저비용)
    OCR → PaddleOCR (태그명, 사양 텍스트)
    이미지 + OCR 텍스트 → Claude/GPT-4o (정밀 분석)
    │
    ▼
[3] 추출 결과
    ┌─────────────────────────────────────┐
    │  설비 목록: P-101(펌프), V-201(밸브) │
    │  관계: P-101 --feeds-→ AHU-01      │
    │  위치: P-101은 도면 A의 (x,y) 영역  │
    │  사양: P-101, 2.2kW, DN50          │
    │  신뢰도: 0.85                       │
    └─────────────────────────────────────┘
    │
    ▼
[4] 검증 + 저장
    온톨로지 검증 엔진 → 규칙 위반 체크
    엔지니어 간소 확인 (필요 시)
    → PostgreSQL 구조화 저장
    │
    ▼
[5] 검색/조회 UI
    태그 검색: "P-101" → 사양, 위치, 연결 설비
    계통 조회: "냉수 계통" → 소속 설비 목록
    도면 뷰어: 설비 클릭 → 정보 팝업
```

### 4.3 구현 우선순위

| 순서 | 작업 | 산출물 | 검증 포인트 |
|:----:|------|--------|-----------|
| 1 | DWG→DXF 변환 + 이미지 생성 | 도면 이미지 파일 | ODA 변환 성공률 |
| 2 | AI 멀티모달 분석 (태그 추출) | 설비 목록 JSON | 추출 정확도 (목표 80%+) |
| 3 | OCR 보강 | 태그명, 사양 텍스트 | 태그 인식률 |
| 4 | 구조화 DB 저장 | 설비/관계 테이블 | 데이터 정합성 |
| 5 | 검색 UI | 태그 검색 + 계통 조회 | 사용자 검색 체감 |
| 6 | 도면 뷰어 연동 | 설비 위치 하이라이트 | 직관성 |

## 5. 서비스 제공 방식

### 5.1 고객에게 어떻게 서비스하나

```
[고객이 하는 것]
1. 기존 도면 파일(DWG/PDF) 제공
2. FMS 웹 화면에서 설비 검색, 도면 조회

[우리가 하는 것]
1. 도면 파일 수집 → AI 자동 분석 → 설비/관계 추출
2. 추출 결과 검증 → 구조화 DB 저장
3. FMS 내 도면관리 기능으로 통합 제공
```

### 5.2 FMS 통합 형태

```
기존 FMS 시스템
┌────────────────────────────────────────┐
│  설비 관리 | 작업 관리 | 점검 관리 | ...│
│                                        │
│  ┌──────────────────────────────────┐  │
│  │  [신규] 도면 관리 기능            │  │
│  │                                  │  │
│  │  ■ 도면 등록/열람                │  │
│  │  ■ 설비 태그 검색                │  │
│  │  ■ 계통별 설비 조회              │  │
│  │  ■ 도면 뷰어 (설비 위치 표시)    │  │

└────────────────────────────────────────┘
         ↕ API
┌────────────────────────────────────────┐
│  도면 분석 백엔드 (우리가 제공)          │

└────────────────────────────────────────┘
```

## 6. 지식 스튜디오와의 관계

### 6.1 원칙

> **FMS 고도화는 지식 스튜디오의 "고객 적용 사례"이다.**
> 지식 스튜디오의 기술을 가져다 쓰지만, 지식 스튜디오의 전체 방향(기술 경로, 아키텍처)을 이 사업이 결정하지 않는다.

- FMS에서 TypeDB가 필요 없다고 해서 지식 스튜디오에서 TypeDB를 버리는 것이 아님
- FMS에서 Phase C가 불필요해도 지식 스튜디오의 Phase C 계획은 유지
- FMS 사업의 기술 결정은 **이 문서 안에서만** 관리

### 6.2 피드백 루프

FMS 구현 과정에서 검증되는 것:

| 검증 항목 | 지식 스튜디오에 환류 |
|----------|-------------------|
| DWG→DXF 변환 성공률 | Phase B 파이프라인 신뢰도 |
| AI 멀티모달 분석 정확도 | Multi-Pass 전략 유효성 |
| OCR 태그 인식률 | OCR 앙상블 전략 |
| 온톨로지 표준 적합성 | 스키마 실사용 검증 |
| 고객이 실제로 원하는 기능 | 서비스 방식(A vs B) 판단 근거 |

> FMS에서 얻는 실전 데이터가 지식 스튜디오 전체 전략의 **검증 수단**이 된다.
